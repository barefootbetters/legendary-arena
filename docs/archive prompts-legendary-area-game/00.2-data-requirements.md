# Prompt 00.2 — Data Requirements & Canonical Data Contract

> **REFERENCE DOCUMENT** — Not an execution prompt. No output required.
> This document defines the canonical data model for the Legendary Arena SPA.
> It is the authoritative reference for schemas, lookups, image URLs, field names,
> validation rules, and data pipeline behavior across all execution prompts (01–13).
>
> All implementation prompts must use the field names and structures defined here.
> When this document conflicts with a prompt's body, this document wins for
> field naming and data shape. When an execution prompt adds constraints
> (e.g., only store rules data in PostgreSQL), those constraints still apply.

---

## Purpose

Legendary Arena is **data-driven**: every UI behavior derives from card data
and metadata lookups. This document:

- Documents the real data that exists on disk (not aspirational)
- Defines canonical field names for all card and metadata types
- Specifies image URL construction rules from real R2 examples
- Clarifies what lives where (R2 JSON vs. PostgreSQL vs. runtime state)
- Serves as a shared contract between the data pipeline, game server, and SPA

## 1. Card Data

### 1.1 Data Shape on Disk (R2 / `data/metadata/{abbr}.json`)

Card data is stored per-set. Each JSON file is an object with top-level arrays
organized by card category. The structure is **hierarchical**, not flat:

```json
{
  "id": 37,
  "abbr": "mdns",
  "exportName": "MidnightSons",
  "heroes": [ ... ],
  "masterminds": [ ... ],
  "villains": [ ... ],
  "henchmen": [ ... ],
  "schemes": [ ... ],
  "bystanders": [ ... ],
  "wounds": [ ... ],
  "other": [ ... ]
}
```

### 1.2 Hero Deck Shape

Heroes are organized as **decks** — each hero is a named set of cards.
Do not treat hero cards as independent flat records.

```json
{
  "id": 260,
  "name": "Blade, Daywalker",
  "slug": "blade-daywalker",
  "team": "midnight-sons",
  "cards": [
    {
      "name": "Hunt High And Low",
      "displayName": "Hunt High And Low",
      "slug": "hunt-high-and-low",
      "rarity": 1,
      "rarityLabel": "Common 1",
      "slot": 1,
      "hc": "instinct",
      "cost": 3,
      "attack": "2",
      "recruit": null,
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-hr-blade-daywalker-3c1.webp",
      "abilities": [
        "[keyword:Patrol]: If it's empty, reveal the top card of your deck. You may KO it."
      ]
    }
  ]
}
```

**Hero card field reference:**

| Field | Type | Notes |
|---|---|---|
| `name` | `string` | Card name (individual card, not hero name) |
| `displayName` | `string` | Usually same as `name`; use for UI display |
| `slug` | `string` | URL-safe card identifier — used in image URLs |
| `rarity` | `number` | 1 = Common, 2 = Uncommon, 3 = Rare |
| `rarityLabel` | `string` | "Common 1", "Common 2", "Common 3", "Uncommon", "Rare" |
| `slot` | `number` | Card position within hero deck (1–4) |
| `hc` | `string` | Hero class abbreviation — matches `abbr` in `hero-classes.json` |
| `cost` | `number` | Recruit cost to buy this card |
| `attack` | `string \| null` | Fight value — may be `"2"`, `"2+"`, `"0+"` etc. (string, not number) |
| `recruit` | `string \| null` | Recruit value — same format as `attack`; null if none |
| `imageUrl` | `string` | Full R2 URL — do not reconstruct from parts if `imageUrl` is present |
| `abilities` | `string[]` | Rule text lines using ability markup (see §5) |

### 1.3 Mastermind Shape

```json
{
  "id": 91,
  "name": "Lilith, Mother of Demons",
  "slug": "lilith-mother-of-demons",
  "alwaysLeads": ["lilin"],
  "vp": 6,
  "cards": [
    {
      "name": "Lilith, Mother of Demons",
      "slug": "lilith-mother-of-demons",
      "tactic": false,
      "vAttack": "8+",
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-mm-lilith-mother-of-demons.webp",
      "abilities": [ ... ]
    },
    {
      "name": "Epic Lilith",
      "slug": "epic-lilith",
      "tactic": false,
      "vAttack": "10+",
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-me-lilith-mother-of-demons.webp",
      "abilities": [ ... ]
    },
    {
      "name": "Connoisseur of Souls",
      "slug": "connoisseur-of-souls",
      "tactic": true,
      "vAttack": 8,
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-mt-lilith-mother-of-demons-connoisseur-of-souls.webp",
      "abilities": [ ... ]
    }
  ]
}
```

**Mastermind field reference:**

| Field | Type | Notes |
|---|---|---|
| `id` | `number` | Stable numeric ID |
| `name` | `string` | Mastermind display name |
| `slug` | `string` | URL-safe identifier |
| `alwaysLeads` | `string[]` | Villain/henchman group slugs this mastermind always leads |
| `vp` | `number` | Victory points when defeated |
| `cards` | `array` | Includes the mastermind card, epic variant, and all tactics |

**Mastermind card field reference:**

| Field | Type | Notes |
|---|---|---|
| `name` | `string` | Card name |
| `slug` | `string` | Used in image URL |
| `tactic` | `boolean` | `false` = mastermind/epic card; `true` = tactic card |
| `vAttack` | `string \| number` | Fight value — may be string (`"8+"`) or number (`8`) |
| `imageUrl` | `string` | Full R2 URL |
| `abilities` | `string[]` | Rule text |

### 1.4 Villain Group Shape

Villain groups are collections of villain cards, linked to a mastermind via
the `alwaysLeads` / `ledBy` relationship:

```json
{
  "id": 112,
  "name": "Lilin",
  "slug": "lilin",
  "ledBy": ["lilith-mother-of-demons"],
  "cards": [
    {
      "name": "Meatmarket",
      "slug": "meatmarket",
      "vp": "3",
      "vAttack": "3+",
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-vi-lilin-meatmarket.webp",
      "abilities": [ ... ]
    }
  ]
}
```

**Villain group field reference:**

| Field | Type | Notes |
|---|---|---|
| `id` | `number` | Stable numeric ID |
| `name` | `string` | Group display name |
| `slug` | `string` | URL-safe identifier |
| `ledBy` | `string[]` | Mastermind slugs that always lead this group (inverse of `alwaysLeads`) |
| `cards` | `array` | Villain cards in this group |

**Villain card fields:** `name`, `slug`, `vp` (`string | number`), `vAttack` (`string | number`), `imageUrl`, `abilities[]`

### 1.5 Scheme Shape

Schemes may have one or more cards (some are double-sided transform cards):

```json
{
  "id": 170,
  "name": "Sire Vampires at the Blood Bank",
  "slug": "sire-vampires-at-the-blood-bank",
  "imageUrl": "https://images.barefootbetters.com/mdns/mdns-sc-sire-vampires-at-the-blood-bank.webp",
  "cards": [
    { "abilities": [ ... ] }
  ]
}
```

Transform schemes have `cardType: "scheme-transform"` and a separate card on
the reverse side (e.g., `great-old-one-chthon`). Their `id` may be `null`
for the reverse side — do not require a non-null `id` for transform cards.

## 2. Metadata Lookups

All metadata files live in `data/metadata/`. They are loaded **before** any
card rendering and used for display labels, filtering, and validation.

Use IDs and lookups throughout — never duplicate a display string in card records.

### 2.2 Sets (`sets.json`)

**Actual shape:**
```json
{
  "id": 36,
  "abbr": "mdns",
  "pkgId": 36,
  "slug": "midnight-sons",
  "name": "Midnight Sons",
  "releaseDate": "2023-05-10",
  "type": "32nd Expansion"
}
```

**Fields:**
| Field | Type | Notes |
|---|---|---|
| `id` | `number` | Numeric ID |
| `abbr` | `string` | Short 4-char set abbreviation — used in image URLs and file names |
| `pkgId` | `number` | Package ordering ID (same as `id` currently) |
| `slug` | `string` | URL-safe full name |
| `name` | `string` | Display name |
| `releaseDate` | `string` | ISO date string (`YYYY-MM-DD`) |
| `type` | `string` | Descriptive type — e.g., `"1st Core Set"`, `"32nd Expansion"`, `"Promo / Trading Card Mini-Expansion"` |

**Note:** `type` is a free-form string, not an enum. Do not validate it as
`"Core" | "Expansion" | "Promo"` — use pattern matching if categorization is needed.

**Current sets:** 40 sets from Core Set (2012) through Weapon X (2024). See `sets.json` for the complete list.

### 2.3 Hero Teams (`hero-teams.json`)

**Actual shape:**
```json
{
  "id": 12,
  "slug": "guardians-of-the-galaxy",
  "name": "Guardians of the Galaxy"
}
```

**Fields:** `id` (number), `slug` (string), `name` (string)

Note: There is no `abbr` or `iconKey` in the actual data. Team data includes
both hero teams and villain teams/factions (e.g., `hydra`, `brotherhood`).
`id: 0` = `unaffiliated`.

**25 teams currently defined.** Representative sample: Avengers, S.H.I.E.L.D.,
Spider Friends, X-Men, Fantastic Four, Marvel Knights, X-Force, Crime Syndicate,
Sinister Six, HYDRA, Brotherhood, Guardians of the Galaxy, Champions, Warbound.

### 2.4 Hero Classes (`hero-classes.json`)

**Actual shape:**
```json
{
  "id": 2,
  "abbr": "instinct",
  "slug": "instinct",
  "name": "Instinct"
}
```

**Fields:** `id` (number), `abbr` (string — same as slug), `slug` (string), `name` (string)

Note: There is no `color` or `iconKey` in the actual data. Hero class is
referenced in card data as `hc: "instinct"` — this matches `abbr`/`slug`.

**5 hero classes:** Covert, Instinct, Ranged, Strength, Tech

### 2.5 Rarities

Rarities are **not defined in a standalone metadata file**. They are encoded
in card records as a numeric `rarity` field with a companion `rarityLabel` string.

**Rarity encoding in hero card data:**

| `rarity` (number) | `rarityLabel` values | Display meaning |
|---|---|---|
| `1` | `"Common 1"`, `"Common 2"`, `"Common 3"` | Common — 3 copies in hero deck |
| `2` | `"Uncommon"`, `"Uncommon 2"` | Uncommon — 1 copy in hero deck |
| `3` | `"Rare"` | Rare — 1 copy in hero deck |

There is no `Legendary` or `Epic` rarity at the card level. "Epic" refers to
a card type (mastermind-epic, mastermind-adapting-epic) not a rarity.

**Implementation note:** If a rarity lookup table is needed for UI purposes
(e.g., colors, filter chips), it must be constructed from this encoding pattern —
there is no source file to import from.

### 2.6 Mastermind → Villain Group Leads (`leads.json`)

The `leads.json` file captures the "always leads" relationship between masterminds
and villain groups across all sets:

```json
{ "set": "mdns", "mastermind": "lilith-mother-of-demons", "villainGroups": ["lilin"] }
```

Some entries also include `henchmen` arrays for masterminds that always lead
henchman groups instead of villain groups:

```json
{ "set": "rvlt", "mastermind": "mandarin", "villainGroups": [], "henchmen": ["mandarins-rings"] }
```

Special cases:
- Some masterminds always lead "any villain group" (`"_anyVillainGroup": true`)
- Some sets have unassigned groups noted with `"_unassigned"` metadata
- Some masterminds have no required groups (`villainGroups: []` with no henchmen)

This file is the canonical relationship reference for deck construction rules.

### 2.7 Icons / Stat Symbols (`icons-meta.json`)

Used for ability text rendering — inline icons in `[icon:X]` markup tokens.

```json
{ "id": 1, "value": "attack",  "label": "Attack" },
{ "id": 2, "value": "recruit", "label": "Recruit" },
{ "id": 3, "value": "cost",    "label": "Cost" },
{ "id": 4, "value": "vp",      "label": "Victory Points" },
{ "id": 5, "value": "focus",   "label": "Focus" },
{ "id": 6, "value": "piercing","label": "Piercing Energy" },
{ "id": 7, "value": "token",   "label": "Token Card" }
```

**Fields:** `id` (number), `value` (string — used in `[icon:X]` markup), `label` (string)

### 2.8 Keywords

Keywords are referenced in ability text via `[keyword:X]` markup tokens.
There is no standalone keywords metadata file in `data/metadata/` as of this
writing. Keywords encountered in the data include:

`Patrol`, `Blood Frenzy`, `Moonlight and Sunlight`, `Hunt for Victims`, `Haunt`,
`Transform`, `Ambush`, `Fight`, `Escape`, `Master Strike`, `Always Leads`

**Implementation note:** Until a formal keywords file exists, maintain a
keywords map as part of the rules glossary. The PostgreSQL schema has a
`legendary.keywords` table — populate it from observed usage.

### 2.9 Rules / Glossary

#### Source Format

Rules data originates from the `@master-strike/data` npm package as **TypeScript
objects** — not JSON files. There are two files involved:

**`src/metadata/rules.ts`** — the rules index. Defines 15 named rules, each with:
```ts
{
  id: number,       // stable numeric ID (1–15, -1 for NONE)
  value: string,    // machine-readable slug (e.g. "shards", "traps")
  label: string,    // display name (e.g. "Shards", "Traps")
  cardTypes: number[] // card type IDs this rule applies to
}
```

The 15 rules currently defined: Shards, Multiclass Cards, Multiple Masterminds,
Divided Card, Traps, Locations, Double-Sided Transforming Schemes, Adapting
Masterminds, Villainous Weapons, Asterisk Symbol, Sidekicks, Choose a Villain
Group, Veiled Schemes, Unveiled Schemes, Wounds on Villains.

**`src/definitions/rules/*.ts`** — one file per rule (e.g. `shards.ts`, `traps.ts`).
Each exports a `RuleDefinition` — a structured array of rich content segments:

```ts
// Example shape from shards.ts:
[
  [ "Intro paragraph as a plain string..." ],
  {
    points: [
      [
        { bold: "Players:" },
        " When you gain a Shard, put it in front of you. ",
        { icon: 1 },   // icon ID maps to icons-meta.json
        " ...more text"
      ],
      // ... more bullet points
    ]
  }
]
```

Content segments are arrays of strings and objects: `{ bold: string }` for bold
text, `{ icon: number }` for inline stat icons (using `icons-meta.json` IDs).

#### Seeding Into PostgreSQL

No manual JSON conversion is needed. The package compiles to `dist/` as standard
JS modules, so the seeder can import directly:

```js
import { rulesArray } from '@master-strike/data/dist/metadata/rules.js';
import { shards }     from '@master-strike/data/dist/definitions/rules/shards.js';
// ...etc
```

The seeder serializes `RuleDefinition` arrays to JSON and inserts them into
PostgreSQL:

| Table | What gets seeded |
|---|---|
| `legendary.rules` | Index rows — `id`, `value`, `label`, `card_types` |
| `legendary.rule_docs` | Full rule text as JSON + generated `search_tsv` FTS vector |

PostgreSQL is required (not optional) for two reasons:
1. The `search_tsv` column on `rule_docs` is a PostgreSQL native full-text search
   vector — it enables rules lookup queries (e.g. "what does Shards do?").
2. `[rule:X]` markup tokens in card ability text are resolved server-side by
   querying `legendary.rules` on the `value`/`code` field.

#### Runtime Usage in the Game Engine

Rules are **never queried inside boardgame.io move functions**. They are loaded
from PostgreSQL into a `rulesCache` object once at server startup (via
`src/rules/loader.mjs`), then accessed in memory throughout the session.
This keeps moves pure and deterministic.

#### PDF Rulesheets

Expansion-specific rulesheets (e.g. `2099-rulesheet.pdf`) live in
`card-images-staging/rule-sheets/` in the setup repo. These are reference
material only — not the source for seeding. The `@master-strike/data` package
is the canonical rules source.

### 3.2 Image URL Construction

All card images use WebP format. The `imageUrl` field is present directly in
the card JSON — always prefer using the stored `imageUrl` over constructing
the URL manually.

**When construction is required** (e.g., for validation scripts or when
`imageUrl` is absent), use this pattern:

```
{baseUrl}/{setAbbr}/{setAbbr}-{prefix}-{slug}{suffix}.webp
```

**Real examples from Midnight Sons:**

| Card type | Example URL fragment | Pattern |
|---|---|---|
| Hero card (Common slot 1, cost 3) | `mdns-hr-blade-daywalker-3c1.webp` | `{abbr}-hr-{heroSlug}-{cost}c{slot}` |
| Hero card (Common slot 2, cost 4) | `mdns-hr-blade-daywalker-4c2.webp` | `{abbr}-hr-{heroSlug}-{cost}c{slot}` |
| Hero card (Uncommon, cost 5) | `mdns-hr-blade-daywalker-5u.webp` | `{abbr}-hr-{heroSlug}-{cost}u` |
| Hero card (Rare, cost 7) | `mdns-hr-blade-daywalker-7r.webp` | `{abbr}-hr-{heroSlug}-{cost}r` |
| Mastermind | `mdns-mm-lilith-mother-of-demons.webp` | `{abbr}-mm-{slug}` |
| Mastermind Epic | `mdns-me-lilith-mother-of-demons.webp` | `{abbr}-me-{slug}` |
| Mastermind Tactic | `mdns-mt-lilith-mother-of-demons-connoisseur-of-souls.webp` | `{abbr}-mt-{mastermindsSlug}-{tacticSlug}` |
| Villain | `mdns-vi-lilin-meatmarket.webp` | `{abbr}-vi-{groupSlug}-{cardSlug}` |
| Scheme | `mdns-sc-sire-vampires-at-the-blood-bank.webp` | `{abbr}-sc-{slug}` |
| Scheme Transform | `mdns-sx-great-old-one-chthon.webp` | `{abbr}-sx-{slug}` |

**Suffix legend for hero cards:**

| rarity | slot | suffix |
|---|---|---|
| Common | 1 | `{cost}c1` |
| Common | 2 | `{cost}c2` |
| Common | 3 | `{cost}c3` |
| Uncommon | — | `{cost}u` |
| Rare | — | `{cost}r` |

**Note:** The suffix encodes both cost and rarity — `3c1` means cost 3, Common 1.
The card prefix (`hr`) is always the same for all hero rarities.

### 3.3 Registry Config

`registry-config.json` at the R2 root lists all set abbreviations. Fetch this
first to know which sets are available before fetching per-set metadata.

### 3.4 UI Icons

Card type prefixes, hero class abbreviations, and icon values from `icons-meta.json`

## 4. PostgreSQL Schema

The PostgreSQL schema (`data/legendary_library_schema.sql`) normalizes card
and metadata JSON into relational tables in the `legendary` schema.

### 4.1 Key Tables

| Table | Purpose |
|---|---|
| `legendary.sets` | Set lookup — `set_code` is the 4-char abbreviation |
| `legendary.card_types` | Card type lookup — `code` is the slug |
| `legendary.teams` | Team lookup |
| `legendary.hero_classes` | Hero class lookup |
| `legendary.keywords` | Keyword lookup |
| `legendary.icons` | Icon/stat symbol lookup |
| `legendary.rarities` | Rarity lookup |
| `legendary.rules` | Rules glossary index |
| `legendary.rule_docs` | Full rules text + full-text search vector |
| `legendary.hero_decks` | Hero deck groups (each hero is a deck) |
| `legendary.villain_groups` | Villain group collections |
| `legendary.henchman_groups` | Henchman group collections |
| `legendary.masterminds` | Mastermind entities |
| `legendary.cards` | All card records — FK to sets, card_types, hero_decks, etc. |
| `legendary.card_keywords` | Many-to-many: card ↔ keyword |
| `legendary.card_icons` | Many-to-many: card ↔ icon |
| `legendary.card_rules` | Many-to-many: card ↔ rules reference |
| `legendary.source_files` | Raw JSON audit trail |

### 4.2 What PostgreSQL Stores (Rules Engine Data Only)

PostgreSQL is the **rules engine backing store**, not a card display mirror.

**Store in PostgreSQL:**
- Mastermind strike counts, victory points
- Scheme twist/epic counts, setup rules text
- Villain group → mastermind relationships (`alwaysLeads` / `ledBy`)
- Hero deck compositions (which cards belong to which deck)
- Card type lookups for rules logic
- Full-text searchable rules glossary

**Do NOT store in PostgreSQL:**
- Card `imageUrl` values — these are R2 CDN URLs, fetch from R2 JSON at runtime
- Full ability text strings — these are display data, not rules logic
- Live game state (`G`, `ctx`) — boardgame.io manages this in memory

### 4.3 Key Foreign Keys

```
cards → sets (via set_id)
cards → card_types (via card_type_id)
cards → hero_decks (via hero_deck_id) — null for non-hero cards
cards → villain_groups (via villain_group_id) — null for non-villain cards
cards → henchman_groups (via henchman_group_id)
cards → masterminds (via mastermind_id) — for tactic and epic cards
cards → rarities (via rarity_id)
card_keywords → cards, keywords
hero_decks → sets, teams, hero_classes
```

### 4.4 External ID Field

The `cards` table uses `ext_id TEXT UNIQUE NOT NULL` as a stable external
identifier from the source JSON — separate from the auto-incremented `card_id`.
Use `ext_id` when referencing cards from the SPA or game server — it is stable
across re-seeds. Use `card_id` only within PostgreSQL queries.

## 5. Ability Text Markup Language

Card ability text uses a lightweight markup language for inline tokens.
All tokens use bracket notation:

| Token format | Meaning | Example |
|---|---|---|
| `[keyword:X]` | Named keyword | `[keyword:Patrol]` |
| `[icon:X]` | Stat icon | `[icon:attack]`, `[icon:recruit]`, `[icon:vp]` |
| `[hc:X]` | Hero class icon | `[hc:instinct]`, `[hc:covert]`, `[hc:ranged]` |
| `[team:X]` | Team icon | `[team:marvel-knights]`, `[team:x-men]` |
| `[rule:X]` | Rules glossary reference | `[rule:additional Mastermind]`, `[rule:Transforms]` |

The `X` value in each token corresponds to:
- `keyword:` → `name` field in keywords lookup (lowercase, hyphenated for multi-word)
- `icon:` → `value` field in `icons-meta.json`
- `hc:` → `abbr`/`slug` field in `hero-classes.json`
- `team:` → `slug` field in `hero-teams.json`
- `rule:` → `code` field in `legendary.rules` table (may include spaces)

**Parsing note:** The `abilities` array in raw card JSON may contain:
- Plain text strings
- Strings with one or more inline tokens
- Strings that are `"[object Object]"` — a known data quality issue from
  the source pipeline; treat these as blank/unknown

## 6. Mastermind → Villain Group Relationship

The relationship between masterminds and villain groups is maintained at
three levels:

**Level 1 — Per-set card JSON** (`data/metadata/{abbr}.json`):
- Mastermind has `alwaysLeads: string[]` (villain/henchman group slugs)
- Villain group has `ledBy: string[]` (mastermind slugs — inverse)

**Level 2 — Leads reference file** (`data/metadata/leads.json`):
- Canonical cross-set reference for all mastermind → group relationships
- Includes `henchmen` arrays for mastermind → henchman group relationships
- Notes unassigned groups and special "any villain group" cases

**Level 3 — PostgreSQL**:
- Seeded from the above sources via `load_legendary_data.mjs`
- Queryable at runtime by the rules engine

When seeding or validating, check all three levels for consistency.

## 7. Deck Data (User-Generated)

Deck data is user-generated and not part of the canonical card data.

### 7.1 Deck Shape

```json
{
  "id": "string (UUID)",
  "name": "string",
  "createdAt": "string (ISO 8601)",
  "updatedAt": "string (ISO 8601)",
  "cards": [
    { "cardId": "string (ext_id from PostgreSQL)", "count": 1 }
  ]
}
```

Cards are referenced by `ext_id` — not by display name or slug. This ensures
deck records remain valid across metadata changes.

### 7.2 Derived Deck Validation

Computed at runtime — never persisted:
- Total card count
- Count by card type category (Hero, Villain, Mastermind, Scheme)
- Scheme / mastermind legality (at most 1 of each)
- Rule violations (warnings only — never block export)

## 9. Search and Filter (Derived at Runtime)

Search and filter state is **derived from card data and metadata lookups** —
it is never separately authored or persisted.

### 9.1 Filterable Fields

All filter values must use IDs from lookup tables — not display strings:

| Filter dimension | Source field | Lookup |
|---|---|---|
| Set | card's set `abbr` | `sets.json` by `abbr` |
| Card type | card type `slug` | `card-types.json` by `slug` |
| Card type category | derived from type slug | derived category grouping (§2.1) |
| Team | `team` field (hero slug) | `hero-teams.json` by `slug` |
| Hero class | `hc` field | `hero-classes.json` by `abbr` |
| Keyword | `[keyword:X]` tokens in abilities | keywords lookup by name |
| Cost | `cost` field | numeric range |
| Attack | `attack` field | string → numeric (strip `+`) |
| Recruit | `recruit` field | string → numeric (strip `+`) |
| Rarity | `rarity` field | 1/2/3 encoding |

### 9.2 Sort Options

| Sort mode | Source field |
|---|---|
| Name (A–Z) | `name` |
| Set (release order) | set `pkgId` |
| Cost (low to high) | `cost` (numeric) |
| Attack (high to low) | `attack` (parse numeric from string) |
| Recruit (high to low) | `recruit` (parse numeric from string) |
| Rarity | `rarity` (numeric) |

### 9.3 Parsing Numeric Values from String Stats

`attack` and `recruit` in hero cards are stored as strings (`"2"`, `"2+"`, `"0+"`).
For sort and range filtering, parse the numeric base value:
- Remove trailing `+` before parsing
- `null` values sort last (or are excluded from range filters)
- Do not throw on non-parseable values — return `null` and exclude from range

## 13. Data Quality Notes

Known data quality issues encountered in the raw JSON — handle these in
seeder scripts and validation tools rather than failing:

| Issue | Location | Handling |
|---|---|---|
| `"[object Object]"` strings in `abilities` | Some set JSON files | Treat as blank; log a warning |
| Scheme `id: null` for transform reverse-side cards | Transform schemes | Accept null ID; use `slug` as key |
| `_unassigned` villain group entries in `leads.json` | Some Secret Wars sets | Skip; document as needing manual review |
| `_PLACEHOLDER_DELETE_THIS` entry at top of `leads.json` | `leads.json` first entry | Skip any entry where `mastermind` = "PLACEHOLDER_DELETE_THIS" |
| `vAttack` as string or number inconsistently | Mastermind/villain cards | Always parse to number for comparison |
| `vp` as string or number inconsistently | Villain cards | Always parse to number |
| `attack` and `recruit` as string `"2+"` | Hero cards | Strip `+` for numeric use |

## 14. Design Principles

These principles are non-negotiable across all execution prompts:

1. **IDs and slugs are the keys** — never use display strings as lookup keys.
   Always resolve display names through lookup tables at render time.

2. **Display data stays in R2** — card images, flavor text, and ability strings
   are fetched from R2 JSON. PostgreSQL stores only rules-engine-relevant data.

3. **Derived data is never persisted** — filter state, search indexes, deck
   validation results, and stat ranges are computed at runtime from canonical data.

4. **All UI must be reproducible from data** — given the same data, the UI
   must produce the same output. No hard-coded display strings beyond labels.

5. **Mock data must match real schemas exactly** — fallback sample data in any
   module must use the exact field names and value types defined in this document.
   Simplified or renamed fields are not acceptable.

6. **Treat `imageUrl` as primary** — always prefer the stored `imageUrl` field
   over constructing image paths from parts. Construct only as a fallback for
   validation or when `imageUrl` is genuinely absent.

7. **Parse, don't trust numeric strings** — `attack`, `recruit`, and `vAttack`
   are stored as strings in the raw data. Parse them to numbers where comparison
   or arithmetic is needed. Never assume they are already numbers.
