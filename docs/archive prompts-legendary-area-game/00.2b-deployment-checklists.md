## Checklist A ŌĆö Cloudflare R2 Data & Image Assets

R2 is the primary CDN for all card data JSON and all card images. The SPA
and game server both fetch from R2 at runtime. Nothing in PostgreSQL replaces
R2 ŌĆö they serve different purposes.

**R2 base URL:** `https://images.barefootbetters.com`

---

### A.0 ŌĆö How to Use `pnpm validate`

`packages/registry/scripts/validate.ts` covers all Checklist A verifications
in five sequential phases. Run it in one of two modes:

**Local mode (default)** ŌĆö validates JSON files in `data/metadata/` against
their Zod schemas. Does not check image reachability unless `R2_BASE_URL` is
also set. Uses `sets.json` as the set index (since `registry-config.json` is
an R2 artifact, not a source file).

```powershell
# From monorepo root ŌĆö local validation, no images:
CARDS_DIR=data/metadata pnpm registry:validate

# Local + skip image checks explicitly:
CARDS_DIR=data/metadata SKIP_IMAGES=1 pnpm registry:validate
```

**R2 mode** ŌĆö fetches and validates everything from the live R2 bucket,
including `registry-config.json`, all metadata files, all per-set card JSON,
and image spot-checks (3 HEAD requests per set).

```powershell
# Full R2 validation including image spot-checks:
R2_BASE_URL=https://images.barefootbetters.com CARDS_DIR=data/metadata pnpm registry:validate

# R2 validation without image checks (faster):
R2_BASE_URL=https://images.barefootbetters.com SKIP_IMAGES=1 CARDS_DIR=data/metadata pnpm registry:validate
```

**What the script checks in each phase:**

| Phase | What it validates | Error vs Warning |
|---|---|---|
| Phase 1 ŌĆö Registry Config | `registry-config.json` exists, valid array of abbrs (R2 mode only) | Error if missing/invalid |
| Phase 2 ŌĆö Metadata Files | `sets.json`, `card-types.json`, `hero-classes.json`, `hero-teams.json`, `icons-meta.json`, `leads.json` | Error if missing; warning if low count |
| Phase 3 ŌĆö Per-Set Card JSON | All `{abbr}.json` files pass Zod schema; imageUrl domain check; `[object Object]` detection | Error if schema fails; warning for data quality |
| Phase 4 ŌĆö Cross-References | `alwaysLeads` slugs resolve to villain groups; no duplicate slugs across sets | Warning only (may be henchmen or cross-set) |
| Phase 5 ŌĆö Image Spot-Checks | HEAD request to first mastermind/villain/hero image per set returns HTTP 200 | Counted as errors in exit code |

**Output files:**

- `dist/registry-health.json` ŌĆö full validation report with all findings,
  image failures, and summary counts
- Console output ŌĆö phase-by-phase Ō£ō/Ō£Ś/ŌÜĀ lines and final SUMMARY block

**Exit codes:**
- `0` ŌĆö passed (warnings do not cause failure)
- `1` ŌĆö one or more errors found, or any image returned non-200

| Wound | `{abbr}-wd-{slug}.webp` | ŌĆö |

**Image existence checks (Phase 5 of `pnpm validate`):**

Phase 5 issues HEAD requests (not GET ŌĆö no image data downloaded) against
the `imageUrl` stored directly in each card's JSON. It samples 3 URLs per set:
first mastermind card, first villain card, first hero slot-1 card.

- [ ] At least one image folder exists per set abbreviation (40 folders total on R2)
- [ ] All sampled mastermind image URLs return HTTP 200
- [ ] All sampled villain image URLs return HTTP 200
- [ ] All sampled hero slot-1 image URLs return HTTP 200
- [ ] `dist/registry-health.json` field `summary.imagesFailed` is `0`

**Verify with:** `pnpm validate` in R2 mode (Phase 5 runs automatically unless
`SKIP_IMAGES=1`). Expected Phase 5 console output for a healthy set:
```
ŌöĆŌöĆ Phase 5 ŌĆö Image Spot-Checks (120 URLs, 3 per set) ŌöĆŌöĆ
  Ō£ō 120 of 120 images returned HTTP 200
```

If any images fail:
```
  Ō£Ś https://images.barefootbetters.com/mdns/mdns-mm-...webp ŌĆö HTTP 404
  Ō£Ś 1 image(s) not reachable
```

**Delay between requests:** 50ms by default (`IMAGE_DELAY_MS` env var) to stay
under R2 rate limits during bulk spot-checks.

### A.6 ŌĆö Uploading New Set Data

When a new expansion is ready, run these steps in order:

- [ ] Add the new set JSON to `data/metadata/{abbr}.json` locally
- [ ] Run `pnpm validate` in local mode ŌĆö confirm zero errors for the new set

      ```powershell
      rclone copy images/{abbr}/ r2:legendary-images/{abbr}/
      ```
- [ ] Add the new abbreviation to `registry-config.json`
- [ ] Upload the updated config:
      ```powershell
      rclone copy registry-config.json r2:legendary-images/
      ```
- [ ] Run `pnpm validate` in R2 mode to confirm the live bucket is correct:
      ```powershell

      ```

- [ ] Run `pnpm seed` to update PostgreSQL with the new set's rules data
- [ ] Verify the new set appears in the Registry Viewer at `cards.barefootbetters.com`

## Checklist B ŌĆö Render.com PostgreSQL Database

PostgreSQL stores **rules engine data only** ŌĆö not display data or game state.
The distinction matters: all card images and display text stay in R2; only
game-mechanical fields (strike counts, twist counts, group relationships) go
in the database.

**Connection:** `DATABASE_URL` environment variable in Render dashboard
**Schema:** All tables live in the `legendary` PostgreSQL schema

---

### B.1 ŌĆö Pre-conditions Before Running Migrations

- [ ] `DATABASE_URL` is set in Render dashboard (auto-wired from managed PostgreSQL)
- [ ] `DATABASE_URL` is set in local `.env` for development use
- [ ] `EXPECTED_DB_NAME` is set to `legendary_arena` (used by health check)
- [ ] `pnpm check` passes the PostgreSQL connection test (Prompt 00.4)
- [ ] Node.js v22+ is installed (`node --version`)
- [ ] `pg` npm package is installed in the project

---

### B.2 ŌĆö Migration Execution Order

Migrations are applied by `scripts/migrate.mjs`. They must be run in filename
order. Each is idempotent ŌĆö safe to re-run.

**Run:** `pnpm migrate`

- [ ] `001_initial_schema.sql` ŌĆö creates `legendary` schema + all lookup tables:
      `sets`, `card_types`, `teams`, `hero_classes`, `keywords`, `icons`,
      `rarities`, `rules`, `rule_docs`
- [ ] `002_seed_base_set.sql` ŌĆö seeds Galactus mastermind (FK relationship proof)
- [ ] `003_add_game_sessions.sql` ŌĆö creates `game_sessions` table for lobby
- [ ] `004_upsert_indexes.sql` ŌĆö adds UNIQUE constraints on `slug` columns
      required for `ON CONFLICT (slug) DO UPDATE` upserts
- [ ] `005_lobby_columns.sql` ŌĆö adds lobby metadata columns to `game_sessions`:
      `join_code`, `status`, `player_count`, `filled_seats`,
      `masterminds_slug`, `scheme_slug`, `seats`

**Verify:** `pnpm migrate` output shows `[migrate] all migrations up to date`
with exit code 0

### B.7 ŌĆö Game-Specific Data Verification

These checks confirm the rules engine data is correct, not just populated:

      ```sql
      SELECT m.name, vg.name AS leads
      FROM legendary.masterminds m
      JOIN legendary.villain_groups vg
      ```

      ```sql
      -- Each hero should have 4 cards
      ```

      ```sql
      SELECT COUNT(*) FROM legendary.cards WHERE ext_id IS NULL;
      -- Expected: 0
      ```

## Quick Reference: Who Provides What

| Data | Provider | When updated |
|---|---|---|
| Card JSON metadata | R2 `/metadata/{abbr}.json` | When new set ships |
| Card images | R2 `/{abbr}/*.webp` | When new set ships |
| Lookup tables (sets, card_types, etc.) | R2 `/metadata/*.json` + PostgreSQL | On seed |
| Rules engine data (strike counts, relationships) | PostgreSQL | On seed |
| Live game state (hand, deck, city) | boardgame.io (in-memory) | Each game turn |
| Player preferences | Browser localStorage | Per session |
| Board visual structure | `boardLayout.js` constants | On code deploy |
| Card images (canvas) | R2 via `cardImageCache.js` | Lazily on demand |
| Lookup display names (modal) | `boardLookups.js` | On page load |
