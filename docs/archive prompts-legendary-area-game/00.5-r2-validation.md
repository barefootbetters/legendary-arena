# Prompt 00.5 — R2 Data & Image Validation

> **FULL CONTENTS MODE** — Output: (1) files changed, (2) full contents of every
> changed file, (3) commands to run with expected output, (4) acceptance checklist.
> Never output diffs. Read-only validation only — do not modify R2 data.
> ESM only. Node v22+. No environment variables required (R2 is public).
> Human-style code: explicit, readable, junior-maintainable. See `00.6-code-style.md`.

## Purpose

Diagnostic tool, not a build step. Run before Prompt 03 (seeding) and any time
a new set is uploaded to R2. Catches data problems early — before they cause
silent failures in the rules engine, seeder, or game UI.

## Assumes

- R2 bucket is publicly readable at `https://images.barefootbetters.com`
- `registry-config.json` exists at the R2 root listing all set abbreviations
- Per-set metadata JSON lives at `/metadata/{abbr}.json`
- Card images live at `/{abbr}/{typePrefix}_{slug}.webp`
  - `mm` = mastermind, `vi` = villain, `hr` = hero (+ sequence suffix `_1`–`_4`)
- `patch-relationships.mjs` has added `alwaysLeads` and `ledBy` to affected sets

---

## Role

You are writing a Node.js diagnostic script that validates the health of a
Cloudflare R2 card data store. The script must be opinionated about what
"correct" looks like — it does not just check files exist, it checks that
the data inside them matches the expected schema and cross-references.

## Code Style Mandate

All output must follow `00.6-code-style.md`. Key rules for this prompt:

- **The script is a linear top-to-bottom sequence** — four named async functions
  (one per phase), called in order from a `main()` function. No class, no factory,
  no runner abstraction.
- **Phase functions are named by what they check**, not by their phase number:
  `checkRegistry()`, `checkMetadataForAllSets()`, `spotCheckImages()`,
  `findCrossSetDuplicateSlugs()`.
- **Validation checks are explicit `if` blocks** — each check is its own `if`
  with a full-sentence message pushed to the errors or warnings array.
  ```js
  // BAD — compressed into one expression
  if (!setJson.heroes?.length) errors.push(`[${abbr}] missing heroes`);

  // GOOD — separate check with a clear message
  if (!setJson.heroes) {
    errors.push(`[${setAbbreviation}] metadata JSON is missing the "heroes" key entirely.`);
  } else if (setJson.heroes.length === 0) {
    errors.push(`[${setAbbreviation}] "heroes" array is present but empty.`);
  }
  ```
- **Slug checks use explicit `for...of` loops**, not `.filter()` chains.
- **The 50ms delay between sets is a named constant with a `// why:` comment**:
  ```js
  const DELAY_BETWEEN_SETS_MS = 50;
  // why: R2 rate-limits aggressive parallel fetches. A 50ms pause between
  // sequential set fetches keeps us well under the limit for 40+ sets.
  await sleep(DELAY_BETWEEN_SETS_MS);
  ```
- **`// why:` comments** on the HEAD request strategy, the sample-only approach
  for image checks, and the exit code behavior.

## Deliverables

### 1. Validation script (`scripts/validate-r2.mjs`)

A standalone Node.js ESM script. Read-only throughout.
Written as four named async phase functions called from `main()`.

#### Phase 1 — `checkRegistry()`
- Fetch `registry-config.json`
- Verify valid JSON containing an array of set abbreviations
- Print total set count and list of abbreviations found

#### Phase 2 — `checkMetadataForAllSets(setAbbreviations)`

For each abbreviation, fetch `/metadata/{abbr}.json` and check:

**Structural checks (ERROR if missing):**
- Top-level keys present: `heroes`, `villains`, `masterminds`, `schemes`
- Each hero has: `slug`, `cards` (array, length 1–4), `setAbbr`
- Each villain has: `slug`, `setAbbr`
- Each mastermind has: `slug`, `strikeCount` (or equivalent numeric field), `setAbbr`
- Each scheme has: `slug`, `twistCount`, `setAbbr`

Each check is its own `if` block with a full-sentence error message that
includes the set abbreviation and the field name that is missing or invalid.

**Slug format checks (WARN if violated):**
- All slugs are lowercase, hyphens only (no spaces, underscores, or uppercase)
- No duplicate slugs within a single set's JSON
- No slug is an empty string

**Relationship checks (WARN if missing):**
- Any mastermind appearing in the `RELATIONSHIPS` map from `patch-relationships.mjs`
  should have a non-empty `alwaysLeads` array
- If `alwaysLeads` is present, each entry must match a villain slug in that set's
  `villains` array

#### Phase 3 — `spotCheckImages(setAbbreviations)`

For each set, HEAD-check one image per type:
- First mastermind: `/{abbr}/mm_{slug}.webp`
- First villain: `/{abbr}/vi_{slug}.webp`
- First hero's first card: `/{abbr}/hr_{slug}_1.webp`

Do NOT check every image — that is 1000+ requests. Sample only.
Report any 404s found and the total sample size.

The HEAD request for each image is its own `try/catch` block with a
`// why:` comment explaining why HEAD is used instead of GET.

#### Phase 4 — `findCrossSetDuplicateSlugs(allSetMetadata)`

Collect all slugs across all sets (heroes, villains, masterminds, schemes)
using an explicit `for...of` loop for each card type — not `.flatMap()`.
Report any slug appearing in more than one set — causes upsert collisions in PostgreSQL.

## Hard Constraints

- Read-only — no writes to R2, no database connections, no file creation
- ESM only; Node.js built-in `fetch` only (no axios, node-fetch)
- HEAD requests for image checks — do not download image data
- Process sets sequentially with 50ms delay between sets (avoid R2 rate limits)
- Must complete in under 90 seconds for 43 sets
- Must run with no environment variables (R2 is publicly readable)
- **Four named async phase functions called from `main()`** — no class, no factory
- **Each validation check is its own `if` block** with a full-sentence message
- **No `.flatMap()` or `.reduce()`** for collecting slugs — use explicit `for...of` loops
- **No abbreviated variable names** — `setAbbreviation`, `mastermindsData`, not `abbr`, `mm`
- **No nested ternaries** — including the exit code logic
- **The 50ms delay is a named constant** with a `// why:` comment
- **`// why:` comments** on HEAD requests, sample-only strategy, and exit codes
- **All error and warning messages are full sentences** including the set abbreviation
  and the specific field or value that failed
