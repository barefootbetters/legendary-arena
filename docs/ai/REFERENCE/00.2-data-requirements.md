## Purpose

Legendary Arena is **data-driven**: every UI behavior, game rule, and match
configuration derives from card data and metadata lookups. This document:

- Documents the real data that exists on disk (not aspirational)
- Defines canonical field names for all card and metadata types
- Specifies image URL construction rules from real R2 examples
- Clarifies what lives where (R2 JSON vs. PostgreSQL vs. runtime state)
- Serves as the shared contract between the data pipeline, game server, and SPA

## 1. Card Data

### 1.1 Data Shape on Disk (R2 / `data/metadata/{abbr}.json`)

Card data is stored per-set. Each JSON file is an object with top-level arrays
organized by card category. The structure is **hierarchical**, not flat:

```json
{
  "id": 37,
  "abbr": "mdns",
  "exportName": "MidnightSons",
  "heroes": [ ... ],
  "masterminds": [ ... ],
  "villains": [ ... ],
  "henchmen": [ ... ],
  "schemes": [ ... ],
  "bystanders": [ ... ],
  "wounds": [ ... ],
  "other": [ ... ]
}
```

### 1.2 Hero Deck Shape

Heroes are organized as **decks** — each hero is a named set of cards.
Do not treat hero cards as independent flat records.

```json
{
  "id": 260,
  "name": "Blade, Daywalker",
  "slug": "blade-daywalker",
  "team": "midnight-sons",
  "cards": [
    {
      "name": "Hunt High And Low",
      "displayName": "Hunt High And Low",
      "slug": "hunt-high-and-low",
      "rarity": 1,
      "rarityLabel": "Common 1",
      "slot": 1,
      "hc": "instinct",
      "cost": 3,
      "attack": "2",
      "recruit": null,
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-hr-blade-daywalker-3c1.webp",
      "abilities": [
        "[keyword:Patrol]: If it's empty, reveal the top card of your deck. You may KO it."
      ]
    }
  ]
}
```

**Hero card field reference:**

| Field | Type | Notes |
|---|---|---|
| `name` | `string` | Card name (individual card, not hero name) |
| `displayName` | `string` | Usually same as `name`; use for UI display |
| `slug` | `string` | URL-safe card identifier — used in image URLs |
| `rarity` | `number` | 1 = Common, 2 = Uncommon, 3 = Rare |
| `rarityLabel` | `string` | "Common 1", "Common 2", "Common 3", "Uncommon", "Rare" |
| `slot` | `number` | Card position within hero deck (1–4) |
| `hc` | `string` | Hero class abbreviation — matches `abbr` in `hero-classes.json` |
| `cost` | `number` | Recruit cost to buy this card |
| `attack` | `string \| null` | Fight value — `"2"`, `"2+"`, `"0+"` etc. — always a string |
| `recruit` | `string \| null` | Recruit value — same format as `attack`; null if none |
| `imageUrl` | `string` | Full R2 URL — prefer this over reconstructing |
| `abilities` | `string[]` | Rule text lines using ability markup (see §5) |

### 1.3 Mastermind Shape

```json
{
  "id": 91,
  "name": "Lilith, Mother of Demons",
  "slug": "lilith-mother-of-demons",
  "alwaysLeads": ["lilin"],
  "vp": 6,
  "cards": [
    {
      "name": "Lilith, Mother of Demons",
      "slug": "lilith-mother-of-demons",
      "tactic": false,
      "vAttack": "8+",
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-mm-lilith-mother-of-demons.webp",
      "abilities": [ ... ]
    },
    {
      "name": "Epic Lilith",
      "slug": "epic-lilith",
      "tactic": false,
      "vAttack": "10+",
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-me-lilith-mother-of-demons.webp",
      "abilities": [ ... ]
    },
    {
      "name": "Connoisseur of Souls",
      "slug": "connoisseur-of-souls",
      "tactic": true,
      "vAttack": 8,
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-mt-lilith-mother-of-demons-connoisseur-of-souls.webp",
      "abilities": [ ... ]
    }
  ]
}
```

**Mastermind field reference:**

| Field | Type | Notes |
|---|---|---|
| `id` | `number` | Stable numeric ID |
| `name` | `string` | Mastermind display name |
| `slug` | `string` | URL-safe identifier |
| `alwaysLeads` | `string[]` | Villain/henchman group slugs this mastermind always leads |
| `vp` | `number` | Victory points when defeated |
| `cards` | `array` | Includes the mastermind card, epic variant, and all tactics |

**Mastermind card field reference:**

| Field | Type | Notes |
|---|---|---|
| `name` | `string` | Card name |
| `slug` | `string` | Used in image URL |
| `tactic` | `boolean` | `false` = mastermind/epic card; `true` = tactic card |
| `vAttack` | `string \| number` | Fight value — may be string (`"8+"`) or number (`8`) |
| `imageUrl` | `string` | Full R2 URL |
| `abilities` | `string[]` | Rule text |

### 1.4 Villain Group Shape

```json
{
  "id": 112,
  "name": "Lilin",
  "slug": "lilin",
  "ledBy": ["lilith-mother-of-demons"],
  "cards": [
    {
      "name": "Meatmarket",
      "slug": "meatmarket",
      "vp": "3",
      "vAttack": "3+",
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-vi-lilin-meatmarket.webp",
      "abilities": [ ... ]
    }
  ]
}
```

`ledBy` is the inverse of mastermind `alwaysLeads`.

**Villain card fields:** `name`, `slug`, `vp` (`string | number`), `vAttack`
(`string | number`), `imageUrl`, `abilities[]`

### 1.5 Scheme Shape

```json
{
  "id": 170,
  "name": "Sire Vampires at the Blood Bank",
  "slug": "sire-vampires-at-the-blood-bank",
  "imageUrl": "https://images.barefootbetters.com/mdns/mdns-sc-sire-vampires-at-the-blood-bank.webp",
  "cards": [ { "abilities": [ ... ] } ]
}
```

Transform schemes have `cardType: "scheme-transform"`. Their reverse-side card
may have `id: null` — do not require a non-null `id` for transform cards.

## 2. Metadata Lookups

All metadata files live in `data/metadata/`. Load them before any card
rendering. Use IDs and slugs as lookup keys — never duplicate display strings
in card records.

### 2.2 Sets (`sets.json`)

**Actual shape:**
```json
{ "id": 36, "abbr": "mdns", "pkgId": 36, "slug": "midnight-sons", "name": "Midnight Sons", "releaseDate": "2023-05-10", "type": "32nd Expansion" }
```

**Fields:**

| Field | Type | Notes |
|---|---|---|
| `id` | `number` | Numeric ID |
| `abbr` | `string` | 4-char abbreviation — used in image URLs and file names |
| `pkgId` | `number` | Package ordering ID (same as `id` currently) |
| `slug` | `string` | URL-safe full name |
| `name` | `string` | Display name |
| `releaseDate` | `string` | ISO date string (`YYYY-MM-DD`) |
| `type` | `string` | Free-form — e.g., `"1st Core Set"`, `"32nd Expansion"` |

`type` is a free-form string, not an enum. Use pattern matching if
categorization is needed; do not validate as `"Core" | "Expansion" | "Promo"`.

### 2.3 Hero Teams (`hero-teams.json`)

**Actual shape:**
```json
{ "id": 12, "slug": "guardians-of-the-galaxy", "name": "Guardians of the Galaxy" }
```

No `abbr` or `iconKey` fields. Team data covers both hero teams and villain
factions (e.g., `hydra`, `brotherhood`). `id: 0` = unaffiliated.

### 2.4 Hero Classes (`hero-classes.json`)

**Actual shape:**
```json
{ "id": 2, "abbr": "instinct", "slug": "instinct", "name": "Instinct" }
```

`abbr` and `slug` are identical. Card data references class as `hc: "instinct"` —
this matches `abbr`. No `color` or `iconKey` fields.

**5 hero classes:** Covert, Instinct, Ranged, Strength, Tech

### 2.5 Rarities

No standalone rarities file exists. Rarities are encoded on card records:

| `rarity` (number) | `rarityLabel` values | Meaning |
|---|---|---|
| `1` | "Common 1", "Common 2", "Common 3" | 3 copies in hero deck |
| `2` | "Uncommon", "Uncommon 2" | 1 copy in hero deck |
| `3` | "Rare" | 1 copy in hero deck |

If a rarity lookup table is needed for UI purposes, construct it from this
encoding pattern — there is no source file to import from.

### 2.6 Mastermind → Villain Group Leads (`leads.json`)

```json
{ "set": "mdns", "mastermind": "lilith-mother-of-demons", "villainGroups": ["lilin"] }
```

Some entries include a `henchmen` array for masterminds leading henchman groups:
```json
{ "set": "rvlt", "mastermind": "mandarin", "villainGroups": [], "henchmen": ["mandarins-rings"] }
```

**Special cases — handle explicitly:**
- `"_anyVillainGroup": true` — mastermind can lead any villain group
- `"_unassigned"` entries — skip and flag for manual review
- Skip any entry where `mastermind` equals `"PLACEHOLDER_DELETE_THIS"`

### 2.7 Icons / Stat Symbols (`icons-meta.json`)

```json
{ "id": 1, "value": "attack",   "label": "Attack" },
{ "id": 2, "value": "recruit",  "label": "Recruit" },
{ "id": 3, "value": "cost",     "label": "Cost" },
{ "id": 4, "value": "vp",       "label": "Victory Points" },
{ "id": 5, "value": "focus",    "label": "Focus" },
{ "id": 6, "value": "piercing", "label": "Piercing Energy" },
{ "id": 7, "value": "token",    "label": "Token Card" }
```

`value` is used in `[icon:X]` ability markup.

### 2.8 Keywords

No standalone keywords file exists. Keywords appear in ability text via
`[keyword:X]` tokens. The PostgreSQL schema has a `legendary.keywords`
table — populate from observed usage in the card data.

### 2.9 Rules / Glossary

#### Source Format

Rules originate from the `@master-strike/data` npm package (TypeScript,
compiles to `dist/`). Two files:

**`src/metadata/rules.ts`** — rules index. 15 named rules, each with:
```ts
{ id: number, value: string, label: string, cardTypes: number[] }
```

**`src/definitions/rules/*.ts`** — one file per rule. Each exports a
`RuleDefinition` array of rich content segments:
```ts
[
  [ "Intro paragraph as a plain string..." ],
  { points: [ [{ bold: "Players:" }, " When you gain a Shard, put it in front of you..."] ] }
]
```

Segments: plain strings, `{ bold: string }` for bold text, `{ icon: number }`
for inline stat icons (IDs map to `icons-meta.json`).

#### Seeding Into PostgreSQL

Import directly from the compiled `dist/`:
```js
import { rulesArray } from '@master-strike/data/dist/metadata/rules.js';
import { shards }     from '@master-strike/data/dist/definitions/rules/shards.js';
```

| Table | What gets seeded |
|---|---|
| `legendary.rules` | Index rows — `id`, `value`, `label`, `card_types` |
| `legendary.rule_docs` | Full rule text as JSON + `search_tsv` FTS vector |

#### Runtime Usage

Rules are **never queried inside boardgame.io move functions**. Load from
PostgreSQL into a `rulesCache` once at server startup via `src/rules/loader.mjs`.
This keeps moves pure and deterministic.

### 3.2 Image URL Construction

Always prefer the stored `imageUrl` field. Construct URLs only as a fallback
for validation scripts or when `imageUrl` is genuinely absent.

**Construction pattern:**
```
{baseUrl}/{setAbbr}/{setAbbr}-{prefix}-{slug}{suffix}.webp
```

**Hero suffix encoding:**

| Rarity | Slot | Suffix |
|---|---|---|
| Common | 1 | `{cost}c1` |
| Common | 2 | `{cost}c2` |
| Common | 3 | `{cost}c3` |
| Uncommon | — | `{cost}u` |
| Rare | — | `{cost}r` |

The suffix encodes both cost and rarity — `3c1` means cost 3, Common slot 1.

**Other card type patterns:**

| Card type | Pattern | Example |
|---|---|---|
| Hero | `{abbr}-hr-{heroSlug}-{suffix}` | `mdns-hr-blade-daywalker-3c1.webp` |
| Mastermind | `{abbr}-mm-{slug}` | `mdns-mm-lilith-mother-of-demons.webp` |
| Mastermind Epic | `{abbr}-me-{slug}` | `mdns-me-lilith-mother-of-demons.webp` |
| Mastermind Tactic | `{abbr}-mt-{mastermindsSlug}-{tacticSlug}` | `mdns-mt-lilith-mother-of-demons-connoisseur-of-souls.webp` |
| Villain | `{abbr}-vi-{groupSlug}-{cardSlug}` | `mdns-vi-lilin-meatmarket.webp` |
| Scheme | `{abbr}-sc-{slug}` | `mdns-sc-sire-vampires-at-the-blood-bank.webp` |
| Scheme Transform | `{abbr}-sx-{slug}` | `mdns-sx-great-old-one-chthon.webp` |

## 4. PostgreSQL Schema

The PostgreSQL schema (`data/legendary_library_schema.sql`) normalizes card
and metadata JSON into relational tables in the `legendary` schema.

### 4.1 Key Tables

| Table | Purpose |
|---|---|
| `legendary.sets` | Set lookup — `set_code` is the 4-char abbreviation |
| `legendary.card_types` | Card type lookup — `code` is the slug |
| `legendary.teams` | Team lookup |
| `legendary.hero_classes` | Hero class lookup |
| `legendary.keywords` | Keyword lookup |
| `legendary.icons` | Icon/stat symbol lookup |
| `legendary.rarities` | Rarity lookup |
| `legendary.rules` | Rules glossary index |
| `legendary.rule_docs` | Full rules text + full-text search vector |
| `legendary.hero_decks` | Hero deck groups |
| `legendary.villain_groups` | Villain group collections |
| `legendary.henchman_groups` | Henchman group collections |
| `legendary.masterminds` | Mastermind entities |
| `legendary.cards` | All card records |
| `legendary.card_keywords` | Many-to-many: card ↔ keyword |
| `legendary.card_icons` | Many-to-many: card ↔ icon |
| `legendary.card_rules` | Many-to-many: card ↔ rules reference |
| `legendary.source_files` | Raw JSON audit trail |

### 4.2 Boundary: What PostgreSQL Stores (Rules Engine Data Only)

**Store in PostgreSQL:**
- Mastermind strike counts, victory points
- Scheme twist/epic counts, setup rules text
- Villain group → mastermind relationships (`alwaysLeads` / `ledBy`)
- Hero deck compositions
- Card type lookups for rules logic
- Full-text searchable rules glossary

**Never store in PostgreSQL:**
- Card `imageUrl` values — fetch from R2 JSON at runtime
- Full ability text strings — display data, not rules logic
- `G` (game state) or `ctx` (boardgame.io context) — ever

### 4.3 Key Foreign Keys

```
cards → sets (via set_id)
cards → card_types (via card_type_id)
cards → hero_decks (via hero_deck_id) — null for non-hero cards
cards → villain_groups (via villain_group_id) — null for non-villain cards
cards → masterminds (via mastermind_id) — for tactic and epic cards
cards → rarities (via rarity_id)
card_keywords → cards, keywords
hero_decks → sets, teams, hero_classes
```

### 4.4 External ID Field

`cards` uses `ext_id TEXT UNIQUE NOT NULL` as a stable external identifier
from the source JSON. Use `ext_id` when referencing cards from the SPA or
game server — it is stable across re-seeds. Use `card_id` only within
PostgreSQL queries.

## 5. Ability Text Markup Language

| Token format | Maps to |
|---|---|
| `[keyword:X]` | `name` in keywords lookup |
| `[icon:X]` | `value` in `icons-meta.json` |
| `[hc:X]` | `abbr`/`slug` in `hero-classes.json` |
| `[team:X]` | `slug` in `hero-teams.json` |
| `[rule:X]` | `code` in `legendary.rules` table |

**Known data quality issue:** `abilities` may contain `"[object Object]"` —
treat as blank, log a warning, never throw.

### 8.2 Match Runtime State

Managed by boardgame.io — **never persisted to PostgreSQL**:
- `G` — game state (deck contents, hand, city, scores)

- Match phases: `Lobby` → `Setup` → `In Progress` → `Completed`

## 9. Search and Filter (Derived at Runtime — Never Persisted)

All filter values must use IDs from lookup tables — not display strings.

| Filter dimension | Source field | Lookup |
|---|---|---|
| Set | card's set `abbr` | `sets.json` by `abbr` |
| Card type | card type `slug` | `card-types.json` by `slug` |
| Card type category | derived from type slug | derived grouping in §2.1 |
| Team | `team` field | `hero-teams.json` by `slug` |
| Hero class | `hc` field | `hero-classes.json` by `abbr` |
| Keyword | `[keyword:X]` tokens in abilities | keywords lookup by name |
| Cost | `cost` field | numeric range |
| Attack | `attack` field | strip `+`, parse to number |
| Recruit | `recruit` field | strip `+`, parse to number |
| Rarity | `rarity` field | 1/2/3 encoding |

**Parsing rule:** Strip trailing `+` before parsing. Return `null` on failure —
never throw. `null` values sort last and are excluded from range filters.

## 12. Data Quality Notes

Known issues in raw JSON — handle in seeder scripts, do not fail:

| Issue | Location | Handling |
|---|---|---|
| `"[object Object]"` in `abilities` | Some set JSON files | Treat as blank; log warning |
| `id: null` on transform reverse-side cards | Transform schemes | Accept null; use `slug` as key |
| `_unassigned` entries in `leads.json` | Some Secret Wars sets | Skip; flag for manual review |
| `mastermind: "PLACEHOLDER_DELETE_THIS"` | `leads.json` first entry | Skip unconditionally |
| `vAttack` / `vp` as string or number | Mastermind/villain cards | Always parse to number for comparison |
| `attack` / `recruit` as `"2+"` | Hero cards | Strip `+` for numeric use |

## 13. Design Principles

Non-negotiable across all execution prompts:

1. **IDs and slugs are the keys** — never use display strings as lookup keys.
   Always resolve display names through lookup tables at render time.
2. **Display data stays in R2** — card images, flavor text, and ability strings
   are fetched from R2 JSON. PostgreSQL stores only rules-engine-relevant data.
3. **Derived data is never persisted** — filter state, search indexes, deck
   validation results, and stat ranges are computed at runtime from canonical data.
4. **All UI must be reproducible from data** — given the same data, the UI
   must produce the same output. No hard-coded display strings beyond labels.
5. **Mock data must match real schemas exactly** — fallback sample data in any
   module must use the exact field names and value types defined in this document.
   Simplified or renamed fields are not acceptable.
6. **`imageUrl` is primary** — always prefer the stored `imageUrl` field over
   constructing image paths from parts. Construct only as a fallback.
7. **Parse, don't trust numeric strings** — `attack`, `recruit`, and `vAttack`
   are stored as strings in raw data. Parse to numbers for comparison or arithmetic.
