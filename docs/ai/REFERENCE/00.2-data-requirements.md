## 1. Card Data Shapes

Card data is stored per-set in JSON files. Each file is an object with
top-level arrays organized by card category. The structure is **hierarchical**,
not flat.

### 1.1 Per-Set JSON Structure

```json
{
  "id": 37,
  "abbr": "mdns",
  "exportName": "MidnightSons",
  "heroes": [ ... ],
  "masterminds": [ ... ],
  "villains": [ ... ],
  "henchmen": [ ... ],
  "schemes": [ ... ],
  "bystanders": [ ... ],
  "wounds": [ ... ],
  "other": [ ... ]
}
```

Validated by `SetDataSchema` in `schema.ts`.

### 1.2 Hero Deck Shape

Heroes are organized as **decks** — each hero is a named set of cards.
Do not treat hero cards as independent flat records.

**Hero (deck-level) fields:**

| Field | Type | Notes |
|---|---|---|
| `id` | `number \| null \| undefined` | Nullable/optional: `3dtc` has null; `shld` omits entirely; `msp1` uses `-1` |
| `name` | `string` | Hero display name |
| `slug` | `string` | URL-safe identifier |
| `team` | `string` | Team slug — matches `slug` in `hero-teams.json` |
| `cards` | `HeroCard[]` | 3-7 cards per hero deck |

Validated by `HeroSchema` in `schema.ts`.

**Hero card fields:**

| Field | Type | Notes |
|---|---|---|
| `name` | `string \| undefined` | Card name (optional: `anni` cards have only slug+imageUrl) |
| `displayName` | `string \| undefined` | Usually same as `name`; `amwp` omits entirely |
| `slug` | `string` | URL-safe card identifier — used in image URLs |
| `rarity` | `1 \| 2 \| 3 \| undefined` | 1=Common, 2=Uncommon, 3=Rare |
| `rarityLabel` | `string \| undefined` | "Common 1", "Common 2", "Common 3", "Uncommon", "Rare" |
| `slot` | `number \| undefined` | Position within hero deck (1-7; no upper bound — `mgtg` has 7-card decks) |
| `hc` | `string \| undefined` | Hero class — matches `abbr` in `hero-classes.json` |
| `cost` | `string \| number \| undefined` | Recruit cost. String when modifier present: `"2*"` (star-cost in `amwp`) |
| `attack` | `string \| null \| undefined` | Fight value — `"2"`, `"2+"`, `"0+"`. Always a string when present; null if none |
| `recruit` | `string \| null \| undefined` | Recruit value — same format as `attack`; null if none |
| `imageUrl` | `string` | Full R2 URL — prefer this over reconstructing |
| `abilities` | `string[] \| undefined` | Rule text lines using ability markup (see §5) |

Validated by `HeroCardSchema` in `schema.ts`.

**Card field type convention** (from WP-003): `cost`, `attack`, `recruit`, and
`vAttack` are `string | number | undefined`. Modifier strings like `"2*"` and
`"2+"` exist in real data. Strip the modifier and parse the integer base; return
0 on unexpected input. See `ARCHITECTURE.md` "Card Field Data Quality" for the
parsing rule.

**Real example (from `mdns.json`):**

```json
{
  "name": "Hunt High And Low",
  "displayName": "Hunt High And Low",
  "slug": "hunt-high-and-low",
  "rarity": 1,
  "rarityLabel": "Common 1",
  "slot": 1,
  "hc": "instinct",
  "cost": 3,
  "attack": "2",
  "recruit": null,
  "imageUrl": "https://images.barefootbetters.com/mdns/mdns-hr-blade-daywalker-3c1.webp",
  "abilities": [
    "[keyword:Patrol]: If it's empty, reveal the top card of your deck. You may KO it."
  ]
}
```

### 1.3 Mastermind Shape

**Mastermind (entity-level) fields:**

| Field | Type | Notes |
|---|---|---|
| `id` | `number` | Stable numeric ID |
| `name` | `string` | Mastermind display name |
| `slug` | `string` | URL-safe identifier |
| `alwaysLeads` | `string[]` | Villain/henchman group slugs this mastermind leads |
| `vp` | `number \| null` | Victory points when defeated; null in `mgtg` |
| `cards` | `MastermindCard[]` | Includes mastermind card, epic variant, and tactics |

Validated by `MastermindSchema` in `schema.ts`.

**Mastermind card fields:**

| Field | Type | Notes |
|---|---|---|
| `name` | `string` | Card name |
| `slug` | `string` | Used in image URL construction |
| `tactic` | `boolean \| undefined` | `false` = mastermind/epic card; `true` = tactic card |
| `vAttack` | `string \| number \| null` | Fight value — may be string (`"8+"`) or number (`8`); null in `msmc`/`dstr`/`bkpt` |
| `imageUrl` | `string` | Full R2 URL |
| `abilities` | `string[]` | Rule text using ability markup (see §5) |

Validated by `MastermindCardSchema` in `schema.ts`.

**Tactic vs Epic distinction:** Cards with `tactic: true` are tactic cards
(shuffled into the villain deck). Cards with `tactic: false` are the main
mastermind or epic variant. The image prefix encodes this: `mm` (mastermind),
`me` (epic), `mt` (tactic). See §3 for image URL patterns.

**Real example (from `mdns.json`):**

```json
{
  "id": 91,
  "name": "Lilith, Mother of Demons",
  "slug": "lilith-mother-of-demons",
  "alwaysLeads": ["lilin"],
  "vp": 6,
  "cards": [
    {
      "name": "Lilith, Mother of Demons",
      "slug": "lilith-mother-of-demons",
      "tactic": false,
      "vAttack": "8+",
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-mm-lilith-mother-of-demons.webp",
      "abilities": ["..."]
    },
    {
      "name": "Epic Lilith",
      "slug": "epic-lilith",
      "tactic": false,
      "vAttack": "10+",
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-me-lilith-mother-of-demons.webp",
      "abilities": ["..."]
    },
    {
      "name": "Connoisseur of Souls",
      "slug": "connoisseur-of-souls",
      "tactic": true,
      "vAttack": 8,
      "imageUrl": "https://images.barefootbetters.com/mdns/mdns-mt-lilith-mother-of-demons-connoisseur-of-souls.webp",
      "abilities": ["..."]
    }
  ]
}
```

### 1.4 Villain Group Shape

| Field | Type | Notes |
|---|---|---|
| `id` | `number` | Stable numeric ID |
| `name` | `string` | Group display name |
| `slug` | `string` | URL-safe identifier |
| `ledBy` | `string[]` | Mastermind slugs — inverse of mastermind `alwaysLeads` |
| `cards` | `VillainCard[]` | Villain cards in this group |

Validated by `VillainGroupSchema` in `schema.ts`.

**Villain card fields:**

| Field | Type | Notes |
|---|---|---|
| `name` | `string` | Card name |
| `slug` | `string` | URL-safe identifier |
| `vp` | `string \| number \| null` | Victory points; null in `wpnx` |
| `vAttack` | `string \| number \| null` | Fight value; may be string or number |
| `imageUrl` | `string` | Full R2 URL |
| `abilities` | `string[]` | Rule text using ability markup |

Validated by `VillainCardSchema` in `schema.ts`.

### 1.5 Scheme Shape

| Field | Type | Notes |
|---|---|---|
| `id` | `number \| null` | Nullable: transform card reverse sides have `id: null` |
| `name` | `string` | Scheme display name |
| `slug` | `string` | URL-safe identifier |
| `imageUrl` | `string` | Full R2 URL |
| `cards` | `array` | Each entry has `abilities: string[]` |

Validated by `SchemeSchema` in `schema.ts`.

Transform schemes have `cardType: "scheme-transform"`. Their reverse-side card
may have `id: null` — do not require a non-null `id` for transform cards.

### 1.6 Henchman, Bystander, Wound, and Other Shapes

These categories use `z.array(z.unknown())` in `schema.ts` — their internal
structure is not yet formally constrained by Zod. Henchman groups follow a
pattern similar to villain groups. Bystanders, wounds, and other cards are
simple card objects with `name`, `slug`, `imageUrl`, and `abilities`.

### 1.7 Rarities

No standalone rarities file exists. Rarities are encoded on hero card records:

| `rarity` (number) | `rarityLabel` values | Meaning |
|---|---|---|
| `1` | "Common 1", "Common 2", "Common 3" | 3 copies in hero deck |
| `2` | "Uncommon", "Uncommon 2" | 1 copy in hero deck |
| `3` | "Rare" | 1 copy in hero deck |

## 2. Metadata Lookups

All metadata files live in `data/metadata/`. Load them before any card
rendering. Use IDs and slugs as lookup keys — never duplicate display strings
in card records.

**Critical distinction:** `sets.json` and `card-types.json` have similar names
but completely different shapes and purposes. See `ARCHITECTURE.md` "Registry
Metadata File Shapes" for the full explanation and the silent failure mode that
occurs when they are confused.

### 2.2 Sets (`sets.json`) — 40 sets

```json
{ "id": 36, "abbr": "mdns", "pkgId": 36, "slug": "midnight-sons", "name": "Midnight Sons", "releaseDate": "2023-05-10", "type": "32nd Expansion" }
```

| Field | Type | Notes |
|---|---|---|
| `id` | `number` | Numeric ID |
| `abbr` | `string` | 4-char abbreviation — used in image URLs and file names |
| `pkgId` | `number` | Package ordering ID (same as `id` currently) |
| `slug` | `string` | URL-safe full name |
| `name` | `string` | Display name |

| `name` | `string` | Display name |

Validated by `HeroTeamEntrySchema` in `schema.ts`.

No `abbr` or `iconKey` fields. Team data covers both hero teams and villain

### 2.6 Mastermind-Villain Leads (`leads.json`)

```json
{ "set": "mdns", "mastermind": "lilith-mother-of-demons", "villainGroups": ["lilin"] }
```

| Field | Type | Notes |
|---|---|---|
| `set` | `string \| undefined` | Set abbreviation |
| `mastermind` | `string \| undefined` | Mastermind slug |
| `villainGroups` | `string[] \| undefined` | Villain group slugs led by this mastermind |
| `henchmen` | `string[] \| undefined` | Henchman group slugs (for masterminds that lead henchmen) |

Validated by `LeadsEntrySchema` in `schema.ts`. Uses `.catchall(z.unknown())`

(`_set`, `_note`, `_unassigned`, etc.).

**Special cases — handle explicitly:**
- `"_anyVillainGroup": true` — mastermind can lead any villain group
- `"_unassigned"` entries — skip and flag for manual review
- Skip any entry where `mastermind` equals `"PLACEHOLDER_DELETE_THIS"`

Some entries include a `henchmen` array:
```json
{ "set": "rvlt", "mastermind": "mandarin", "villainGroups": [], "henchmen": ["mandarins-rings"] }
```

## 3. Image & Asset Conventions

### 3.1 R2 Base URL

```
https://images.barefootbetters.com
```

All card images are WebP format hosted on Cloudflare R2.

### 3.2 Image URL Construction

Always prefer the stored `imageUrl` field on card objects. Construct URLs only
as a fallback for validation scripts or when `imageUrl` is genuinely absent.

**Construction pattern:**
```
{baseUrl}/{setAbbr}/{setAbbr}-{prefix}-{slug}{suffix}.webp
```

Image URLs use **hyphens**, not underscores.

### 3.3 Hero Card Suffix Encoding

The hero card image suffix encodes both cost and rarity:

| Rarity | Slot | Suffix |
|---|---|---|
| Common | 1 | `{cost}c1` |
| Common | 2 | `{cost}c2` |
| Common | 3 | `{cost}c3` |
| Uncommon | — | `{cost}u` |
| Rare | — | `{cost}r` |

Example: `3c1` means cost 3, Common slot 1.

### 3.4 Image Naming Convention by Card Type

| Card Type | Pattern | Example |
|---|---|---|
| Hero | `{abbr}-hr-{heroSlug}-{suffix}` | `mdns-hr-blade-daywalker-3c1.webp` |
| Mastermind | `{abbr}-mm-{slug}` | `mdns-mm-lilith-mother-of-demons.webp` |
| Mastermind Epic | `{abbr}-me-{slug}` | `mdns-me-lilith-mother-of-demons.webp` |
| Mastermind Tactic | `{abbr}-mt-{mastermindSlug}-{tacticSlug}` | `mdns-mt-lilith-mother-of-demons-connoisseur-of-souls.webp` |
| Villain | `{abbr}-vi-{groupSlug}-{cardSlug}` | `mdns-vi-lilin-meatmarket.webp` |
| Scheme | `{abbr}-sc-{slug}` | `mdns-sc-sire-vampires-at-the-blood-bank.webp` |
| Scheme Transform | `{abbr}-sx-{slug}` | `mdns-sx-great-old-one-chthon.webp` |

### 3.5 Registry Manifest

Fetch `metadata/sets.json` from R2 first to enumerate available set
abbreviations before fetching per-set metadata. `metadata/sets.json` is
the authoritative registry manifest. See `ARCHITECTURE.md` "Registry
Metadata File Shapes" for the critical distinction between `sets.json` and
`card-types.json`.

## 4. PostgreSQL Schema Reference

> **Note:** PostgreSQL is **server-layer** infrastructure. See `ARCHITECTURE.md`
> "Layer Boundary (Authoritative)" for layer ownership rules. The game engine
> must never query PostgreSQL directly. Rules text is loaded from PostgreSQL at
> server startup and served via an in-memory cache — moves remain pure and
> deterministic.

All tables live in the `legendary.*` schema namespace. PKs use `bigserial`.
Cross-service identifiers use `ext_id text` columns.

### 4.1 Table Inventory

| Table | Purpose |
|---|---|
| `legendary.sets` | Set lookup — `set_code` is the 4-char abbreviation |
| `legendary.card_types` | Card type lookup — `code` is the slug |
| `legendary.teams` | Team lookup |
| `legendary.hero_classes` | Hero class lookup |
| `legendary.keywords` | Keyword lookup |
| `legendary.icons` | Icon/stat symbol lookup |
| `legendary.rarities` | Rarity lookup |
| `legendary.rules` | Rules glossary index |
| `legendary.rule_docs` | Full rules text + full-text search vector |
| `legendary.hero_decks` | Hero deck groups |
| `legendary.villain_groups` | Villain group collections |
| `legendary.henchman_groups` | Henchman group collections |
| `legendary.masterminds` | Mastermind entities |
| `legendary.cards` | All card records |
| `legendary.card_keywords` | Many-to-many: card-to-keyword |
| `legendary.card_icons` | Many-to-many: card-to-icon |
| `legendary.card_rules` | Many-to-many: card-to-rules reference |
| `legendary.source_files` | Raw JSON audit trail |

### 4.2 What PostgreSQL Stores vs What Stays in R2

**Store in PostgreSQL:**
- Mastermind strike counts, victory points
- Scheme twist/epic counts, setup rules text
- Villain group-to-mastermind relationships (`alwaysLeads` / `ledBy`)
- Hero deck compositions
- Card type lookups for rules logic
- Full-text searchable rules glossary

**Never store in PostgreSQL:**
- Card `imageUrl` values — fetch from R2 JSON at runtime
- Full ability text strings — display data, not rules logic
- `G` (game state) or `ctx` (boardgame.io context) — see `ARCHITECTURE.md`
  "Persistence Boundaries"

### 4.3 Key Foreign Keys

```
cards -> sets (via set_id)
cards -> card_types (via card_type_id)
cards -> hero_decks (via hero_deck_id) — null for non-hero cards
cards -> villain_groups (via villain_group_id) — null for non-villain cards
cards -> masterminds (via mastermind_id) — for tactic and epic cards
cards -> rarities (via rarity_id)
card_keywords -> cards, keywords
hero_decks -> sets, teams, hero_classes
```

### 4.4 External ID Convention

`cards` uses `ext_id TEXT UNIQUE NOT NULL` as a stable external identifier from
the source JSON. Use `ext_id` when referencing cards from the game server or
engine setup — it is stable across re-seeds. Use `card_id` (bigserial PK) only
within PostgreSQL queries.

For engine runtime concepts (`CardExtId`, zones, piles), see `ARCHITECTURE.md`
"Zone & Pile Structure".

## 5. Ability Text Markup Language

Card ability text uses bracket notation tokens that resolve to metadata
lookups at display time. This is a **data contract for parsing** — the parser
is a display-layer concern, not an engine concern.

### 5.1 Token Types

| Token Format | Resolves To | Lookup Source |
|---|---|---|
| `[keyword:X]` | `name` in keywords lookup | `legendary.keywords` table |
| `[icon:X]` | `value` in `icons-meta.json` | See §2.5 (7 icons) |
| `[hc:X]` | `abbr`/`slug` in `hero-classes.json` | See §2.4 (5 classes) |
| `[team:X]` | `slug` in `hero-teams.json` | See §2.3 (25 teams) |
| `[rule:X]` | `code` in `legendary.rules` table | PostgreSQL rules glossary |

### 5.2 Parsing Notes

- Tokens appear inline within ability text strings in the `abilities` array
- A single ability string may contain multiple tokens
- Tokens must be resolved at display time against the appropriate metadata
  lookup — not at engine runtime

### 5.3 Data Quality Warning

`abilities` arrays may contain `"[object Object]"` entries — this is a known
data quality issue from the source JSON conversion pipeline. Treat as blank,
log a warning, never throw.

## 6. Mastermind-Villain Group Relationship

The mastermind-to-villain-group relationship exists at three levels of
authority. Check all three for consistency when seeding data.

### 6.1 Three-Level Model

| Level | Source | Field |
|---|---|---|
| 1 — Per-set JSON | `data/metadata/{abbr}.json` | `alwaysLeads[]` on mastermind; `ledBy[]` on villain group |
| 2 — Cross-set index | `data/metadata/leads.json` | Canonical multi-set relationship reference |
| 3 — PostgreSQL | Seeded from levels 1 and 2 | Queryable by rules engine at runtime |

Level 1 provides per-set relationships. Level 2 (`leads.json`) provides the
canonical cross-set view. Level 3 is the queryable form seeded from both.

### 6.2 Special Cases

- **Any villain group:** Entries with `"_anyVillainGroup": true` indicate the
  mastermind can lead any villain group, not just the ones listed
- **Henchmen leads:** Some masterminds lead henchman groups instead of (or in
  addition to) villain groups, indicated by the `henchmen` array in `leads.json`
- **Unassigned groups:** `"_unassigned"` entries in `leads.json` represent
  villain groups not currently assigned to any mastermind — skip and flag for
  manual review
- **Placeholder entries:** Skip any entry where `mastermind` equals

## 7. Match Configuration

Match configuration uses the `MatchSetupConfig` payload with **9 locked fields**
(defined in WP-005A, documented in `ARCHITECTURE.md` "Match Lifecycle"):

`schemeId`, `mastermindId`, `villainGroupIds`, `henchmanGroupIds`,
`heroDeckIds`, `bystandersCount`, `woundsCount`, `officersCount`,
`sidekicksCount`

These field names are **locked**. Do not rename, abbreviate, or add fields.
All ID fields use `ext_id` strings. Count fields are numbers.

For the full match lifecycle (setup validation, state initialization, turn
flow, endgame evaluation), see `ARCHITECTURE.md` Section 2 "Match Lifecycle:
From Config to Game State". For engine runtime state (`G`, zones, piles),
see `ARCHITECTURE.md` Section 3 "Persistence Boundaries".

## 8. Authority Notes

This document is a **human-readable reference** for card data, metadata shapes,
image URL conventions, and PostgreSQL schema structure.

### 8.1 Subordination

- **`packages/registry/src/schema.ts`** is the machine-enforced source of truth
  for all field shapes, nullable/optional constraints, and validation rules. If
  this document and `schema.ts` differ on a field type, shape, or constraint,
  `schema.ts` wins.
- **`docs/ai/ARCHITECTURE.md`** governs persistence boundaries, layer
  ownership, engine contracts, and package import rules. This document must not
  contradict or restate architectural rules — it references them.
- **Individual Work Packets** own the engine contracts they define (`CardExtId`,
  `MatchSetupConfig`, `MoveResult`, zone structure, etc.). This document
  references those contracts but does not redefine them.

### 8.2 What This Document Does Not Cover

The following are explicitly out of scope:

- **Engine runtime state** (`G`, `ctx`, moves, phases, turn stages) — see
  `ARCHITECTURE.md` and the relevant Work Packets
- **User deck data** — future feature, not a current data contract
- **Search/filter logic** — UI-layer concern; derived at runtime
- **User preferences** — UI-layer; not a data contract
- **App configuration / feature flags** — UI-layer
- **Export/interoperability** — future feature
- **Deployment procedures** — see WP-042

### 8.3 Data Quality Notes

Known issues in raw JSON — handle in seeder scripts, do not fail:

| Issue | Location | Handling |
|---|---|---|
| `"[object Object]"` in `abilities` | Some set JSON files | Treat as blank; log warning |
| `id: null` on transform reverse-side cards | Transform schemes | Accept null; use `slug` as key |
| `_unassigned` entries in `leads.json` | Some Secret Wars sets | Skip; flag for manual review |
| `mastermind: "PLACEHOLDER_DELETE_THIS"` | `leads.json` first entry | Skip unconditionally |
| `vAttack` / `vp` as string or number | Mastermind/villain cards | Always parse to number for comparison |
| `attack` / `recruit` as `"2+"` | Hero cards | Strip `+` for numeric use |
