## Purpose

A **post-mortem checklist** is an execution audit that runs **after** the EC
acceptance criteria pass but **before** the pre-commit review. It exists to
catch second-order failures that passing tests alone cannot detect:

- silent boundary leaks
- "it passed tests but broke architecture"
- future coupling between layers
- replay and determinism regressions
- governance decay or documentation drift

Tests prove **correctness**. This checklist proves **safety and longevity**.

This is NOT a substitute for the EC. The EC confirms the WP was built to
spec. The post-mortem confirms the WP was built **safely and durably**.

This checklist is a **normative execution gate** when applicable. If a Work
Packet meets the criteria in "When Post-Mortem Is Required", completion of
this checklist is mandatory before the WP may be marked complete, regardless
of EC success.

---

## When Post-Mortem Is Required

Post-mortem is **mandatory** when the WP introduces any of:

- A new contract or type consumed by future WPs
- A new projection, view model, or derived representation
- A new setup-time artifact or builder
- A new long-lived abstraction or extension seam
- New runtime wiring or lifecycle integration
- A new code category directory
- Any WP that adds a new execution path or representation that did not
  previously exist (even if small)

Post-mortem is **recommended but optional** for:

- Single-file bug fixes or micro-patches
- Documentation-only WPs
- Test-only WPs with no new contracts

When in doubt, run the post-mortem. The cost is low; the cost of skipping
it when it matters is high.

## Workflow Position

This checklist is **step 4** in the WP execution workflow:

| Step | Gate | When | Document |
|---|---|---|---|
| 0 | Session context | Start of session, before pre-flight | `docs/ai/session-context/session-context-wpXXX.md` |
| 1 | Pre-flight (`01.4`) | Before session prompt | `docs/ai/REFERENCE/01.4-pre-flight-invocation.md` |
| 1b | Copilot check (`01.7`) | After pre-flight READY verdict, before session prompt | `docs/ai/REFERENCE/01.7-copilot-check.md` |
| 2 | Session prompt | Before execution | `docs/ai/invocations/session-wpXXX-*.md` |
| 3 | Execution | New session, EC-mode | Runs the session prompt under EC-mode |
| **4** | **Post-mortem (`01.6`)** | **After EC passes, same session** | **This template** |
| 5 | Pre-commit review | Separate session, gatekeeper | `docs/ai/prompts/PRE-COMMIT-REVIEW.template.md` |
| 6 | Commit | Commit hygiene (`01.3`) | `docs/ai/REFERENCE/01.3-commit-hygiene-under-ec-mode.md` |
| 7 | Lessons learned | After commit, same or new session | `01.4` §Lessons Learned Template |
| 8 | Session context | After lessons, before next WP | `docs/ai/session-context/session-context-wpXXX.md` |

Step 0 loads the session context file from the prior WP — the bridge that
carries test baselines, artifacts, and decisions across session boundaries.
Post-mortem (step 4) runs in the **same session** as execution — it has
full context and can actively inspect code, run greps, and trigger fixes.
The pre-commit review (step 5) runs in a **separate session** as a
non-implementing gatekeeper. Steps 7-8 run after the commit to capture
institutional knowledge and produce the session context that becomes the
next WP's step 0 input.

### 3. Boundary Integrity Check (Critical)

Confirm by grep/search and inspection.

> Tailor this section to the WP's primary layer (engine, server, UI,
> registry, infrastructure). The boundary principles apply universally;
> the specific systems to audit vary by layer.

#### Framework Boundaries

- [ ] No `boardgame.io` imports outside authorized framework files
- [ ] No framework context (`ctx`) leaking into pure helpers
- [ ] No lifecycle coupling (new code not wired into `game.ts`, moves,
  or phase hooks unless the WP explicitly authorizes it)

**How to verify:**
```bash
# Check for boardgame.io imports in new files
grep -r "from 'boardgame.io'" <new-file-paths>
# or use Select-String on Windows
```

#### Registry / IO Boundaries

- [ ] No registry imports where forbidden
- [ ] No filesystem, network, or persistence access unless explicitly allowed

#### Code Category Compliance

- [ ] New files live in directories matching their Code Category
  (per `docs/ai/REFERENCE/02-CODE-CATEGORIES.md`)
- [ ] No setup-time code placed in engine-only directories
- [ ] No execution logic placed in data/type-only files

---

### 4. Representation & Determinism Audit

- [ ] New representations are **data-only** and JSON-serializable
  - No functions, closures, Maps, Sets, Dates, or classes
- [ ] Execution logic is deterministic
  - Same input always produces same output
  - No hidden state, caching, memoization, or mutation
- [ ] Unknown or future values fail safely
  - Warn + skip, never throw (unless in `Game.setup()`)

**How to verify:**
- Inspect new types for non-serializable members
- Check for module-level `let` or `const` that accumulates state
- Run the determinism test (if the WP includes one)

---

### 5. Mutation, Aliasing, & Reference Safety (High-Risk)

Aliasing bugs are among the most common sources of silent corruption and
are rarely detected by standard test suites. This section is considered
**high-risk** and should be reviewed deliberately.

- [ ] All mutation occurs only in authorized lifecycle contexts
- [ ] No helpers mutate `G` or `ctx`
- [ ] No mutation during rendering, projection, or evaluation
- [ ] Pure helpers are actually pure (confirmed by test or inspection)
- [ ] **No aliasing bugs**: returned objects do not share references with
  mutable arrays or maps in `G`. Shallow copies (`[...array]`,
  `{...object}`) are used where the source is mutable.

**Why aliasing matters:** If a projection returns a reference to a mutable
array in `G` (e.g., `G.cardKeywords[id]`), the consumer can accidentally
mutate `G` through the returned reference. This violates purity and
determinism. Always spread-copy mutable sources.

**How to verify:**
- Read the implementation line by line
- For each returned array or object, trace its source
- If the source is a field of `G`, confirm a copy is made

**Precedent (WP-028):** Post-mortem discovered that `cardKeywords` in the
city projection returned a direct reference to `G.cardKeywords[space]`.
Fixed with spread operator during post-mortem. This class of bug is
invisible to tests unless a specific aliasing test exists.

---

### 6. Hidden-Coupling Detection

Check for accidental coupling risks:

- [ ] No engine internals exposed through public projections or types
- [ ] No enum or union widened unintentionally
- [ ] No implicit knowledge of upstream implementation details
- [ ] No ordering assumptions not explicitly documented or tested
- [ ] No dependency on non-exported functions or internal module state

---

### 7. Test Adequacy Review

Tests should act as **boundary enforcement**, not examples.

- [ ] Tests fail if architectural boundaries are violated
- [ ] Determinism is explicitly tested (same input -> same output)
- [ ] Serialization is tested if representation is public
- [ ] Non-mutation is tested if the function receives mutable state
- [ ] Tests do NOT depend on unrelated engine behavior
- [ ] No tests weakened to "make things pass"

**Rule:** If a test fails, the implementation is incorrect. Do NOT weaken
assertions to make tests pass — fix the implementation instead.

---

### 8. Documentation & Governance Updates

Confirm documentation reflects reality **after code passed** (not before —
documenting intent that was never implemented is a governance failure):

- [ ] **DECISIONS.md** — new decisions added where new patterns or
  constraints were introduced (with rationale, not just outcome)
- [ ] **ARCHITECTURE.md** — new fields, flows, layers, or boundaries
  reflected (if architecturally significant)
- [ ] **STATUS.md** — phase or system status updated accurately
- [ ] **WORK_INDEX.md** — WP checked off with actual completion date

---

### 9. Forward-Safety Questions (Answer YES to All)

- [ ] Can this code survive future refactors without touching unrelated
  layers?
- [ ] Can replay/debugging reconstruct behavior from stored data?
- [ ] Would removing upstream data still fail safely?
- [ ] Is it impossible for this WP's output to influence gameplay
  unintentionally?
- [ ] Is the contract stable enough to be referenced by future WPs?

Any NO requires one of the following before the WP may be closed:

- a documented mitigation in the Notes section below, OR
- an explicit follow-up WP reference in WORK_INDEX.md, OR
- a DECISIONS.md entry explaining why the risk is accepted

Undocumented NO answers are considered post-mortem failures.

## Precedent Log

### WP-028 — UI State Contract (2026-04-14)

**Post-mortem finding:** `cardKeywords` array in city projection returned a
direct reference to `G.cardKeywords[space]` instead of a spread copy. This
meant UIState consumers could accidentally mutate `G` through the returned
array reference, violating the purity guarantee.

**How discovered:** Section 5 (Mutation & Aliasing Safety) — line-by-line
inspection of returned objects and their source traces.

**Fix:** Replaced `gameState.cardKeywords[space] ?? []` with
`cardKeywords !== undefined ? [...cardKeywords] : []`.

**Lesson:** Aliasing bugs are invisible to standard tests (including
`JSON.stringify` equality checks, which compare values, not references).
They require explicit inspection of return paths. Future WPs that produce
projections from mutable `G` data should treat aliasing as a mandatory
check.

**ARCHITECTURE.md gap (non-blocking):** ARCHITECTURE.md states the UI
projection principle (line 66) but does not mention `UIState` or
`buildUIState` by name. The principle is correct and the implementation
is documented in STATUS.md and DECISIONS.md. Can be addressed in a future
ARCHITECTURE.md refresh.

## Closing Guidance

Post-mortem exists to catch what tests cannot: architectural drift,
boundary erosion, and subtle safety violations.

If you are unsure whether a finding is blocking:

- If it could cause data corruption, replay divergence, or boundary
  leakage: it is blocking.
- If it is cosmetic, stylistic, or hypothetical: it is non-blocking
  (log it under Notes).

When in doubt, fix it now. The cost of a post-mortem fix is low. The cost
of discovering the issue three WPs later is high.
