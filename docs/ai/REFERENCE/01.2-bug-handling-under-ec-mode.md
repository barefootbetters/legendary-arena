## Purpose

Once Execution Checklists are active, every bug is treated as an
**execution contract violation**. This protocol replaces ad-hoc debugging
and guarantees that all packages remain deterministic, drift-proof, and
fully traceable to Work Packets, ARCHITECTURE.md invariants, and locked
contracts.

---

## Core Principle (Non-Negotiable)

Every observed bug belongs to exactly one of two categories:

1. **Implementation violation** — code fails to satisfy a valid EC clause.
2. **Specification error** — the EC, source WP, or ARCHITECTURE.md contains
   an inconsistency.

There is no third category. "Unexpected behavior," "edge case," or "it
worked before" are not valid diagnoses.

---

## Scope

This protocol applies to all packages under EC-driven execution:
- `packages/game-engine`
- `packages/registry`
- `apps/server`
- Any future package that has a corresponding EC

It activates the moment a Work Packet's Execution Checklist exists in
`docs/ai/execution-checklists/EC_INDEX.md` with status `Draft` or `Done`.

---

## Bug Intake Rule (Mandatory First Step)

When any anomaly is observed (test failure, runtime error, incorrect game
state, desync, etc.):

1. Open `EC_INDEX.md` and locate the governing Execution Checklist
2. **Do not** open the code first
3. **Do not** add logs, console statements, or debug helpers
4. **Do not** attempt any code change

All bugs begin as EC investigations.

---

## Diagnosis Sequence (Strict Order)

Follow these steps exactly. Stop at the first applicable clause.

### Step 1 — EC Clause Violation Check (Default Path)

- Identify the exact EC clause that governs the failing behavior
  (Locked Value, Guardrail, Required Comment, File list, After Completing)
- Verify the violation against the **literal text** of the EC
- If the EC has a **Common Failure Smells** section, check symptoms there

If a clause is violated → this is an **implementation bug**.
Proceed to "Implementation Fix" below.

### Step 2 — Specification Consistency Check

Only if Step 1 yields no violation. Re-read, in this exact order:

1. The EC itself
2. The source Work Packet (WP)
3. Relevant sections of `docs/ai/ARCHITECTURE.md`
4. Any linked decision in `docs/ai/DECISIONS.md`

Ask:
- Does the EC contradict its source WP?
- Does the WP contradict an ARCHITECTURE.md invariant?
- Is a locked value in the EC different from the WP?

If an inconsistency is confirmed → this is a **specification bug**.
Stop coding immediately and follow "Specification Fix" below.

### Step 3 — Escalation

If no violated clause can be identified after Steps 1 and 2, the bug is
**not yet well-understood**. Do not fix it. Escalate by updating the
relevant WP or EC with clearer language before proceeding.

---

## Permitted Fix Types

### Implementation Fix (99% of cases)

**When:** Code violates a valid EC clause.

Rules:
- Modify code only — never weaken EC language
- Never edit the EC file during an implementation fix
- The fix must trace to a specific EC clause
- Re-run the full EC checklist after fixing

### Specification Fix (Rare, Controlled)

**When:** The EC cannot be satisfied without breaking the WP or
ARCHITECTURE.md.

Rules:
- Stop all coding immediately
- Update the WP, EC, or ARCHITECTURE.md first
- Record the change in `docs/ai/DECISIONS.md` with a new D-XXXX entry
- Re-tighten affected ECs using the auto-tighten prompt
- Re-run the full EC checklist
- Update `docs/ai/DECISIONS_INDEX.md` with the new decision reference

---

## Prohibited Responses (Zero Tolerance)

- Adding logs or debug code to "discover" behavior
- Hot-patching or "quick fixes" that pass tests but violate EC intent
- Editing an EC to accommodate existing code
- Fixing symptoms instead of the violated clause
- Relying on tribal knowledge ("we know what it should do")
- Bypassing `EC_INDEX.md` lookup
- Guessing at the root cause without tracing to an EC clause

---

## Diagnosis Workflow

```
Bug observed
  │
  ▼
Open EC_INDEX.md → locate governing EC
  │
  ▼
Step 1: EC Clause Violation Check
  │
  ├─ Clause violated → Implementation Fix → re-run EC checklist → done
  │
  ▼
Step 2: Specification Consistency Check
  │
  ├─ Inconsistency found → Specification Fix → re-tighten EC → done
  │
  ▼
Step 3: Not yet understood → escalate (update WP/EC language)
```

---

## Regression Rule

When a previously resolved bug reappears:

- The EC was insufficiently tight
- **Strengthen the EC** first (add tighter guardrail, new failure smell,
  or more precise After Completing check)
- Then apply the normal fix process

Regressions always improve the ECs, never just the code.

---

## Completion Rule

A bug is resolved only when **all** of the following are true:

- [ ] Every clause of the governing EC is satisfied
- [ ] All After Completing checks pass
- [ ] No other EC is incidentally violated
- [ ] The change is traceable in git to the EC clause (or to the
      specification update)

Passing tests alone is not sufficient.
