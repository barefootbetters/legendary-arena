# 00.1 — Master Coordination Prompt
# Legendary Arena · AI Development System
# REFERENCE DOCUMENT — Coordination & Operating Manual

> **Why this file exists:**
> Context drift is the #1 failure mode when developing across 15+ Claude Code
> sessions. Each session starts with no memory of prior sessions. The fix is to
> treat the *repository* as the memory — not the chat history. This file is the
> operating manual for that system.

---

## The Core Principle: Repo as Memory

Claude Code has no memory between sessions. The solution is **spec-driven
development**: a small set of authoritative Markdown files that Claude reads at
the start of every session, plus a scoped Work Packet that defines exactly what
the current session must accomplish.

**The repo is the memory. The chat is disposable.**

This means:
- Never rely on "the last session established X" — write X to a doc
- Never assume Claude remembers a decision — log it in `DECISIONS.md`
- Never let scope expand mid-session — it belongs in the next Work Packet
- Never end a session without updating the shared docs

## Authoritative Reference Documents

These files are the **single source of truth** for their domains.
Claude must read the relevant ones at the start of every session.

| File | Domain | Overrides |
|------|--------|-----------|
| `docs/ai/PROJECT_BRIEF.md` | Product scope, MVP definition, non-goals | Everything on scope |
| `docs/ai/ARCHITECTURE.md` | System structure, package boundaries, data flow | Implementation details |
| `docs/ai/DECISIONS.md` | Why things were chosen — prevents re-litigating | All prompt suggestions |
| `docs/ai/STATUS.md` | Current progress snapshot — what works today | Work Packet assumptions |
| `docs/ai/work-packets/WORK_INDEX.md` | Ordered list of all Work Packets + completion status | Session sequencing |
| `docs/ai/REFERENCE/00.2-data-requirements.md` | Canonical data contract — field names and shapes | **Every prompt on data** |
| `docs/ai/REFERENCE/00.3-prompt-lint-checklist.md` | Rules every Work Packet must pass before execution | All prompt drafts |
| `docs/ai/REFERENCE/00.6-code-style.md` | Human-style code rules for junior maintainability | All generated code |

### Override Hierarchy (highest to lowest)

```
00.2-data-requirements.md   ← wins on field names and data shape
ARCHITECTURE.md             ← wins on system boundaries
DECISIONS.md                ← wins on previously settled choices
PROJECT_BRIEF.md            ← wins on scope questions
Active Work Packet          ← wins on session-specific details
Chat conversation           ← lowest priority — never authoritative
```

If any prompt, suggestion, or generated output conflicts with a higher-ranked
document, the higher-ranked document wins. Do not deviate without explicitly
updating the higher-ranked document first.

## Work Packet System

Each Claude Code session targets **exactly one Work Packet**.
Work Packets live in: `docs/ai/work-packets/WP-NNN-<topic>.md`

### Work Packet Naming

```
WP-001-foundation.md
WP-002-game-skeleton.md
WP-005A-auth-contracts.md   ← use A/B suffixes when splitting a large task
WP-005B-auth-implementation.md
```

### Required Work Packet Structure

Every Work Packet must contain all of the following sections.
A packet that is missing any section fails the lint checklist (§00.3) and
must not be executed.

```markdown
# WP-NNN — [Short Title]

## Goal
One paragraph. What will exist after this session that did not exist before?

## Assumes
- Prior packets complete: WP-001 through WP-NNN-1
- Files that must already exist (with the exports/shapes they must have)
- External state that must be true (DB seeded, R2 populated, etc.)
- Any assumption left implicit here is a hidden dependency that will break silently
- If any assumption is false, this Work Packet is BLOCKED and must not proceed.

## Context (Read First)
List the specific docs and sections Claude must read before starting:
- docs/ai/ARCHITECTURE.md §[relevant section]
- docs/ai/REFERENCE/00.2-data-requirements.md §[relevant section]
- docs/ai/DECISIONS.md (scan for decisions touching this area)

## Scope (In)
Bullet list. Be explicit. If it is not listed here, it is out of scope.
Refactors, cleanups, or "while I'm here" improvements are OUT OF SCOPE
unless explicitly listed here.

## Out of Scope
Bullet list. Explicitly name things that might seem related but must not be touched.
This is the primary defense against scope creep.

## Files Expected to Change
List every file that will be created or modified:
- `path/to/file.ts` — [new | modified] — one-line description

## Contract
> **Output contract for this session:**
> - Full file contents for every new or modified file (no diffs, no snippets)
> - List of exact commands to run with expected output
> - ESM only, Node v22+
> - Human-style code — see `docs/ai/REFERENCE/00.6-code-style.md`
> - [Any session-specific constraints, e.g. "No Math.random()", "Read-only"]

## Acceptance Criteria
6–12 binary, observable, specific checks. Each must be pass/fail with no
partial credit. Reference actual file names, function names, or output values.

- [ ] `packages/game-engine/src/game.ts` exports a `Game` object
- [ ] `Game.setup()` accepts a payload matching §8.1 of 00.2 without throwing
- [ ] etc.

## Verification Steps
Exact commands (using pnpm) with expected output shown:

```pwsh
pnpm --filter game-engine test
# Expected: 3 passing
```

## Definition of Done
This packet is complete when ALL of the following are true:
- [ ] All acceptance criteria pass
- [ ] `docs/ai/STATUS.md` updated with what changed
- [ ] `docs/ai/DECISIONS.md` updated with any new decisions made
- [ ] `docs/ai/work-packets/WORK_INDEX.md` has this packet checked off
- [ ] No files outside the "Files Expected to Change" list were modified
```

## Session Execution Protocol (Mandatory)

Follow these steps in order at the start of **every** Claude Code session.
Do not skip steps. Do not reorder steps.

### Step 1 — Identify the Active Packet
If the Work Packet is not specified in the session prompt:
```
Which Work Packet should I execute? Please confirm the WP number.
Do not guess. Do not infer. Wait for confirmation.
```
Do not proceed until the packet is confirmed.

### Step 2 — Load Context
Read in this order:
1. The active Work Packet
2. Every doc listed in the packet's `## Context (Read First)` section
3. `docs/ai/STATUS.md` — verify assumptions in `## Assumes` are true
4. `docs/ai/DECISIONS.md` — scan for decisions that touch the current area

If any assumption in `## Assumes` is false, **stop** and report:
```
BLOCKED: Assumption failed — [describe what is missing].
This packet requires WP-NNN to be complete first.
```

### Step 3 — Plan Before Coding
Before writing any code, output:
1. Confirmation that all assumptions passed
2. A numbered list of implementation steps
3. The exact list of files that will be created or modified
4. Any ambiguities that need resolution before starting

Wait for approval if any ambiguity exists.

### Step 4 — Execute Within Scope
- Implement only what is listed under `## Scope (In)`
- Do not touch files outside `## Files Expected to Change`
- If you encounter something that needs fixing but is out of scope:
  → Note it as a future Work Packet, do not fix it now

### Step 5 — Verify
Run every command listed in `## Verification Steps`.
If a verification step fails, fix it before proceeding to Step 6.

### Step 6 — Close Out the Session
Update these files before the session ends:

**`docs/ai/STATUS.md`** — Add a dated entry:
```markdown
## [DATE] — WP-NNN Complete
- What was built
- What is now working end-to-end
- Any known limitations or deferred items
```

**`docs/ai/DECISIONS.md`** — Add any decision made during this session:
```markdown
## [DATE] — [Short Title]
**Decision:** [What was chosen]
**Rationale:** [Why — not just what]
**Alternatives rejected:** [What was not chosen and why]
**Packet:** WP-NNN
```

**`docs/ai/work-packets/WORK_INDEX.md`** — Check off the completed packet:
```markdown
- [x] WP-NNN — [title] — Completed [DATE]
```

## Context Drift Detection

Context drift occurs when generated code, suggestions, or architecture deviate
from the authoritative documents without those documents being updated first.

### Warning Signs of Context Drift
- Claude suggests a field name that differs from `00.2-data-requirements.md`
- Claude proposes an architecture that contradicts `ARCHITECTURE.md`
- Claude "remembers" a decision from a prior chat session (it cannot)
- A file outside the Work Packet scope is being modified
- The session has been running long and Claude starts hedging or contradicting itself

### Response to Detected Drift
Stop immediately. Do not accept the output. Say:
```
DRIFT DETECTED: [describe the contradiction].
The authoritative source is [document name, section].
Please re-read that section and revise your output to match it.
```

### Preventing Drift in Long Sessions
If a session is running long (many files changed, complex logic):
1. Explicitly ask Claude to re-read the relevant reference doc section
2. Ask Claude to confirm the canonical field names or architecture before continuing
3. Consider ending the session and starting fresh with a new Work Packet

## Splitting Work Packets

If a task is too large for a single coherent session, split it.

### When to Split
- The `## Files Expected to Change` list exceeds ~8 files
- The task requires both designing a contract AND implementing it
- The task touches multiple architectural layers (e.g., DB schema + game logic + UI)

### How to Split
```
WP-005A — [topic] Contracts    ← Define interfaces, types, API shapes
WP-005B — [topic] Implementation ← Write the code that fulfills the contracts
WP-005C — [topic] Integration  ← Wire it together and verify end-to-end
```

**Architecture and contracts always come before implementation.**
Never let implementation define the contract — that is the path to drift.

## Non-Negotiable Constraints

These rules apply to every session, every packet, every file. They cannot be
overridden by a Work Packet or by chat conversation.

### boardgame.io Rules
- boardgame.io is authoritative for: match lifecycle, turn order, phases, game
  state (`G`, `ctx`)
- All game moves must be **pure and deterministic**
- `G` must always be JSON-serializable (no class instances, no functions)
- All randomness uses `ctx.random.*` — never `Math.random()`
- PostgreSQL is **not** used for live game state — never store `G` or `ctx` in DB

### Backend Rules
- `pg` pool for all DB connections (never a single client)
- No DB queries inside boardgame.io moves
- No WebSocket logic outside `boardgame.io Server()`

### Frontend Rules
- Components do not fetch R2 data independently — use `cardRegistry.js`
- Components contain no game logic
- boardgame.io client is in `src/game/client.js`, not in Vue components
- `boardgame.io/react` is never imported
- All moves dispatched through `gameStore.move()`

### Technology Rules
- ESM only (`.mjs` or `"type": "module"`) — no CommonJS
- Node v22+ built-ins — no `axios`, no `node-fetch`
- No ORMs — use `pg` only
- No Jest/Vitest/Mocha — use `node:test` only
- No Passport/Auth0/Clerk — use `jsonwebtoken` or credentials-only
- All commands use `pnpm` — never `npm run`

### Windows / PowerShell Rules
- Shell scripts use `pwsh` (PowerShell 7+) — never `bash` or `sh`
- rclone config path uses `process.env.APPDATA` — never hardcode `~/.config/rclone`
- Node scripts needing `.env` use `node --env-file=.env` (Node v22 built-in)

### CLAUDE.md
`CLAUDE.md` at the project root is loaded automatically by Claude Code at the
start of every session. It must stay **short** (under 60 lines). Its job is to
point Claude at this coordination system — not to duplicate it.

Suggested CLAUDE.md content:
```markdown
# Legendary Arena — Claude Code Rules

## Coordination System
This project uses spec-driven development. Read before coding:
- docs/ai/REFERENCE/00.1-master-coordination-prompt.md (this system)
- docs/ai/STATUS.md (current state)
- docs/ai/work-packets/WORK_INDEX.md (what's next)

## One Rule Above All Others
The file docs/ai/REFERENCE/00.2-data-requirements.md defines all field names
and data shapes. It overrides everything else. Do not rename fields.

## Always
- One Work Packet per session. Do not expand scope.
- Full file contents only — no diffs, no snippets.
- pnpm, not npm. pwsh, not bash.
- ESM only. Node v22+.
- Human-style code per docs/ai/REFERENCE/00.6-code-style.md.
```
