## Purpose

A **copilot check** is a second-opinion review pass that runs **after** a
pre-flight (`01.4`) produces a `READY TO EXECUTE` verdict and **before** the
session execution prompt is generated.

Its job is to audit the Execution Checklist (EC), Work Packet (WP), and
pre-flight report against a fixed list of **30 failure modes that cause
long-term pain in Legendary Arena's stack** (pnpm monorepo, boardgame.io
`^0.50.0`, TypeScript ESM, PostgreSQL, Cloudflare R2).

The lens is **strictly preventative**. It verifies that architectural,
typing, governance, and execution-discipline decisions are locked **before
code exists** — when fixes are cheap, deterministic, auditable, and
enforceable.

If a problem is only caught during implementation, it was caught **too
late**.

Pre-flight (`01.4`) confirms the WP is **sequenced and scoped**. The
copilot check confirms the WP is **shaped to prevent known failure
categories**. The two are complementary: a WP can pass pre-flight and
still fail the copilot check, which surfaces governance gaps that
pre-flight is not structured to detect.

---

## When Copilot Check Is Required

The copilot check is **mandatory** whenever `01.4` returns
`READY TO EXECUTE`, prior to generating the session execution prompt, for
any WP of class:

- Runtime Wiring
- Behavior / State Mutation
- Infrastructure & Verification

For **Contract-Only** WPs it is **recommended but optional**. Contract WPs
rarely introduce determinism, persistence, or mutation risks — the lens
still applies, but most issues collapse to rapid `PASS` verdicts.

The copilot check may also be run:

- Against a draft EC/WP **before** pre-flight, as an early sanity check
- Against a migration of legacy prompts into governed artifacts
- As a standing review for any document that claims to lock execution
  scope (invocation prompts, templates, governance clauses)

It is **not** a substitute for pre-flight. It is not a code review. It
does not re-execute build or tests.

---

## Workflow Position

The copilot check is **step 1b** in the Work Packet execution workflow. It
sits between pre-flight and session prompt generation:

| Step | Gate | When | Document |
|---|---|---|---|
| 0 | Session context | Start of session | `docs/ai/session-context/session-context-wpXXX.md` |
| 1 | Pre-flight | Before session prompt | `docs/ai/REFERENCE/01.4-pre-flight-invocation.md` |
| **1b** | **Copilot check** | **After `READY TO EXECUTE`, before session prompt** | **This template** |
| 2 | Session prompt | Before execution | `docs/ai/invocations/session-wpXXX-*.md` |
| 3 | Execution | New session, EC-mode | Runs the session prompt |
| 4 | Post-mortem | After EC passes, same session | `docs/ai/REFERENCE/01.6-post-mortem-checklist.md` |
| 5 | Pre-commit review | Separate session | `docs/ai/prompts/PRE-COMMIT-REVIEW.template.md` |
| 6 | Commit | Commit hygiene | `docs/ai/REFERENCE/01.3-commit-hygiene-under-ec-mode.md` |
| 7 | Lessons learned | After commit | `01.4` §Lessons Learned Template |
| 8 | Session context | After lessons, before next WP | `docs/ai/session-context/session-context-wpXXX.md` |

Sequencing rules:

- Step 1b may only begin after step 1 produces `READY TO EXECUTE`.
- Step 1b runs in the **same session** as step 1 (no fresh-session gap
  required — the copilot check does not execute code).
- If step 1b escalates to `BLOCK`, the pre-flight verdict is **suspended**
  until the blockers are resolved. The session prompt must not be
  generated.
- If step 1b escalates to `RISK`, fixes may be applied in-place and the
  copilot check re-run without re-running pre-flight, provided the fixes
  do not change scope. Scope-changing fixes require re-running pre-flight.

## Inputs

The copilot check reviews **all three** of the following together:

1. The Execution Checklist at
   `docs/ai/execution-checklists/EC-###-*.checklist.md`
2. The Work Packet at `docs/ai/work-packets/WP-###-*.md`
3. The pre-flight report produced by `01.4` (in the same session, not yet
   committed)

The lens is applied to the **union** of these artifacts. A finding may
cite any of the three as the location of the gap.

## Discipline

**Strict discipline.** The copilot check has a fixed scope that must not
drift into general review.

- Stay **exclusively** inside the 30 issues listed below. Findings outside
  this list must not appear in the review.
- Do not add stylistic comments, feature ideas, refactors, or
  architectural proposals.
- Prefer **locking semantics** ("must", "forbidden", "explicitly
  required") over open-ended language.
- Governance fixes (`docs/ai/DECISIONS.md` entries,
  `docs/ai/REFERENCE/02-CODE-CATEGORIES.md` registry updates, new
  `.claude/rules/*.md` enforcement, WORK_INDEX entries, test baselines)
  are explicitly permitted and expected as remediation.
- Violations of this discipline must result in a `RISK` or `BLOCK`
  judgment, not advisory commentary.

Reviewers must scan **all 30 issues** even when many result in a rapid
`PASS`.

For each of the 30 issues below, assign one of:

- `PASS` — The artifact **explicitly prevents** the issue.
- `RISK` — Prevention is implicit, weak, or missing.
- `FIX` — Provide **only** a scope-neutral improvement (contracts,
  wording, tests, governance rules, verification steps). A `FIX` is
  attached to a `RISK` or `BLOCK` — it is never a standalone category.

`BLOCK` is reserved for the overall verdict (see "Required Output
Format") and applies when any `RISK` finding would cause architectural or
determinism damage if execution proceeded without remediation.

## Legendary Arena Anchors (Required Reading for Reviewers)

Before applying the lens, the reviewer must have the following documents
in working memory. The 30 issues below reference them by name.

- `docs/ai/ARCHITECTURE.md` — layer boundaries, determinism, persistence
- `.claude/rules/architecture.md` — enforced invariants
- `.claude/rules/code-style.md` — canonical arrays, drift detection,
  naming discipline, `.reduce()` bans, pure-helper rules
- `.claude/rules/work-packets.md` — dependency discipline, scope lock
- `docs/ai/DECISIONS.md` — settled governance decisions
- `docs/ai/REFERENCE/02-CODE-CATEGORIES.md` — directory classification
  registry
- `docs/ai/REFERENCE/00.2-data-requirements.md` — canonical field names
- `docs/ai/REFERENCE/00.6-code-style.md` — human-facing style guide

If the reviewer does not have these anchors loaded, `RISK` judgments
are likely to be underweighted.

## The Top 30 "Could've Been Solved Early" Issues

### 1. Separation of Concerns & Boundaries

**1. Engine vs UI / App Boundary Drift**
**Failure mode:** UI or orchestration logic leaks into the engine over
time.
**Early prevention signals:** Explicit import prohibitions or barrel-file
restrictions, clear ownership statements ("engine is unaware of X"),
file-level or grep-based enforcement, alignment with the Layer Boundary
section of `ARCHITECTURE.md`.

**9. UI Re-implements or Re-interprets Engine Logic**
**Failure mode:** UI logic diverges from gameplay truth.
**Early prevention signals:** Authoritative projection / view layer
defined, UI explicitly forbidden from reading raw engine state,
audience-filtered views required, intents-not-outcomes rule cited.

**16. Lifecycle Wiring Creep**
**Failure mode:** Helpers get "helpfully" wired into `game.ts`, phases,
or moves.
**Early prevention signals:** Explicit lifecycle prohibition language,
verification steps (grep for forbidden imports), clear separation of
orchestration vs execution, alignment with
`01.5-runtime-wiring-allowance.md` when wiring is unavoidable.

**29. Assumptions Leaking Across Layers**
**Failure mode:** Engine assumes UI behavior or UI assumes engine
internals.
**Early prevention signals:** One-directional knowledge rules, explicit
"engine does not know X" statements, enforcement via imports and tests.

### 2. Determinism & Reproducibility

**2. Non-Determinism Introduced by Convenience**
**Failure mode:** Replay and desync bugs caused by time, randomness, or
ordering.
**Early prevention signals:** Explicit bans on `Math.random`, `Date.now`,
locale behavior; all randomness via `ctx.random.*`; determinism declared
non-negotiable; replay identified as the single source of debugging
truth; `// why:` comment required on every `ctx.random.*` use.

**8. No Single Debugging Truth Artifact**
**Failure mode:** Debugging relies on logs instead of reproducible
inputs.
**Early prevention signals:** Replay or deterministic input declared
canonical, reconstruction story documented, no hidden or ambient state
allowed.

**23. Lack of Deterministic Ordering Guarantees**
**Failure mode:** Object iteration order affects results.
**Early prevention signals:** Explicit ordering rules, arrays preferred
over object maps, tests asserting stable order, `for...of` over
`Object.keys` for order-sensitive paths.

### 3. Immutability & Mutation Discipline

**3. Confusion Between Pure Functions and Immer Mutation**
**Failure mode:** Helpers mutate inputs; moves return new objects
incorrectly. (boardgame.io uses Immer for `G` mutations; helpers must
remain pure.)
**Early prevention signals:** Clear rule (moves mutate Immer drafts;
helpers return new values), explicit "pure helper" designation,
`zoneOps.ts`-style separation, tests asserting non-mutation.

**17. Hidden Mutation via Aliasing**
**Failure mode:** Returned objects share array/object references.
**Early prevention signals:** Explicit copy discipline, tests asserting
deep immutability, language calling out aliasing risks, zone helpers
returning new `CardExtId[]` rather than splicing.

### 4. Type Safety & Contract Integrity

**4. Contract Drift Between Types, Tests, and Runtime**
**Failure mode:** Types evolve silently and desync from callers or
persistence.
**Early prevention signals:** Verbose contract copying into ECs,
drift-detection tests, canonical readonly arrays (`MATCH_PHASES`,
`TURN_STAGES`, `CORE_MOVE_NAMES`, `RULE_TRIGGER_NAMES`,
`RULE_EFFECT_TYPES`, `REVEALED_CARD_TYPES`) asserted equal to their
union types, re-exports without shape re-definition.

**5. Optional Field Ambiguity (`exactOptionalPropertyTypes`)**
**Failure mode:** Inconsistent handling of optional fields causes TS
churn.
**Early prevention signals:** Documented conditional construction
pattern, tests exercising all optional combinations, explicit ban on
inline `T | undefined` hacks, alignment with tsconfig strictness.

**6. Undefined Merge Semantics (Replace vs Append)**
**Failure mode:** Overrides behave differently depending on unspoken
assumptions.
**Early prevention signals:** Merge semantics explicitly locked in text,
tests pinning exact behavior, no implied or emergent behavior allowed.

**10. Stringly-Typed Outcomes and Results**
**Failure mode:** Free-form strings create fragile logic.
**Early prevention signals:** Discriminated unions required, exhaustive
switches enforced, typed outcomes propagated end-to-end, no hardcoded
stage/phase/trigger/effect strings.

**21. Type Widening at Boundaries**
**Failure mode:** `string`, `any`, or `unknown` used where unions are
expected.
**Early prevention signals:** Narrow types enforced at all boundaries,
`CardExtId` used instead of raw `string` for card identity, EC-locked
signatures.

**27. Weak Canonical Naming Discipline**
**Failure mode:** Near-synonyms proliferate (`winner` vs `outcome`).
**Early prevention signals:** Locked field names in ECs matching
`00.2-data-requirements.md` exactly, EC-copied canonical shapes,
drift-checked constants, no abbreviations in new names
(`00.6-code-style.md` Rules 4, 14).

### 5. Persistence & Serialization

**7. Persisting Runtime State by Accident**
**Failure mode:** `G`, `ctx`, or execution artifacts leak into storage.
**Early prevention signals:** Clear data-class taxonomy, explicit "must
never be persisted" lists, JSON-roundtrip tests required, snapshots
defined as derived records, `G` declared runtime-only.

**19. Weak JSON-Serializability Guarantees**
**Failure mode:** Functions, `Map`, `Set`, or class instances leak into
data.
**Early prevention signals:** JSON-roundtrip tests mandated, "data-only"
assertions required, enumeration-based field checks,
`ImplementationMap` kept out of `G`, `G.hookRegistry` declared
data-only.

**24. Mixed Persistence Concerns**
**Failure mode:** Config, runtime, and snapshot concepts blur.
**Early prevention signals:** Clear persistence class taxonomy, strong
prohibition language, dedicated persistence tests, snapshots storing
counts only rather than zone contents.

### 6. Testing & Invariant Enforcement

**11. Tests Validate Behavior, Not Invariants**
**Failure mode:** Tests pass while invariants silently break.
**Early prevention signals:** Invariant-focused tests required, "must
never happen" assertions, baseline test counts locked in the pre-flight,
`makeMockCtx` used instead of `boardgame.io/testing`, no silent passes.

### 7. Scope & Execution Governance

**12. Scope Creep During "Small" Packets**
**Failure mode:** Harmless packets balloon unexpectedly.
**Early prevention signals:** Explicit allowed/forbidden file lists,
`git diff --name-only` verification step, "anything not explicitly
allowed is forbidden" rule cited, scope lock enforced by the pre-flight.

**13. Unclassified Directories and Ownership Ambiguity**
**Failure mode:** New directories appear without architectural
classification.
**Early prevention signals:** Code-category registry required
(`02-CODE-CATEGORIES.md`), mandatory classification before execution,
`DECISIONS.md` entries for any structural change.

**30. Missing Pre-Session Governance Fixes**
**Failure mode:** Structural issues discovered but not resolved before
execution.
**Early prevention signals:** Explicit Pre-Session Actions (PS-n),
blocking vs non-blocking classification, statement that fixes are
scope-neutral governance only, resolution logged in the pre-flight
"Authorized Next Step" section.

### 8. Extensibility & Future-Proofing

**14. No Extension Seams for Future Growth**
**Failure mode:** Adding new conditions/rewards requires refactors.
**Early prevention signals:** Discriminated unions with exhaustive
handling, MVP variant locking, rule-hook and effect-type registries
designed for open-ended extension.

**28. No Upgrade or Deprecation Story**
**Failure mode:** Changes require breaking old data later.
**Early prevention signals:** Version-aware thinking,
backward-compatibility statements, "no migration needed" explicitly
stated when true.

### 9. Documentation & Intent Clarity

**15. Missing "Why" for Invariants and Boundaries**
**Failure mode:** Future contributors violate critical rules unknowingly.
**Early prevention signals:** Required `// why:` comments on every
`ctx.events.setPhase()`, `ctx.events.endTurn()`, and `ctx.random.*`
call, plus every catch-block swallow and every non-obvious constant;
decisions recorded in `DECISIONS.md`; rationale tied to trust and
determinism.

**20. Ambiguous Authority Chain**
**Failure mode:** Unsure which document wins when conflicts arise.
**Early prevention signals:** Explicit authority order, "read order"
sections, clear override hierarchy (`CLAUDE.md` >
`00.1-master-coordination-prompt.md` > `ARCHITECTURE.md` >
`.claude/rules/` > WPs > conversation).

**26. Implicit Content Semantics**
**Failure mode:** "Convention-based" meaning not written down.
**Early prevention signals:** Semantics locked in prose, tests asserting
meaning, no reliance on names alone.

### 10. Error Handling & Failure Semantics

**18. Outcome Evaluation Timing Ambiguity**
**Failure mode:** Results evaluated at inconsistent lifecycle moments.
**Early prevention signals:** Clear "before vs after game end" rules,
typed separation of runtime vs post-game logic, stage transitions
locked in `turnPhases.logic.ts`.

**22. Silent Failure vs Loud Failure Decisions Made Late**
**Failure mode:** Unclear whether to throw, warn, or return fallbacks.
**Early prevention signals:** Explicit fail-fast vs fail-soft policy
(moves never throw; only `Game.setup()` may throw; unknown effects
warn and continue), consistent patterns across helpers, tests asserting
failure mode, full-sentence error messages per
`00.6-code-style.md` Rule 11.

### 11. Single Responsibility & Logic Clarity

**25. Overloaded Function Responsibilities**
**Failure mode:** Helpers do merging, validation, evaluation, and
mutation in one place.
**Early prevention signals:** Single-responsibility descriptions, narrow
input/output contracts, explicit prohibition of extra work, move
validation contract followed (validate args -> check stage gate ->
mutate via helpers -> return `void`).

## What a Strong Artifact Demonstrates

A strong EC, WP, or pre-flight:

- Prevents failure **by construction**, not by reviewer vigilance
- Locks intent **before code exists**
- Leaves minimal room for interpretation
- Converts architectural risk into checklist-verifiable rules
- Treats governance as a first-class engineering concern
- Cites the Legendary Arena anchors above when establishing rules,
  rather than re-deriving them

## Required Output Format

Use **this exact structure** for every copilot check. The review is
saved alongside the pre-flight report for the same WP (for example,
appended to `docs/ai/pre-flights/pre-flight-wpXXX.md` under a
`## Copilot Check` heading, or as a separate file if the pre-flight
artifact has not yet been spun out).

```
# Copilot Check — WP-XXX (<short title>)

**Date:** YYYY-MM-DD
**Pre-flight verdict under review:** READY TO EXECUTE (<pre-flight date>)
**Inputs reviewed:**
- EC: docs/ai/execution-checklists/EC-###-*.checklist.md
- WP: docs/ai/work-packets/WP-###-*.md
- Pre-flight: <in-session report or file path>

## Overall Judgment

[PASS | RISK | BLOCK]

<3-6 sentences: whether the pre-flight verdict still holds, which
categories surfaced findings, whether any finding would cause
architectural or determinism damage if unaddressed.>

## Findings

1. [PASS | RISK] — <one-sentence verdict>
   FIX (if RISK): <scope-neutral remediation>

2. [PASS | RISK] — <one-sentence verdict>
   FIX (if RISK): <scope-neutral remediation>

... (continue for every issue that is not a clean PASS; group by
category if more than 15 findings.)

## Mandatory Governance Follow-ups (if any)

- DECISIONS.md entry: <...>
- 02-CODE-CATEGORIES.md update: <...>
- .claude/rules/*.md update: <...>
- WORK_INDEX.md update: <...>

## Pre-Flight Verdict Disposition

- [ ] CONFIRM — Pre-flight READY TO EXECUTE verdict stands. Session
      prompt generation authorized.
- [ ] HOLD — Apply listed FIXes in-place, re-run copilot check. No
      pre-flight re-run required (scope unchanged).
- [ ] SUSPEND — Pre-flight verdict suspended. Blockers must be resolved;
      if scope changes, re-run pre-flight before re-running copilot
      check.
```

`HOLD` is used when fixes are scope-neutral (wording, missing `// why:`
comments, missing test assertions, missing governance entries).
`SUSPEND` is used when the fix would change the WP's allowlist, move
the mutation boundary, or introduce a new contract.

## Rule Against Drift

This lens is **stable and repeatable**. Changes to the 30 issues
require:

1. A `DECISIONS.md` entry justifying the change
2. A lessons-learned entry (`01.4` §Lessons Learned Template) citing the
   WP that surfaced the gap
3. Explicit human approval — Claude may not unilaterally add, remove,
   reorder, or rephrase the 30 issues

Reviewers may not silently re-interpret an issue. If the project
encounters a failure mode that does not map to any of the 30, raise it
as a governance update rather than stretching an existing issue to
cover it.

---

## Final Instruction

The copilot check exists to **prevent premature execution and
preventable long-term pain**.

If there is uncertainty, missing context, or unresolved ambiguity after
the 30-issue scan:

**DO NOT AUTHORIZE SESSION PROMPT GENERATION.**

Escalate via a `RISK` or `BLOCK` verdict, apply the scope-neutral fix,
or re-run pre-flight. The cost of holding is low; the cost of shipping
a WP that trips a known failure mode is high.
