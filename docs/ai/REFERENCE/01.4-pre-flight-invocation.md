## Purpose

A **pre-flight invocation** is a mandatory readiness gate that runs **before**
a session execution prompt is generated for a Work Packet.

It exists to:

- Confirm the WP is **sequenced correctly** (all prerequisites complete)
- Confirm the repo is **green** (build/tests pass) at the starting point
- Lock **scope boundaries**, allowed files, and forbidden behavior
- Identify and resolve **risks / ambiguities** *before* implementation begins
- Produce a **binary verdict**: READY / NOT READY
- Authorize (or deny) generation of the WP's **session execution prompt**

Pre-flight is an **audit artifact**. It must be written so a reviewer can
understand why execution was authorized without re-reading transcripts.

"Green" means all relevant package builds succeed and all tests pass
with no pending EC violations.

---

## When Pre-Flight Is Required

Pre-flight should be run for every WP.

It is **mandatory** when the WP:

- Mutates runtime state (`G`) or implements moves / phase hooks
- Adds or consumes runtime-visible fields (often triggers 01.5)
- Wires behavior into `game.ts` or other runtime orchestration
- Introduces new "engine-wide" contracts that future WPs depend on
- Changes test expectations or introduces integration tests

---

## Work Packet Execution Workflow

This template is **step 1** of a 5-step workflow. Each step must complete
before the next may begin.

| Step | Gate | Document | Location |
|---|---|---|---|
| 1 | Pre-flight | This template (`01.4`) | `docs/ai/REFERENCE/01.4-pre-flight-invocation.md` |
| 2 | Session prompt | Generated invocation | `docs/ai/invocations/session-wpXXX-*.md` |
| 3 | Execution | New Claude Code session | Runs the session prompt under EC-mode |
| 4 | Pre-commit review | Review template | `docs/ai/prompts/PRE-COMMIT-REVIEW.template.md` |
| 5 | Commit | Commit hygiene rules | `docs/ai/REFERENCE/01.3-commit-hygiene-under-ec-mode.md` |

Step 2 may only begin after step 1 produces a **READY TO EXECUTE** verdict.
Step 3 runs in a **new Claude Code session** — not the session that generated
the prompt. Step 4 runs after execution completes but before any commit.
Step 5 follows commit hygiene rules (`EC-###:` prefix, hook validation).

### Runtime Readiness Check (Mutation & Framework)

**Required when Work Packet Class = Runtime Wiring or Behavior / State Mutation.**

(See also: `docs/ai/REFERENCE/01.5-runtime-wiring-allowance.md`)

Answer **YES / NO** to each:

- [ ] The expected runtime touchpoints are known (e.g., `game.ts`, phase hooks, moves map)
- [ ] Framework context requirements are understood (e.g., `ctx.events.*`, `ctx.random.*`)
- [ ] Existing test infrastructure can support required mocks without modifying locked helpers
- [ ] Any runtime wiring allowance needs are anticipated (01.5 referenced if applicable)
- [ ] No architecture boundary violations are expected (engine vs framework separation)

Any **NO** must be explained. If any answer is **NO**, pre-flight is **NOT READY**.

### Established Patterns to Follow (Locked Precedents)

The following patterns are validated by prior WPs and must be reused rather
than reinvented:

- Inline mock contexts for tests requiring framework features (e.g.,
  `ctx.events`, `ctx.random`) — do NOT modify shared helpers
- Passing `ctx` directly where it is structurally compatible with narrower
  interfaces (e.g., `ShuffleProvider`)
- Pure helper functions return new values; moves assign results into `G`
  under Immer
- Mutating `G` occurs only inside move implementations or
  framework-authorized contexts (phase hooks, `onBegin`)
- No changes to locked helpers unless a WP explicitly authorizes it

**Deviations from these patterns require explicit justification** and may
convert pre-flight to **NOT READY** if they imply scope creep.

### Scope Lock (Critical)

#### WP-XXX Is Allowed To

List only what the WP is permitted to implement, and the expected file targets.

Examples:

- Create: `src/moves/zoneOps.ts` — pure helper (no framework imports)
- Create: `src/moves/coreMoves.impl.ts` — move implementations (framework ctx allowed via parameter)
- Modify: `src/game.ts` — wire implementations into `moves`
- Add: integration tests required by the WP
- Update required governance docs

#### WP-XXX Is Explicitly Not Allowed To

List forbidden changes.

Examples:

- No modification of prior contract files (unless explicitly authorized)
- No new gameplay systems beyond this WP
- No randomness via `Math.random()`
- No `.reduce()` if disallowed by rules
- No forbidden imports (e.g., `boardgame.io` in pure helpers)
- No touching non-allowlisted files

**Rule:** Anything not explicitly allowed is out of scope.

---

### Test Expectations (Locked Before Execution)

Define the testing intent for this Work Packet before execution begins.
These expectations become part of the scope lock — deviations require
pre-flight escalation or a WP split.

Include:

- **New tests:** expected count and file location(s)
- **Existing test changes:** whether any existing tests are expected to
  require updates (e.g., structural assertion value changes under
  `01.5-runtime-wiring-allowance.md`)
- **Prior test baseline:** confirmation that all prior tests must continue
  to pass, with the current passing count recorded
- **Test boundaries:** what kinds of test changes are forbidden (e.g., no
  new logic in existing test files, no `boardgame.io/testing` imports)

Example (WP-008B style):

```
- 9 new integration tests in src/moves/coreMoves.integration.test.ts
- All 80 existing tests must continue to pass
- game.test.ts may require structural assertion updates (move count)
  under 01.5 runtime wiring allowance — value-only, no new logic
- No boardgame.io imports in test files
- No modifications to makeMockCtx or other shared test helpers
```

**Rule:** Test expectations are locked for execution. If the implementation
discovers that more or fewer tests are needed, or that existing tests
require logic changes (not just value updates), the deviation must be
documented in the execution summary and justified in the pre-commit review.

---

### Mutation Boundary Confirmation

**Required when Work Packet Class = Behavior / State Mutation.**
Skip this section for Contract-Only and Runtime Wiring WPs.

Confirm all of the following:

- [ ] `G` mutations occur only inside move implementations (framework-authorized)
- [ ] No mutation occurs in helpers or validators
- [ ] Helpers return new values; move assigns results into `G`
- [ ] No mutation occurs outside the boardgame.io move context

Any violation is a **blocking pre-flight failure**.

---

### Risk & Ambiguity Review (Resolve Now, Lock for Execution)

List risks, ambiguities, or edge cases that could cause execution drift.

For each item, include:

- **Risk / ambiguity**
- **Impact**
- **Mitigation**
- **Decision / pattern to follow**

**Locking rule:** Any ambiguity resolved here is considered **locked for
execution**. The execution session must not revisit or reinterpret these
decisions.

Examples of risks often relevant to Behavior WPs:

- Mock context shape for tests (do not modify shared helpers)
- Interface compatibility (ctx structurally assignable to narrower provider)
- Framework function names vs move names (`endTurn` move vs `ctx.events.endTurn()`)
- Immer mutation model and helper assignment pattern

If risks exist, recommend **mitigation or clarification** before execution.

---

### Pre-Flight Verdict (Binary)

Choose **exactly one**:

- ✅ **READY TO EXECUTE** — WP-XXX is properly sequenced, scoped, and ready
  for execution.
- ❌ **DO NOT EXECUTE YET** — WP-XXX has blocking issues that must be
  resolved first.

Provide 3-6 sentences of justification grounded in:

- dependency readiness
- scope lock clarity
- risks resolved
- architectural boundary confidence

---

### Authorized Next Step

If and only if the verdict is **READY TO EXECUTE**:

> You are authorized to generate a **session execution prompt** for WP-XXX
> to be saved as:
> `docs/ai/invocations/session-wpxxx-<short-name>.md`

**Guard:** The session prompt **must conform exactly** to the scope, constraints,
and decisions locked by this pre-flight. No new scope may be introduced.

If the verdict is **DO NOT EXECUTE YET**, stop. Do not generate a session
prompt.

---

### Final Instruction

Pre-flight exists to **prevent premature execution and scope drift**.

If there is uncertainty, missing context, or unresolved ambiguity:

**DO NOT PROCEED TO EXECUTION.**

Escalate, clarify, or split the WP instead.
