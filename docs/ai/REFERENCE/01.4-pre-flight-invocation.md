## Purpose

This template defines a **pre-flight validation gate** that must be run
before generating any Work Packet session execution prompt. It exists to:

- Confirm the target WP is ready to execute
- Verify all prerequisites are satisfied
- Lock scope boundaries and exclusions
- Detect sequencing or dependency errors
- Decide whether it is safe to generate a session execution prompt

This step fills the gap between "a new WP is next" and "Claude generated
a concrete execution session prompt." Without it, decisions are implied
during prompt generation rather than validated explicitly.

---

## When to Use

Run this template **before** generating any `docs/ai/invocations/session-wpXXX-*.md`
file. The pre-flight verdict must be **READY TO EXECUTE** before the session
prompt may be created.

---

## Work Packet Execution Workflow

This template is **step 1** of a 5-step workflow. Each step must complete
before the next may begin.

| Step | Gate | Document | Location |
|---|---|---|---|
| 1 | Pre-flight | This template (`01.4`) | `docs/ai/REFERENCE/01.4-pre-flight-invocation.md` |
| 2 | Session prompt | Generated invocation | `docs/ai/invocations/session-wpXXX-*.md` |
| 3 | Execution | New Claude Code session | Runs the session prompt under EC-mode |
| 4 | Pre-commit review | Review template | `docs/ai/prompts/PRE-COMMIT-REVIEW.template.md` |
| 5 | Commit | Commit hygiene rules | `docs/ai/REFERENCE/01.3-commit-hygiene-under-ec-mode.md` |

Step 2 may only begin after step 1 produces a **READY TO EXECUTE** verdict.
Step 3 runs in a **new Claude Code session** — not the session that generated
the prompt. Step 4 runs after execution completes but before any commit.
Step 5 follows commit hygiene rules (`EC-###:` prefix, hook validation).

### Structural Readiness Check

Answer **YES / NO** to each:

- [ ] All prior WPs compiled and tests pass
- [ ] No known EC violations remain open
- [ ] Required types/contracts already exist where referenced
- [ ] No architectural boundary conflicts are anticipated
- [ ] WP-XXX does not overlap another planned WP

Any **NO** must be explained.

### Scope Lock (Critical)

#### WP-XXX Is Allowed To:

*List only what this WP may implement.*

#### WP-XXX Is Explicitly Not Allowed To:

*List forbidden actions.*

Anything not listed in the allowed section is **out of scope**.

---

### Risk & Ambiguity Review

Identify any **known risks**:

- Ambiguous specs
- Gaps between WP text and EC wording
- Edge cases likely to cause Claude hesitation
- Potential temptation to overreach scope

If risks exist, recommend **mitigation or clarification** before execution.

---

### Pre-Flight Verdict (Binary)

Choose **exactly one**:

- **READY TO EXECUTE** — WP-XXX is properly sequenced, scoped, and ready
  for execution.
- **DO NOT EXECUTE YET** — WP-XXX has blocking issues that must be
  resolved first.

Provide a short justification (3-6 sentences).

---

### Authorized Next Step

If and only if the verdict is **READY TO EXECUTE**:

> You are authorized to generate a **session execution prompt** for WP-XXX
> to be saved as:
> `docs/ai/invocations/session-wpxxx-<short-name>.md`

If the verdict is **DO NOT EXECUTE YET**, stop. Do not generate a session
prompt.

---

### Final Instruction

This pre-flight invocation exists to **prevent premature execution**.

If there is uncertainty, missing context, or scope ambiguity:

**DO NOT PROCEED TO EXECUTION.**
