## Purpose

A **pre-flight invocation** is a mandatory readiness gate that runs **before**
a session execution prompt is generated for a Work Packet.

It exists to:

- Confirm the WP is **sequenced correctly** (all prerequisites complete)
- Confirm the repo is **green** (build/tests pass) at the starting point
- Lock **scope boundaries**, allowed files, and forbidden behavior
- Identify and resolve **risks / ambiguities** *before* implementation begins
- Verify **contract fidelity** between dependency WPs and the current WP
- Produce a **binary verdict**: READY / NOT READY
- Authorize (or deny) generation of the WP's **session execution prompt**

Pre-flight is an **audit artifact**. It must be written so a reviewer can
understand why execution was authorized without re-reading transcripts.

"Green" means all relevant package builds succeed and all tests pass
with no pending EC violations.

---

## Review Order & Authority (Normative)

Pre-flight review must proceed in the following order:

1. **Execution Checklist (EC)**
2. **Work Packet (WP)**
3. **Invocation / Session Execution Prompt**

Rules:

- The **EC is the primary authority** for execution correctness.
- The **WP must justify and conform to the EC**, not reinterpret it.
- The **invocation prompt is a derived artifact** and must not introduce
  new constraints, scope, or interpretations beyond EC + WP.

If an inconsistency is found:
- EC errors must be resolved first
- Then WP inconsistencies
- Invocation prompt issues are addressed last

Pre-flight fails if any lower-level artifact contradicts a higher one.
Lower-authority documents must be corrected; they may not "work around"
or compensate for higher-level errors.

---

## When Pre-Flight Is Required

Pre-flight should be run for every WP.

It is **mandatory** when the WP:

- Mutates runtime state (`G`) or implements moves / phase hooks
- Adds or consumes runtime-visible fields (often triggers 01.5)
- Wires behavior into `game.ts` or other runtime orchestration
- Introduces new "engine-wide" contracts that future WPs depend on
- Changes test expectations or introduces integration tests
- Touches infrastructure, deployment, or database migrations

**Foundation Prompt prerequisite:** All 4 Foundation Prompts
(FP-00.4 -> FP-00.5 -> FP-01 -> FP-02) must have completed
successfully before any WP may execute. Foundation Prompts are
**stop-on-failure gates** — if any FP fails, ALL downstream work is
blocked. This is distinct from the per-WP lint gate, which checks
individual WP quality. Pre-flight may assume FPs are complete unless
the WP touches infrastructure that an FP established (database
schema, R2 data, server wiring, environment variables).

## Work Packet Execution Workflow

This template is **step 1** of a 9-step workflow (steps 0-8). Each step
must complete before the next may begin.

| Step | Gate | When | Document |
|---|---|---|---|
| 0 | Session context | Start of session, before pre-flight | `docs/ai/session-context/session-context-wpXXX.md` |
| 1 | Pre-flight (`01.4`) | Before session prompt | `docs/ai/REFERENCE/01.4-pre-flight-invocation.md` |
| 2 | Session prompt | Before execution | `docs/ai/invocations/session-wpXXX-*.md` |
| 3 | Execution | New session, EC-mode | Runs the session prompt under EC-mode |
| 4 | Post-mortem (`01.6`) | After EC passes, same session | `docs/ai/REFERENCE/01.6-post-mortem-checklist.md` |
| 5 | Pre-commit review | Separate session, gatekeeper | `docs/ai/prompts/PRE-COMMIT-REVIEW.template.md` |
| 6 | Commit | Commit hygiene (`01.3`) | `docs/ai/REFERENCE/01.3-commit-hygiene-under-ec-mode.md` |
| 7 | Lessons learned | After commit, same or new session | `01.4` §Lessons Learned Template (below) |
| 8 | Session context | After lessons, before next WP | `docs/ai/session-context/session-context-wpXXX.md` |

**Step sequencing rules:**

- **Step 0 -> 1:** Step 0 loads the session context file produced by the
  **prior** WP's step 8. This file contains test baselines, dependency
  chain status, key artifacts, architectural patterns, and relevant
  decisions — everything needed to begin pre-flight without re-deriving
  project state from scratch. If no session context file exists (e.g.,
  first WP in a phase, or the prior WP predates this workflow), skip
  step 0 and begin pre-flight from the authority documents directly.
  Session context files live in `docs/ai/session-context/`.
- **Step 1 -> 2:** Step 2 may only begin after step 1 produces a
  **READY TO EXECUTE** verdict.
- **Step 2 -> 3:** Step 3 runs in a **new Claude Code session** — not the
  session that generated the prompt.
- **Step 3 -> 4:** Step 4 (post-mortem) runs in the **same session** as
  execution, after EC passes. It actively inspects code and may trigger
  fixes. Mandatory for WPs that introduce new contracts, projections,
  setup artifacts, or long-lived abstractions; recommended but optional
  for simpler WPs (see `01.6` for criteria).
- **Step 4 -> 5:** Step 5 runs in a **separate session** as a
  non-implementing gatekeeper. It assumes verification has passed and
  does not re-run commands.
- **Step 5 -> 6:** Step 6 follows commit hygiene rules (`EC-###:` prefix,
  hook validation).
- **Step 6 -> 7:** Step 7 captures lessons learned from the WP execution
  that should inform future pre-flight templates, session prompts, or
  governance. Updates the pre-flight precedent log and any affected
  REFERENCE documents. May also update `.claude/rules/` if a new
  enforcement pattern was discovered.
- **Step 7 -> 8:** Step 8 produces a session context file for the **next**
  WP in the dependency chain. Saved to
  `docs/ai/session-context/session-context-wpXXX.md` where `XXX` is the
  next WP. This file is the bridge between sessions — it captures the
  completed WP's state so the next session starts with full context
  instead of re-deriving it from scratch. Step 8's output becomes the
  next WP's step 0 input.

### Dependency Contract Verification

**Required when the WP consumes types, field names, helper functions, or
constants defined by a prior WP.**

For each dependency contract consumed by this WP, verify:

- [ ] **Type/field names match exactly** between the dependency WP's code
  and this WP's text. Read the actual source files — do not trust the WP
  text alone.
- [ ] **Function signatures are compatible** with how this WP intends to
  call them. Verify parameter types, return types, and whether the function
  is a pure helper vs a move function (move functions take `MoveContext`
  and perform validation/gating — they cannot be called from helpers).
- [ ] **Move classification is correct** — if the WP adds new moves, verify
  whether they are core moves (registered in `CoreMoveName` / `CORE_MOVE_NAMES`)
  or domain moves with internal stage gating. Do not expand `CoreMoveName`
  without explicit authorization. Domain moves use `G.currentStage` checks
  internally — this is the established pattern from WP-014A/016.
- [ ] **Field paths in G are verified** against `LegendaryGameState` in
  `types.ts`. Do not assume field paths from WP text — read the actual
  type definition.
- [ ] **Helper return patterns are understood** — pure helpers return new
  values (not void). The caller must assign the return value back to `G`.
  Treating a pure helper as a void mutator is a silent no-op.
- [ ] **Optional fields are identified** — if a consumed type has optional
  fields (e.g., `effects?: HeroEffectDescriptor[]`), the WP must handle
  `undefined` explicitly.
- [ ] **Data source identity verified** — if the WP loads files from disk,
  R2, or database, verify that the correct file is being loaded. Files with
  similar names but incompatible schemas (e.g., `sets.json` vs
  `card-types.json`) can be silently confused — Zod validation may pass
  with zero results instead of throwing. Check that loader code references
  the correct source by name, not just by path pattern (WP-003 Defect 1,
  D-1203).
- [ ] **TypeScript types accommodate real data quirks** — if the WP
  consumes card data, verify that TypeScript types are wide enough for
  actual data. Card fields may contain modifier strings (`"2+"`, `"2*"`),
  nulls, or unexpected types that compile against Zod schemas but fail
  narrow TypeScript interfaces. Read `packages/registry/src/schema.ts`
  comments for known quirks before defining or consuming card types
  (WP-003 Defect 2, D-1204).
- [ ] **Schema-engine alignment verified** — if the WP touches configuration
  schemas (JSON, Zod, or similar), verify that schema field names, types,
  and constraints match the corresponding TypeScript contract exactly.
  Schema-engine drift is a stop-ship class failure: mismatched field names
  or missing fields break replay determinism (D-1244 through D-1248).
- [ ] **Handler ownership is explicit** — if the WP defines data-only
  contracts (e.g., hook definitions, effect descriptors), verify that the
  WP explicitly states where runtime handler functions live. Data stored in
  `G` must never contain functions. Handler code belongs in an
  `ImplementationMap` or equivalent runtime-only structure outside `G`
  (D-1232).
- [ ] **Persistence classification is clear** — if the WP adds new fields
  to `G`, verify that ARCHITECTURE.md Section 3 documents which persistence
  class each field belongs to: runtime state (in G, never persisted),
  configuration (MatchSetupConfig, persisted separately), or snapshot
  (audit record with counts only). Derived fields (e.g., endgame outcome)
  must never be stored. Setup-only data (e.g., `hookRegistry`) must be
  excluded from snapshots.

- [ ] **WP scope pre-validated against ARCHITECTURE.md** — if the WP's
  scope section specifies data structures, field layouts, or storage
  locations, verify they match ARCHITECTURE.md. WP scope sections are
  subordinate to architecture — if they contradict, the WP must be
  corrected before execution (D-1212: WP-005B specified flat global
  piles but ARCHITECTURE.md required nested `G.piles`).
- [ ] **Framework API workarounds are documented** — if the WP requires
  a pattern that works around a framework limitation (e.g., module-level
  holders, CJS bridge imports), the workaround must be documented in
  DECISIONS.md as known tech debt with a `// why:` comment in code.
  Do not hide framework friction — make it visible and flagged for
  future resolution (D-1213, D-1206).
- [ ] **New types defined in dedicated contract files** — if the WP
  introduces types that will be consumed by multiple files or future WPs,
  they must be defined in a dedicated `*.types.ts` file, not inline in
  implementation code. Inline definitions create consolidation debt when
  the next WP needs the same types (D-1216: `PlayerZones` and
  `CardExtId` defined inline in WP-005B, had to be moved to
  `zones.types.ts` during WP-006A).

- [ ] **Immutable file designations respected** — if the WP references
  files marked immutable by prior WPs (e.g., `schema.ts`, `shared.ts`,
  `localRegistry.ts` per WP-003), verify the WP does not modify them.
  Changes to immutable files require WP-level authorization and a
  DECISIONS.md entry — not a "fix it while I'm here" change.
- [ ] **Cross-service identifiers use ext_id strings** — if the WP
  introduces or consumes card/entity references that cross package or
  storage boundaries, they must use `ext_id` text strings, never numeric
  foreign keys. Numeric IDs couple to seeding order and break on
  re-seeds (D-1201, D-1202).

- [ ] **Locked value string literals match actual code** — if the WP's
  locked values table contains string literals (condition types, keyword
  names, enum values, discriminant tags), verify them against the code
  that *produces* those strings, not just the code that consumes them.
  WP text may use aspirational or renamed labels that never appear in
  `G` at runtime. An evaluator built against WP names instead of actual
  code strings will silently fail to match any real data
  (WP-023 precedent, D-2304).
- [ ] **Runtime data availability verified for all evaluators** — if the
  WP introduces evaluators, dispatchers, or condition checkers that read
  data from `G`, verify that the required data fields *actually exist*
  in `G` at the time they will be read. Check the actual type definitions
  (`LegendaryGameState`, `CardStatEntry`, etc.), not the WP's claims
  about what data is available. A WP may assume setup-time resolution
  that was never implemented by prior WPs. If data is missing and the
  scope lock does not permit adding it, the WP must be narrowed or split
  (WP-023 precedent, D-2302).
- [ ] **Setup-time data source verified for all extractors** — if the WP
  introduces setup-time functions that extract structured data from
  registry card fields (e.g., keyword classification from ability text),
  verify that the source data actually contains the expected information.
  Read actual card JSON files — do not trust WP claims about data
  availability. A WP may assume structured metadata exists when only
  free-text ability strings are available, or may assume a keyword
  appears in the data when it refers to a different mechanic with the
  same name. If data is partially available, use the safe-skip pattern:
  implement the full extraction structure but return empty results for
  data-absent types (WP-025 precedent, D-2504).
- [ ] **WP function signatures match established conventions** — if the
  WP specifies function signatures for setup-time functions that accept
  registry data, verify they match the established convention for that
  pattern. The engine convention is `registry: unknown` with a local
  structural interface and runtime type guard (e.g.,
  `CardStatsRegistryReader` in `economy.logic.ts`). WP text may use
  direct registry types (e.g., `CardRegistry`) that violate the layer
  boundary. Correct the WP before execution (WP-025 precedent, D-2504).

- [ ] **WP file paths verified against actual filesystem** — if the WP's
  "Files Expected to Change" or scope sections reference file paths, verify
  those files actually exist in the codebase. WP text may reference intended
  file locations that were never created, or code that was later consolidated
  into a different file. Read the actual filesystem — do not trust WP path
  claims. If a WP says to create or modify `src/setup/foo.ts` but `foo()`
  actually lives in `src/rules/bar.ts`, the WP must be corrected before
  execution (WP-024 precedent, D-2405).
- [ ] **WP file paths verified against code category rules** — if the WP
  creates new files, verify the target directory matches the code category
  defined in `docs/ai/REFERENCE/02-CODE-CATEGORIES.md`. A WP may place a
  setup-time builder in a domain directory (e.g., `src/scheme/`) when code
  categories require it in `src/setup/`. Category violations create
  precedent drift that compounds across future WPs. Correct the WP file
  paths before execution (WP-026 precedent, D-2603).
- [ ] **WP gating assumptions verified as true** — if the WP's Assumes
  section contains statements marked as blocking (e.g., "if any of the
  above is false, this packet is BLOCKED"), verify each assumption against
  the actual codebase and data. A WP may assert that structured metadata
  exists in the registry when only free-text ability strings are available.
  If a gating assumption is provably false, the WP must be corrected
  before execution — either remove the blocking gate, convert the
  assumption to a safe-skip MVP constraint, or defer the WP entirely.
  Do not execute a WP whose own blocking assumptions are false (WP-026
  precedent: "Scheme card data includes setup instruction metadata"
  was false — converted to safe-skip with builder returning `[]`).
- [ ] **Decision IDs not referenced before creation** — if the WP or EC
  references a DECISIONS.md entry by ID (e.g., D-0603), verify that entry
  actually exists. WP/EC text may reference placeholder decision IDs that
  were never assigned. Use title-based references (e.g., "Representation
  Before Execution" decision) with the ID as a navigational aid, not a
  hard dependency. If the decision doesn't exist yet, either create it
  before execution or decouple the reference from the specific ID
  (WP-026 precedent: EC-026 referenced D-0603 which did not exist;
  resolved by creating D-2601 and using title-first references).

- [ ] **Functions the WP calls are actually exported** — if the WP says
  to "call", "import", or "use" a function, verify that function is
  exported from its module. Local/non-exported functions cannot be
  imported by other files. If a function is not exported, the WP must
  document an alternative approach: call the underlying helper directly,
  reconstruct equivalent behavior, or request the function be exported
  (which requires modifying the source file and updating the WP scope).
  Read the actual source file — do not trust WP claims about
  importability (WP-027 precedent: `advanceStage` was a local function
  in `game.ts`, not exported. Resolved by calling `advanceTurnStage`
  from `turnLoop.ts` directly, D-2705).
- [ ] **WP approach does not require forbidden imports** — if the WP
  describes calling framework APIs (e.g., `Game.setup()`,
  `boardgame.io/testing`) or package-level APIs (e.g., `CardRegistry`
  from `@legendary-arena/registry`), verify whether the calling file is
  permitted to import those APIs under its code category rules. If the
  file must be pure (no `boardgame.io` imports) or must not cross layer
  boundaries (no registry imports in engine code), the WP must use
  alternative approaches (direct builder functions, local structural
  interfaces, mock contexts). This is a transitive import check — even
  if the WP text doesn't say `import boardgame.io`, the approach may
  require it implicitly (WP-027 precedent: `replayGame` could not call
  `Game.setup()` because replay files must not import boardgame.io;
  resolved by calling `buildInitialGameState` directly, D-2702.
  WP-025/026 precedent: `CardRegistry` replaced with `registry: unknown`
  + local structural interface).

If any name, field, or signature in the WP text does not match the actual
code, the WP must be corrected **before** execution. This is a blocking
pre-flight finding.

**Precedent (WP-022):** Pre-flight discovered keyword literal drift
(`gainAttack` vs `'attack'`, `amount` vs `magnitude`), a helper-vs-move
confusion (`drawCards` is a move function, not a callable helper), a
missing `playerID` parameter, and an unverified field path (`G.ko`).
All were corrected before execution. Without this verification, the
executing agent would have used wrong names and produced broken code.

**Precedent (WP-016):** Pre-flight v1 identified a design blocker —
the WP said to add moves to `CoreMoveName` but also said not to modify
that contract type. Pre-flight v2 resolved this by recognizing that
WP-014A already established the domain-move pattern (internal stage
gating without expanding `CoreMoveName`). The blocker dissolved once
the existing pattern was applied. Lesson: check whether prior WPs
already solved the design problem before declaring a blocker.

**Precedent (WP-023):** Pre-flight discovered two blocking issues.
(1) **String literal drift in locked values:** the WP's condition type
table used `requiresColor` but `heroAbility.setup.ts` produces
`heroClassMatch`. The evaluator would have silently failed to match any
real condition data. (2) **Data availability gap:** the WP assumed
team/class data was available in `G.cardStats` for condition evaluation,
but `CardStatEntry` only has `attack/recruit/cost/fightCost`. The data
exists in the registry but was never resolved into `G` at setup time,
and the WP's "Files Expected to Change" didn't include the files needed
to add it. Resolution: Option A (narrow scope) — implement all 4
evaluators but `heroClassMatch` and `requiresTeam` return `false`
(safe-skip placeholder) until a follow-up WP resolves the data into G.
`requiresKeyword` and `playedThisTurn` were fully functional with
existing G data. Lesson: verify locked value strings against the code
that *produces* them, and verify that all data read by evaluators
actually exists in G — not just that the G fields the WP names exist.

Answer **YES / NO** to each:

- [ ] All prior WPs compile and tests pass (record counts if known)
- [ ] No known EC violations remain open
- [ ] Required types/contracts exist and are exported as referenced by `WP-XXX`

### Runtime Readiness Check (Mutation & Framework)

**Required when Work Packet Class = Runtime Wiring or Behavior / State Mutation.**
Skip this section for Contract-Only WPs.
(See also: `docs/ai/REFERENCE/01.5-runtime-wiring-allowance.md`)

Answer **YES / NO** to each:

- [ ] The expected runtime touchpoints are known (e.g., `game.ts`, phase hooks, moves map)
- [ ] Framework context requirements are understood (e.g., `ctx.events.*`, `ctx.random.*`)
- [ ] Existing test infrastructure can support required mocks without modifying locked helpers
- [ ] Any runtime wiring allowance needs are anticipated (01.5 referenced if applicable)
- [ ] No architecture boundary violations are expected (engine vs framework separation)
- [ ] Integration point code has been read and the call site is confirmed
  (e.g., which function destructures what, what parameters are available)
- [ ] If the WP adds a new move or hook: stage gating requirements are
  identified (which `G.currentStage` values allow the action)
- [ ] If the WP performs multi-step mutations: the ordering is safe
  (state is not removed from a source before the destination is confirmed
  valid — avoid silent data loss)
- [ ] If the WP uses registry data: it flows through setup-time resolution
  into `G`, not runtime registry access (registry boundary)
- [ ] If the WP involves phase transitions (`ctx.events.setPhase`): any
  observability flags in `G` are set **before** the transition call, not
  after. Reversed ordering causes race conditions in multiplayer (D-1238).
- [ ] If the WP handles simultaneous or conflicting conditions: evaluation
  order is explicitly defined and documented (e.g., loss conditions checked
  before victory — D-1235).
- [ ] If the WP introduces a degradation path for unknown/unsupported
  types: the strategy is explicit (warn + continue vs skip vs throw).
  Default: warn via `G.messages` and continue — never crash the game loop
  (D-1234).
- [ ] If the WP calls move functions outside boardgame.io's framework
  context (e.g., replay harness, simulation, testing infrastructure):
  required mock context shapes are identified. Move functions destructure
  `FnContext<LegendaryGameState> & { playerID }` from boardgame.io — a
  local structural interface must be defined with the minimum fields moves
  actually use (G, playerID, ctx, events, random). Events may be no-ops;
  random must provide deterministic Shuffle. Identify which moves the WP
  will dispatch and verify their destructuring patterns in the actual
  source code (WP-027 precedent: `ReplayMoveContext` local interface).
- [ ] If the WP assumes specific mock or PRNG capabilities: verify that
  `makeMockCtx` (or the intended mock) actually supports the assumed
  behavior. `makeMockCtx` provides a deterministic reverse-shuffle, NOT
  a seeded PRNG — different seed values produce identical shuffle output.
  If the WP requires seed-faithful determinism, that is a capability gap
  requiring either new test infrastructure or a safe-skip MVP where the
  seed field is stored but not used. Document the gap explicitly
  (WP-027 precedent: MVP uses mock shuffle; seed stored for future
  seed-faithful replay, D-2704).

Any **NO** must be explained. If any answer is **NO**, pre-flight is **NOT READY**.

### Established Patterns to Follow (Locked Precedents)

The following patterns are validated by prior WPs and must be reused rather
than reinvented:

- Inline mock contexts for tests requiring framework features (e.g.,
  `ctx.events`, `ctx.random`) — do NOT modify shared helpers
- Passing `ctx` directly where it is structurally compatible with narrower
  interfaces (e.g., `ShuffleProvider`)

## Precedent Log

This section documents recurring failure modes discovered during prior
phases. Each entry describes what went wrong, why, and what pre-flight
check now prevents it. Entries are ordered by discovery date.

### Foundation Phase — Environment & Infrastructure (FP-00.4 through FP-02)

**FP-1: Stop-on-failure gate semantics (FP ordering)**
- **What it establishes:** Foundation Prompts run once in strict order
  (00.4 -> 00.5 -> 01 -> 02). If any FP fails, ALL downstream work is
  blocked — not just the next FP, but all Work Packets. This is a
  harder gate than per-WP lint checks.
- **Why this matters for pre-flight:** Pre-flight may assume FPs are
  complete for normal WPs. But if a WP touches infrastructure that an
  FP established (database schema, R2 data, server wiring, environment
  variables), pre-flight should verify the relevant FP's assumptions
  still hold.

**FP-2: R2 validation error vs warning distinction (FP-00.5)**
- **What happened:** R2 validation produces both errors (structural
  failures like missing sets, invalid JSON) and warnings (known data
  quirks like `[object Object]` abilities, cross-set duplicate slugs,
  missing VP on some masterminds). Early in development, warnings were
  sometimes treated as blockers, unnecessarily halting work.
- **Root cause:** No explicit severity distinction in validation output.
- **Fix:** `pnpm validate` exit code 0 = no errors (warnings ok),
  exit code 1 = errors exist. Warnings are informational and documented
  in `03-DATA-PIPELINE.md` Known Data Quality Items.
- **Prevention:** Structural Readiness now distinguishes errors from
  warnings for R2 validation.

**FP-3: Idempotent migration pattern (FP-01/02)**
- **What it establishes:** All database DDL uses
  `CREATE TABLE IF NOT EXISTS`, all inserts use
  `INSERT ... ON CONFLICT DO NOTHING`. Migrations can be re-run safely.
  There is no rollback mechanism — this is intentional simplicity.
  Manual recovery via `psql` if needed.
- **Why this matters for pre-flight:** Any WP that adds migrations must
  follow the same idempotency pattern. Non-idempotent migrations will
  fail on re-deployment.

**FP-4: Environment assumptions validated once (FP-00.4)**
- **What it establishes:** 9 required environment variables, Node v22+,
  boardgame.io `^0.50.0`, rclone configured with R2 remote, database
  seeded. These are checked once by FP-00.4 and assumed valid for all
  subsequent work.
- **Why this matters for pre-flight:** WPs that depend on specific
  environment capabilities (database access, R2 connectivity, specific
  Node APIs) can assume FP-00.4 has validated them. But if a WP adds
  new environment requirements (new env vars, new external services),
  the FP-00.4 check should be updated — note this as a post-execution
  task.

### Phase 0 — Coordination & Game Skeleton (WP-001 through WP-004)

**P0-1: Prior sessions drifted without coordination system (WP-001)**
- **What happened:** 15+ prior Claude Code sessions had re-derived
  settled decisions from chat history, producing conflicting field names
  (`strikeCount` vs `vp`), stale table names (`expansions` vs
  `legendary.sets`), wrong migration paths, and inconsistent image URL
  patterns.
- **Root cause:** No single source of truth existed. Each session started
  fresh and guessed at conventions.
- **Fix:** WP-001 created the full `docs/ai/REFERENCE/` coordination
  system with override hierarchy, WP template, and session protocol.
  All REFERENCE documents were audited and corrected.
- **Prevention:** The override hierarchy (00.1 > ARCHITECTURE.md > WPs >
  conversation) is now the foundational governance pattern. Pre-flight
  reads all authority documents before making a verdict.

**P0-2: ext_id vs numeric FK decision cascaded everywhere (D-1201, D-1202)**
- **What happened:** The database schema initially used `bigint` foreign
  keys for card references. This would couple match records to seeding
  order — a re-seed would invalidate all saved configurations and break
  replay targeting.
- **Root cause:** Standard relational database practice (numeric FKs) is
  wrong for a system where the seed data can be regenerated.
- **Fix:** D-1201 and D-1202 locked `ext_id` text strings for all
  cross-service identifiers. `MatchSetupConfig` uses string IDs, not
  numeric ones. This cascaded into zone contents (CardExtId), virtual
  card ext_ids, and all downstream WPs.
- **Prevention:** Dependency Contract Verification now checks that
  cross-service identifiers use ext_id strings.

**P0-3: boardgame.io CJS-only server bundle (D-1206)**
- **What happened:** `boardgame.io/server` only ships CommonJS. The
  project is ESM-only (Node v22+). Direct `import` failed at runtime
  with no helpful error message.
- **Root cause:** boardgame.io's package exports don't include ESM for
  the server subpackage. This was not discoverable until WP-004 tried
  to wire the server.
- **Fix:** D-1206 introduced `createRequire(import.meta.url)` bridge.
  Documented as known tech debt with `// why:` comment.
- **Prevention:** Established Patterns documents framework workarounds
  as visible tech debt. Runtime Readiness checks framework API
  compatibility.

**P0-4: Immutable file pattern established (WP-003)**
- **What happened:** After fixing the two registry defects, WP-003
  designated `schema.ts`, `shared.ts`, and `localRegistry.ts` as
  immutable — confirmed correct and locked against casual modification.
- **Root cause:** Registry code is the data input foundation. Incorrect
  changes here cause silent failures that are extremely hard to trace
  (zero sets loaded with no error).
- **Fix:** Immutable file designations recorded in
  `.claude/rules/registry.md`. Future changes require WP-level
  authorization and DECISIONS.md entry.
- **Prevention:** Dependency Contract Verification now checks immutable
  file designations.

**P0-5: End-to-end proof before feature work (WP-004)**
- **What happened:** WP-004 was a pure wiring packet — no game logic,
  no new features. It loaded the registry, wired `LegendaryGame` into
  `boardgame.io Server()`, handled the CJS bridge, and proved the
  architecture works end-to-end with real packages.
- **Why this mattered:** If the wiring had failed (wrong API shape,
  incompatible versions, missing exports), every Phase 1+ WP would
  have been blocked. Proving the architecture before building features
  prevented cascading failures.
- **Prevention:** Established Patterns documents that new architectural
  layers should be proven with a wiring WP before feature WPs build on
  them.

### Phase 1 — Foundation Contracts & Setup (WP-005A through WP-006B)

**P1-1: WP scope contradicted ARCHITECTURE.md (D-1212)**
- **What happened:** WP-005B's scope section specified global piles
  (`bystanders`, `wounds`, `officers`, `sidekicks`) as flat top-level

  against ARCHITECTURE.md. The authority hierarchy (ARCHITECTURE.md >
  Work Packets) was not consulted before scope was finalized.
- **Fix:** D-1212 documented that the nested structure is correct and
  established the override principle: architecture always wins.
- **Prevention:** Dependency Contract Verification now includes WP scope
  pre-validation against ARCHITECTURE.md.

**P1-2: Framework API workaround required (D-1213)**
- **What happened:** boardgame.io's `setup()` function signature only
  accepts `(context, setupData)` — it has no way to pass the registry.
  The engine needed registry data at setup time but the framework API
  didn't support it.
- **Root cause:** Framework constraints not discovered until
  implementation. The `setup()` API was not inspected during WP design.
- **Fix:** D-1213 invented `setRegistryForSetup(registry)` — a
  module-level registry holder in `game.ts`. Documented as known tech
  debt, explicitly flagged for future factory pattern replacement.
- **Prevention:** Runtime Readiness now checks framework API
  compatibility. Established Patterns documents that framework
  workarounds must be visible tech debt with DECISIONS.md entries.

**P1-3: Inline type definitions created consolidation debt (D-1216)**
- **What happened:** WP-005B defined `PlayerZones`, `GlobalPiles`, and
  `CardExtId` inline in `types.ts`. When WP-006A needed zone contracts,
  these types already existed in two places, creating duplication risk.
  WP-006A had to retroactively consolidate them into
  `src/state/zones.types.ts` with re-exports for backward compatibility.
- **Root cause:** Types were defined inline during implementation for
  convenience, without considering that future WPs would need them as
  standalone contracts.
- **Fix:** D-1216. Types moved to dedicated `*.types.ts` file. Re-exports
  preserved backward compatibility.
- **Prevention:** Dependency Contract Verification now requires new types
  consumed by multiple files to be defined in dedicated contract files.
  Established Patterns documents this as project norm.

**P1-4: Test helper state pollution risk (WP-005B revision)**
- **What happened:** `makeMockCtx` used a module-level registry holder
  that could leak state between tests if not explicitly reset.
- **Root cause:** Shared test helpers with module-level state were
  designed without considering cross-test isolation.
- **Fix:** Added `clearRegistryForSetup()` test-only reset hook during
  the same WP's revision pass.
- **Prevention:** Established Patterns now documents that test helpers
  must include state isolation mechanisms.

**P1-5: A/B packet pattern emerged as project norm**
- **What happened:** Phase 1 organically split every major feature into
  A packets (contracts/types) and B packets (implementation). This was
  not planned — it emerged because contracts needed to be stable before
  logic could safely build on them.
- **Root cause:** Without the split, implementation would lock API
  surfaces prematurely, and subsequent WPs couldn't depend on stable
  types.
- **Fix:** Adopted as project norm. All future phases follow A/B
  packet pattern for features that define new contracts.
- **Prevention:** Established Patterns documents this explicitly.

### Phase 2 — Core Turn Engine (WP-003 through WP-008B)

**P2-1: Silent data source confusion (WP-003 Defect 1)**
- **What happened:** `httpRegistry.ts` fetched `card-types.json` (card
  type taxonomy) instead of `sets.json` (set index). Zod validation
  silently succeeded because `card-types.json` entries are valid objects —
  but they lack `abbr` and `releaseDate` fields. Result: zero sets loaded
  with no error. Every downstream match creation call would fail.
- **Root cause:** Two registry files with similar names (`sets.json`,
  `card-types.json`) but incompatible schemas. Generic Zod parsing
  accepted both without distinguishing them.
- **Fix:** D-1203. Added explicit `// why:` comment explaining the
  distinction. Loader validates data source identity, not just shape.
- **Prevention:** Dependency Contract Verification now includes data
  source identity checks. Risk examples include data source confusion.

**P2-2: TypeScript type too narrow for real card data (WP-003 Defect 2)**
- **What happened:** `FlatCard.cost` was typed as `number | undefined`
  but real card data includes star-cost strings like `"2*"` (Wasp) and
  `"3*"`. The Zod schema (`HeroCardSchema.cost`) already accepted both
  `number` and `string`, but the TypeScript interface rejected valid data
  silently during registry load.
- **Root cause:** TypeScript interfaces were defined from the "ideal"
  data model, not the actual data. Real Legendary card data has modifier
  strings, nulls, and formatting quirks that the Zod schema accommodates
  but narrow types do not.
- **Fix:** D-1204. Widened `FlatCard.cost` to `string | number | undefined`.
- **Prevention:** Dependency Contract Verification now checks that
  TypeScript types accommodate real data quirks. Risk examples include
  type narrowing vs real data.

**P2-3: Registry schema gaps forced WP split (WP-014A/B)**
- **What happened:** WP-014 was originally one packet covering the full
  villain deck (reveal pipeline + deck composition). During pre-flight,
  4 critical schema gaps were discovered: henchmen have no individual
  card entries, scheme twists have no ext_ids, deck composition counts
  depend on rules not config, and mastermind strike identification was
  implicit. These gaps couldn't be resolved without design decisions.
- **Root cause:** Registry schema was designed for card display (registry
  viewer), not for gameplay engine consumption. Gameplay needs (individual
  card tracking, deterministic instancing) exceed the display schema.
- **Fix:** D-1410 through D-1413. Split into WP-014A (reveal pipeline,
  independent of deck construction) and WP-014B (deck composition with
  virtual card conventions). Virtual cards generated at setup time with
  locked ext_id formats.
- **Prevention:** Established Patterns now documents virtual card
  conventions and WP splitting when registry gaps block implementation.

**P2-4: Fail-closed vs silent no-op confusion (cross-WP pattern)**
- **What happened:** Early WPs inconsistently handled error cases —
  some threw, some returned silently, some logged warnings. The
  distinction between "structural invariant violation" (something is
  wrong with the game setup) and "unmet gameplay condition" (player
  doesn't have enough attack) was not formalized.
- **Root cause:** No explicit failure mode taxonomy. Each WP invented
  its own error handling strategy.
- **Fix:** Formalized across Phase 2: structural violations produce
  fail-closed behavior with `G.messages` entry. Unmet gameplay conditions
  produce silent no-op (return void). Moves never throw.
  `Game.setup()` is the only context that may throw.
- **Prevention:** Established Patterns now documents the fail-closed vs
  silent no-op distinction with specific examples.

### Phase 3 — MVP Multiplayer Infrastructure (WP-009A through WP-013)

**P3-1: Schema-engine drift in Match Setup (Phase 3 exit gate)**
- **What happened:** The Match Setup JSON schema used wrong field name
  `heroIds` instead of `heroDeckIds`, was missing 5 count fields, and
  had `playerCount` max of 10 vs engine's `maxPlayers = 5`. Drift
  accumulated silently from WP-005A through all of Phase 3.
- **Root cause:** The JSON schema and the TypeScript `MatchSetupConfig`
  contract were derived from the same requirements document but never
  cross-validated. No executable test compared schema constraints against
  type constraints.
- **Fix:** D-1244 through D-1248 locked composition block to 9 fields
  matching `MatchSetupConfig` exactly, enforced `additionalProperties:
  false`, and corrected `playerCount` max.
- **Prevention:** Dependency Contract Verification now includes
  schema-engine alignment. Established Patterns now documents
  fail-closed configuration with `additionalProperties: false`.

**P3-2: Handler ownership ambiguity (WP-009A/009B boundary)**
- **What happened:** WP-009A defined `HookDefinition` as data-only but
  did not specify where handler functions should live. WP-009B had to
  invent the `ImplementationMap` pattern during implementation.
- **Root cause:** The contract WP (009A) focused on the data shape but
  not the runtime architecture. Handler location is an architectural
  concern that should have been resolved before the contract was locked.
- **Fix:** D-1232 locked handlers in `ImplementationMap` outside `G`.
  D-1233 established the two-step execute/apply pipeline.
  ARCHITECTURE.md Section 4 documents both.
- **Prevention:** Dependency Contract Verification now requires explicit
  handler ownership documentation for data-only contracts. Established
  Patterns documents the two-step pipeline and data/handler separation.

**P3-3: Persistence field classification discovered during implementation (WP-013)**
- **What happened:** WP-013 had to discover at implementation time which
  `G` fields belong in snapshots. `hookRegistry` is setup data (not
  runtime state) and must be excluded. `outcome` is derived and must
  never be stored. Zone snapshots store counts only, not card arrays.
- **Root cause:** ARCHITECTURE.md did not have a field-level persistence
  classification table before WP-013. The WP had to build it during
  execution, consuming time and introducing risk.
- **Fix:** ARCHITECTURE.md Section 3 now documents explicit field tables
  for runtime, configuration, and snapshot persistence classes.
- **Prevention:** Dependency Contract Verification now checks persistence
  classification for any new `G` fields.

**P3-4: Lobby observability flag ordering (WP-011)**
- **What happened:** UI needed to detect lobby completion, but querying
  `ctx.phase` from the UI violates the engine/framework boundary.
  Solution: set `G.lobby.started = true` before calling
  `ctx.events.setPhase('setup')`. Order is non-negotiable — if reversed,
  multiplayer clients may miss the event due to race conditions.
- **Root cause:** Bridging framework state (`ctx.phase`) and game state
  (`G`) requires explicit coordination. The pattern is subtle and was
  not documented before WP-011 encountered it.
- **Fix:** D-1238 locked the "flag then transition" pattern.
- **Prevention:** Runtime Readiness now checks phase transition ordering.
  Established Patterns documents the flag-before-transition rule.

**P3-5: Endgame evaluation order ambiguity (WP-010)**
- **What happened:** Both loss and victory conditions could be met in
  the same turn. Initial implementation did not specify evaluation order,
  creating a determinism violation.
- **Root cause:** The WP described conditions independently without
  addressing the simultaneous case. Legendary tabletop rules specify
  loss overrides victory, but this was not in the WP text.
- **Fix:** D-1235 locked loss-before-victory evaluation order. Explicit
  test covers the simultaneous edge case.
- **Prevention:** Runtime Readiness now checks for simultaneous/conflicting
  condition handling. Risk examples include this pattern.

**P3-6: Unknown rule effect type handling (WP-009B)**
- **What happened:** Future WPs will add new `RuleEffect` types. The
  runtime needed a strategy for encountering unknown types — crash,
  ignore, or degrade gracefully?
- **Root cause:** Forward compatibility requires an explicit degradation
  strategy. Without one, the first new effect type added by a future WP
  could crash the game loop.
- **Fix:** D-1234 established: unknown effect types push a warning to
  `G.messages` and execution continues without throwing.
- **Prevention:** Runtime Readiness now checks for degradation strategies
  when unknown/unsupported types are possible. Established Patterns
  documents the warn-and-continue approach.

### Phase 4 — Core Gameplay Loop (WP-014A through WP-020)

**P4-1: Stage gating gap (WP-015A patch of WP-014A)**
- **What happened:** `revealVillainCard` had no check for
  `G.currentStage === 'start'`, allowing cards to be revealed in the
  wrong turn phase.
- **Root cause:** WP-014A focused on the reveal pipeline but did not
  specify stage gating. The omission was not caught in pre-flight.
- **Fix:** WP-015A added an early-return stage gate. A micro-patch
  packet was created rather than modifying WP-014A.
- **Prevention:** Runtime Readiness now requires identifying stage
  gating requirements for any new move or hook.

**P4-2: Mutation ordering — silent card loss (WP-015A patch)**
- **What happened:** Cards were removed from the villain deck before
  City placement was confirmed valid. A malformed `G.city` caused the
  card to vanish — removed from deck but never placed.
- **Root cause:** Synchronous mutation ordering was not considered.
  The "remove then place" sequence is unsafe when the destination
  can reject the card.
- **Fix:** Deferred deck removal until City placement was confirmed.
- **Prevention:** Runtime Readiness now checks mutation ordering for
  multi-step operations. Scope Lock now forbids removing from source
  before confirming destination validity.

**P4-3: Move classification ambiguity (WP-016 v1 blocker)**
- **What happened:** WP-016 said to add `fightVillain` and `recruitHero`
  as moves, implying expansion of `CoreMoveName`. But `CoreMoveName` is
  a locked contract. Pre-flight v1 declared this a blocking conflict.
- **Root cause:** The WP did not distinguish between core moves
  (registered in `CoreMoveName`) and domain moves (with internal stage
  gating). WP-014A had already established the domain-move pattern with
  `revealVillainCard`, but this wasn't documented as a reusable pattern.
- **Fix:** Pre-flight v2 recognized the WP-014A precedent. Domain moves
  enforce stage gating internally without expanding `CoreMoveName`.
  ARCHITECTURE.md was updated to document the pattern.
- **Prevention:** Dependency Contract Verification now includes move
  classification. Established Patterns now documents the domain-move
  pattern explicitly.

**P4-4: Registry boundary at runtime (prevented across Phase 4)**
- **What happened:** Never reached production, but was a consistent risk.
  Multiple WPs needed card data during moves (type classification,
  stat values, ability hooks).
- **Root cause:** The natural impulse is to query the registry when you
  need card data. The architecture forbids this — moves must use
  pre-resolved data in `G`.
- **Fix:** Every Phase 4 WP that needed card data added a setup-time
  resolution step: `G.villainDeckCardTypes` (WP-014B),
  `G.cardStats` (WP-018), `G.heroAbilityHooks` (WP-021). Import
  audits in verification steps catch violations.
- **Prevention:** Runtime Readiness now explicitly checks registry
  data flow. Scope Lock forbids runtime registry access from moves.

**P4-5: Narrow mock crashes (WP-014B runtime guard)**
- **What happened:** Test mocks satisfying `CardRegistryReader` (narrow
  interface with only `listCards()`) crashed when `buildCardStats`
  called `listSets()` and `getSet()` — methods only on the broader
  `CardStatsRegistryReader` interface.
- **Root cause:** Structural typing means a narrow mock compiles but
  crashes at runtime when broader methods are called.
- **Fix:** Added `isCardStatsRegistryReader` runtime type guard that
  returns an empty result for narrow mocks instead of crashing.
- **Prevention:** Dependency Contract Verification now checks for
  optional fields and interface compatibility. Test Expectations now
  includes defensive guard guidance.

**P4-6: Downstream invalidation requiring re-audit (WP-014A/014B)**
- **What happened:** After WP-015 changed villain/henchman routing from
  discard to City, WP-014A's reveal pipeline invariants needed
  re-validation. Pre-flight v2/v3 revisions were created as "Mode B
  audit" prompts to verify invariants still held.
- **Root cause:** Downstream WPs can modify shared state paths that
  earlier WPs established. Without re-validation, invariant violations
  accumulate silently.
- **Mitigation:** When a WP modifies state paths established by a prior
  WP, note the risk explicitly. If the change is significant, a Mode B
  audit session may be warranted after execution.
- **Note:** This is a post-execution concern, not a pre-flight check.
  Pre-flight can flag the risk; the audit happens after the downstream
  WP completes.

### Phase 5 — Hero Abilities (WP-021, WP-022, WP-023)

**P5-1: Contract field drift in WP text (WP-022 pre-flight)**
- **What happened:** WP-022 used `gainAttack`, `gainRecruit`, `koCard`
  as keyword literals and `amount` as the descriptor field. The actual
  WP-021 contracts defined `'attack'`, `'recruit'`, `'ko'` and
  `magnitude`.
- **Root cause:** WP-022 was written before WP-021 was implemented.
  The WP author used intuitive names rather than the exact literals
  that WP-021 later defined.
- **Fix:** Pre-flight read the actual WP-021 source files and corrected
  all names before execution.
- **Prevention:** Dependency Contract Verification now mandates reading
  actual source files, not trusting WP text alone.

**P5-2: Helper-vs-move confusion (WP-022 pre-flight)**
- **What happened:** WP-022 listed `drawCards` as a helper to call for
  the `'draw'` keyword. But `drawCards` is a full move function that
  takes `MoveContext` and performs args validation and stage gating.
  Calling it from `executeHeroEffects(ctx: unknown)` would not compile.
- **Root cause:** The WP text described the *intent* (draw cards) but
  not the *mechanism* (which function to call and whether it's callable
  from a helper context).
- **Fix:** Pre-flight identified the incompatibility and locked a
  resolution: extract the draw algorithm using zone-ops primitives
  instead of calling the move function.
- **Prevention:** Dependency Contract Verification now requires verifying
  whether functions are pure helpers vs move functions.

**P5-3: Missing parameter in function signature (WP-022 pre-flight)**
- **What happened:** `executeHeroEffects(G, ctx, cardId)` had no
  `playerID` parameter, but the `'draw'` keyword needs to know whose
  deck/hand zones to modify.
- **Root cause:** The original signature was designed around the effect
  descriptors (which reference a card) without considering that some
  effects need player context.
- **Fix:** Pre-flight added `playerID: string` to the signature after
  reading the integration point code and understanding what data the
  caller has available.
- **Prevention:** Runtime Readiness now requires reading the integration
  point code and confirming what parameters are available.

**P5-4: Locked value string literals using aspirational names (WP-023 pre-flight)**
- **What happened:** WP-023's locked condition types table used
  `requiresColor` but `heroAbility.setup.ts` produces `heroClassMatch`.
  An evaluator built against the WP's name would never match any real
  condition in `G.heroAbilityHooks` at runtime.
- **Root cause:** The WP was written with intended semantics
  (`requiresColor` = "requires a color match") rather than the actual
  string produced by WP-021's setup code (`heroClassMatch`). The WP
  author described what the condition *means*, not what the code *says*.
- **Fix:** Pre-flight read `heroAbility.setup.ts` and found the actual
  string at line 123. Locked the session prompt to use `heroClassMatch`.
- **Prevention:** Dependency Contract Verification now includes a
  "Locked value string literals match actual code" check that requires
  verifying string literals against the code that *produces* them, not
  just the code that *consumes* them.

**P5-5: Data availability gap — G field exists but lacks needed subfields (WP-023 pre-flight)**
- **What happened:** WP-023 said condition evaluators would look up
  team/class data from `G.cardStats`. The field `G.cardStats` exists,
  but `CardStatEntry` only has `attack`, `recruit`, `cost`, `fightCost`
  — no `team` or `heroClass`. The data exists in the registry schema
  (`HeroGroupSchema.team`, `HeroCardSchema.hc`) but was never resolved
  into G at setup time.
- **Root cause:** The WP assumed a richer `CardStatEntry` than WP-018
  actually built. The WP's "Files Expected to Change" didn't include
  `economy.types.ts`, `types.ts`, or setup code — so there was no
  authorized path to add the missing data.
- **Fix:** Pre-flight proposed Option A (narrow scope): implement all
  4 evaluators but `heroClassMatch` and `requiresTeam` return `false`
  (safe-skip placeholder) until a follow-up WP resolves team/class data
  into G. Human authorized this approach.
- **Prevention:** Structural Readiness and Dependency Contract
  Verification now include checks that verify subfield availability
  inside G types, not just top-level field existence. The safe-skip
  placeholder pattern is now a locked precedent for handling data gaps
  without blocking execution or expanding scope.

**P5-6: Scope lock vs data infrastructure mismatch (WP-023 pre-flight)**
- **What happened:** Two of four condition types required data that
  didn't exist in G, and the WP's scope didn't include the files needed
  to add it. This created a design gap — the WP described behavior that
  was unimplementable within its own scope boundaries.
- **Root cause:** The WP was designed top-down from game mechanics
  (what conditions should exist) rather than bottom-up from available
  data (what conditions can be evaluated with current G fields).
- **Fix:** Narrow scope with human authorization. The session prompt
  made blocked evaluators explicit and documented them as safe-skip
  placeholders with `// why:` comments.
- **Prevention:** Risk & Ambiguity Review now includes "Scope lock vs
  data infrastructure mismatch" as a named risk pattern.

### Phase 6 — Scheme & Mastermind Abilities (WP-024)

**P6-1: WP file paths referencing non-existent files (WP-024 pre-flight)**
- **What happened:** WP-024's scope section and "Files Expected to Change"
  listed `src/setup/buildDefaultHookDefinitions.ts` (modify) and

  exists. Both `buildDefaultHookDefinitions` and

- **Root cause:** The WP was authored before WP-009B was implemented.
  The WP author assumed file paths based on intended architecture
  (`setup/` directory for setup-time functions) rather than the actual
  structure that WP-009B created (`rules/` directory, consolidated
  into one file). The paths were never validated against the actual
  filesystem after WP-009B was complete.
- **Fix:** Pre-flight read the actual codebase, found the functions in
  `ruleRuntime.impl.ts`, and corrected both WP-024 and EC-024 before
  execution. D-2405 logged as a permanent decision.
- **Prevention:** Dependency Contract Verification now includes a
  "WP file paths verified against actual filesystem" check. Risk
  examples include "WP file paths referencing non-existent files."

**P6-2: Stub replacement cascading into integration tests (WP-024 pre-flight)**
- **What happened:** WP-009B created stub handlers
  (`defaultSchemeImplementation`, `defaultMastermindImplementation`)
  with placeholder triggers (`onTurnStart`, `onTurnEnd`) and specific
  message strings (`'Scheme: turn started.'`, `'Mastermind: turn ended.'`).
  Integration tests in `ruleRuntime.integration.test.ts` asserted these
  exact values. WP-024 replaced stubs with real handlers using real
  triggers (`onSchemeTwistRevealed`, `onMastermindStrikeRevealed`),

- **Root cause:** The integration tests were designed to prove the
  pipeline worked with stubs. When the stubs were replaced, the tests'
  value assertions became stale. This is a predictable consequence of
  stub-to-real replacement, but it was not anticipated in the WP's
  "Files Expected to Change" section.
- **Fix:** Pre-flight identified the integration tests as requiring
  value-only assertion updates and authorized the changes under the
  01.5 runtime wiring allowance. The session prompt explicitly listed
  each test's required update. D-2404 logged.
- **Prevention:** Established Patterns now documents "stub replacement
  cascades into integration tests" as a locked pattern. Risk examples
  include this scenario.

**P6-3: Effect type vocabulary gap limiting MVP behavior (WP-024 pre-flight)**
- **What happened:** WP-024 specified that the mastermind strike handler
  should cause "each player gains 1 wound." This requires moving a card
  from `G.piles.wounds` to a player's discard pile. But the existing
  `RuleEffect` types (`queueMessage`, `modifyCounter`, `drawCards`,
  `discardHand`) have no wound-gain effect. Adding a new `'gainWound'`
  type would require modifying `ruleHooks.types.ts` (WP-009A contract).
- **Root cause:** The WP described desired gameplay behavior without
  checking whether the existing effect type vocabulary could express it.
  The `RuleEffect` union was designed for WP-009B's MVP and was never
  expanded. The WP's "Non-Negotiable Constraints" said to use existing
  effect types, creating a tension between the stated behavior and the
  stated constraints.
- **Fix:** Pre-flight resolved the tension by extending the safe-skip
  pattern from condition evaluators (WP-023) to effect production:
  implement the handler structure using available effect types
  (`modifyCounter` + `queueMessage`) for tracking and observability,
  defer actual wound card movement to a future WP that adds the
  `'gainWound'` effect type. D-2403 logged.
- **Prevention:** Established Patterns now documents the "safe-skip for
  effect type gaps" pattern. Risk examples include "effect type
  vocabulary gap."

### Phase 5 continued — Board Keywords (WP-025)

**P5c-1: Effect type gap resolved by inline precedent instead of deferred
tracking (WP-025 pre-flight)**
- **What happened:** WP-025 specified `getAmbushEffects(): RuleEffect[]`
  applied via `applyRuleEffects` for the Ambush keyword's "each player
  gains 1 wound" behavior. Like WP-024, no `gainWound` RuleEffect type
  exists. However, unlike WP-024 (which deferred to counter-only
  tracking), the codebase already had functional inline wound-gain code:
  escape wounds in `villainDeck.reveal.ts` (lines 124-137) call
  `gainWound` directly.
- **Root cause:** The WP described the ideal architecture (pipeline-routed
  effects via RuleEffect) without checking whether the effect vocabulary
  could express it, AND without checking whether an inline precedent
  already existed for the same behavior.
- **Fix:** Pre-flight identified both the gap and the inline precedent.
  Resolution: use the inline `gainWound` pattern (same as escape wounds)
  rather than counter-only tracking. The WP's scope B and E were
  corrected — `getAmbushEffects(): RuleEffect[]` became
  `hasAmbush(): boolean` with inline wound logic. D-2503 logged.
- **Prevention:** Established Patterns now documents "inline behavior as
  effect type workaround." Risk examples include "effect type gap with
  existing inline precedent." Dependency Contract Verification refined
  to check for inline precedents when effect type gaps are found.
- **Key insight:** The D-2403 safe-skip pattern has two modes: (1) defer
  to counter-only tracking when no inline precedent exists (WP-024), or
  (2) use the inline precedent when one does (WP-025). Mode 2 delivers
  functional MVP behavior; mode 1 delivers tracking only.

**P5c-2: Setup-time data source missing for declared keywords (WP-025
pre-flight)**
- **What happened:** WP-025 declared three board keywords (Patrol,
  Ambush, Guard) with `buildCardKeywords` extracting keyword data from
  the registry. Pre-flight investigated the actual card JSON data across
  all 40 sets and found: Ambush appeared 304 times (identifiable by
  `"Ambush:"` prefix in ability text), Patrol appeared 4 times but as
  a completely different mechanic (`[keyword:Patrol the Bank]` in Secret
  Wars Vol 2 — named location patrols, not the generic fight-cost
  modifier WP-025 describes), and Guard had zero occurrences.
- **Root cause:** The WP's Assumes section said "Card data includes
  keyword information in ability text or structured metadata" without
  verifying this against actual data. No structured keyword metadata
  exists in the registry schema. Only Ambush is parseable from ability
  text. Patrol in the data is semantically different from the WP's
  concept. Guard simply doesn't exist.
- **Fix:** Pre-flight corrected the WP's Assumes section and scope C.
  `buildCardKeywords` extracts Ambush from ability text. Patrol and
  Guard produce empty results (safe-skip per D-2302). All three
  mechanics are fully implemented and tested with synthetic data.
  D-2504 logged.
- **Prevention:** Dependency Contract Verification now includes
  "setup-time data source verified for all extractors." Risk examples
  include "setup-time data source gap." Established Patterns now
  documents "setup-time data extraction with safe-skip for missing
  sources."

**P5c-3: WP function signature using direct registry type instead of
`unknown` convention (WP-025 pre-flight)**
- **What happened:** WP-025 scope C specified
  `buildCardKeywords(registry: CardRegistry, matchConfig: MatchSetupConfig)`.

  the game engine must not import. The established convention (used by

- **Root cause:** The WP author used the conceptual type name rather
  than the established convention. This is a variation of "locked value
  string literal drift" (D-2304) applied to function signatures.
- **Fix:** Pre-flight corrected the WP to use `registry: unknown` with
  `CardKeywordsRegistryReader` local interface per the `buildCardStats`
  pattern.
- **Prevention:** Dependency Contract Verification now includes
  "WP function signatures match established conventions." Risk examples
  include "WP function signatures using direct registry types."
