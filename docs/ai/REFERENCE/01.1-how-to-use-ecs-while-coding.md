## Purpose

Execution Checklists (ECs) are the **execution contract** for implementing
Work Packets (WPs). They convert design intent into binary, observable
requirements that prevent drift, reduce debugging time, and enforce
determinism.

**Rule:** No EC → no implementation.

---

## Authority Model

| Document | Authority |
|---|---|
| Work Packet (WP) | Design authority — what to build and why |
| Execution Checklist (EC) | Execution authority — what must be satisfied exactly |
| Code | Implementation — must satisfy both |

If behavior conflicts:
- WP wins over EC for **design intent**
- EC wins over code for **execution correctness**

---

## Standard Workflow

### 1. Before Writing Any Code

Confirm all of the following:

- [ ] A corresponding `EC-NNN` exists in `EC_INDEX.md`
- [ ] EC status is **Draft** or **Done** (not **Blocked**)
- [ ] All **Before Starting** items in the EC pass
- [ ] You have read the EC (first), then the full WP (second)

If any item fails, **stop**.

### 2. While Coding

- Implement **only** what appears in Files to Produce
- Treat **Locked Values** as immutable — never re-derive, rename, or reorder
- Enforce every item in **Guardrails**
- Add all `// why:` comments exactly where specified in **Required Comments**

If you find yourself wanting to add new fields, relax a rule, reorder logic,
or introduce helpers not in scope — stop and revisit the WP. Do not improvise.

### 3. After Coding (Before Declaring Done)

Every item in **After Completing** must be checked off:

- [ ] Build passes
- [ ] Tests pass
- [ ] No forbidden patterns (confirmed with verification commands from WP)
- [ ] Required docs updated (STATUS.md, DECISIONS.md)
- [ ] WORK_INDEX.md updated

If even one item fails, the WP is **not complete**.

## Debugging with ECs

When something breaks, do not start with logs. Instead:

1. Identify the relevant EC
2. Locate the violated clause:
   - Locked Value mismatch?
   - Guardrail violated?
   - Required file missing or modified outside scope?
3. If the EC has a **Common Failure Smells** section, check symptoms there
4. Fix the **specific clause violation**
5. Re-run build and tests
6. Re-check the EC

This turns debugging from "what happened?" into "which rule was violated?"

---

## Status Transitions

| From | To | Condition |
|---|---|---|
| Draft | Done | WP completed and validated per WORK_INDEX.md |
| Draft | Blocked | WP dependencies discovered to be unmet |
| Blocked | Draft | Dependencies resolved |

Only change EC status when WORK_INDEX.md reflects the corresponding change.

---

## Anti-Patterns

- Coding without reading the EC first
- Treating EC items as suggestions rather than gates
- Adding logic not covered by the EC scope
- Modifying an EC during coding to make code pass
- Debugging by intuition instead of tracing to EC clauses
- Re-deriving locked values instead of copying them

---

## Completion Rule

A WP is considered correctly implemented only when:

- The WP acceptance criteria pass, **and**
- The EC is fully satisfied without exceptions

Both are required. Neither alone is sufficient.
