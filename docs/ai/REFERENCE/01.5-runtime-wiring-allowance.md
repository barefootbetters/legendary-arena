## Purpose

Some Work Packets introduce or consume new **runtime-visible structure**
(e.g., required fields, exported moves, or phase wiring) on
`LegendaryGameState` or other shared contracts. When such structural
requirements change, files that construct, expose, or assert on those
structures may fail to compile or test unless they are updated — even if
those files are not listed in the WP's "Files Expected to Change" section.

This clause defines the **narrow, explicit allowance** under which files
outside a WP allowlist may be modified **solely for wiring purposes**.

This is an allowance, not a default right.

---

## When to Include This Clause

This clause applies **only when explicitly referenced** in a session
invocation prompt. It does **not** apply automatically.

Include a reference to this document in a session invocation prompt
(`docs/ai/invocations/session-wpXXX-*.md`) when the WP:

- Adds a **required field** to `LegendaryGameState` or another shared type
- Changes the **shape** of a return type produced by
  `buildInitialGameState` or similar setup orchestrators
- Adds a **new move** to `LegendaryGame.moves` that affects structural
  assertions in existing tests
- Adds a **new phase hook** that alters the expected structural shape of
  gameplay initialization or test scaffolding

If a WP is purely additive (new files only, no type or shape changes),
this clause does not apply.

---

## Runtime Wiring Allowance (Explicit)

This Work Packet introduces or consumes new runtime-visible fields.
As a result, limited edits to the following files may be required
*solely to satisfy type completeness or structural test correctness*:

- `buildInitialGameState.ts` — to provide the new field in the returned
  `LegendaryGameState` object
- Existing runtime tests affected by **structural** changes
  (e.g., move-count assertions, presence of required fields,
  phase-shape assertions — not behavioral sequencing or logic tests)
- Existing structural assertions that enumerate or assume fixed contract
  surface area (e.g., move list length, exported keys)

This allowance applies **only** to wiring changes required by the
**specific field(s) or structural contract introduced by the invoking WP**.
It does **not** generalize to unrelated wiring, cleanup, or refactoring.

**Reviewer shortcut:** if the change can be reverted by deleting the new
contract element (field, move, phase hook) with no remaining diff, it is
likely eligible under this clause.

---

### Allowed Modifications (All Must Apply)

Edits performed under this clause **must**:

- Be **minimal** — limited strictly to restoring type completeness or
  assertion validity, without reformatting, refactoring, or reorganizing
  surrounding code
- Be **strictly dependency-driven** — directly traceable to a new required
  field or structural contract introduced by the WP
- Be limited to **updating expected literal values** in existing assertions
  when the underlying contract surface has legitimately expanded
- Introduce **no new behavior** beyond wiring:
  - no logic
  - no branching
  - no new helpers
  - no new test cases in existing files

---

### Prohibited Modifications

Edits performed under this clause **must not**:

- Add gameplay logic, rules, effects, or side effects
- Introduce new exports, abstractions, or helpers
- Modify the behavioral contract of the affected file
- Touch files unrelated to the structural change
- Introduce new conditional logic or branching inside existing tests
- Modify integration tests introduced by the WP itself
  (this clause applies only to *existing* tests)

---

## Reporting Requirement

Any file modified outside the WP allowlist under this clause must be
**explicitly documented** in:

1. The **execution summary** at the end of the session
2. The **pre-commit review** (Runtime Boundary Check)

Each call-out must state:

- Which file was modified
- Why the modification was required
- What structural change was applied
- An explicit statement confirming that no new gameplay or runtime behavior
  was introduced as part of the wiring change

---

## Escalation

If a required modification cannot be classified as **minimal wiring**:

- **STOP** — do not force-fit the change
- Report the issue in the execution summary
- The WP must be **split**, clarified, or its allowlist revised

This clause **must be invoked in the session prompt**.
It may **not** be cited retroactively in execution summaries or
pre-commit reviews to justify undeclared changes.

---

## Standard Reference Block for Session Prompts

Use the following block **verbatim** when invoking this allowance:

```markdown
### Runtime Wiring Allowance

This WP adds `<field>` to `<type>`. Per
`docs/ai/REFERENCE/01.5-runtime-wiring-allowance.md`, minimal wiring
edits to the following files are permitted solely to restore type and
assertion correctness:

- buildInitialGameState.ts
- <explicitly list affected existing tests>

No new behavior may be introduced.
```

Replace `<field>`, `<type>`, and the file list with the actual values.

## Precedent

This clause was formalized after **WP-007B**, which added
`currentStage: TurnStage` to `LegendaryGameState`.

That WP required minimal, dependency-driven updates to:

- `buildInitialGameState.ts` — adding `currentStage` to the return object
- `game.test.ts` — updating a move-count assertion after introducing
  `advanceStage`

Both edits were minimal, structural, and introduced no new behavior.

This clause was also exercised in **WP-008B**, which added a new core move
(`drawCards`) to `LegendaryGame.moves`. This required a minimal update to
an existing structural assertion in `game.test.ts` (move count increased
from 3 to 4). The update changed assertion values only and introduced no
new logic or behavior.

Together, WP-007B (required field addition) and WP-008B (contract surface
expansion) define the two primary scenarios this clause addresses.

This clause ensures future WPs can accommodate similar evolution
without ambiguity or scope drift.

### Final Note

This document exists to **enable safe progress without eroding scope
discipline**. When in doubt, escalation is always correct.
