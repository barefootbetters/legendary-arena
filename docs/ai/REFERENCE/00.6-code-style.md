## Purpose

This project is maintained by a small team that includes contributors who are
learning web development. Every piece of code Claude writes must be readable
and modifiable by a junior developer — someone with 1–2 years of experience
who can read JavaScript confidently but is not yet fluent in advanced patterns.

The goal is **human-style code**: explicit, boring, and obviously correct.
Clever code is a liability. When in doubt, write it out.

---

## Rule 1 — Duplication Over Abstraction

**Duplicate first. Abstract only when a third copy appears.**

If two places need similar logic, copy it. Do not create a shared helper until
you have three or more identical uses. A wrong abstraction is harder to maintain
than a copy-paste.

```js
// BAD — abstracted too early (only used twice)
function validateSlug(slug, context) {
  if (!slug || typeof slug !== 'string') throw new Error(`[${context}] slug is required`);
}
validateSlug(mastermind.slug, 'mastermind');
validateSlug(scheme.slug, 'scheme');

// GOOD — explicit, obvious, junior-readable
if (!mastermind.slug || typeof mastermind.slug !== 'string') {
  throw new Error('mastermind.slug is required and must be a non-empty string.');
}
if (!scheme.slug || typeof scheme.slug !== 'string') {
  throw new Error('scheme.slug is required and must be a non-empty string.');
}
```

Enforced by: **00.3 §16.1**

## Rule 2 — No Factory Functions for One-Time Setup

Factory functions (functions that return objects or other functions) hide
what is being created. If an object is only built in one place, build it inline.

```js
// BAD — factory hides what the config actually is
function makePoolConfig(url) {
  return { connectionString: url, max: 10, idleTimeoutMillis: 30000 };
}
const pool = new Pool(makePoolConfig(process.env.DATABASE_URL));

// GOOD — the shape is right there, obvious at a glance
const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  max: 10,
  idleTimeoutMillis: 30000,
});
```

Enforced by: **00.3 §16.1**

---

## Rule 3 — No Nested Ternaries

A single ternary for a simple true/false is fine. Nested or chained ternaries
are forbidden. Use `if/else if/else` blocks instead.

```js
// BAD
const status = isActive ? 'active' : isWaiting ? 'waiting' : 'complete';

// GOOD
let status;
if (isActive) {
  status = 'active';
} else if (isWaiting) {
  status = 'waiting';
} else {
  status = 'complete';
}
```

This applies everywhere — including exit code logic, default value selection,
and conditional string building.

Enforced by: **00.3 §16.2**

## Rule 4 — Readable Variable Names

Full English words only. No abbreviations, no single letters (except `i` in a
classic `for` loop where the index is only used once, directly).

**Forbidden abbreviations and their correct replacements:**

| Forbidden | Use instead |
|---|---|
| `abbr` | `setAbbreviation` |
| `cfg` | `config` |
| `vg` | `villainGroup` |
| `mm` | `mastermind` |
| `sch` | `scheme` |
| `pg` | `pool` or `dbPool` |
| `ctx` (shortened) | `gameContext` if clarity is needed |
| `res`, `req` | `response`, `request` |
| `e` | `error` |
| `cb` | `callback` |
| `fn` | `handler` or a descriptive name |
| `msg` | `message` or `errorMessage` |
| `ver` | `versionString` |
| `fix` | `remediationMessage` |

**Note on boardgame.io context:** `G` and `ctx` are boardgame.io's own parameter
names inside move functions. Do not rename them there — the framework requires
these exact names. Outside of move functions, use descriptive names.

**Boolean names** must start with `is`, `has`, or `can`:
- `isConnected`, `hasAlwaysLeadsGroups`, `canFight`, `isPlaceholderValue`

**Loop variables** must be descriptive:
```js
// BAD
for (const v of villainGroups) { ... }

// GOOD
for (const villainGroup of villainGroups) { ... }
```

Enforced by: **00.3 §16.3**

## Rule 5 — Small Functions (One Screen Rule)

Each function should fit on one screen — roughly 20–30 lines of code,
not counting JSDoc. If a function is longer, break it into named sub-functions.

Named sub-functions must have descriptive names that read like a sentence:
- `buildVillainDeckFromGroups(villainGroups, scheme, playerCount)`
- `validateMastermindsSlug(mastermind)`
- `insertGameSessionRow(client, matchID, joinCode, playerCount)`

No anonymous functions assigned to variables unless they are event handlers
or callbacks that are immediately obvious in purpose.

Every function must have a JSDoc comment:
```js
/**
 * Fetches and parses the per-set metadata JSON from R2.
 * @param {string} setAbbreviation - The 4-char set abbreviation (e.g. "mdns").
 * @returns {Promise<object>} The parsed metadata object.
 * @throws {Error} If the fetch fails or the response is not valid JSON.
 */
async function fetchSetMetadata(setAbbreviation) { ... }
```

Enforced by: **00.3 §16.4**

## Rule 6 — Comments Explain WHY, Not WHAT

Code explains what it does. Comments explain why a decision was made, why an
edge case exists, or why a less-obvious approach was chosen.

```js
// BAD — restates what the code obviously does
// Increment counter
errorCount++;

// GOOD — explains a non-obvious decision
// why: R2 rate-limits aggressive parallel fetches. Sequential processing with
// a 50ms pause keeps us well under the limit even for 40+ sets.
for (const setAbbreviation of setAbbreviations) {
  await fetchAndValidateSet(setAbbreviation);
  await sleep(DELAY_BETWEEN_SETS_MS);
}
```

**When `// why:` comments are required** (not optional):
- Any constant whose value is not self-evident (`DELAY_BETWEEN_SETS_MS = 50`)
- Any catch block that swallows an error instead of re-throwing
- Any `HEAD` request where a `GET` might seem more natural
- Any place where a CJS path is checked in an ESM project
- Any `process.env.APPDATA` usage (Windows-specific path resolution)
- Any game move that uses `ctx.random.*` instead of `Math.random()`

Enforced by: **00.3 §16.5**

## Rule 7 — No Dynamic Property Access for Known Keys

If you know the property names ahead of time, write them out. Do not use
bracket notation to access known properties dynamically.

```js
// BAD — hides which fields are being accessed; not greppable
const fields = ['slug', 'vp', 'alwaysLeads'];
const extracted = {};
for (const field of fields) {
  extracted[field] = mastermind[field];
}

// GOOD — explicit, greppable, shape is obvious at a glance
const extracted = {
  slug: mastermind.slug,
  vp: mastermind.vp,
  alwaysLeads: mastermind.alwaysLeads,
};
```

Enforced by: **00.3 §16.2**

## Rule 8 — No `Array.reduce()` for Multi-Step Operations

`reduce()` is acceptable for simple accumulation (summing numbers, joining
strings). For anything with branching logic, use an explicit `for...of` loop
with a descriptive variable and intermediate results.

```js
// BAD — branching logic inside reduce is hard to read and hard to debug
const groupMap = villainGroups.reduce((acc, group) => {
  if (group.slug) acc[group.slug] = group;
  return acc;
}, {});

// GOOD — a junior can add a console.log or breakpoint anywhere in this loop
const groupMap = {};
for (const villainGroup of villainGroups) {
  if (villainGroup.slug) {
    groupMap[villainGroup.slug] = villainGroup;
  }
}
```

Enforced by: **00.3 §16.2**

## Rule 9 — Explicit Imports

No barrel re-exports. No `import * as`. Import exactly what you use by name.

```js
// BAD
import * as lobbyUtils from './lobbyUtils.mjs';
lobbyUtils.generateJoinCode();

// GOOD
import { generateJoinCode } from './lobby/joinCode.mjs';
generateJoinCode();
```

If a module exports many things and you need most of them, that is a sign
the module is too large — split it up.

Enforced by: **00.3 §16.6**

---

## Rule 10 — No Higher-Order Functions Unless the Framework Requires Them

A function that returns another function obscures what is happening. Avoid
currying, closures-as-configuration, and middleware factories unless the
surrounding framework (boardgame.io, Express) explicitly requires them.

```js
// BAD — returns a function, hides the real logic behind a closure
function makeVillainPhaseHandler(rules) {
  return (G, ctx) => {
    // villain logic using rules
  };
}

// GOOD — pass rules as a parameter, keep the shape flat and readable
function runVillainPhase(G, ctx, rules) {
  // villain logic using rules
}
```

Enforced by: **00.3 §16.1**

## Rule 11 — Error Messages Are Full Sentences

Error messages must tell the reader what went wrong and, where possible,
what to do about it. Single-word or terse error messages are forbidden.

```js
// BAD
throw new Error('missing field');
throw new Error('invalid');

// GOOD
throw new Error(
  `Mastermind "${mastermind.slug}" is missing the required "vp" field. ` +
  'Check the R2 metadata JSON for this set and re-run the seed script.'
);
```

Every error message must include:
1. What failed (the entity name and field, or the operation that failed)
2. What to check or do (where possible)

Enforced by: **00.3 §16.7**

## Rule 12 — Flat File Structure

Avoid deeply nested folder hierarchies. New modules go in the most obvious
existing folder. Do not create a new sub-folder for a single file.

Acceptable: `src/game/deckBuilder.mjs`, `src/lobby/routes.mjs`
Not acceptable: `src/game/deck/builder/core/deckBuilder.mjs`

Not enforced by 00.3 (structural decision — enforce at code review).

## Rule 13 — ESM Only, No CommonJS

This project is ESM-only. Every generated JavaScript file must use ES module
syntax. CommonJS `require()` is forbidden.

```js
// BAD — CommonJS
const { readFile } = require('fs');
const pg = require('pg');

// GOOD — ESM
import { readFile } from 'node:fs/promises';
import pg from 'pg';
```

**Use the `node:` prefix for all Node.js built-in module imports.** This
makes it immediately clear that a module is a Node built-in, not an npm package:

```js
// BAD — ambiguous: is 'fs' a built-in or a package named 'fs'?
import { readFile } from 'fs';
import { execSync } from 'child_process';

// GOOD — unambiguous: the node: prefix marks these as built-ins
import { readFile } from 'node:fs/promises';
import { execSync } from 'node:child_process';
import { randomBytes } from 'node:crypto';
import { join } from 'node:path';
```

**File extensions** — use `.mjs` for new standalone scripts. For packages with
`"type": "module"` in `package.json`, `.js` is acceptable. Never `.cjs`.

**Generating secret values** — use the ESM-compatible form:
```js
// BAD — CommonJS in an ESM project
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

// GOOD — ESM-compatible
node --input-type=module -e "import { randomBytes } from 'node:crypto'; console.log(randomBytes(32).toString('hex'))"
```

Not currently in 00.3 as a standalone check — this is enforced via the contract
block (ESM only, Node v22+) in every Work Packet.

## Rule 14 — Field Names Must Match the Data Contract

All field names used in generated code must match `00.2-data-requirements.md`
exactly. Do not rename, abbreviate, or "improve" field names from the canonical
data contract.

```js
// BAD — renamed field not in 00.2
const mastermindsVictoryPoints = mastermind.victoryPoints; // field is "vp"
const leadGroups = mastermind.leadsGroups;                 // field is "alwaysLeads"

// GOOD — exact field names from 00.2 §1.3
const victoryPoints = mastermind.vp;
const alwaysLeadsGroups = mastermind.alwaysLeads;
```

**This rule applies to:**
- Database column names (must match 00.2 §4)
- Setup payload field names (must match 00.2 §8.1 exactly)
- API request/response shapes
- Any object destructuring from card JSON

**Setup payload field names are locked** (from 00.2 §8.1):
`schemeId`, `mastermindId`, `villainGroupIds`, `henchmanGroupIds`,
`heroDeckIds`, `bystandersCount`, `woundsCount`, `officersCount`,
`sidekicksCount`

Do not rename these in any Work Packet, even if a different name seems cleaner.

**If a field name in 00.2 seems wrong**, do not silently rename it in code.
Instead: stop, raise it as a question, and if a change is agreed upon, update
00.2 first with a DECISIONS.md entry, then update the code.

Not currently in 00.3 as a standalone check — enforced via 00.3 §6
(Naming Consistency) and 00.2 override authority.

## Rule 15 — Async Error Handling Is Always Explicit

Every `async` function that performs I/O (network, database, file system) must
handle errors explicitly. Do not let errors bubble up silently.

```js
// BAD — unhandled rejection if fetch fails
async function loadRegistryConfig() {
  const response = await fetch(`${R2_BASE_URL}/metadata/sets.json`);
  return response.json();
}

// GOOD — explicit error handling with a full-sentence message
async function loadRegistryConfig() {
  let response;
  try {
    response = await fetch(`${R2_BASE_URL}/metadata/sets.json`);
  } catch (error) {
    throw new Error(
      `Failed to fetch metadata/sets.json from R2. ` +
      `Network error: ${error.message}. ` +
      'Check that R2_PUBLIC_URL is set correctly and R2 is reachable.'
    );
  }
  if (!response.ok) {
    throw new Error(
      `metadata/sets.json returned HTTP ${response.status}. ` +
      'The file may be missing from R2 or the bucket may not be public.'
    );
  }
  return response.json();
}
```

**Never swallow errors silently** in a catch block without at minimum logging
a warning. If an error is intentionally ignored, a `// why:` comment is required
explaining why it is safe to do so.

Not enforced by 00.3 as a standalone rule — covered by Rule 11 (full-sentence
error messages) and Rule 6 (`// why:` on catch blocks that swallow errors).

## Enforcement

These rules are checked by **00.3 §16** of the Prompt Lint Checklist.
The mapping is:

| 00.3 section | Rules enforced |
|---|---|
| §16.1 — No premature abstraction | Rules 1, 2, 10 |
| §16.2 — Explicit control flow | Rules 3, 7, 8 |
| §16.3 — Readable names | Rule 4 |
| §16.4 — Small functions | Rule 5 |
| §16.5 — Comments explain WHY | Rule 6 |
| §16.6 — No magic imports | Rule 9 |
| §16.7 — Error messages are full sentences | Rules 11, 15 |
| Not in §16 (structural) | Rules 12, 13, 14 |

Rules 12, 13, and 14 are enforced through the Work Packet contract block
(ESM only, Node v22+), the 00.2 override authority, and 00.3 §6
(Naming Consistency) respectively.

**When asking Claude to fix a violation, say:**
> "This violates Code Style Rule N from 00.6. Please rewrite [function/block]
> to follow that rule."
