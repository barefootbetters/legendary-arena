### BLOCKING CONTRACT FINDINGS

**Finding 1: Condition type name drift**

The WP-023 locked condition types table specifies:

| WP-023 name | Actual code (heroAbility.setup.ts) |
|---|---|
| `requiresColor` | `heroClassMatch` |
| `requiresTeam` | *not generated by any setup code* |
| `requiresKeyword` | *not generated by any setup code* |
| `playedThisTurn` | *not generated by any setup code* |

The setup code (`heroAbility.setup.ts`, line 122-126) produces conditions
with `type: 'heroClassMatch'`, not `type: 'requiresColor'`. The WP-022
test file (`heroEffects.execute.test.ts`, line 214) also uses
`'heroClassMatch'`.

The WP's condition type names are **aspirational labels** that do not match
the condition type strings actually produced by the existing setup code.
The evaluator must match the strings that exist in `G.heroAbilityHooks`
at runtime — otherwise no condition will ever evaluate to `true`.

**Resolution required:** Either:
- (A) Update WP-023 locked values table to use `heroClassMatch` (the actual
  string), and acknowledge the other 3 types are not yet produced by setup
  code, OR
- (B) Add condition-type generation for the other 3 types to setup code —
  but this is out of WP-023 scope (setup code is not in Files Expected to
  Change) and requires WP-level authorization

**Finding 2: Team and hero class data not available in G for condition evaluation**

The WP says:
- `requiresTeam` evaluates using `G.playerZones[player].inPlay` + "team data"
- `requiresColor` evaluates using `G.playerZones[player].inPlay` + "`hc` field"

However, the data needed to resolve team/class for cards in `inPlay` does
**not exist in G**:

- `G.cardStats` (`CardStatEntry`) has only: `attack`, `recruit`, `cost`,
  `fightCost` — **no team, no heroClass** (`economy.types.ts`, lines 36-49)
- `G.heroAbilityHooks` (`HeroAbilityHook`) has: `cardId`, `timing`,
  `keywords`, `conditions?`, `effects?` — **no team, no heroClass**
  (`heroAbility.types.ts`, lines 31-49)
- `HeroAbilityFlatCard` (setup-time reader) has: `key`, `cardType`,
  `setAbbr`, `abilities` — **no team, no heroClass**
  (`heroAbility.setup.ts`, lines 33-42)

The registry schema **does** have this data:
- `hc` (hero class) is a per-card field on `HeroCardSchema` (schema.ts:117)
- `team` is a per-hero-group field on `HeroGroupSchema` (schema.ts:134)

But this data is never resolved into G at setup time. To evaluate
`requiresColor`/`requiresTeam` conditions, the evaluator would need a
lookup like `G.cardTeams: Record<CardExtId, string>` and
`G.cardHeroClasses: Record<CardExtId, string>` — but:

1. These fields don't exist in `LegendaryGameState`
2. Adding them requires modifying `types.ts` and setup code — neither is
   in WP-023's "Files Expected to Change"
3. `economy.types.ts` (`CardStatEntry`) is also not in scope

**This is a design gap, not a code gap.** The WP describes behavior that
requires data infrastructure that hasn't been built yet.

**Resolution required:** Either:
- (A) Add a prerequisite WP (or expand WP-023 scope) to resolve team/class
  data into G at setup time (e.g., extend `CardStatEntry` or add new G fields)
- (B) Narrow WP-023 to only implement condition types that CAN be evaluated
  with existing G data:
  - `heroClassMatch` — **BLOCKED** (needs class data lookup)
  - `requiresKeyword` — **FEASIBLE** via `G.heroAbilityHooks` keyword lookup
  - `playedThisTurn` — **FEASIBLE** via `G.playerZones[player].inPlay.length`
  - `requiresTeam` — **BLOCKED** (needs team data lookup)

### Risk & Ambiguity Review

**Risk 1: Condition type name drift (BLOCKING)**

- **Risk:** WP-023 specifies `requiresColor` but actual code produces
  `heroClassMatch`. Evaluator built against WP names would never match
  actual condition data.
- **Impact:** HIGH — condition evaluation would silently fail on all
  hero-class conditions
- **Mitigation:** WP must be updated to use actual condition type string
  `heroClassMatch`, or setup code must be updated (scope violation)
- **Decision:** Update WP-023 locked values to use `heroClassMatch`

**Risk 2: Missing team/class data in G (BLOCKING)**

- **Risk:** Two of four condition types (`requiresTeam`, `requiresColor`/
  `heroClassMatch`) cannot be evaluated because the data they need (team
  name, hero class per card) is not in G.
- **Impact:** HIGH — 50% of the WP's MVP condition types are
  unimplementable with current data
- **Mitigation options:**
  - (A) Add a prerequisite "data expansion" WP to extend `CardStatEntry`
    or add new G fields with team/class data
  - (B) Narrow WP-023 to implement only `playedThisTurn` (feasible) and
    `requiresKeyword` (feasible via `G.heroAbilityHooks`), deferring
    `heroClassMatch` and `requiresTeam` until data is available
  - (C) Expand WP-023 scope to include adding team/class fields to G and
    modifying setup code — but this changes "Files Expected to Change"
    significantly
- **Decision:** BLOCKED — requires human decision on approach

**Risk 3: Only one condition type is currently generated by setup code**

- **Risk:** Only `heroClassMatch` conditions are generated by
  `heroAbility.setup.ts`. The other 3 types (`requiresTeam`,
  `requiresKeyword`, `playedThisTurn`) have no setup-time generation code.
  Even if evaluators are implemented, no real game data would exercise
  3 of 4 condition types.
- **Impact:** MEDIUM — evaluators would work but only be testable via
  hand-crafted mocks, not real game state
- **Mitigation:** Accept for MVP — evaluators are correct by construction
  and tested with synthetic data. Future WPs will add setup-time parsing
  for additional condition types as card data supports them.
- **Decision:** Acceptable — does not block execution

**Risk 4: `HeroCondition.value` is always `string` but `playedThisTurn`
needs a number**

- **Risk:** `HeroCondition.type: string; value: string` — the `value` is
  always a string. For `playedThisTurn`, the evaluator needs to parse
  `value` as a number (`parseInt`). This is fragile but workable.
- **Impact:** LOW — parsing is straightforward and testable
- **Mitigation:** Evaluator parses `value` to number for `playedThisTurn`;
  returns `false` if parse fails. Document in `// why:` comment.
- **Decision:** Acceptable pattern — consistent with Legendary data
  where numeric values are sometimes stored as strings
