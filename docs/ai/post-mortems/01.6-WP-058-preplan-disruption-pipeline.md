## Section 1 — Overview

WP-058 lands the disruption pipeline that closes the pre-planning layer's
detect → invalidate → rewind → notify workflow. Four new source files
(`disruption.types.ts`, `disruptionDetection.ts`, `disruptionPipeline.ts`,
`preplanEffectTypes.ts`) introduce five new public runtime functions, four
new public types, a canonical readonly effect-type array, and a compile-
time drift-check proof locked to `PrePlan.invalidationReason.effectType`.
The public surface is appended to `packages/preplan/src/index.ts` below
the WP-056 + WP-057 blocks; both prior blocks are unchanged verbatim. The
WP-056 `PrePlan` / `RevealRecord` contracts and the WP-057 sandbox +
speculative-operation implementations are consumed read-only with zero
modification.

The pipeline is a pure integration-adapter boundary: detection is binary
per-player (DESIGN-CONSTRAINT #4), invalidation returns a full-spread
fresh `PrePlan` with `status: 'invalidated'` without incrementing
`revision`, source restoration is derived exclusively from
`revealLedger` (DESIGN-CONSTRAINT #3 — the reveal ledger is the sole
authority for rewind), and the notification carries literal-typed
`requiresImmediateNotification: true` that encodes Constraint #7 at the
type level. The programming-error throw in `buildDisruptionNotification`
is the sole throw in the package.

All four 01.6 triggers fire:

1. **New long-lived abstractions.** `isPrePlanDisrupted`,
   `invalidatePrePlan`, `computeSourceRestoration`,
   `buildDisruptionNotification`, `executeDisruptionPipeline`,
   `PREPLAN_EFFECT_TYPES` + derived `PrePlanEffectType` +
   `PlayerAffectingMutation` / `DisruptionNotification` /
   `SourceRestoration` / `DisruptionPipelineResult` — the canonical
   disruption surface for the lifetime of the project.
2. **First runtime consumer of `invalidationReason.effectType` closed
   union.** `PREPLAN_EFFECT_TYPES` canonicalizes the array side of the
   union, deferred from WP-056 per `preplan.types.ts:101-106` JSDoc; the
   `NonNullable<>`-wrapped compile-time drift-check and the runtime
   set-equality test lock parity.
3. **First implementation of DESIGN-CONSTRAINT #3 "reveal ledger is sole
   rewind authority."** `computeSourceRestoration` reads only
   `revealLedger`; the ledger-sole test (Copilot Issue 11 FIX) enforces
   this mechanically by constructing a plan whose sandbox disagrees with
   the ledger and asserting restoration follows the ledger.
4. **First full-spread 42/42 pattern applied to a status-transition
   operation.** `invalidatePrePlan` changes `status` and
   `invalidationReason` — not `sandboxState` / `revealLedger` /
   `planSteps` — yet still returns fresh copies for all three. WP-057
   applied the pattern to sandbox-mutation operations; WP-058 is the
   first case where the pattern is applied to a field-set the operation
   does not semantically change.

## Section 5 — Representation & Determinism Audit

- All four new types (`PlayerAffectingMutation`, `DisruptionNotification`,
  `SourceRestoration`, `DisruptionPipelineResult`) are JSON-serializable
  by construction: plain object shapes with `string` / `CardExtId`
  (alias for `string`) / `Record<string, CardExtId[]>` / literal-true
  fields only. Zero functions, closures, Maps, Sets, Dates, or classes.
- `PREPLAN_EFFECT_TYPES` is a frozen-by-type-system `as const` readonly
  tuple; the derived `PrePlanEffectType` union is a closed set.
- Five runtime functions are all pure: given the same inputs they
  produce the same outputs byte-for-byte. No module-level state, no
  memoization, no caching.
- `isPrePlanDisrupted`, `invalidatePrePlan`, `computeSourceRestoration`,
  `executeDisruptionPipeline` all return `null` / `false` on expected
  failure paths (null-on-inactive convention from WP-057 RS-8 extended
  to the detection layer and the pipeline layer). Only
  `buildDisruptionNotification` throws, and only on programming-error
  status mismatch.
- Unknown `RevealRecord.source` values are handled by the open-union
  fallback path in `computeSourceRestoration` — `record.source !==
  'player-deck'` drops into the lazily-initialized
  `sharedSourceReturns[record.source]` bucket. No throw, no warning,
  no special case.

## Section 7 — Invariant Enforcement

### Null-on-inactive uniform convention (inherited from WP-057 RS-8)

- `isPrePlanDisrupted` returns `false` on `prePlan === null` OR `status
  !== 'active'`. Verified by detection tests 3, 4, 5.
- `invalidatePrePlan` returns `null` on `status !== 'active'`. Verified
  by pipeline Test 4 (both `'invalidated'` and `'consumed'` branches).
- `executeDisruptionPipeline` returns `null` when `invalidatePrePlan`
  returns `null`. Verified by pipeline Test 18 (both non-active
  branches).
- `computeSourceRestoration` never fails — empty ledger produces empty
  buckets. Verified by Test 6.

### Programming-error throw — sole exception

`buildDisruptionNotification` throws exactly when `invalidatedPlan.status
!== 'invalidated'` (verbatim template per WP-058 §F lines 400-403).
Verified by Test 16 (both `'active'` and `'consumed'` branches, both
matching the `/expected 'invalidated'/` error pattern). No other
function in this package throws.

### Revision non-increment in `invalidatePrePlan`

`PrePlan.revision` is NOT bumped on invalidation per
`preplan.types.ts:36-38` — status and `invalidationReason` are not
revision-bumping fields. The `...prePlan` top-level spread carries
`revision` through unchanged. Verified by Test 5
(`result.revision === originalRevision`) and by grep gate 19
(`revision: prePlan.revision +` → zero hits in `disruptionPipeline.ts`).

### Ledger-sole rewind (DESIGN-CONSTRAINT #3)

`computeSourceRestoration` reads only `revealLedger`. Verified
mechanically by Test 11, which constructs a plan whose `sandboxState`
disagrees with the ledger and asserts the restoration follows the
ledger. The implementing `for (const record of revealLedger)` loop at
`disruptionPipeline.ts:114` is the only input source.

### Source partitioning — open-union fallback

`RevealRecord.source` is an open union (`'player-deck' | 'officer-stack'
| 'sidekick-stack' | 'hq' | string`). Non-`'player-deck'` values drop
into `sharedSourceReturns[record.source]`, lazily initialized to `[]` on
first encounter. Iteration order preserves ledger insertion order within
each bucket. Verified by Tests 8, 9, 10.

### `requiresImmediateNotification: true` literal-type encoding

`DisruptionPipelineResult.requiresImmediateNotification` is typed as
literal `true` (not `boolean`) per Copilot Issue 15 FIX. The JSDoc +
`// why:` at `disruption.types.ts:118-125` explains that widening to
`boolean` would delete the type-level Constraint #7 encoding. Verified
at runtime by pipeline Test 17
(`result.requiresImmediateNotification === true`).

### `PREPLAN_EFFECT_TYPES` drift check

Compile-time proof at `preplanEffectTypes.ts:34-41` uses
`NonNullable<PrePlan['invalidationReason']>['effectType']` — the
`NonNullable<>` wrapper is mandatory because `invalidationReason` is
optional on `PrePlan`. Without it, the indexed type is `undefined`-
contaminated and the check would assert the wrong set. Runtime set-
equality test at `preplanEffectTypes.test.ts` locks the array order and
values (`['discard', 'ko', 'gain', 'other']`).

### First-mutation-wins

Multiple mutations targeting the same player resolve via the status
guard: the first call returns a result (moving status to
`'invalidated'`), the second call receives a plan whose status is
already `'invalidated'` and short-circuits to `null`. Zero caller dedup
logic required. Verified by pipeline Test 22.

Follow-ups for future WPs (non-blocking, not part of WP-058 scope):

- **Integration-adapter boundary.** WP-058 produces instructions
  (`SourceRestoration`) — it does not execute returns against
  authoritative engine state. The integration layer that executes
  returns is out of scope for the pre-planning package and will be
  addressed by a future WP (candidate: a client-controller or adapter
  layer). The boundary is locked by `SourceRestoration` being plain
  JSON-serializable data and by the pipeline producing no side effects.
- **`PrePlanStep.isValid` per-step invalidation.** WP-058 does not flip
  individual `planSteps[i].isValid` flags on invalidation — the whole
  plan transitions to `'invalidated'` and is discarded. Per-step
  invalidation (a richer UI affordance that would highlight which steps
  survived) is deferred and intentionally out of scope here.
- **Multi-plan coordination.** WP-058 operates on one pre-plan at a
  time. A future coordination layer that dispatches disruption events
  to multiple waiting players' plans simultaneously is the natural
  next layer up from this pipeline.
