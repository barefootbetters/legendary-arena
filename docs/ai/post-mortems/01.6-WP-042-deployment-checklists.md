### Reality-reconciliation findings (authored into checklist prose)

Three drifts between the WP-042 / EC-042 / session-prompt locked
values and the reality on disk were discovered during execution.
Each was resolved by documenting reality accurately in the checklist
body rather than by documenting the wrong locked value:

1. **Environment variable list for `validate.ts` — `SETS_DIR`, not
   `CARDS_DIR`.** The session prompt's RS-3 locked list named
   `CARDS_DIR` as the local-mode env var for the card JSON path.
   The actual script at `packages/registry/scripts/validate.ts`
   reads `SETS_DIR` (default: `data/cards`). Two additional env vars
   the script reads — `METADATA_DIR` and `HEALTH_OUT` — were omitted
   from the RS-3 list entirely. The R2 checklist §A.1 documents
   reality: a full six-variable table (`METADATA_DIR`, `SETS_DIR`,
   `HEALTH_OUT`, `R2_BASE_URL`, `SKIP_IMAGES`, `IMAGE_DELAY_MS`).
   Using the incorrect `CARDS_DIR` would have made the checklist's
   own §A.1 invocation commands fail the checklist's pass criteria,
   since the script would not pick up the variable. Same discipline
   as PS-5 applied at pre-flight for migration filenames: document
   what ships, not what the paper spec assumed would ship.

2. **`EXPECTED_DB_NAME` is not read by `scripts/migrate.mjs`.** The
   session prompt's RS-3 list and EC-042 §Locked Values both named
   `EXPECTED_DB_NAME` as a pre-condition env var. The migrate runner
   only reads `DATABASE_URL`. The PostgreSQL checklist §B.1 resolves
   the discrepancy by documenting an **operator-level** eye-check
   (parse the database name out of `DATABASE_URL` and compare against
   the environment's documented database name) rather than
   inventing a non-functional env var contract. The operator-level
   check accomplishes the same intent — catch a cross-environment
   connection-string leak — without drifting from the runner's
   actual env var surface.

3. **Session prompt §B.7 mention of `legendary.card_types` /
   `legendary.cards` / slug indexes.** The session prompt's Goal
   item 1 and §Files Expected to Change indirectly referenced
   verification targets drawn from the pre-amendment five-migration
   draft (Foundation Prompt 03 scope). Those tables and slug columns
   do not exist in the three shipped migrations (`001_server_schema`
   creates `legendary.source_files` / `sets` / `masterminds` /
   `villain_groups` / `schemes` / `rules` / `rule_docs`; no
   `card_types`, no `cards`, no `slug` columns anywhere). §B.7 of
   the produced checklist verifies what actually ships, and includes
   a block-quote note at §B.7.5 that explains why the slug-index
   expectation is intentionally not checked at this scope and flags
   it as WP-042.1 work. Verification Step 6 confirms the
   pre-amendment filenames (`001_initial_schema`, `002_seed_base_set`,
   `003_add_game_sessions`, `004_upsert_indexes`, `005_lobby_columns`)
   do not appear anywhere in the checklist prose.

None of these three findings is a scope violation — each is a
documentation-accuracy correction that preserves the D-4203
documentation-only invariant while ensuring the produced checklists
actually match the code they document. The post-mortem flags them
for the lessons-learned cycle (step 7 in the workflow) so future
pre-flights can add a "cross-check env-var list against actual
script source" pattern alongside the P6-46 grep-archaeology pattern
introduced at WP-035.
