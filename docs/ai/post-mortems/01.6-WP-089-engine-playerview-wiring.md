## Section 13 — Findings

### Finding 13.1 — RS-3 locked cast failed under `exactOptionalPropertyTypes: true`; authorized refinement to `NonNullable<Game<...>['playerView']>`

**Surface:** `packages/game-engine/src/game.ts` `playerView` field assignment.

**Symptom:** Session prompt + EC-089 + WP-089 RS-3 §Local Type Assertion Lock mandated exactly one cast form:

```ts
playerView: buildPlayerView as unknown as Game<LegendaryGameState>['playerView'],
```

Applying that cast produced TS2375:

```
src/game.ts(129,14): error TS2375: Type '{ ...; playerView: ((context: {...}) => any) | undefined; ... }' is
not assignable to type 'Game<LegendaryGameState, Record<string, unknown>, MatchSetupConfig>' with
'exactOptionalPropertyTypes: true'. Consider adding 'undefined' to the types of the target's properties.
```

**Root cause:** `Game<G>['playerView']` is an indexed-access type into an **optional** property, so it resolves to `((context) => any) | undefined`. The `as unknown as` double-cast carries the `| undefined` onto the object-literal field. Under `exactOptionalPropertyTypes: true` (enabled in `packages/game-engine/tsconfig.json`), an explicit `undefined` cannot be assigned to a field declared with the `?:` marker — only omission is permitted. The RS-3 cast was designed against the looser `exactOptionalPropertyTypes: false` semantics.

**Secondary signal:** boardgame.io 0.50.2's `Game<G>['playerView']` is declared as `(context: { G, ctx, playerID }) => any` — a **single-context-object** signature. The RS-3 / EC-089 / WP-089 three-positional-argument signature `(gameState, ctx, playerID) => UIState` is type-compatible only via a double-cast — and, critically, is **wrong at runtime**: boardgame.io invokes `playerView({ G, ctx, playerID })`, so a three-arg function would receive the whole context object as its first arg and `undefined` for the other two. The cast would have hidden this runtime bug behind a TS-only contract.

**Detection:** Typecheck via `pnpm --filter @legendary-arena/game-engine exec tsc --noEmit` immediately after applying the RS-3 cast verbatim. User raised the RS-3 "STOP and ask the human" rule; explicit user authorization to refine was requested and granted.

**Resolution (user-authorized):**
1. `buildPlayerView` signature changed to match boardgame.io's actual single-context-object shape: `({ G, ctx, playerID }: { G: LegendaryGameState; ctx: Ctx; playerID: PlayerID | null }) => UIState`.
2. Field assignment uses the narrowest `NonNullable<Game<LegendaryGameState>['playerView']>` refinement — preserves RS-3's intent ("anchor to boardgame.io's `Game<...>.playerView` property type; do NOT modify the `Game<...>` generic") while stripping the `| undefined` that poisons the assignment.
3. Test file calls use the matching single-object form: `LegendaryGame.playerView!({ G: gameState, ctx: ctxLike as any, playerID: '0' })`.
4. The three required `// why:` comments were re-scoped to match the new shape — (a) null/non-string → spectator (on the audience-derivation ternary), (b) projection authority (on the `playerView:` field), (c) cast anchor rationale (on the `NonNullable<...>` assertion).

**Cost:** zero additional SPEC commits; zero additional test cases; zero change to test counts; zero ripple through other files. Documented as a refinement of RS-3, not an invention of a new cast pattern.

**Lesson:** RS-3's three-positional-arg signature assumed a looser variant of boardgame.io 0.50.x's Game type than is actually declared in 0.50.2. Future pre-flight passes for runtime-wiring WPs should **grep the boardgame.io `.d.ts`** for the locked property's actual declared type before fixing a signature in the WP. The `Game<G>['playerView']` type in `node_modules/.pnpm/boardgame.io@0.50.2/node_modules/boardgame.io/dist/types/src/types.d.ts:268-272` is the authoritative source.

**No DECISIONS entry required.** This is TypeScript language / library-version variance, not an architectural shift. D-8901 remains the architectural decision (projection authority); the cast form is an implementation detail below D-8901.

## Section 17 — Lessons Learned (for future Runtime-Wiring WPs)

1. **Pre-flight must verify the locked cast compiles, not just typechecks-in-isolation.** The RS-3 cast pattern was derived from WP-089's analysis of `Game<...>['playerView']` without running `tsc --noEmit` on a sample wiring. TypeScript's `exactOptionalPropertyTypes: true` semantics only surface at the object-literal assignment site — a cast-in-isolation analysis passes. Future pre-flight for runtime-wiring WPs should stand up the cast pattern in a local scratch file and compile it against the project's actual tsconfig before locking it in the WP.
2. **Grep `node_modules/.pnpm/<library>/dist/types/**/*.d.ts` for the declared property type before locking a function signature.** boardgame.io 0.50.2's `playerView` is single-context-object; RS-3 locked three-positional-args. The mismatch was hidden until execution. A single `grep -n "playerView" node_modules/.pnpm/boardgame.io@0.50.2/...` call at pre-flight time would have surfaced it.
3. **When RS-3 says STOP-and-ask, ask.** RS-3's language ("do not improvise a third option") is load-bearing. The `NonNullable<...>` refinement was authorized explicitly by the user in-session; without that authorization, the correct action would have been to halt and escalate to a SPEC amendment. Precedent preserved for the next runtime-wiring WP where a locked cast fails at execution time.
4. **`STRIP_SECRETS` in `// why:` prose trips the `grep STRIP_SECRETS` guardrail.** Initial field comment explicitly named the boardgame.io helper by its constant name; the grep AC caught this and the comment was rewritten to a paraphrase ("built-in secret-stripping helper"). Future runtime-wiring WPs that document "we intentionally do not use X" should paraphrase X if X is also a grep guardrail target — the DECISIONS.md entry is the authoritative named reference; in-code comments can defer to it.
5. **Dual-context test fixture (RS-2) scales.** `makeMockCtx` for `SetupContext` + inline `ctxLike` for runtime `Ctx` is the pattern `uiState.build.test.ts:52-56` established and `game.playerView.test.ts` extends. Future WPs that wire additional `Game<>` hooks (`onEnd`, `endIf`, etc.) should follow the same pattern rather than growing `makeMockCtx` into a kitchen-sink fixture.
