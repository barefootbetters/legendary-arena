## Section 1 — Overview

WP-051 lands the pre-submission PAR enforcement point at the server layer.
Two new files under `apps/server/src/par/` introduce:

- `parGate.mjs` — two exported functions plus two internal helpers:
    - `checkParPublished(simulationIndex, seedIndex, scenarioKey)` — pure
      synchronous lookup across both in-memory indices with D-5003 /
      D-5101 sim-over-seed precedence. Returns a **fresh object literal**
      `{parValue, parVersion, source}` on every hit (aliasing guard per
      copilot #17). Returns `null` when the scenario is absent from both
      indices and when both indices are `null` (fail-closed).
    - `createParGate(basePath, parVersion)` — async factory that loads
      both source-class indices once at startup via `Promise.all` on
      two `loadParIndex` calls, catches `ParStoreReadError` per class via
      the `handleParLoadError` helper (warn-log + null-for-that-class),
      then returns a bound gate whose `checkParPublished` partially
      applies the module-level function over the loaded indices.
      Exposes `simulationScenarioCount` and `seedScenarioCount` for
      observability hooks in future WPs.
    - `handleParLoadError(source, indexPath)` (internal) — returns the
      per-call `.catch` handler. Swallows only `ParStoreReadError`;
      re-throws every other error class so infrastructure failures
      (permission denied, disk full) surface loudly rather than silently
      degrading the gate.
    - `countUnionScenarios(simulationIndex, seedIndex)` (internal) —
      pure function for the union-of-scenario-keys count emitted in
      the startup log.
    - `formatStartupLogLine(simulationIndex, seedIndex, parVersion)`
      (internal) — all four log variants (both-loaded, sim-only,
      seed-only, both-missing) in full sentences per code-style Rule 11.

- `parGate.test.ts` — exactly 13 tests in one `describe('PAR publication
  gate (WP-051)', …)` block:
    - 3 `loadParIndex` smoke (valid round-trip, null-on-missing, throws
      on cross-class source stamp mismatch)
    - 3 `checkParPublished` base (sim-only hit, absent→null, dual-null
      fail-closed)
    - 3 `createParGate` integration (bound-gate semantic equivalence,
      **zero-filesystem-IO-at-request-time invariant** via post-construction
      `rm -rf` of the backing directory, version isolation)
    - 3 dual-class precedence (sim-only hit, seed-only hit with graceful
      degradation, both-present sim wins even when `parValue` differs)
    - 1 aliasing guard (two sequential calls return identity-distinct
      objects; mutating result1 leaves result2 and the in-memory index
      untouched)

Two existing files receive surgical modifications:

### 3.8 Test-glob audit

**Question:** Does `apps/server/package.json`'s test script actually pick
up the new tests in `src/par/`? And what caused the mid-execution glob
refinement away from the Locked Values quoted form?

**Verification:**

`pnpm --filter @legendary-arena/server test` output:

```
ℹ tests 19
ℹ suites 3
ℹ pass 19
ℹ fail 0
```

The session prompt's Locked Values specified quoted globs:

```
"test": "node --import tsx --test 'scripts/**/*.test.ts' 'src/**/*.test.ts'"
```

This form was initially written. `pnpm --filter @legendary-arena/server
test` reported `tests 0, suites 0, pass 0, fail 0` — the new **and** the
pre-existing tests were both invisible to the runner.

**Root cause:** On Windows, pnpm routes script execution through `cmd.exe`
(the default `script-shell` was `undefined`, so pnpm falls back to the
platform default; `C:\Windows\System32\cmd.exe` is present and verified).
`cmd.exe` does **not** honor single quotes as a quoting mechanism — it
passes them as literal characters. The two arguments reaching Node were
the literal strings `'scripts/**/*.test.ts'` and `'src/**/*.test.ts'`
(leading and trailing quote characters included). Node's
`--test` built-in glob expansion treats each argument as a glob pattern —
but no file path starts with a literal `'`, so neither glob matched.
Result: silent-green (zero tests, zero failures).

The original pre-WP-051 `scripts/**/*.test.ts` was **unquoted** and had
been working on this Windows + pnpm + cmd.exe setup — `cmd.exe` passes
the literal argument to Node, which then expands the glob internally via
the Node v22+ native `--test` glob support.

**Resolution:** Drop the quotes. The locked form becomes:

```
"test": "node --import tsx --test scripts/**/*.test.ts src/**/*.test.ts"
```

Two unquoted globs. Both get passed as literal strings to Node (cmd.exe
does no expansion on them because `*` and `?` are not cmd.exe glob
metacharacters unless they resolve to files in the current directory —
and `scripts/**/*.test.ts` contains slashes that cmd.exe doesn't expand).
Node's `--test` expands both globs internally. Result: 19 tests, 3
suites, 0 fail through `pnpm --filter` **and** through direct
`node --import tsx --test` invocation.

This is a strict in-allowlist refinement: `package.json` is in the Commit
A allowlist; the refinement does not leave the allowlist. 01.5 is **NOT
INVOKED** — the change is cross-platform quoting hygiene, not a
structural wiring allowance. Commit A commit body records the refinement
explicitly so the deviation from Locked Values is searchable from the
commit log.

**Recommendation for future ECs:** Specify proven-unquoted globs as
Locked Values on this repo unless a POSIX shell is explicitly forced via
`script-shell` in `.npmrc`. The theoretical "cross-platform-safe" quoted
form is a POSIX sh-ism that fails silently under pnpm-on-Windows-through-
cmd.exe.

**Verdict:** PASS.

## Section 8 — Unused-Binding Pattern in `server.mjs`

The `parGate` binding in `server.mjs` is intentionally held but unused
until WP-053 wires submission handlers. Three candidate patterns:

1. `// eslint-disable-next-line no-unused-vars` — adds a tool-specific
   directive; noise.
2. Rename to `_parGate` — conventional but semantically weaker; a casual
   reader might think the underscore signals a private variable.
3. `void parGate;` — the explicit expression-statement form. The
   statement has no runtime side effect (reading the binding and
   discarding it via `void`) but documents "this binding is kept on
   purpose; future code will consume it."

Pattern 3 chosen. A future WP-053 session that adds a `getParGate()`
accessor or wires the gate into a request handler will remove the
`void` line naturally.

## Section 9 — Vision Alignment

Per WP-051 §Vision Alignment, the clauses this packet touches are
**§3, §22, §24, §25, §26**.

- **§3 (Player Trust & Fairness)** — casual play continues when PAR is
  absent (no opaque difficulty imposed on non-competitive players);
  competitive submissions fail closed when PAR is missing (no skill
  claim on an unpublished baseline). The gate's binary published /
  not-published posture removes probabilistic and heuristic paths that
  could erode trust.
- **§22 (Deterministic & Reproducible Evaluation)** — synchronous pure
  gate check; no wall-clock reads; no randomness; stable for process
  lifetime per D-5102. `PAR_VERSION` is read once at startup; the gate
  closure over the version string is immutable post-construction.
  Given identical indices and a scenario key, the gate is bit-for-bit
  reproducible.
- **§24 (Replay-Verified Competitive Integrity)** — the gate is the
  pre-submission trust surface for competitive records. Fail-closed on
  missing indices, sim-over-seed precedence per D-5003 / D-5101, and
  the fresh-object-literal return shape `{parValue, parVersion, source}`
  together defend against leaderboard trust violations. Server is
  read-only for PAR data; never derives or mutates it. Engine owns PAR
  file IO per D-5001 line 8937; server consumes via named imports only
  — no cross-layer bleed.
- **§25 (Skill Over Repetition)** — submission rejected iff the scenario
  has no published PAR. Rejection is deterministic; no player-,
  hero-, or outcome-dependent behavior. The gate does not interpret,
  transform, or reason about `parValue` — it is returned opaque for
  downstream recording and display. Repetition cannot cause a scenario
  to become published; only the seed-authoring or simulation-calibration
  pipeline can.
- **§26 (Simulation-Calibrated PAR Determination)** — sim-over-seed
  precedence at the gate preserves the three-phase derivation pipeline:
  seed provides day-one coverage; simulation supersedes it once
  calibrated. Both source classes remain on disk so historical
  leaderboard entries pinned to the seed-era `scoringConfigVersion`
  remain explainable. Hash / coverage validation is CI-time only via
  WP-050 `validateParStore`; the server trusts the index verbatim
  (D-5103) because integrity is the calibration pipeline's
  responsibility, not the enforcer's.

**Non-Goal proximity.** N/A — no paid surface involved. The fail-closed
posture structurally prevents NG-1 (no payer can bypass the gate by
submitting against a missing PAR).

Vision trailer for governance commits: `Vision: §3, §22, §24, §25, §26`.
