## Section 1 — Overview

WP-050 lands the PAR artifact storage and indexing layer — Phase 2/3 of the three-phase PAR derivation chain documented in `docs/12-SCORING-REFERENCE.md`. One new source file (`par.storage.ts`) and one new test file (`par.storage.test.ts`) introduce:

- The dual source-class storage layout: `data/par/seed/{parVersion}/` (content-authored Phase 1 baseline) and `data/par/sim/{parVersion}/` (WP-049 simulation-calibrated Phase 2 output), versioned independently. Both roots share identical shard (first two characters of the scheme slug) and filename (ScenarioKey with `::` → `--`, `+` → `_`, `.json` suffix) conventions. Directory-name construction is funneled through `sourceClassRoot` — the only place `seed/` and `sim/` strings are produced.
- The `ParArtifactSource` discriminated union (`'seed' | 'simulation'`) + the canonical `PAR_ARTIFACT_SOURCES` readonly array drift-pinned against the union via test #6.
- Immutable artifact shapes: `SeedParArtifact` (8 top-level fields including a full `ParBaseline` + `authoredAt` / `authoredBy` / `rationale` provenance) and `SimulationParArtifact` (9 top-level fields including `percentileUsed` / `sampleSize` / the `simulation.policyTier: 'T2'` guard / `simulation.policyVersion` / `simulation.seedSetHash`). Both carry `scoring.scoringConfigVersion` + `scoring.rawScoreSemanticsVersion` version pins and an `artifactHash` self-hash.
- Writers (`writeSimulationParArtifact`, `writeSeedParArtifact`) with immutability enforcement at the write layer: both refuse to overwrite an existing file via an `fs.access` pre-check and throw a full-sentence `Error` on violation. Both compute `artifactHash` themselves — caller-supplied hashes are ignored (drift prevention). The seed writer additionally runs a consistency guard: `artifact.parValue` must equal `computeParScore(scoringConfig with parBaseline=artifact.parBaseline)`; mismatches throw before any disk write.
- Readers (`readSimulationParArtifact`, `readSeedParArtifact`) that return `null` on file-not-found (ENOENT) and throw `ParStoreReadError` on malformed content or structural shape violations. File-not-found and corrupt-read are deliberately distinct signals.
- `buildParIndex(basePath, source, parVersion)` — atomic index builder. Scans `{class-root}/scenarios/**/*.json`, extracts per-artifact `parValue` + `artifactHash`, sorts scenarios map keys alphabetically, writes via temp-file + `fs.rename` atomicity. Indices are not immutable (rebuildable from the artifact set); individual artifact files are.
- `resolveParForScenario(scenarioKey, basePath, parVersion)` — the ONLY sanctioned cross-class reader. Applies the locked simulation-over-seed precedence rule: load `sim/{parVersion}/index.json` first; if it covers the scenario, return the simulation entry. Otherwise load `seed/{parVersion}/index.json`. Otherwise return `null`. Missing index files are treated as "class has no coverage" (the resolver advances to the next class); truncated / malformed / JSON-parse-failing indices throw `ParStoreReadError` — never silent fall-through.
- `validateParStore(basePath, source, parVersion)` — read-only structured validator. Reports every inconsistency (completeness, exclusivity, parValue match, hash integrity, filename-scenarioKey mismatch, cross-class `source` drift, non-T2 policy tier for simulation, seed baseline field completeness). Never silently repairs.
- `validateParStoreCoverage(basePath, parVersion, expectedScenarios)` — cross-class coverage reporter. Answers "do we have PAR for every scenario we plan to ship?" in one call so the pre-release gate (WP-051) has a single oracle.
- `computeArtifactHash(artifact)` — SHA-256 via `node:crypto.createHash('sha256')` over canonical JSON with the `artifactHash` field excluded (self-hash exclusion avoids circular dependency). Output format `sha256:{64-hex}`.

One new test suite (`par.storage.test.ts`) with exactly 34 tests in one `describe` block covers every contract surface: 6 path helpers, 7 simulation I/O, 7 seed I/O, 4 index building + lookup, 5 cross-class resolver, 4 store validation, 1 hashing canonicalization.

## Section 3 — Mandatory Checks (01.6 §1-7)

### 3.1 Aliasing Check (mandatory)

**Question:** Do `readSeedParArtifact` and `readSimulationParArtifact` return fresh objects that do not alias any caller-held reference? Does `writeSeedParArtifact` / `writeSimulationParArtifact` retain a reference to the caller's artifact after returning?

**Verification approach:**
- Both readers read bytes from disk via `fs.readFile(path, 'utf8')`, parse via `JSON.parse(raw)`, and return the parsed result. `JSON.parse` produces a fresh object graph with no aliasing to any previously-held reference. This is the same aliasing posture WP-049's `ParSimulationResult` carries through its JSON-roundtrip determinism test (par.aggregator.test.ts test #8).
- The writers never retain references. `writeSimulationParArtifact` builds a local `artifact` via object spread (`{ ...artifactWithoutHash, artifactHash }`), serializes to a local string via `canonicalJsonStringify`, and calls `writeFile(fullPath, serialized, 'utf8')`. After `writeFile` resolves, the local `artifact` and `serialized` values fall out of scope and are garbage-collectable. No module-level cache, no Map, no Set, no closure retains the reference.
- `writeSeedParArtifact` explicitly enumerates each non-hash field when constructing `withoutHash` (eight named fields) rather than spreading the caller's artifact. This means any property the caller might add beyond the declared schema is dropped — defensive against accidentally persisting extra fields that the TypeScript signature would otherwise permit under structural typing.

**Verdict:** PASS. Both readers produce JSON.parse-fresh objects with no aliasing to caller state. Both writers hold no reference after the write resolves.

### 3.4 Determinism Audit (mandatory)

### 3.7 Hash Integrity Audit (mandatory)

**Question:** Does `computeArtifactHash` correctly exclude the `artifactHash` field itself from the hash input (self-hash exclusion)? Does it correctly detect any non-hash field change?

**Verification approach:**
- `computeArtifactHash` calls `stripArtifactHashField(artifact)` before canonical JSON serialization. The stripper iterates `Object.entries(artifact)` and omits the key `artifactHash` from the output. This means inserting, removing, or modifying the `artifactHash` field of the input does NOT change the hash — the hash depends only on non-hash fields.
- Test #34 explicitly asserts this two-part invariant:
  1. `computeArtifactHash({...content, artifactHash: '<placeholder>'})` === `computeArtifactHash({...content})` — adding `artifactHash` does not change the output.
  2. `computeArtifactHash({...content, parValue: 100})` !== `computeArtifactHash({...content, parValue: 101})` — any non-hash field change produces a different hash.
- Test #34 also asserts canonicalization: `computeArtifactHash(objA)` === `computeArtifactHash(objB)` when `objA` and `objB` have the same content but different key insertion order.
- Writer flow: `writeSimulationParArtifact` computes the hash over `artifactWithoutHash` (built via `buildSimulationArtifactFromResult`, which never includes `artifactHash`), then constructs the final artifact via `{ ...artifactWithoutHash, artifactHash }`, then serializes and writes. The written artifact contains the correct hash, and recomputing `computeArtifactHash` on the written artifact (which now has `artifactHash`) reproduces the same value — self-hash exclusion holds end-to-end.
- Validator flow: `validateParStore` reads each artifact, recomputes `computeArtifactHash(parsed)` (self-hash automatically excluded), and compares to `parsed.artifactHash`. Mismatch flags as `artifact-hash-mismatch`.

**Verdict:** PASS. Self-hash exclusion is structurally correct (via `stripArtifactHashField`) and test-asserted (test #34). Any field change produces a different hash. Any tampering with `artifactHash` itself is detectable by recomputation.

## Section 9 — Lessons & Handoffs

### What WP-050 did well

- The PS-1..PS-6 pre-flight resolution cycle was load-bearing — every PS item landed a concrete signature or contract change that would otherwise have required mid-execution escalation. PS-5 in particular (the `scoringConfig` parameter on `writeSeedParArtifact`) would have blocked the consistency guard entirely if discovered during coding.
- The 34-test breakdown (6/7/7/4/5/4/1) mapped directly to contract surfaces: the tests were written against the locked surface rather than against implementation details, so the test file is a second specification of the module.
- `sourceClassRoot` as a single choke-point for directory-name construction is the cleanest isolation guarantee this module has. Per-source-class isolation (§3.5) falls out of the fact that `seed/` and `sim/` literal strings appear exactly once in the module, inside that function's body.

### What future WPs should inherit

- **WP-051 (server publication gate)** consumes `resolveParForScenario` + `validateParStoreCoverage` as the two read-side entry points. The server MUST NOT probe artifact files directly; the index is the oracle. WP-051 should add a server-side cache layer IF and ONLY IF measurement shows the per-request index load is a latency issue; prematurely caching the index would introduce a cross-cutting consistency concern.
- **WP-054 (public leaderboards)** consumes the `ParResolution.path` and `ParResolution.indexPath` values to render "which PAR did this leaderboard entry compete against?" — the resolver's job is to produce enough breadcrumbs that the UI layer never needs to re-derive paths.
- **Any future simulation-category IO** MUST NOT cite D-5001 as precedent. D-5001 is deliberately scoped to PAR artifact storage. New IO needs land through their own architectural review (or move out of `packages/game-engine/` entirely, following the `vue-sfc-loader` Shared Tooling precedent).

### What to watch for

- **`rawScoreSemanticsVersion` is a locked constant (`1`) in `par.storage.ts`.** If a future WP changes the raw-score formula in `scoring.logic.ts`, it MUST bump `CURRENT_RAW_SCORE_SEMANTICS_VERSION` in lockstep. A DECISIONS entry + cross-WP check should accompany any formula change.
- **Index non-determinism on `generatedAt`.** The index carries an ISO-8601 timestamp from `new Date().toISOString()`, so two regenerations from the same artifact set produce different `generatedAt` values. If future WPs need whole-file byte-identity for indices (e.g., for CDN `Cache-Control: immutable` headers), they should add a `generatedAtOverride` parameter to `buildParIndex` following WP-049's `generatedAtOverride` precedent on `ParSimulationConfig`.
- **Cross-class `source` drift is only caught by `validateParStore`, not `buildParIndex`.** The index builder extracts `scenarioKey`, `parValue`, and `artifactHash` from artifacts but does not inspect the `source` field. This is deliberate — `validateParStore` is the read-only auditor, and the builder should not reject a structurally-valid artifact (even if it's cross-class-corrupt). Operators MUST run `validateParStore` on each class as the final gate before publishing.
