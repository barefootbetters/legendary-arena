## Section 1 — Overview

WP-056 establishes the pre-planning state contract as types-only scaffolding
in a new non-authoritative package (`packages/preplan/`). Four public types —
`PrePlan`, `PrePlanSandboxState`, `RevealRecord`, `PrePlanStep` — are
exported from a single source file with JSDoc lifecycle invariants inline.
Zero runtime code, zero tests, zero wiring into the game engine. The package
consumes `CardExtId` from `@legendary-arena/game-engine` via a type-only
import and carries the engine as a workspace peer — no runtime dependency.
This contract is the foundation for WP-057 (sandbox execution) and WP-058
(disruption detection), which will own the first runtime consumers of the
closed unions (`status`, `invalidationReason.effectType`) and therefore own
the canonical readonly arrays + drift-detection tests per Finding #4
deferral lock.

## Section 6 — Hidden-Coupling / Aliasing Audit

**N/A at the types layer.** WP-056 introduces no function returns, no spread
copies, no projections, no mutable-array handoffs. The aliasing risk
surface is inherited by WP-057 when sandbox execution begins constructing
snapshots from real player state — at that point §6 becomes load-bearing
(spread-copy mutable sources; never hand back references to authoritative
state). WP-056 pre-emptively defends against this downstream risk by:

1. Making `revealLedger` the **sole** authority for deterministic rewind
   (DESIGN-CONSTRAINT #3; JSDoc-pinned on `RevealRecord`). Rewind logic
   that inspects `sandboxState` directly is defined as invalid up front,
   so aliasing of `sandboxState` fields cannot cross the rewind boundary.
2. Encoding the `victory` omission as a structural invariant on
   `PrePlanSandboxState`, not as a runtime check. Hidden-state leakage is
   impossible by construction — the type does not carry the field.
3. Declaring `PrePlanStep.intent` and `RevealRecord.source` as
   **advisory-open** unions with explicit "never executable" and
   "descriptive-only" language. Consumers are structurally prevented from
   treating these as dispatch keys, which pre-empts a common class of
   coupling bug where UI hints drift into execution paths.

No hidden coupling introduced. No enum or union widened unintentionally.
No ordering assumptions not explicitly documented (`revision` monotonicity,
`revealIndex` monotonicity, lifecycle transition list all JSDoc-pinned).

### Finding 8.1 — `pnpm -r build` pre-existing registry failure

**What happened.** Task 8 Verification Step 3 calls for `pnpm -r build` to
exit 0. It exits 1 with `packages/registry build: ❌ dist/cards.json not
found — run 'pnpm normalize' first`. The registry `build` script chains
`tsc -p tsconfig.build.json && node scripts/build-dist.mjs`; `build-dist.mjs`
reads `dist/cards.json` which is produced by a separate `pnpm normalize`
step that must be run once before `pnpm -r build` will succeed.

**Is it WP-056's fault?** No. Reproduced the failure on clean HEAD (`311c8fe`)
by stashing WP-056 work (`git stash push -u -- packages/preplan pnpm-lock.yaml`,
stash `415798f`), re-running `pnpm install` + `pnpm -r build`. Same registry
failure reproduces with zero WP-056 code present. This is a pre-existing
registry build-pipeline bootstrapping condition that predates the WP-056
branch.

**Why the session prompt didn't catch it.** The session prompt Verification
Step 3 was written under the implicit assumption that `pnpm -r build` is
green at HEAD. The assumption is empirically false today and has been
false since at least the WP-081 planning cycle — the parallel WP-081
session (`.claude/worktrees/`) is explicitly chartered to fix the registry
build-pipeline bootstrapping issue (`WP-081-registry-build-pipeline-cleanup.md`).
Neither pre-flight audit for WP-056 nor the 01.7 copilot check reality-
reconciled against `pnpm -r build`; both assumed the baseline `pnpm test`
repo-wide green implied the build was also green.

**Resolution.** Skip the repo-wide build gate for WP-056 and substitute the
package-scoped build: `pnpm --filter @legendary-arena/preplan build` exits
0, produces valid `dist/*.js` + `*.d.ts` + sourcemaps. This is the correct
scope because WP-056 only introduces `packages/preplan/` and must not touch
registry build artifacts (which would be a scope violation — `packages/registry/**`
is in the §UNCHANGED files tripwire list). Registry build is unblocked
separately by WP-081.

**No action required by WP-056.** No scope amendment. No D-entry needed —
this is a documentation-to-reality drift, not a decision. Recommend a
future pre-flight guardrail: "before citing `pnpm -r build` in a
Verification Step, reality-reconcile against HEAD to confirm it is
green today."

### §9 Forward-Safety Questions (Answer YES to All)

1. **Can this code survive future refactors without touching unrelated layers?** YES. The package imports only `import type { CardExtId }` (a string alias). If the engine's `CardExtId` definition moves or is renamed, TypeScript will surface the break as a compile error at the single import site; no silent semantic drift is possible. No registry / server / apps / framework imports exist, so refactors in those layers cannot affect preplan.
2. **Can replay/debugging reconstruct behavior from stored data?** YES. The `revealLedger` is JSDoc-pinned as the sole rewind authority (DESIGN-CONSTRAINT #3). Every field on every public type is JSON-serializable (`string`, `number`, `string[]`, `Record<string, number>`, discriminated union of string literals, optional structured object). No functions, closures, Maps, Sets, Dates, or class instances. A `PrePlan` instance can be fully round-tripped through `JSON.stringify` → `JSON.parse` with no loss.
3. **Would removing upstream data still fail safely?** YES. The only upstream is the type-only `CardExtId` import. Removing it triggers a compile error, which is a safe failure (build breaks before any runtime executes). There is no runtime data consumption — the package has zero runtime code paths.
4. **Is it impossible for this WP's output to influence gameplay unintentionally?** YES. Zero runtime code. Zero `G` write path (the layer is non-authoritative by construction; there is no access path from preplan types to authoritative game state). Zero engine wiring (the engine does not know preplan exists; WP-028 lifecycle-prohibition precedent applies preemptively). The package cannot be called, instantiated, or mutated to affect gameplay — it is strictly a type surface consumed later by WP-057 / WP-058 runtime code.
5. **Is the contract stable enough to be referenced by future WPs?** YES. This is the explicit purpose: WP-057 (sandbox execution) and WP-058 (disruption detection) are documented downstream consumers, and the closed-union deferrals (Finding #4) name them by number. The open-union `| string` fallbacks on `RevealRecord.source` and `PrePlanStep.intent` (Finding #10) provide extension seams so future content can widen the vocabulary without a breaking change. Null semantics, lifecycle transitions, revision monotonicity, and reveal-ledger-sole-rewind-authority are all JSDoc-pinned as invariants.

All five questions answered YES. No documented mitigations / follow-up
WPs / accepted risks are needed for WP-056 forward-safety. Zero
undocumented NOs.
