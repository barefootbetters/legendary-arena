## Section 3 — Six Mandatory Post-Mortem Items

### 3.1 Aliasing Check (WP-028 D-2801 + WP-029 precedent)

Every outbound object reference returned by the aggregator is a fresh literal — no caller-supplied input flows into the result unchanged.

- `generateScenarioPar` builds `rawScoreDistribution` as a fresh object literal at the return-site; the eight summary-statistic fields are recomputed per call from the sorted copy `[...rawScores].sort(...)`. The internal `rawScores` array is not retained in the result. The sorted copy is local to the function.
- `ParSimulationResult.scenarioKey`, `simulationPolicyVersion`, and `scoringConfigVersion` are primitive copies from `config` — JavaScript primitive assignment produces a copy, not an alias, so no caller-side mutation of `config` can perturb the returned `result`.
- `aggregateParFromSimulation` never returns the input array; it returns a primitive number. `[...rawScores].sort((a, b) => a - b)` copies before sorting, so the caller's array is never mutated — explicit contract verified at par.aggregator.ts lines of `aggregateParFromSimulation` and `generateScenarioPar`.
- `generateSeedSet` returns a freshly-allocated `string[]`. `computeSeedSetHash(seeds)` reads but does not retain `seeds`; the returned string is a new primitive.
- `createCompetentHeuristicPolicy` closes over a mulberry32 instance created at factory-call time. No caller-visible reference to the PRNG state escapes. The returned `decideTurn` reads `playerView` and `legalMoves` but never retains references to them between calls.

Cited precedent: WP-028 D-2801 local-structural-interface + aliasing-prohibition pattern; WP-029 filter-boundary spread-copy pattern. Follow-up: none required — aliasing surface is clean.

### 3.2 JSON-Roundtrip Check (D-4806 + RS-15)

`ParSimulationResult` must survive `JSON.parse(JSON.stringify(result))` with structural equality — no functions, Maps, Sets, Dates, or class instances.

- Every field in `ParSimulationResult` is a primitive (`string`, `number`, `boolean`) or a plain object literal of primitives (`rawScoreDistribution`).
- `ParAggregationError` is thrown, never stored — the result type has no error-valued slot. `ParAggregationError` itself carries `name` as a readonly instance field and `code` as a readonly string, both of which survive `JSON.stringify(error)` as the discriminated `{ name, code, message }` shape.
- Test #15 (`generateScenarioPar is byte-identical + JSON-roundtrip stable`) exercises both `deepStrictEqual(resultA, resultB)` (byte identity with injected `generatedAtOverride`) AND `deepStrictEqual(JSON.parse(JSON.stringify(resultA)), JSON.parse(JSON.stringify(resultB)))` (proves round-trip survives). Both assertions pass.

Cited precedent: D-4806 `ScoreBreakdown` JSON-roundtrip; WP-028 projection-serializability. Follow-up: none required.

## Section 4 — Deviations from the WP / EC

One in-allowlist refinement (strict in-allowlist, no 01.5 invocation per WP-031 precedent):

1. **Added `MAX_MOVES_PER_GAME = 2000` to `simulateOneGame`.** The WP specified a 200-turn safety cap matching WP-036; during execution, a stall trap surfaced where a deterministic T2 policy at `stage === 'start'` against an empty mock villain deck kept choosing `revealVillainCard` (highest score among `revealVillainCard`, `drawCards`, `advanceStage`), which is a no-op on an empty deck — turnsElapsed never incremented and the turn cap never fired. Added a move-level cap that flags the game as stuck in `gameState.messages` and breaks out. No contract change, no public API change, no test signature change.

2. **Rewrote multimodality detector to use cluster-based peak detection.** The initial implementation used "≥ 2 bins each ≥ 20% of max, separated by ≥ 2 bins" which false-positived on unimodal distributions where many adjacent bins all cleared the 20% threshold. The replacement walks the histogram counting contiguous peak clusters (maximal runs of bins at or above 20% of max) and flags multimodal only when `clusterCount >= 2`. This is a strict improvement in smell-test precision without changing the function's public signature or the test it powers.

Both refinements are documented with `// why:` citations in the source. Neither alters any contract, neither is a 01.5 trigger.

## Section 6 — Risks Mitigated

- **Silent non-determinism in reproducibility tests** — mitigated by the `generatedAtOverride` injection pattern (RS-11). Byte-identity claims now require explicit clock pinning.
- **Default lexical sort in percentile computation** — mitigated by the explicit `(left, right) => left - right` comparator in `aggregateParFromSimulation` and `generateScenarioPar`. Locked by the comment and verified by test #1 (rank index 549 on 1000 scores produces 550).
- **`ParAggregationError` message-substring matching** — mitigated by tests #3 and #4 which assert `instanceof ParAggregationError` AND `.code === '<expected union member>'`.
- **Seed silently dropped** — mitigated by the runtime assertion `rawScores.length === config.simulationCount` inside `generateScenarioPar` and by test #17 which asserts `result.sampleSize === config.simulationCount`.
- **Non-serializable fields leaking into result** — mitigated by test #15 which performs both byte-identity and JSON-roundtrip assertions.
- **Tier ordering violated but PAR still published** — mitigated by `validateTierOrdering` returning structured `passed: false` + violations, intended to be called before publication. Enforcement is server-layer per D-4907.

## Section 7 — Risks Accepted (With Rationale)

- **Mock-registry bootstrap runs produce artificial Raw Scores.** Tests #15/#16/#17 use an empty `CardRegistryReader` mock, so every simulated game hits the `MAX_MOVES_PER_GAME` stall cap. The Raw Scores they produce are not meaningful game outcomes; they exist to exercise the aggregation pipeline's shape + determinism + provenance. Publication-grade PAR requires a real registry (WP-050 / WP-051 wire that in). Acceptable because the pipeline contract is what we're testing here.
- **`needsMoreSamples` thresholds are empirical.** `IQR_THRESHOLD = 2000` and `STDEV_THRESHOLD = 1500` were chosen to be loose enough for healthy ±20% distributions and tight enough for real bimodal patterns. A future calibration WP may tune these based on production telemetry. The D-4910 entry in DECISIONS.md locks the "module-level constants, not config inputs" rule; specific values may shift with a DECISIONS.md entry + `scoringConfigVersion` bump.
- **T2 policy does not consult deck composition.** Heuristic 4 (limited deck awareness) is implemented as scoring-constant ordering only — the policy does not count cards or inspect deck contents. This matches the human-play model in WP-049 §A.4 ("No exact card counting, no draw prediction") but means the current T2 is less sophisticated than a "strong" policy would be. T3 (Strong Heuristic) is future work.
