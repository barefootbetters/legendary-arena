## Section 6 — Post-Mortem Findings (Aliasing, Hidden Coupling, `// why:` Gaps)

### 6.1 Aliasing check

`BetaFeedback`, `BetaCohort`, and `FeedbackCategory` are pure type exports — no runtime values, no functions, no module-level state. There is no engine-side constructor, no module-level instance, and no runtime field reachable from any move, phase hook, or setup function. **No aliasing path exists** from a `BetaFeedback` instance into `G`, because the engine never constructs an instance in the first place. Future consumers that construct `BetaFeedback` objects (server layer, ops tooling — explicitly Out of Scope per WP-037) will build them at their own boundary and transport them over the wire as JSON-serializable payloads.

### 6.2 Hidden-coupling check

The beta subdirectory is a leaf of the engine-export DAG: `beta.types.ts` has zero `import` statements. It depends on nothing engine-internal, nothing registry-side, nothing server-side, and nothing in `boardgame.io`. The re-export blocks appended to `types.ts` and `index.ts` add type-only re-exports that are erased at compile time — they produce no runtime coupling, no circular import risk, and no ordering dependency with existing re-export blocks. `FeedbackCategory` and `BetaFeedback.category` share a five-member closed literal union; they were deliberately kept as two independent literal-union declarations rather than unified via `category: FeedbackCategory` so that the shared surface is locked by lock-step review (EC-037 §Locked Values) rather than by a type-level reference that would let one drift silently if the other changed.

### 6.3 `// why:` gap check

The mandatory EC-037 line-65 `// why:` comment sits at the `beta.types.ts` module header verbatim: *"feedback tied to build version for traceability; replay reference enables reproduction."* Three recommended additional `// why:` comments sit above each of the three type declarations, explaining the field-order lock (`BetaFeedback`), the cohort-signal-target ordering (`BetaCohort`), and the five-category verbatim-match rationale (`FeedbackCategory`). No `// why:` comment is required in `types.ts` or `index.ts` because the re-export blocks are additive wiring (no new semantics) — same pattern as the WP-035 ops and WP-036 simulation blocks. No governance gap.

## Section 9 — Lessons Learned (Feeding 01.4 Template)

1. **JSDoc-prose-vs-verification-grep lesson.** When a verification step uses a loosely-scoped regex (no anchoring to `from ['"]` or function-call syntax), the JSDoc module header for pure-type files should reference its D-entry rather than enumerate forbidden tokens verbatim. Restating the forbidden list in prose produces false-positive grep hits and forces a prose-softening reality reconciliation. See Section 8 for the concrete instance.
2. **Pre-landing D-entries scales cleanly through ten precedents.** D-3701 is the 10th engine-subdirectory classification. The pattern — pre-land the D-entry and the `02-CODE-CATEGORIES.md` row in a `SPEC:` commit A0 before session open — removes all ambiguity about whether the category rules apply. The cost of the A0 commit (one commit, two touched files) is dwarfed by the cost of relitigating the classification mid-execution. This is now a well-validated pattern; future subdirectory-adding WPs should default to it.
3. **Metadata-not-state default for ops/feedback surfaces.** Two WPs in a row (WP-035 ops, WP-037 beta) have used the "declare types in engine, construct outside engine" discipline. Both produced clean diffs with zero `G` mutation and zero contract drift. WP-039 post-launch metrics should default to this pattern rather than adding fields to `LegendaryGameState`.
4. **Binary pass/fail exit criteria scale.** The WP-037 exit gate has 12 criteria spread across 4 categories, every one binary with a measurable threshold and a specific source signal. The pattern is re-runnable (two reviewers reach the same verdict given the same signals) and auditable. Future beta / launch / live-ops exit gates should follow this template.
5. **Strategy-document pair reduces scope ambiguity.** Splitting the beta artifact into `BETA_STRATEGY.md` (prose, objectives, scope, rationale) and `BETA_EXIT_CRITERIA.md` (binary gates, thresholds, source signals) kept each document's responsibility focused. Neither document carries both narrative and measurement, so reviewers know exactly where to look. WP-038 / WP-039 should follow this pair pattern.
6. **The `## Vision Alignment` block retrofit timeline (00.3 §17 gate landed 2026-04-22) caught WP-037 cleanly.** The retrofit was completed before session open (`e5b0d67`), so the session prompt's Vision Sanity Check verified against a clause-numbered WP block that actually existed. No confusion, no blocker. The Vision trailer convention on the commit body ensures the commit log remains re-auditable.
