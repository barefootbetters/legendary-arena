## 8. Mid-Execution Amendment Reflection (D-3103 Precedent)

**Surface.** `packages/game-engine/src/economy/economy.logic.ts` `buildCardStats()` was missed by EC §Locked Values "Builder internal-iterator updates (PS-7 LOCKED)" enumeration of four builders. The function consumes `matchConfig.heroDeckIds`, `villainGroupIds`, and `henchmanGroupIds` as bare slugs at three iterator sites (lines 179, 195, 224). After the qualified-ID contract landed in the four EC-enumerated builders, `buildCardStats` would silently produce empty `G.cardStats`, breaking the loadout integration test's locked acceptance criterion "G.cardStats populated for the chosen cards" and breaking runtime move costs (`fightVillain`, `recruitHero`, `fightMastermind`).

**How the gap was discovered.** Mid-execution. After Steps 1-4 (the four enumerated builders) landed and engine tests passed, the source-code structure made the inconsistency visible: a `grep matchConfig\.heroDeckIds` across `packages/game-engine/src/` returned five hits (the four EC-listed builders + `economy.logic.ts`). The fifth hit was confirmed via a 2-minute trace through `buildInitialGameState.ts` line 168 (`buildCardStats(registry as unknown, config)`).

**Why the EC enumeration missed it.** `economy.logic.ts` is sibling to `setup/`, not nested under it. The EC's PS-7 enumeration walked `/setup`-named directories and the two adjacent `villainDeck/` + `mastermind/` directories, but did not include `/economy/`. The "builder" mental model treated setup-builder files as a structural set; `economy.logic.ts`'s `buildCardStats` is functionally a setup builder (called from `buildInitialGameState`) but lives outside the named set.

**Process improvement.** Future WPs introducing contract changes on `MatchSetupConfig` field types or formats should grep ALL source files for `matchConfig.{fieldName}` consumption, not just `/setup`-named directories. Add this to the pre-flight checklist (`01.4-pre-flight-invocation.md`) under §Established Patterns: "When changing a `MatchSetupConfig` field's value semantics or format, the pre-flight enumeration of consumer files MUST be a global grep, not a directory walk."

**Cap excursion.** Authorized inline per the D-3103 precedent (WP-031 mid-execution amendment). The amendment block was added to WP-113 § "Mid-Execution Amendment (Spec Gap Fix)" before resuming code. The hard-cap was raised 16 → 17 in the EC §Guardrails block. EC §Files to Produce was extended with two entries (11a `economy.logic.ts` modified + 11b `economy.logic.test.ts` regression tests).

**Verified completeness.** Independent grep against `main` HEAD confirmed `economy.logic.ts` was the ONLY missed iterator site: `campaign/campaign.logic.ts` treats IDs as opaque strings (spread/merge only); `rules/ruleRuntime.impl.ts:58,65` uses `sourceId` opaquely (no equality checks against bare slugs); `ui/uiState.build.ts:256` is a cosmetic display-string ripple, not a determinism gap; `types.ts` and `matchSetup.types.ts` are type definitions only. The user's verification confirmed the +1 file scope.

**Builder soft-skip semantic.** During fixture migration, the existing test that asserts `gameState.messages` starts empty (`ruleRuntime.integration.test.ts`) surfaced a corollary of the PS-3 widening: with the wider `EMPTY_REGISTRY`, the four orchestration-side guards return `true` (interface complete) but the underlying data is empty (`listSets() === []`). This caused the four builders to traverse their main paths and throw on missing data — a behavior change from pre-WP-113 narrow-mock skip paths. The fix was to convert the throws to soft-skip (return null/empty) per the validator-is-authoritative model: the validator catches misconfigured loadouts upfront via `loadedSetAbbrs` and per-field known-IDs sets; with a real registry, missing data has already been rejected; the builder soft-skip is defense-in-depth. This is documented in D-10014 § "Soft-skip semantic".
