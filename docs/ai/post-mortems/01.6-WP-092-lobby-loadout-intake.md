## Section 2 — What Worked Well

- **Single-template-constant + single-helper discipline extended from registry-side (D-9101) to arena-client (D-9201).** `UNSUPPORTED_HERO_SELECTION_MODE_TEMPLATE` is the only declaration of the WP-093 error template in `apps/arena-client/src/**`. `renderUnsupportedModeMessage(value)` is the only place `.replace('<value>', ...)` happens. Every test asserts byte-for-byte equality via `assert.strictEqual(error.message, renderUnsupportedModeMessage(<value>))` — `assert.match` / `.includes()` / substring checks are absent from the suite. The `Select-String "is not a supported rule mode"` grep returns exactly one match in the parser file (only the const declaration itself).
- **Wrap-don't-rewrite for the WP-090 manual form.** The `<details>` wrapping is purely additive — `git diff` shows new template fragments around the form, never inside it. The `Select-String` grep for the 9 composition-field `v-model` bindings returns exactly 9 matches, identical to the WP-090 baseline. The manual-form's submission path is byte-identical and the existing `lobby-submit-create` test ID still resolves; the new JSON-intake submission path uses a separate `lobby-submit-from-json` test ID.
- **Single-error parser keeps the `field` taxonomy unambiguous.** Returning `{ ok: false, error: { code, message, field? } }` at the first failure (with documented order) makes every test deterministic — no zod-style "multiple issues, dedup the upgraded one" complexity. The locked per-error-code `field` mapping (§3.8) is encoded in the test suite via `assert.strictEqual(error.field, "<expected>")` for every error-bearing case and `assert.strictEqual(error.field, undefined)` for every fieldless case.
- **Hand-rolled type predicates beat external dependencies under the layer rule.** `isPlainObject` (4 lines), `isArrayOfNonEmptyStrings` (~10 lines), `isNonNegativeInteger` (1 line), and a `COMPOSITION_FIELDS` table of 9 specs handle the entire shape-guard surface with zero external dependencies. `git diff apps/arena-client/package.json` is empty under WP-092 — no `zod`, no `ajv`, no `jsonschema`, no schema-runtime fixed cost on the bundle.
- **The pre-flight `createMatch(config, numPlayers)` argument-order lock (RS-12) prevented a class of error.** The session prompt's verbatim transcription of the locked call-site idiom `await createMatch(parsed.composition, parsed.playerCount)` made it unambiguous which way the arguments go. TypeScript would have rejected the reversed order against `MatchSetupConfig`'s strict shape, but the explicit lock + `// why:` comment at the call site reduces the risk that a future refactor accidentally swaps them.
- **Parallel-drift vigilance noticed three out-of-scope file modifications that landed in the working tree mid-session.** `git status` after edits revealed `docs/ai/execution-checklists/EC-052-player-identity-replay-ownership.checklist.md`, `docs/ai/work-packets/WP-052-player-identity-replay-ownership.md`, and `docs/ai/work-packets/WP-053-competitive-score-submission-verification.md` modified — none are in WP-092's scope. Per RS-8 the parallel-session work is preserved (left in working tree; never staged); the pre-commit review session will see them and route them to the appropriate session for commit. ## Section 3 — What Surprised Us

- **`@legendary-arena/registry` mentioned in `// why:` comments tripped the "no forbidden token" parser grep (§11 Step 6).** The first draft of `parseLoadoutJson.ts` mentioned the literal string `@legendary-arena/registry` twice in `// why:` comments to explain the layer-rule rationale. The session-prompt §11 Step 6 grep `Select-String -Pattern "zod|ajv|jsonschema|boardgame.io|@legendary-arena"` is meant to verify there are no forbidden imports, but PowerShell's regex matches the literal token in comments too. Resolved by rewording the comments to "the registry package" / "registry-side" — preserving the layer-rule explanation while satisfying the grep gate. The `// why:` content is unchanged in spirit; only the literal token is avoided. Recorded here as a precedent: **never write the literal `@legendary-arena/<pkg>` token inside arena-client source comments when the pkg is one the layer rule forbids; use prose instead** so the no-forbidden-import grep stays definitive.
- **Pre-existing line-by-line grep false-positive on `SharedScoreboard.vue:6`.** The session prompt §11 Step 4 expects the runtime-engine-import grep to return exactly one match in `bgioClient.ts:16`. In practice, the line-by-line `Select-String ... | Where-Object { $_.Line -notmatch "import type" }` filter returns two matches because `SharedScoreboard.vue` (pre-existing from WP-062 / EC-069) uses a multi-line `import type {…} from '@legendary-arena/game-engine'` form where `import type` is on line 2 but the `from` clause is on line 6. The line-by-line filter sees only line 6, which doesn't contain `import type`, so it isn't filtered out — but it IS a type-only import. WP-090's post-mortem already documented this as an EC-tightening recommendation (multiline-aware grep). WP-092 doesn't change the SharedScoreboard.vue baseline; the false-positive is inherited and unaffected. Logged here so future arena-client WPs continue to know the line-by-line grep is approximate.
- **The "compound failure dedup test" framing didn't map cleanly to a single-error parser.** The session prompt §7.1 item 2 calls for a "compound-failure dedup test" pairing `heroSelectionMode: "HERO_DRAFT"` with another envelope error like an out-of-range `playerCount`. WP-091's parser is zod-based and can produce multiple issues per parse (so dedup matters); WP-092's parser stops at the first failure. Resolved by adding **two** compound tests: (a) HERO_DRAFT + ignored envelope extras → single `unsupported_hero_selection_mode` byte-identical (canonical "dedup" verification); (b) HERO_DRAFT + `playerCount: 99` → first-failure-wins-per-documented-order returns `player_count_out_of_range`, documenting the parser's check ordering. The session-prompt's expectation is met by (a); (b) documents the order so future readers don't re-derive it from code inspection.

## Section 6 — Precedent to Establish

- **Five-file cross-file byte-identity gate as the canonical pattern for consumer-side string discipline.** WP-091 established the four-file gate (DECISIONS.md, MATCH-SETUP-VALIDATION.md, registry-side `setupContract.types.ts`, registry-side `dist/.d.ts`). WP-092 promotes it to five files by adding the arena-client-side `parseLoadoutJson.ts`. Future WPs that add another consumer-surface (e.g., a server-side Stage 1 envelope-validation WP that must surface the same error message to the lobby) extend the gate to six files. Each new copy is justified by a layer-rule firewall (a pkg that cannot import the registry); documentation duplication is justified by the non-overrideable layer rule.
- **Hand-rolled type-predicate parser as the canonical arena-client pattern when authoritative validation lives behind a layer-rule firewall.** When the arena-client needs to validate a document whose authoritative validator lives in a package the arena-client cannot import at runtime, the right pattern is: (a) hand-rolled primitive type predicates only — no semantic checks (uniqueness, ext_id existence, cross-field invariants); (b) byte-for-byte mirror of any locked error templates with a cross-file gate; (c) single-error parser with documented check order; (d) per-error-code locked `field` mapping; (e) tests use `assert.strictEqual` on full messages, not substring matches. WP-092 is the first application; future arena-client WPs (e.g., a future spectator-link parser, a future replay-import parser) should follow this template.
- **Per-error-code `field` mapping enumeration as the canonical taxonomy discipline.** When a parser exposes a typed error code, the `field` property must be specified per-code in the WP / session prompt and pinned in tests via `assert.strictEqual` on both the dot-path string and `undefined` (for fieldless codes). WP-092 §3.8's enumeration becomes the template: a small markdown table mapping each code to a literal `field` value (or "absent"), with a test discipline statement that every error-bearing code asserts the `field` and every fieldless code asserts `field === undefined`.
- **9-field `v-model` byte-for-byte preservation grep gate as the canonical pattern for "wrap, don't rewrite" form refactors.** The §11 Step 10 grep `Select-String "v-model=\"(<9-field-list>)\""` returning exactly 9 matches is a cheap, deterministic guard against silently breaking a previously-tested input form during a UI refactor that wraps the form in a new container. Future WPs that wrap an existing form in a `<details>`, a tab, a modal, or a new layout container should adopt the same grep gate against the form's locked field list.

## Section 7 — Lessons Learned (Feed Forward to 01.4 §Established Patterns)

- **The "no forbidden import" grep needs prose-not-token guidance.** The §11 Step 6 grep can't tell a forbidden import from a `// why:` comment that explains why the import is forbidden. Future ECs should explicitly warn: when explaining the layer-rule rationale in `// why:` comments, never write the literal forbidden import token (`@legendary-arena/registry`, `@legendary-arena/game-engine`, etc.); use prose ("the registry package", "the engine runtime") instead. The grep then stays definitive: any match means a real import, not commentary.
- **Parser ordering should be documented in the WP itself, not just in tests.** WP-092 §Scope (In) A bullet 5 listed the parser's 8 control-flow steps. The check order matters because it determines which error a compound-failure document surfaces. The post-mortem's second compound test documents this; future parsers should likewise have their check order pinned in both the WP and the test suite, so pre-commit reviewers can confirm the lock without re-deriving it from code inspection.
- **`describe`-wrapping discipline for `assert.strictEqual` byte-for-byte tests is as cheap as test count discipline.** Wrapping all 30 parser tests in exactly one `describe('parseLoadoutJson (WP-092)')` block makes the suite-count delta +1 deterministic and the WP-attribution of the suite explicit. Similarly, the new `describe('parseLoadoutJson + createMatch (WP-092)')` block in `lobbyApi.test.ts` (rather than appending tests to the pre-existing `describe('lobbyApi (WP-090)')` block) makes the suite-count delta from `lobbyApi.test.ts` exactly +1 and preserves clear WP attribution. Future regression-test additions to existing test files should default to a new sibling `describe` block when the new tests are part of a different WP.
- **Parallel-session pCloud drift on out-of-scope files is a normal operating mode, not an exception.** The repo lives on pCloud per memory; multi-session work surfaces in the working tree without explicit checkout. The right response is per RS-8: leave the parallel work alone, never stage it from a session that doesn't own it, and document the observation in the post-mortem so the pre-commit review session can route the parallel work to the right commit. WP-091 and WP-092 both observed this; future arena-client + governance WPs should expect it.
