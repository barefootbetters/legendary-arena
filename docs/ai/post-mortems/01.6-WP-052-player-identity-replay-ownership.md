## Section 1 ‚Äî Overview

WP-052 establishes identity as a first-class server concern in Legendary
Arena. The packet lands eight new files: four TypeScript source files
(two `.types.ts` + two `.logic.ts`) under `apps/server/src/identity/`,
two `.test.ts` files in the same directory, and two new SQL migrations
appended at numbers 004 and 005 in `data/migrations/`.

The packet introduces the durable contracts that future submission and
leaderboard WPs (WP-053, WP-054) and any GDPR endpoint will consume:

- `AccountId` ‚Äî branded `string & { readonly __brand: 'AccountId' }`,
  generated as UUID v4 via `node:crypto.randomUUID()` and brand-cast at
  exactly one site (the `createPlayerAccount` insert path). Renamed
  from a draft-time `PlayerId` to avoid collision with the engine
  `PlayerId` per D-5201 / D-8701.
- `PlayerAccount` (7 readonly fields), `GuestIdentity` (3 readonly
  fields with `isGuest: true` discriminant), `PlayerIdentity =
  PlayerAccount | GuestIdentity`.
- `AuthProvider` literal union + `AUTH_PROVIDERS` canonical readonly
  array (drift-detection enforced).
- `ReplayVisibility` literal union + `REPLAY_VISIBILITY_VALUES`
  canonical readonly array.
- `ReplayOwnershipRecord` (7 readonly fields, `expiresAt: string |
  null`), `ReplayRetentionPolicy`, `DEFAULT_RETENTION_POLICY`
  (`{minimumDays: 30, defaultDays: 90, extendedDays: null}`).
- `Result<T>` discriminated union + `IdentityErrorCode` literal union
  for structured fallible operations.

The eight functions on the public surface (`createPlayerAccount`,
`findPlayerByEmail`, `findPlayerByAccountId`, `createGuestIdentity`,
`assignReplayOwnership`, `updateReplayVisibility`,
`listAccountReplays`, `findReplayOwnership`, `deletePlayerData`) are
consumed by their own test files only. WP-052 establishes the library;
no consumer is wired in this packet.

The two SQL migrations create `legendary.players` (UNIQUE on `email`
and `ext_id`) and `legendary.replay_ownership` (UNIQUE on
`(player_id, replay_hash)` for race-safe idempotency), both idempotent
via `CREATE TABLE IF NOT EXISTS`.

### 2.4 Aliasing audit

`pg` returns a fresh JavaScript row object on every query; the
`mapPlayerAccountRow` and `mapOwnershipRow` helpers construct fresh
`PlayerAccount` / `ReplayOwnershipRecord` literals on every call from
those rows. The functions in this packet never see `G`, never accept
or return references to engine state, and never cache results across
calls. The aliasing risk that drove D-2801 / WP-028 (engine projection
aliasing) does not apply here: server-layer DB queries materialize
fresh JavaScript objects, and there is no shared in-memory data
structure that could be aliased through a returned reference.

### 2.7 Idempotency SQL audit

```
$ grep -nE "ON CONFLICT \(player_id, replay_hash\) DO UPDATE" \
    apps/server/src/identity/replayOwnership.logic.ts
144:      'ON CONFLICT (player_id, replay_hash) DO UPDATE SET visibility = legendary.replay_ownership.visibility ' +
$ grep -nE "ON CONFLICT.*DO NOTHING" apps/server/src/identity/replayOwnership.logic.ts

```

The locked SQL pattern from WP-052 ¬ßLocked Values (PS-6) is used
verbatim. The `DO UPDATE SET visibility =
legendary.replay_ownership.visibility` is a no-op self-assignment
that forces `RETURNING` to emit the existing row's columns on
conflict ‚Äî `DO NOTHING` would emit zero rows and the caller could
not distinguish "first insert" from "race-lost idempotent retry"
without a second SELECT. The CTE wrapper resolves
`ext_id ‚Üí player_id` in the same statement so the unknown-account
case is detected by `result.rows.length === 0` rather than by
catching SQLSTATE 23503 after a foreign-key violation. Atomicity
and race-safety are guaranteed by the `UNIQUE (player_id,
replay_hash)` constraint declared in migration 005. Test #2 in
`replayOwnership.logic.test.ts` proves race-safe idempotency by
calling `assignReplayOwnership` twice with identical
`(accountId, replayHash, scenarioKey)` and asserting field-by-field
equality across the returned records.

## Section 3 ‚Äî Lessons & Carry-Forward

### 3.1 Skip-pattern grep gate vs. ternary skip option

The session prompt locked the DB-dependent skip pattern as a
ternary expression `{ skip: !hasTestDatabase ? 'requires test
database' : undefined }` while the verification grep asked for a
literal `skip: 'requires test database'` substring. The two
patterns are mutually exclusive: the ternary form does not contain
the literal substring on a single line.

The reconciliation used in this packet was an inline conditional
options object: `hasTestDatabase ? {} : { skip: 'requires test
database' }`. This puts the literal `skip: 'requires test database'`
on each test line when no test database is configured, satisfies
the grep gate, and preserves the runtime semantics (tests run when
DB is available, skip with the locked reason when not).

A future WP that introduces DB-dependent tests under the same skip
contract should use this exact inline form (or update the gate
text). Recording this here so the next executor does not waste a
round-trip rediscovering the asymmetry.

### 3.2 Handler-injected `Pool` versus engine startup wiring

WP-052 deliberately does **not** wire the identity pool into
`server.mjs`'s `startServer`. The functions accept a caller-injected
`DatabaseClient` (alias for `pg.Pool`); a future request-handler WP
will create the pool at startup and pass it in. This keeps
WP-052's allowlist tight (8 files, no `server.mjs` modification) and
defers the runtime-wiring decision to the WP that owns the
consumer surface. The lifecycle prohibition in the session prompt
(no calls from `game.ts`, no calls from any phase hook, no calls
from `server.mjs` in this packet) is honored by construction
because none of the eight functions are referenced from any file
outside this directory.

### 3.3 `.ts` extension for new identity files

Per WP-052 ¬ßLocked Values, the new identity files are `.ts` rather
than `.mjs`. This diverges from the WP-051 precedent (`parGate.mjs`)
because the identity surface relies on TypeScript-only syntax ‚Äî
branded `AccountId`, discriminated `Result<T>`, literal-union
`IdentityErrorCode`, `readonly` interface fields. The `.ts` files
load cleanly under `node --import tsx --test` (the server's
existing test runner), but plain `node` (used by `server.mjs`'s
runtime path) cannot import them directly. Any future WP that
wires identity into `server.mjs` will need to either add `tsx` to
the production runtime, add a build step, or convert these files
to `.mjs`. WP-052 does not solve that future problem.
