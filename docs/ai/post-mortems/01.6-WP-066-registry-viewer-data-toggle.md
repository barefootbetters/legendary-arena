- [x] **Is the contract stable enough to be referenced by future WPs?** YES. The two-name composable public API (`{ viewMode, toggleViewMode }`) is narrow, documented, and matches existing composable conventions (`useLightbox.ts`). The `cardViewMode` localStorage key is locked in EC-066 §Locked Values.

### Notes

1. **EC/FlatCard field-name gap (non-blocking, resolved in-component):** During context-reading (before any edits), I noticed that the EC-066 §Locked Values "13 FlatCard Display Field Names" table asserts field names (`victoryPoints`, `recruiterText`, `attackerText`) that do not exist on `FlatCard` in `apps/registry-viewer/src/registry/types/types-index.ts` at `HEAD=1623152`. Three other field names (`type`, `heroClass`, `edition`, `abilityText`) map to renamed FlatCard equivalents (`cardType`, `hc`, `setName`, `abilities: string[]`). The copilot-check's 30-lens sweep did not surface this gap.

   **Disposition:** Did NOT escalate, because the EC/WP intent is unambiguous (the display labels on the right side of each row are byte-exact; the left-side "field names" describe intended data access semantics), and the EC's "AND-semantics" rule ("Data view displays attributes only if non-null and non-empty") naturally handles absent fields without runtime errors. I interpreted the mapping as:

   | EC-listed name | Actual FlatCard access | AND-semantics guard |
   |---|---|---|
   | `name` | `card.name` | `v-if="card.name"` |
   | `type` | `card.cardType` | `v-if="card.cardType"` |
   | `edition` | `card.setName` | `v-if="card.setName"` |
   | `heroClass` | `card.hc` | `v-if="card.hc"` |
   | `team` | `card.team` | `v-if="card.team"` |
   | `cost` | `card.cost` | `v-if="card.cost !== undefined && card.cost !== null"` |
   | `attack` | `card.attack` | `v-if="card.attack !== undefined && card.attack !== null && card.attack !== ''"` |
   | `recruit` | `card.recruit` | `v-if="card.recruit !== undefined && card.recruit !== null && card.recruit !== ''"` |
   | `rarity` | `card.rarityLabel` (preferred) or `card.rarity` (fallback) | `v-if="card.rarityLabel"` / `v-else-if="typeof card.rarity === 'number'"` |
   | `victoryPoints` | not on FlatCard today | always omitted by AND-semantics (template never renders this row) |
   | `recruiterText` | not on FlatCard today | always omitted |
   | `attackerText` | not on FlatCard today | always omitted |
   | `abilityText` | `card.abilities: string[]` | rendered as a bulleted list under an "Ability" header when non-empty |

   The mapping is documented verbatim in the top-level JSDoc comment of `CardDataDisplay.vue` (lines 18-39 of that file). The three absent rows are flagged as "contract-aware placeholders that will auto-light-up if/when flattenSet() begins surfacing those fields (a future registry WP)". No runtime errors; no silent misdisplay; no contract-drift risk.

   **Why not escalate under "Locked value contradicts reality"?** The locked values *can* be implemented as stated — the EC author's intent is clear (show 13 fields with these labels when present, omit when absent). The AND-semantics rule is the escape hatch. If I had interpreted "field names" as "must strictly match `FlatCard` type properties", then 4-7 rows would have been non-implementable, which is a strictly worse outcome than the semantic mapping above. Documenting the mapping in-component preserves every locked display label (right-hand side is byte-exact) while honoring the AND-semantics rule for absent fields.

   **Forward-safety:** if a future WP adds `victoryPoints` / `recruiterText` / `attackerText` / `abilityText` (singular) to `FlatCard`, the template's `v-if` guards already read those names... actually no, the current template reads the FlatCard-actual names (`cardType`, `setName`, `hc`, `abilities`). A future WP that surfaces new fields on FlatCard would need to add new `<template v-if>` blocks for those specific new property names — the template does NOT speculatively read `card.victoryPoints` etc., to avoid TypeScript errors. This is documented in the component's top-level comment so a future author knows the mapping table.

2. **Designer choice — omit vs em-dash:** Per EC-066 §Locked Values "Empty/null fields rendered as em-dash (—) or omitted entirely (designer choice — pick one consistently; document which in the component's top-level comment)", I chose **OMIT entirely**. Rationale (documented in `CardDataDisplay.vue` JSDoc): cleaner printable output on cards that only carry a subset of attributes (e.g., schemes have no cost/attack/recruit/team; bystanders have almost no stats). Em-dashes would clutter the printable data view with repeated `—` rows on most card types.

3. **Composable architecture — module-scoped singleton:** I chose a module-scoped `ref` (matching `useLightbox.ts` precedent) rather than a factory that returns fresh state per call. This means all consumers of `useCardViewMode()` share a single source of truth across the app — `ViewModeToggle.vue` and `App.vue` see identical state without prop-passing the toggle itself (only the `viewMode` value is prop-passed to `CardDetail.vue` for cleaner declarative rendering). This matches the WP-066 §Implementation Notes "Composable Option" example structure and keeps the cross-component flip atomic.

4. **Module-scoped init → `onMounted` equivalence:** Because `useCardViewMode.ts` reads `localStorage` at module-import time (module-scoped constants run when the module is first loaded), the "read on mount" requirement from EC-066 §Required `// why:` Comments is satisfied by the module's top-level code running during Vue's `<script setup>` phase of `App.vue`. The `App.vue` `// why:` comment (lines 28-34 of the edited file) documents this equivalence for a future reader: the localStorage read happens before the first render, the user preference persists across reloads, no explicit `onMounted` block is needed in `App.vue` for this feature.

5. **No `setViewMode` on public API:** Per copilot-check §External Review Disposition Log (PS-4) and architecture.md §"Prohibited AI Failure Patterns" (no API without callers), `setViewMode` is deliberately not exposed. `useCardViewMode()` returns exactly `{ viewMode, toggleViewMode }` — no more. Documented in the composable's top-level JSDoc.

6. **Lint delta analysis (10 new cosmetic warnings):** The 184 - 174 = 10 new warnings are all in `CardDataDisplay.vue` and belong to two rule families: `vue/singleline-html-element-content-newline` (multi-row) and they match the same cosmetic families that EC-102's rule-disposition explicitly declined to fix. None are semantic. Consistent with baseline style in `CardGrid.vue`, `ThemeGrid.vue`, `ImageLightbox.vue`, `HealthPanel.vue`.
