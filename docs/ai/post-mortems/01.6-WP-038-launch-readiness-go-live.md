## Section 8 — Reality-Reconciliation Findings

Three reality-reconciliation items surfaced during Verification Step execution and were resolved in-session before Commit A. None affected governance content; all resolved without scope expansion.

### 8.1 "Mostly ready" tripped Verification Step 5 in meta-prose

The original `LAUNCH_READINESS.md` opening paragraph used the literal phrase `"mostly ready,"` inside meta-prose stating that no such state exists for the launch readiness review. Verification Step 5 (`grep -iE "looks good|looks great|mostly ready|good enough|should be fine|probably"`) matches against any occurrence of the forbidden tokens regardless of meta-prose context. The grep does not distinguish "the document forbids this phrase" from "the document uses this phrase."

**Resolution.** The opening paragraph was rephrased to use `partial-readiness state` instead of the forbidden literal `mostly ready` quoted form. The semantic intent (no partial state in the launch readiness review) is preserved; the forbidden literal is no longer present. Verification Step 5 now returns zero matches.

**Lesson (feeds Section 9).** When a verification step uses a loosely-scoped regex against forbidden subjective language, document prose that mentions the forbidden phrases — even to forbid them — must paraphrase rather than quote verbatim. The same lesson surfaced in WP-037 §10 (JSDoc module headers in pure-type files should reference D-entries rather than restate forbidden tokens). The pattern generalizes: any prose constrained by a verbatim grep must paraphrase the forbidden surface.

### 8.2 Four-category headings restructure to satisfy Verification Step 4

The original `LAUNCH_READINESS.md` placed the four readiness-gate categories as `### 1.1 / 1.2 / 1.3 / 1.4` subsections under a wrapping `## 1. Readiness Gate Categories` heading. Verification Step 4 (`grep -E "^## .*(Engine|Determinism|Content|Balance|Beta Exit|Ops|Deployment)"`) requires the four category headings to be at level-2 (`^## `) so the regex anchors match. Subsection headings at level-3 (`### `) do not satisfy the anchor.

**Resolution.** The four categories were promoted to top-level `## ` headings (`## 1. Engine & Determinism Readiness Gates`, `## 2. Content & Balance Readiness Gates`, `## 3. Beta Exit Criteria Readiness Gates`, `## 4. Ops & Deployment Readiness Gates`). The wrapping `## 1. Readiness Gate Categories` heading was absorbed into §1's introductory paragraph. Subsequent sections were renumbered: §2 → §5 (Single Launch Authority Model), §3 → §6 (Final GO / NO-GO Verdict Summary), §4 → §7 (Relationship to Companion Documents), §5 → §8 (What This Document Is Not). Internal cross-references (`§2.1` / `§2.2` / `§2.3` / `§2.4` / `§1.2` / `§1.3` / `§1.4`) were updated to the new numbering (`§5.1` / `§5.2` / `§5.3` / `§5.4` / `§2` / `§3` / `§4`). Per-category verdict labels were updated from `Category 1.1 / 1.2 / 1.3 / 1.4` to `Category 1 / 2 / 3 / 4`. Per-gate IDs were updated from `1.1.1 / 1.1.2 / ...` to `1.1 / 1.2 / ...` for Category 1 and similarly for Categories 2 / 3 / 4.

Verification Step 4 now returns the expected four category headings, and the `^## ` count is 8 (≥ 4 as required).

**Lesson (feeds Section 9).** When a verification step constrains the document's heading hierarchy (e.g., level-2 anchor in a regex), the document should plan its heading hierarchy around the constraint from the start. Wrapping section headings around the gated set adds nesting that breaks the anchor.

### 8.3 Rollback triggers needed verbatim lowercase form for Verification Step 8

The original `LAUNCH_DAY.md` §5.6 listed the four rollback triggers as bullet items with capitalized labels (`**Invariant violation spike.**`, `**Replay hash divergence.**`, `**Migration failure.**`, `**Client desync.**`). Verification Step 8 (`grep -E "invariant violation spike|replay hash divergence|migration failure|client desync"`) is case-sensitive and requires the verbatim lowercase form to match.

**Resolution.** A lead-in sentence was added to §5.6 that names all four triggers verbatim in lowercase: "The following four conditions are **rollback triggers**, named verbatim: invariant violation spike, replay hash divergence, migration failure, client desync." **Lesson (feeds Section 9).** Case-sensitive verification greps require the document to include the matched phrases in their verbatim form somewhere, regardless of the natural-English capitalization the document might otherwise use. Adding a lead-in verbatim sentence is cleaner than down-casing the labels themselves.

## Section 9 — Lessons Learned (Feeding 01.4 Template)

1. **Loosely-scoped verification greps require paraphrased meta-prose.** When a verification step issues a regex without anchoring to syntactic context (e.g., the subjective-language grep in WP-038 Step 5, or the boardgame.io-import grep in WP-037 Step 5b), document prose that *mentions* the forbidden surface must paraphrase rather than quote verbatim. Two consecutive WPs (WP-037 §10, WP-038 §8.1) have surfaced the same lesson in different forms; future WPs that author governance prose under a verbatim-grep gate should treat paraphrase as the default and quote verbatim only when verification permits.
2. **Heading-hierarchy verification constrains document structure.** When a verification step regex anchors on a specific heading depth (e.g., `^## ` for level-2), the document's heading hierarchy must be planned around the anchor from the start. Wrapping section headings around a gated set breaks the anchor and forces a mid-execution restructure (WP-038 §8.2). Future strategy-doc-pair WPs should review the verification step's heading regex before drafting.
3. **Case-sensitive verification greps need verbatim phrase mirrors.** When a verification step is case-sensitive against named tokens (e.g., the rollback-triggers grep in WP-038 Step 8), the document must include the verbatim lowercase form somewhere even if natural-English capitalization is otherwise preferred (WP-038 §8.3). A lead-in verbatim list sentence is the cleanest pattern.
4. **The strategy-document-pair pattern continues to scale.** WP-038 is the second consecutive WP to reuse the WP-037 strategy-document-pair template (strategy doc + binary-gate doc). The pattern produces auditable, re-runnable governance with explicit source signals. WP-039 / WP-040 should default to this pattern.
5. **The §18 prose-vs-grep discipline (codified at `f3e59b2`) is operationally useful.** Both launch documents cite D-entries for engine-determinism requirements rather than enumerating literal tokens (`Math.random` / `Date.now` / `performance.now` / `new Date()`). Verification Step 10 returns zero matches, confirming the discipline holds. The §18 codification — which landed three commits before WP-038's content authoring — caught the discipline at draft time and prevented a Final Gate row 33 FAIL.
6. **Pre-landing the SPEC pre-flight bundle as Commit A0 isolates governance from content.** WP-038 used the established A0 (SPEC pre-flight bundle) → A (EC content) → B (SPEC governance close) three-commit topology. Pre-landing the pre-flight bundle separately makes the readiness review auditable independently of the content commit, and lets the content commit's diff stay focused on the seven allowlisted files. The pattern is now validated across multiple WPs (WP-035, WP-036, WP-037, WP-082, WP-083, WP-084, WP-085, WP-038); future WPs should default to it.
7. **Documentation-only WPs are still 01.6 MANDATORY when they introduce new long-lived abstractions.** The two new launch documents become the canonical launch-readiness surface for the project; both will be referenced indefinitely by subsequent launch cycles and by WP-039 onward. The 01.6 trigger fires for new abstractions regardless of whether they are code or documentation.
