## 4. Representation & Determinism Audit

- [x] **New representations are data-only and JSON-serializable:** the
      only "representations" introduced are `ReplayFixtureName` (a
      string union) and the fixture JSON triplet. `ReplaySnapshotSequence`
      is the engine-owned shape; the client consumes it unchanged.
      No functions, closures, Maps, Sets, Dates, or classes stored in
      any new persisted shape.
- [x] **Execution is deterministic:**
  - `parseReplayJson(raw, source?)` is a pure string -> object function.
    Same input, same output. No I/O, no wall-clock, no cache.
  - `loadReplayFixture('three-turn-sample')` returns the same parsed
    object reference every call. The parsing happens once at module
    load — deterministic by virtue of being a pure function over a
    bundled JSON literal.
  - `<ReplayInspector />` navigation is deterministic: given the same
    `sequence`, `initialIndex`, and the same sequence of user inputs
    (clicks, key events, range scrubs), the rendered position label
    and the store's snapshot reference are stable across runs.
  - **Fixture byte-determinism:** committed `three-turn-sample.json`
    and a fresh regeneration via the committed `.cmd.txt` are
    byte-identical. Verified twice during execution
    (post-initial-generation diff `BYTE_IDENTICAL_OK`; final
    pre-commit diff `STEP10_BYTE_IDENTICAL_OK`).
- [x] **Unknown / future values fail safely:**
  - Unknown `version` values throw the locked D-6303 template. The
    inspector never renders a partially-incompatible newer format.
  - Missing / empty `snapshots` throw locked templates.
  - Stepping past a sequence boundary clamps (no-op), never wraps,
    never throws (verified by the `clamps does not wrap` tests for
    both clicks and keyboard).
  - Unknown keyboard keys are ignored by the `default` arm of
    `onKeyDown`'s switch (no preventDefault, no state change).

**Determinism test coverage:** the determinism-by-output property is
enforced by the deterministic store-update assertions and the fixture
regeneration check, not a separate `innerHTML` snapshot. Per WP-064
§Determinism AC, snapshotting raw `innerHTML` is forbidden because Vue
internal markers and attribute ordering are not part of the
determinism contract; visible textContent + a11y tree (which is what
the existing tests assert via `text()` and `attributes()`) is.

## 5. Mutation, Aliasing, & Reference Safety (High-Risk)

This was the highest-risk section per the WP-028 precedent. Every
returned array or object was traced to its source.

### Audit per file

**`loadReplay.ts`** — pure parser. Returns `candidate as unknown as
ReplaySnapshotSequence`, where `candidate` is the JSON.parse result.
The returned object is a fresh JSON.parse output (new object), not a
reference to anything in `G` or to a shared fixture. Safe.

**`fixtures/replay/index.ts`** — `loadReplayFixture` returns the
**same** `threeTurnSample` reference on every call (parsed once at
module load). This is shared across all consumers in a process. The
risk: if any consumer mutated `threeTurnSample.snapshots[i].log` or
similar, every other consumer would see the mutation. Mitigation:
the type is `ReplaySnapshotSequence` (all `readonly` fields per
WP-063's contract), and consumers respect `readonly`. **Fix
applied during post-mortem (see §10):** `<ReplayInspector />`'s
`currentLog` computed now spread-copies `snapshot.log` before passing
to `<GameLogPanel />`, so a future child-component mutation cannot
corrupt the canonical fixture.

**`GameLogPanel.vue`** — receives `log: readonly string[]` as a prop.
Iterates with `v-for` and renders each entry as text. Performs zero
mutations (no `push`, no `splice`, no assignment to indices). Test
"GameLogPanel does not mutate the supplied log array through render"
explicitly proves this.

**`ReplayInspector.vue`** — local refs `currentIndex` and `isPlaying`
are component-scoped (Vue `ref`s, not promoted to the store).
`applyCurrent()` calls `store.setSnapshot(props.sequence.snapshots[i])`
— passes the original snapshot reference into the store. Pinia wraps
it in a reactive proxy on storage. The component never writes back to
`store.snapshot` (only reads via the proxy). The component never
writes to `props.sequence` either. Safe.

**`ReplayFileLoader.vue`** — `errorMessage` is a local ref. Emits
`loaded` with the freshly-parsed sequence (a new object each call).
No mutation of any input.

### Aliasing-test coverage

- [x] GameLogPanel non-mutation: explicit test
- [x] ArenaHud-style "snapshot not mutated through render cycle": not
      added for the inspector itself because the inspector legitimately
      writes to the store as its primary side effect, but the
      embedded GameLogPanel (the rendering surface) carries the
      non-mutation guarantee
- [x] Spread-copy on `currentLog`: defensive aliasing fix applied
      mid-post-mortem (see §10 Fixes)

### `// why:` comment on the post-mortem fix

The aliasing-fix `// why:` comment in `ReplayInspector.vue` cites
WP-028 / D-2802 by precedent — the same defensive-copy lesson for
projections that read from mutable shared state.

**Verdict:** clean. One in-allowlist refinement applied; no remaining

## 6. Hidden-Coupling Detection

- [x] **No engine internals exposed:** the only engine reach is the
      type import `ReplaySnapshotSequence`. The client does not depend
      on any internal engine helper, constant, or move name.
- [x] **No enum / union widened unintentionally:**
      `ReplayFixtureName = 'three-turn-sample'` is a fresh single-
      member union; future fixtures append to it. No engine union
      widened.
- [x] **No implicit knowledge of upstream implementation details:** the
      consumer-side D-6303 assertion (`version === 1`) is the
      documented contract, not an inferred property. The three locked
      error templates mirror the WP-063 CLI wording at
      `apps/replay-producer/src/cli.ts:83-94` verbatim — operator
      tooling produced by producer (stderr) and consumer (in-browser
      alert region) agree on diagnostic phrasing per D-6303.
- [x] **No ordering assumptions undocumented:** the inspector iterates
      `sequence.snapshots` by array index (0..length-1). The WP-063
      contract guarantees `snapshots[0]` is post-setup and
      `snapshots[i>0]` is the state after the i-th move. The inspector
      does not depend on any intra-snapshot ordering of fields.
- [x] **No dependency on non-exported functions:** the only engine
      symbol consumed is `ReplaySnapshotSequence` (type), which is
      exported from `packages/game-engine/src/index.ts` at the
      `// Replay snapshot sequence (WP-063)` block.

### Hidden-coupling finding (non-blocking)

**JSON import widening forced a runtime-validation re-route in
`fixtures/replay/index.ts`.** TypeScript imports JSON with the
`version` field widened to `number` (not literal `1`), so
`satisfies ReplaySnapshotSequence` rejects the import at compile time
because the contract types `version: 1` as a literal. The cleanest
fix would have been a `as ReplaySnapshotSequence` cast (loses the
satisfies-style structural check) or a static assertion via
`as unknown as ReplaySnapshotSequence`. Instead the executor routed
the import through `parseReplayJson(JSON.stringify(threeTurnSampleJson))`,
which gives belt-and-braces runtime validation but adds a redundant
encode/decode pair. The trade-off is documented in the
`fixtures/replay/index.ts` `// why:` block. Non-blocking but worth
noting for any future fixture that wants to use `satisfies` on a
JSON import: either widen the engine type's `version` from `1` to
`number` (would require a D-6303 amendment), or accept the
re-parse pattern locked here.

## 9. Forward-Safety Questions

- [x] **Can this code survive future refactors without touching
      unrelated layers?** Yes. The only outward dependency is the
      engine type `ReplaySnapshotSequence`; engine refactors that
      preserve this type leave the client untouched. The Pinia store
      contract is stable (set by WP-061 and untouched here).
- [x] **Can replay/debugging reconstruct behavior from stored data?**
      Yes. Every visible state is derived from a `ReplaySnapshotSequence`
      + a current index. Every interaction (click, keypress, scrub)
      is observable via the store snapshot and the position label.
      Bug reports name the fixture and the index — exactly the
      Diagnostics & Debuggability invariant from WP-064 §Debuggability.
- [x] **Would removing upstream data still fail safely?** Yes. If a
      future producer emits an empty snapshot, the inspector renders
      the empty `role="status"` log region. If a future producer emits
      `version: 2`, the parser throws the locked template and the file
      loader renders a `role="alert"` region with the message — no
      silent partial render.
- [x] **Is it impossible for this WP's output to influence gameplay
      unintentionally?** Yes. The client never writes to engine state
      (engine is not even imported at runtime). The only side effect
      is `useUiStateStore().setSnapshot`, which the WP-061 contract
      explicitly authorizes for any client surface and which the
      engine cannot observe.
- [x] **Is the contract stable enough to be referenced by future WPs?**
      Yes. `parseReplayJson(raw, source?)` is the canonical
      consumer-side D-6303 site; future replay consumers (spectator
      surface, shared-match replay, export tools) inherit it directly
      or adopt the same three-template wording. D-64NN locks the
      keyboard focus pattern for all future stepper components.
