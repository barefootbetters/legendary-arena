## Section 3 — Five Mandatory Post-Mortem Items

### 3.1 Aliasing Check (RS-14 + WP-028/029 precedent)

`AIPolicy.decideTurn` receives a `UIState` produced by `filterUIStateForAudience(buildUIState(G, ctx), { kind: 'player', playerId })`. Both `buildUIState` and `filterUIStateForAudience` already perform defensive spread copies at their projection boundaries:

- `buildUIState` at `uiState.build.ts:207` spreads `handCards: [...zones.hand]` so the projection never aliases `G.playerZones[playerId].hand`.
- `filterUIStateForAudience` at `uiState.filter.ts:55-72` and `uiState.filter.ts:135-150` rebuilds every nested structure (`players`, `city.spaces`, `hq.slots`, `mastermind`, `scheme`, `economy`, `log`, `progress`, `gameOver`) as fresh spread copies.

Simulation-side guarantees layered on top:

- `simulation.runner.ts :: simulateOneGame` calls `buildUIState` → `filterUIStateForAudience` on every turn; the `filteredState` passed into `decideTurn` is therefore always a fresh projection the policy cannot mutate to affect `G`.
- `simulation.runner.ts :: shuffleWithPrng` always copies via `[...deck]` before Fisher-Yates in-place — the input array is never mutated.
- `simulation.runner.ts :: aggregateOutcomes` iterates `perGame` with `for...of` and accumulates primitives only; no outbound reference is retained.
- `createRandomPolicy` constructs a new `ClientTurnIntent` object on every `decideTurn` call — no aliasing through a memoized result.

**Conclusion:** no aliasing path exists from `AIPolicy` back into `G`. The spread-copy discipline at the projection boundary (WP-028/029) combined with the policy's inability to import engine internals (D-3601) makes mutation-via-alias structurally impossible.

### 3.2 Extension-Seam Check

`AIPolicy` exposes exactly two surface members — `name: string` and `decideTurn(playerView, legalMoves): ClientTurnIntent`. `LegalMove` exposes exactly two — `name: string` and `args: unknown`. The surface is policy-agnostic by construction:

- A **heuristic policy** (future WP-037) adds its own internal scoring function, closes over a pre-sorted priority list, and returns the best `LegalMove`. No contract change.
- An **MCTS policy** (future WP-040) opens a speculative tree over `LegalMove` nodes, uses the same filtered `UIState` as its world model, and returns the simulation-chosen root-child. No contract change.
- A **neural policy** (future WP-041) runs an inference pass over a featurized `UIState` and returns the argmax `LegalMove`. No contract change.

Three deliberate affordances:

1. `AIPolicy.name` is free-form metadata (not drift-pinned, not a closed union) so policy authors can name their policies freely for logging and leaderboard attribution.
2. `LegalMove.args` is `unknown` (deliberate wide type) — mirrors `ClientTurnIntent.move.args` so the enumerator and the submission type are compatible without narrowing casts. Future move authors can add richer args payloads without forcing every policy to update.
3. `SimulationConfig.policies: AIPolicy[]` is indexed by player seat — a symmetric N-player match can use `[policyA, policyA]`, a mirror match can use `[policyA, policyB]`, and a policy-vs-random benchmark can use `[heuristic, createRandomPolicy('baseline')]`. The simulation runner never inspects the policy type; it dispatches by seat index.

**Conclusion:** the contract surface is open-ended. Adding a heuristic / MCTS / neural policy requires a new `ai.heuristic.ts` / `ai.mcts.ts` / `ai.neural.ts` file under the same D-3601 classification and no contract amendment.

### 3.3 Capability-Gap Documentation (D-2704 + D-3604)

The simulation subsystem requires a seeded PRNG, but the shared test helper `makeMockCtx` is not seeded — it reverse-shuffles arrays (see `test/mockCtx.ts:35-41`). Per D-2704 (PRNG Capability Gap), simulation implements its own mulberry32 instance locally. The implementation choice is locked in D-3604 (Simulation Seed Reproducibility: Two Independent PRNG Domains):

- **djb2 seed hashing** — tiny, deterministic, no crypto dependency, no third-party surface. ~8 lines.
- **mulberry32 PRNG** — same seed + same call sequence = same output sequence, across Node versions. ~10 lines.
- **Two independent domains** — the run-level shuffle PRNG (`runSimulation`) and the policy-level decision PRNG (`createRandomPolicy`) never share state. Tests can reseed one domain without perturbing the other.

The implementation is duplicated across `ai.random.ts` and `simulation.runner.ts` rather than shared via a fifth helper file. Per WP-036 Scope Lock (4 simulation files + 1 test file), introducing a fifth file would require a WP amendment; duplicating ~20 lines of PRNG plumbing is cheaper than the governance overhead. If a third simulation subsystem ever needs the same PRNG, the de-dup becomes a follow-up WP.

**Conclusion:** capability gap documented in D-3604; duplication is intentional and bounded by the Scope Lock.

### 3.4 Forbidden-Behaviors Docstring Block (WP-028 precedent)

`AIPolicy.decideTurn` JSDoc in `ai.types.ts` explicitly enumerates forbidden behaviors:

```
Forbidden behaviors (permanent, not just at MVP):
- Caching or memoizing decisions across calls.
- Closing over G, ctx, or any engine internal.
- Retaining state between calls that would affect the next call's output.
- Reading or writing any external resource (filesystem, network, clock).
```

The phrasing matches the WP-028 projection-purity precedent on `buildUIState` (`uiState.build.ts:162-168` "Forbidden behaviors (do not add later)"). The docstring carries the same "permanent" framing — these are not MVP-only restrictions; they are contractual guarantees of the AI interface.

Mechanical enforcement is layered on top:

- `ai.types.ts` imports only type declarations (`MatchSetupConfig`, `UIState`, `ClientTurnIntent`) — no runtime values, no engine internals. A policy file that follows the import style of `ai.random.ts` physically cannot close over engine internals.
- `decideTurn`'s signature accepts only `(playerView: UIState, legalMoves: LegalMove[])` and returns `ClientTurnIntent`. The policy has no access to `G`, `ctx`, or the registry by construction.

**Conclusion:** forbidden-behaviors block is in place, consistent with WP-028 precedent, reinforced by the import surface.

### 5.3 Session prompt's `cardIndex` vs. real `playCard` `cardId`

Agents) produce two `LegalMove` entries with identical args — consistent with RS-13's "hand zone index ascending, one LegalMove per hand card" intent. No amendment needed — this is a session-prompt pseudocode illustration, with the binding rule "copy the real move args shape" already in the Hard Stops section.

### 5.4 Session prompt's `gameOver.outcome === 'victory'`

The session prompt's RS-12 statistics sourcing referenced `gameOver?.outcome === 'victory'` as the win-count predicate. The actual `evaluateEndgame` outcome literal for a heroes victory is `'heroes-win'` (with `'scheme-wins'` for the villain-side loss). Implementation uses `postEndgameUi.gameOver?.outcome === 'heroes-win'`. The session prompt's `'victory'` was a schema misremembering — the `EndgameOutcome` literal union is the authoritative surface. No amendment needed — the session prompt's free-text description of the predicate is subordinate to the real type.
