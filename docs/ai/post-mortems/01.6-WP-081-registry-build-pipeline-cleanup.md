## Section 1 — Overview

WP-081 is a purely **subtractive** doc-hygiene packet. It resolves the
pre-existing registry build failure that has haunted every monorepo
build since WP-003 landed. The failure surfaced as `pnpm --filter
@legendary-arena/registry build` exiting 1 with
`dist/cards.json not found — run 'pnpm normalize' first.` at the
post-tsc `node scripts/build-dist.mjs` step. The root cause was not a
missing file: the entire `normalize-cards → build-dist →
standardize-images` pipeline was dead code referencing Zod schema
symbols (`CardSchema`, `CardIDSchema`, `CANONICAL_ID_REGEX`,
`CardTypeSchema`) that WP-003 removed and producing JSON artifacts
(`dist/cards.json`, `dist/index.json`, `dist/sets.json`,
`dist/keywords.json`, `dist/registry-info.json`, plus
`dist/image-manifest.json` from `standardize-images.ts`) that no
monorepo consumer reads. The registry runtime path is
`metadata/sets.json` + per-set `metadata/{abbr}.json` fetched directly
from R2 by `httpRegistry.ts` / `localRegistry.ts`.

The fix: delete the three scripts, trim `scripts.build` in
`packages/registry/package.json` to `"tsc -p tsconfig.build.json"`
only, remove the redundant `"Normalize cards"` step from
`.github/workflows/ci.yml` job `build`, and rewrite six anchor regions
in `README.md` plus the stale "Legacy Scripts (Retained for
Reference)" subsection in `docs/03-DATA-PIPELINE.md`. Zero engine
changes, zero new code, zero new tests, zero new dependencies, zero
`packages/registry/src/**` diff, zero `pnpm-lock.yaml` diff, zero
`version` bump.

After WP-081, `pnpm --filter @legendary-arena/registry build` exits 0
for the first time since WP-003 landed, and `pnpm -r build` exits 0
end-to-end.

## Section 17 — Lessons Learned (for future WP-class-doc-hygiene packets)

1. **Pre-flight audit grep must match the Acceptance Criterion pattern
   set.** PS-1 audited only `cards\.json`, but the AC promised "no
   precomputed registry JSON artifact other than `dist/registry-health.json`".
   The mismatch caused PS-2 (`image-manifest.json`) and PS-3
   (`Legacy Scripts` subsection) to surface mid-execution. Future
   doc-hygiene packets should pre-compute the full AC-required grep
   pattern set during pre-flight and run every pattern across every
   targeted doc directory, not just the primary doc.
2. **Docs outside the primary target directory are a blind spot.**
   PS-3's `docs/03-DATA-PIPELINE.md` was top-level documentation,
   not under `docs/ai/**`. Future pre-flight greps for doc-hygiene
   packets should scope to `. --exclude-dir=node_modules
   --exclude-dir=.git --exclude-dir=dist --exclude-dir=.claude` from
   the start, not to specific doc paths.
3. **Session invocations must explicitly name 01.6 + PRE-COMMIT-REVIEW
   gates between verification and commit.** Omission led to both
   gates being skipped in this session. A session invocation cannot
   override governance; if a session invocation does not name a gate,
   the gate still applies and the executor must apply it from the
   governance source. In practice, explicit naming reduces the
   probability of the gate being overlooked to near-zero.
4. **Three-commit topology (pre-execution SPEC + EC execution + SPEC
   close) scales to four commits when mid-session findings trigger
   pre-execution amendments.** The WP-081 topology became
   PS-2 SPEC → PS-3 SPEC → EC-081 → SPEC close (four commits). The
   pattern is a generalization of the PS-1 precedent (single SPEC
   amendment). Future pre-execution findings should follow the same
   SPEC-commit-before-execution pattern rather than bundling
   amendments into the execution commit — this preserves a clean
   governance trail and keeps the execution commit minimal.
5. **Subtractive WPs benefit from running `git diff --stat` on the
   invariant directories at every gate.** `git diff --stat
   packages/registry/src/` was empty at every verification point in
   this session. The same discipline should apply to `apps/server/`,
   `apps/registry-viewer/`, and any other explicitly-OOS directory.
