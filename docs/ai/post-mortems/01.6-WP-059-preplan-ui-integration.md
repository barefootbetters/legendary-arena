## Section 1 — Overview (Summary)

WP-059 lands the **first runtime consumer of `@legendary-arena/preplan` from the arena client**. Before this packet, the preplan package was complete (WP-056 / WP-057 / WP-058) but had no client. The arena client could host `<ArenaHud />` against `useUiStateStore()` (WP-061..WP-064) but had no surface for client-local advisory state. WP-059 ships:

- A new Pinia store `usePreplanStore` (id `'preplan'`) with two state fields (`current: PrePlan | null`, `lastNotification: DisruptionNotification | null`), five actions (`startPlan`, `consumePlan`, `recordDisruption`, `dismissNotification`, `clearPlan`), and one getter (`isActive`).
- Two pure adapter functions in `apps/arena-client/src/preplan/preplanLifecycle.ts` — `startPrePlanForActiveViewer` (wraps `createPrePlan` and installs the result via `store.startPlan`) and `applyDisruptionToStore` (forwards a `DisruptionPipelineResult` to `store.recordDisruption`). The lifecycle file exports two runtime symbols and **no types** — the v1 `PrePlanContext` shape was dropped after pre-flight CV-1 verified `createPrePlan` takes three positional scalars, not a context object.
- Two Vue 3 SFCs — `<PrePlanNotification />` (alert banner with `role="alert"` + `aria-live="assertive"` + dismiss button) and `<PrePlanStepList />` (passive plan-step display with the empty-state literal `"No plan is active."` and a terminal-state status paragraph for `'consumed'` / `'invalidated'`).
- Six named fixtures under `apps/arena-client/src/fixtures/preplan/index.ts` — three `PrePlan` variants (active / consumed / invalidated), two `DisruptionPipelineResult` variants (no-card / with-card), and one `PlayerStateSnapshot` for the lifecycle adapter test.
- Three drift-detection tests in `apps/arena-client/src/fixtures/preplan/index.test.ts` — `PREPLAN_STATUS_VALUES` shape (3 members in canonical order `['active', 'invalidated', 'consumed']`) plus `DisruptionNotification` field-set assertions in both no-card and with-card variants.
- One compile-time drift sentinel at the top of `preplanLifecycle.test.ts` — bidirectional tuple-equality `extends` proof against `Parameters<typeof createPrePlan>` that catches any add/remove/rename/reorder of the three positional parameters at typecheck time.
- A workspace dep promotion: `@legendary-arena/preplan: "workspace:*"` added to `apps/arena-client/package.json` `dependencies` (was absent).
- Layer-boundary carve-out **D-5901**: ARCHITECTURE.md + `.claude/rules/architecture.md` updated in lockstep to remove `preplan` from `apps/arena-client`'s "Must NOT import" column and add `@legendary-arena/preplan (runtime — per D-5901)` to the "May import" column. The carve-out is confined to the arena client; no other app or package gains the runtime-import right.

The engine, registry, preplan, vue-sfc-loader, server, and registry-viewer packages are entirely untouched.

- **The v1 → v2 SPEC bundle paid off immediately.** Pre-flight v1 caught six contract drifts (CV-1..CV-6) by reading `packages/preplan/src/*` directly rather than trusting the WP draft. The v2 spec rewrite folded the corrections; execution then implemented against accurate field names, signatures, and array orders. Zero compile errors, zero test failures, zero drift discovered during execution. The cost of the rewrite (one SPEC commit) saved six compile-fail ripples plus the cost of a stale WP shipping out of sync with the package.
- **Compile-time drift sentinel locks `createPrePlan`'s signature.** The bidirectional `Parameters<typeof createPrePlan> extends [PlayerStateSnapshot, string, number]` proof at the top of `preplanLifecycle.test.ts` is a zero-runtime-cost guard that fails typecheck if WP-057's signature drifts. Mirrors the proven `preplanStatus.ts:25-31` pattern.
- **The §J drift sentinels protect the `DisruptionNotification` field-set in both variants.** Two separate `Object.keys(...).sort().join(',')` assertions — one for the no-card fixture (4 fields), one for the with-card fixture (5 fields) — make a renamed / added / removed field surface as a fixture-load-time failure rather than a component-mount-time crash. The `PREPLAN_STATUS_VALUES` length + order + literal-fixture-key-set assertion completes the drift triplet.
- **`<script setup>` vs `defineComponent({ setup() })` discipline held.** Both new components reference store data in their templates (not just props), so per D-6512 the explicit `defineComponent({ setup() { return {...} } })` form was required. The WP draft suggested `<script setup lang="ts">`, but the actual vue-sfc-loader pipeline (separate-compile under `node:test`, not the inlined-template Vite path) does not expose `<script setup>` top-level bindings on the template's `_ctx`. Using the explicit form matched the BootstrapProbe / PlayerPanel / ArenaHud / LobbyView precedent and produced templates that render store data correctly under both Vite and `node:test`.
- **Pinia test isolation pattern internalized from session start.** Every store-instantiating test creates a fresh Pinia root in a `beforeEach(() => setActivePinia(createPinia()))` hook. Zero nondeterministic store test failures.
- **The `recordDisruption` no-double-invalidation discipline held.** Per CV-2 / WP §A, the store reads `result.invalidatedPlan` directly rather than calling `invalidatePrePlan(current, reason)` itself — the pipeline already produced the invalidated plan with full nested copies. The `// why:` block on `recordDisruption` cites `disruptionPipeline.ts:42-82` so a future maintainer cannot accidentally reintroduce the v1 double-invalidation pattern without first reading the source.
- **Privacy invariant verified by spy.** `preplanLifecycle.test.ts §D #6` monkey-patches `store.$subscribe` with a counter, calls `startPrePlanForActiveViewer`, and asserts `subscribeCallCount === 0`. Production code under `apps/arena-client/src/preplan/**` and `apps/arena-client/src/components/preplan/**` registers no subscriptions — verified at code level by Verification Step 9's grep with `--exclude='*.test.ts'`. Plan existence is never broadcast.
- **Self-contained `sampleDisruptionResultFixture.invalidatedPlan` literal.** Per the §I lock and the WP-028 `cardKeywords` aliasing precedent (pre-flight v2 NEW-RS-2), the fixture's `invalidatedPlan` field is an inline `PrePlan` literal — deliberately not aliased to `invalidatedPrePlanFixture`. A future refactor of either fixture cannot silently couple the two.

## Section 3 — What Surprised Us

- **`<script setup>` is not a one-size-fits-all default under vue-sfc-loader.** The WP draft (and the EC) named `<script setup lang="ts">` as the form for both new components. Verification against the actual codebase (BootstrapProbe.vue's D-6512 comment, PlayerPanel.vue's identical citation, the PlayerPanelList.vue precedent) confirmed that `<script setup>` works only when the template references nothing other than `defineProps` results. Both new components in WP-059 reference store-derived refs, so the explicit `defineComponent({ setup() { return {...} } })` form was required. This is not a contract drift in the strict sense — both forms render the same `aria-live="assertive"` / `role="alert"` / step-list output — but the WP's nominal form would have produced templates rendering against an undefined `_ctx` proxy under `node:test`. Lesson 7.5 from session-context-wp059 (verify against actual source) caught this before the first component was written.
- **`as const satisfies <Type>` works fine on the fixture trees despite mutable-array fields in `PrePlan.sandboxState`.** The expected friction (readonly tuple from `as const` not assignable to mutable `CardExtId[]`) didn't surface — TypeScript's contextual typing under `satisfies` resolves the literal narrowing without complaint. The fixture file passes `vue-tsc --noEmit` cleanly with `as const satisfies PrePlan` on every `PrePlan` literal and `as const satisfies DisruptionPipelineResult` on the `DisruptionPipelineResult` literals.
- **`pnpm-lock.yaml` is a mechanical companion to the `package.json` workspace promotion.** Adding `@legendary-arena/preplan: "workspace:*"` to `apps/arena-client/package.json` triggered `pnpm install` to add three lines to the root lockfile — the workspace link entry. The lockfile change is not in the WP §Files Expected to Change allowlist (which lists 16 human-edited files), but it is a strict, mechanically-derived consequence of the package.json edit. Treating it as a scope violation would require excluding it and accepting a stale lockfile, which would diverge from the committed `package.json` on every clone. Included it in Commit A as a companion to `apps/arena-client/package.json`.

## Section 5 — Precedent to Establish

- **Client-local advisory-state pattern.** The WP-059 store (Pinia Options API + two state keys + five actions + one getter + module-header invariant block + per-action `// why:` blocks) is the canonical pattern for any future arena-client feature that needs client-local non-authoritative state. Future features that need to track speculative gestures, draft-mode hand reorderings, or tutorial overlays should follow this shape rather than extending `useUiStateStore` (which is the authoritative engine projection — mixing client-local state into it would violate the Layer Boundary).
- **Lifecycle adapter pattern: thin pass-through with a `// why:` block stating "the value is the name, not the behavior".** `applyDisruptionToStore` is one line of code with a multi-line `// why:` block. The point is to freeze a stable function identifier that future middleware can be written against without restating the assignment idiom at every call site. Any future "single named integration seam between an external producer and a Pinia store action" should follow this shape.
- **Compile-time drift sentinel at the top of an adapter test file.** The bidirectional `Parameters<typeof X> extends [...] ? [...] extends Parameters<typeof X> ? true : never : never` pattern is the canonical way to lock a third-party signature. Future adapters that wrap an external function whose signature changes would drift the tests should add an analogous sentinel at the top of their test file. Mirrors `preplanStatus.ts:25-31`.
- **Field-set drift sentinels in fixture test files.** `Object.keys(fixture.shape).sort().join(',') === 'expected,key,list'` is a fast and high-signal way to lock the field set of an external type. Two variants (with-optional, without-optional) cover both render paths in a single grep. Future fixture modules that mirror an external package's type shape should include analogous drift tests.
- **Self-contained fixture literals (no aliasing).** `sampleDisruptionResultFixture.invalidatedPlan` is an inline `PrePlan` literal, not a reference to `invalidatedPrePlanFixture`. Per the WP-028 `cardKeywords` aliasing precedent (pre-flight v2 NEW-RS-2), aliased fixtures are an aliasing-bug surface — a refactor that mutates one inadvertently mutates the other. Future fixture modules should keep nested fixture trees self-contained even at the cost of duplication.
- **`defineComponent({ setup() { return {...} } })` — not `<script setup>` — for any Vue SFC with template-referenced non-prop bindings under vue-sfc-loader.** D-6512 / P6-30 precedent now extended across BootstrapProbe, PlayerPanel, PlayerPanelList, ArenaHud, ReplayFileLoader, LoadoutBuilder (WP-091), App.vue (WP-090), LobbyView.vue (WP-090), and now PrePlanNotification.vue + PrePlanStepList.vue (WP-059). `<script setup>` is fine for props-only leaf components (GameLogPanel, SharedScoreboard, TurnPhaseBanner); anything that consumes local refs, computeds, or methods from the template must use the explicit form.

## Section 6 — Lessons Learned (Feed Forward to 01.4 §Established Patterns)

- **Lesson 7.5 from session-context-wp059 carried into execution.** The pre-flight already verified `createPrePlan`'s signature against `packages/preplan/src/preplanSandbox.ts:37-41` and the `DisruptionNotification` shape against `disruption.types.ts:57-72`. Execution re-verified these inline at every fixture / template / adapter site by reading the cited source files first, then writing the consuming code. Zero drift discovered. The general lesson: when a WP cites a field name or function signature from a dependency package, the executor's first action on opening that file is to read the cited source line range and verify it matches the WP. The WP draft IS a contract, and the contract MUST match reality.
- **Pre-existing form-precedent overrides spec-suggested form when the runtime constrains it.** WP §G suggested `<script setup lang="ts">` for both components. D-6512 + four working precedents (BootstrapProbe, PlayerPanel, PlayerPanelList, ArenaHud) said the explicit form is required for any template that references store data. Choosing the form by precedent-evidence rather than by spec-text was the right call. Future executors should treat WP form-suggestions as *defaults the WP author thought would work* — when the actual runtime constrains the choice, the runtime wins.
- **Pinia's reactive proxy interacts with `assert.deepStrictEqual` cleanly when the comparison target is a literal.** `assert.deepStrictEqual(store.current, sampleDisruptionResultFixture.invalidatedPlan)` works because `deepStrictEqual` walks structural content rather than checking proxy identity. (`assert.equal(store.current, sample.invalidatedPlan)` would fail because Pinia wraps `store.current` in a proxy.) The WP-090 §Section 3 surprise about reactive proxies foreshadowed this — execution defaulted to `deepStrictEqual` for every store-state assertion and avoided the trap.
- **The `// why:` blocks live where the action happens, not in the JSDoc.** `recordDisruption`'s "do not call invalidatePrePlan twice" rationale is in a multi-line `// why:` block immediately above the action body, not in the JSDoc summary. A future maintainer reading `recordDisruption`'s body sees the rationale at the line they'd be tempted to change. JSDoc summaries are for callers; `// why:` blocks are for editors.

## Section 9 — Mutation, Aliasing, & Reference Safety

- [x] All store mutation occurs only inside the five Pinia actions. No component, fixture, or adapter writes to `store.current` or `store.lastNotification` directly — verified by inspection. Components read via `storeToRefs(store)` (the documented pattern in `defineComponent + setup()`); the dismiss button calls `store.dismissNotification()` rather than touching `lastNotification.value = null`.
- [x] No aliasing risk on `current` mutation. `consumePlan` constructs a new `PrePlan` literal via `{ ...this.current, status: 'consumed' }` rather than mutating in place — preserves JSON serializability and avoids aliasing shared references that may have been captured by components via `storeToRefs`. `recordDisruption` assigns `result.invalidatedPlan` directly; the disruption pipeline (WP-058) already produced a fresh nested copy per `disruptionPipeline.ts:42-82`, so the assignment carries no aliasing risk into the store.
- [x] Fixtures are deeply literal trees (string + number primitives only). The `as const satisfies <Type>` pattern locks the fixture shapes at compile time. No fixture references a runtime function or another fixture; `sampleDisruptionResultFixture.invalidatedPlan` is a self-contained literal per the §I lock (do-not-alias-to-invalidatedPrePlanFixture).
- [x] No engine `G` mutation. The arena client cannot reach `G`; the preplan runtime functions do not touch `G`; the store is entirely client-local. Engine-side replay determinism is preserved by construction (the engine cannot observe the preplan store, and the preplan store cannot observe `G`).

## Section 13 — Forward-Safety Questions

- [x] **Can this code survive future refactors without touching unrelated layers?** Yes. The store is a leaf module that reads only from `@legendary-arena/preplan` types. The lifecycle adapter is two functions whose signatures are locked by the compile-time drift sentinel. The components are leaf SFCs that read only from `usePreplanStore`. None of the new files reference `useUiStateStore`, the engine, the registry, the server, or any other UI surface.
- [x] **Can replay/debugging reconstruct behavior from stored data?** Yes (and N/A in the strict sense — preplan state is intentionally never persisted). The store state is JSON-serializable; the determinism tests assert byte-equal serialization across runs; a future debug-export feature could capture `JSON.stringify(store.$state)` and restore via `JSON.parse(...) → setActivePinia + store.$patch`.
- [x] **Would removing upstream data still fail safely?** Yes. The store starts in `{ current: null, lastNotification: null }` — every action handles the null case explicitly. The components render the empty state on `null` (`<PrePlanStepList />`) or render nothing (`<PrePlanNotification />`). No upstream-data-required assumption.
- [x] **Is it impossible for this WP's output to influence gameplay unintentionally?** Yes. The store is client-local; nothing it does propagates to the engine. The components do not invoke moves; they cannot reach `submitMove`. Plan privacy is preserved by the absence of a broadcast surface (Verification Step 9: no `$subscribe` registration in production code).
- [x] **Is the contract stable enough to be referenced by future WPs?** Yes. The two-state-key / five-action / one-getter shape is locked by §B test #1 (`Object.keys(store.$state).length === 2`) and the per-action tests. The two adapter signatures are locked by the compile-time drift sentinel + the WP-locked args-object shape. The two component DOM shapes are locked by class-selector assertions. Future WPs can write middleware against these surfaces without breaking the WP-059 contract.
