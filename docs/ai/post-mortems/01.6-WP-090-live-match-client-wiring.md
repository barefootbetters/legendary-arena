## Section 2 — What Worked Well

- **Session Protocol live-server verification caught the real CLI-script disagreement.** The two existing scripts (`create-match.mjs` and `join-match.mjs`) disagree on the join endpoint's request body and response field. Rather than choose one and hope, the Session Protocol booted the actual server, POSTed both body shapes, and inspected the raw responses. Outcome: the server accepts `{ playerID, playerName }` and returns `{ playerID, playerCredentials }`. The minimal form `{ playerName }` alone is also accepted (server auto-assigns the next open seat), but `create-match.mjs`'s shape is canonical and `join-match.mjs` is buggy (reads the non-existent `result.credentials`). D-9001 records the evidence and flags the CLI fix as a follow-up WP.
- **Single runtime engine-import site discipline held.** Grep verification (multiline-aware regex filtering for `^import\s+\{[^}]*\}\s+from\s+'@legendary-arena/game-engine'`) returns exactly one match, at `bgioClient.ts:16`. The line-by-line form of the grep (the version in EC-090 §Verification Steps) does surface a false positive on `SharedScoreboard.vue` line 6 — that line is the continuation of a multi-line `import type { ... }` block — but the runtime grep is clean. Documented in §Section 6 so future packets use the multiline form.
- **Split `import type` / `import` for the engine carve-out.** `bgioClient.ts:16` imports `LegendaryGame` via a separate `import { LegendaryGame } from '@legendary-arena/game-engine'` statement, and `UIState` via `import type { UIState } from '@legendary-arena/game-engine'` on the next line. This two-file split matches the FIX-10 pre-flight guidance so a merged `import { type UIState, LegendaryGame }` cannot mask the runtime hit from the grep gate (RS-1 defense).
- **Factory-injection test seam keeps the public surface minimal.** `createLiveClient` has exactly three keys in its return shape. The testing seam (`setClientFactoryForTesting`, `getLiveClientCallLog`, `resetLiveClientCallLog`) lives as *additional* module exports — not on the returned handle — so production callers never see them. App-routing tests use the seam to assert that the fixture path invokes no factory and the live path invokes exactly one with the right args.
- **Route precedence `fixture > live > lobby` plus partial-param fallback.** The fixture short-circuit preserves WP-061's offline/testing UX and the reproducible bug-report mechanism (`?fixture=mid-turn` regresses to the committed snapshot path regardless of any live-route params also in the URL). Missing or empty `match`/`player`/`credentials` fall back to the lobby silently — never a half-mounted live branch with undefined strings propagating into boardgame.io.
- **FIX-22 malformed-frame guard is load-bearing defense, not ceremony.** The subscribe callback rejects any `state.G` that is neither `null`/`undefined` nor an object, emits a one-shot `console.warn`, and coalesces to `null` via `setSnapshot(null)`. The HUD's empty-state render (WP-062) handles `null` cleanly. Without the guard, a malformed server frame would cast a primitive to `UIState` and crash the HUD subtree at `snapshot.game.phase`.
- **LobbyMatchSummary normalization hides the raw server shape from the UI.** The server returns `players: Array<{ id: number, name?: string }>` with `id` as a number and open seats as `{ id: N }` (no `name` key). `listMatches()` normalizes to `{ id: string; name?: string }[]` in exactly one place; callers of `LobbyView.isOpenSeat(seat)` simply check `typeof seat.name !== 'string'`. Gameover is normalized to explicit `null` when absent. The UI never sees the raw boardgame.io shape.

## Section 3 — What Surprised Us

- **`import.meta.env` is undefined under `node:test`.** lobbyApi.ts reads `VITE_SERVER_URL` at module load via `import.meta.env.VITE_SERVER_URL ?? 'http://localhost:8000'`. Under Vite, `import.meta.env` is inlined statically and this works. Under `node --import tsx --test`, `import.meta.env` is the literal `undefined`, and property access throws `TypeError: Cannot read properties of undefined (reading 'VITE_SERVER_URL')` at the top of the file — before a single test executes. Fix: optional chaining (`import.meta.env?.VITE_SERVER_URL`) in both `lobbyApi.ts` and `App.vue` (`import.meta.env?.DEV`). `// why:` comment on both sites.
- **boardgame.io v0.50 proxy-dir imports fail under tsx, AND the obvious CJS-default workaround fails under Vite.** This was a two-stage rabbit hole. Stage 1: `import { Client } from 'boardgame.io/client'` resolves cleanly under Vite (which honors the `module` field in the proxy-dir `package.json`), but tsx's resolver rejects the directory-import path and falls back to trying `index.jsx`. The ESM dist path (`dist/esm/client.js`) loads, but tsx transforms the parent package as CJS (no `"type": "module"` on `boardgame.io`'s root `package.json`) and named imports fail with "does not provide an export named 'Client'". Stage 1 fix: default-import the CJS bundle (`import boardgameioClient from 'boardgame.io/dist/cjs/client.js'`) — works under tsx because Node's CJS→ESM interop wraps `module.exports` and exposes it on the default binding. Stage 2 (caught only during the live browser smoke test, not by tests or production build): under Vite 5, the same default import resolves to `undefined`. Vite's interop shim reads `__esModule: true` on the CJS module (the bundle sets it via `Object.defineProperty(exports, '__esModule', { value: true })`), then exposes the default as `module.exports.default` — but boardgame.io exports `Client` as a *named* property, never sets a `default` property, so the chain collapses to `undefined`. Final fix: use a **namespace import** (`import * as boardgameioClient from 'boardgame.io/dist/cjs/client.js'`) and read through a fallback chain `pkg.Client ?? pkg.default?.Client`. The namespace import gives the full exports object under both runtimes, and the fallback chain handles whichever binding shape the runtime presents. Belt-and-suspenders: a `throw new Error(...)` if neither branch resolves, surfacing the failure with a descriptive message rather than `Cannot destructure property 'Client' of undefined` at the call site.
- **jsdom's `window.location` is non-configurable.** The first draft of App-routing tests used `window.history.replaceState({}, '', '/?match=...')` to drive App.vue's route parsing. jsdom's default URL is `about:blank`, and `replaceState` rejects any relative or HTTP URL against that base with a `SecurityError`. `Object.defineProperty(window, 'location', ...)` also fails with `TypeError: Cannot redefine property: location`. Final fix: App.vue exposes an optional `searchOverride: string | null` prop (default `null`); tests pass the query string directly via `mount(App, { props: { searchOverride: '?match=...' } })`. The prop is a testing seam — production callers never pass it — and the `// why:` comment documents intent.
- **The game-engine barrel pulls in Node-only `par.storage.js`.** Adding the first runtime engine import to arena-client exposed a pre-existing issue: `packages/game-engine/src/index.ts` re-exports `writeSimulationParArtifact` / `readSimulationParArtifact` / etc. from `./simulation/par.storage.js`, which imports `node:crypto`, `node:fs/promises`, and `node:path` at module scope. Vite externalizes `node:*` for browser builds, but its stub does not expose the named bindings (`mkdir`, `access`, …) that `par.storage.js` destructures, so Rollup's pre-bundle analysis fails before tree-shaking can drop the module. Game-engine is off-limits per the session abort conditions. Fix: a narrow Vite plugin (§Section 8) replaces `par.storage`'s contents at load time with an inert stub that advertises the same export names and throws descriptive errors if anything ever calls them. Tree-shaking then elides the stub entirely from the arena-client production bundle. The stub is browser-only; the server package still gets the real `par.storage` from its own build graph.
- **Pinia's reactive snapshot breaks `assert.equal` reference checks.** The subscribe-callback test initially used `assert.equal(store.snapshot, sampleUiState)` (strict equality). Pinia wraps state in a reactive proxy, so `store.snapshot !== sampleUiState` even though they share the same structural content. Fix: `assert.deepEqual` is the right granularity for this contract — the invariant being exercised is "the store received the projection's content", not "no proxy wrapping occurred".

## Section 4 — Fixes Applied During Post-Mortem

- **Vite CJS interop fix for boardgame.io imports (§Section 3 stage 2).** Caught only when the user opened the running dev server in a browser and DevTools surfaced `Uncaught TypeError: Cannot destructure property 'Client' of 'boardgameioClient' as it is undefined`. The earlier verification gates (`pnpm test`, `pnpm build`, `pnpm typecheck`, programmatic smoke test against a running server) all passed because the Vite-transformed module had not yet been *executed in a browser*: the build verifies bundling and tree-shaking but not runtime evaluation, and the programmatic smoke test exercised only the HTTP lobby endpoints, not the WebSocket subscribe path that drives `bgioClient.ts:createLiveClient`. The five-step manual browser smoke test is the gate that surfaced this. Fix lives in [bgioClient.ts](apps/arena-client/src/client/bgioClient.ts) — namespace import + fallback chain + descriptive throw, documented inline.
- **Lesson:** any first-time-runtime-import to a CJS-only third-party SDK in a Vite-bundled app must be smoke-tested in a real browser before commit. `pnpm build` succeeding is necessary but not sufficient; Vite's CJS interop shim has runtime branches the compiler cannot statically verify. Future packets that introduce a new CJS dependency should treat the browser smoke test as the definitive runtime gate, not a courtesy step.

## Section 5 — Precedent to Establish

- **The single runtime engine-import carve-out pattern.** Future arena-client gameplay WPs (WP-092 Lobby Loadout Intake, any WP-09N move-submission UI) must route runtime engine imports through `bgioClient.ts` or the `submitMove` pass-through. A second runtime engine-import site in arena-client would violate the single-site invariant recorded in the EC. If a new site is genuinely required, the burden is on the executor to amend the EC first.
- **Split `import type` + `import` rather than mixed imports for runtime engine carve-outs.** `import { type UIState, LegendaryGame } from '@legendary-arena/game-engine'` and `import { LegendaryGame } from '@legendary-arena/game-engine'; import type { UIState } from '@legendary-arena/game-engine'` compile to identical runtime behavior, but the mixed form can mask a runtime import from a `| grep -v "import type"` filter. Split-form only.
- **Factory-injection test seam for external-service modules.** `bgioClient.ts`'s `setClientFactoryForTesting(factory | null)` + call-log accumulator is the canonical pattern for a module whose default implementation opens a network connection. Any future module that calls into an external SDK at runtime should expose an analogous seam so tests can exercise delegation without real I/O. Export names with an `__` / `setXforTesting` convention to signal "testing-only; do not call from production code".
- **Optional chaining on `import.meta.env` for any file consumed by `node:test`.** `import.meta.env?.FOO` rather than `import.meta.env.FOO`. Otherwise the file blows up at module load in any non-Vite runtime.
- **`defineComponent({ setup() { return {...} } })` — not `<script setup>` — for any Vue SFC with template-referenced non-prop bindings under vue-sfc-loader.** D-6512 / P6-30 precedent held again. `<script setup>` is fine for props-only leaf components; anything that consumes local refs, computed, or methods from the template must use the explicit form. This is now confirmed across `ArenaHud`, `ReplayFileLoader`, `LoadoutBuilder` (WP-091), `App.vue` (WP-090), and `LobbyView.vue` (WP-090).
- **boardgame.io v0.50 interop via namespace import + fallback chain.** Canonical pattern for any future arena-client file that imports from `boardgame.io/*`: `import * as pkg from 'boardgame.io/dist/cjs/<entry>.js'` plus `(pkg as any).Foo ?? pkg.default?.Foo`. Works under Vite 5 (where the `__esModule` interop shim collapses default lookup to undefined for CJS modules that export named bindings without a `default` property), tsx (where CJS→ESM interop wraps `module.exports` under `default`), and pure Node ESM (where namespace import gives a structured exports object). The default-only form is a trap.
- **Stub-plugin pattern in `vite.config.ts` for unavoidable Node-only modules in shared packages.** Dropped when the game-engine barrel re-exports Node-only code and the consuming app is browser-only. The plugin's `load` hook replaces the module body with same-named inert exports. Tree-shaking drops them from production bundles. Keeps the consuming app building without modifying the shared package.

## Section 6 — Lessons Learned (Feed Forward to 01.4 §Established Patterns)

- **Multiline-aware greps for multi-line `import type` declarations.** The EC-090 §Verification Step 4 grep is a line-by-line filter: `Select-String -Pattern "from '@legendary-arena/game-engine'" | Where-Object { $_.Line -notmatch "import type" }`. SFC files frequently break `import type { A, B, C } from '...'` across multiple lines; the closing `} from '...'` line does not contain "import type" and surfaces as a false positive. Future EC-authoring should use a multiline-aware pattern (ripgrep `--multiline-dotall`, or `grep -Pz` with newline anchors) to avoid the ambiguity.
- **Any first-runtime-engine-import to an app surfaces the game-engine barrel's Node-only re-exports.** The game-engine barrel exports from every submodule including `par.storage.js` (Node-only) and the `ops/` subtree. Before WP-090, arena-client had only `import type` imports, so no `par.storage.js` code was ever loaded at build time; the transitive issue was invisible. WP-090 crossed the threshold. Future apps (and any future re-introduction of a runtime engine import elsewhere) must anticipate a similar mitigation — either a stub plugin, a narrower subpath export on the game-engine package, or a `sideEffects` hint. Registry precedent: WP-091's `./setupContract` subpath export. Game-engine has no such subpath yet.
- **jsdom's URL constraints are a per-test concern, not a setup concern.** The shared `testing/jsdom-setup.ts` installs a minimal jsdom. Tests that need URL manipulation should not reach for `history.replaceState` against `about:blank` (rejected) or `Object.defineProperty(window, 'location', ...)` (non-configurable). The right pattern is component-level props or explicit function seams. Documented in App.vue's `searchOverride` prop comment.
- **Test-runner `import.meta.env` mismatch with Vite is load-order, not value-order.** The breakage is at module load, before any test runs. `node --import tsx --test` exposes this immediately on any file that reads `import.meta.env` without an optional chain. Any new file that reads Vite env vars must include the optional chain on first write, not as a follow-up fix.
