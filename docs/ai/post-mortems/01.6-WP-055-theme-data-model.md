## 6. Hidden-Coupling / Aliasing Audit

Traced `validateTheme` return value and confirmed WP-028 / D-2802
compliance:

- Zod's `safeParse` on an object schema produces a fresh top-level
  object (`result.data`). Test #1 pins this guarantee via
  `assert.notStrictEqual(result.theme, fullTheme)`.
- Nested references (arrays like `villainGroupIds`, nested objects
  like `references.primaryStory`) may share references with the
  input `data`. This is documented in the `validateTheme` JSDoc with
  the concrete clone recipe (`structuredClone(result.theme)`) and
  a WP-028 / D-2802 back-pointer.
- Nested-reference semantics remain **documented-but-untested** by
  design — asserting them would pin Zod internals that future
  upgrades might change, and the concrete guarantee callers depend
  on (top-level distinctness) is covered by test #1.

No hidden coupling surfaced. Future consumers that need nested
isolation will clone; those that don't will benefit from the
single-allocation baseline.

## 8. Reality-Reconciliation vs Session Prompt Locked Values

Three drift-or-observation items surfaced during execution. Each is
small and each is documented here for audit traceability rather than
treated as a blocker. None required a pre-flight amendment.

1. **Paraphrase-discipline self-catch.** The initial draft of
   `theme.schema.ts` contained the literal string
   `packages/game-engine/src/matchSetup.types.ts` inside a `// why:`
   comment documenting the source of the `setupIntent` field names.
   Verification Step 4 (`grep -r "game-engine" packages/registry/src/theme.*.ts`)
   flagged the match. The comment was rewritten to reference the
   WP-005A MatchSetupConfig contract by name only, which both
   satisfies the grep gate and matches the P6-50 paraphrase-discipline
   precedent. Token-only fix; no scope impact.
2. **Step 9a grep returns 69, not 68.** The session prompt's Step 9a
   expected `grep -c '"themeSchemaVersion": 2' content/themes/*.json | grep -v ':0' | wc -l`
   to return 68. Observed: 69. Reason: the new `content/themes/minimal-example.json`
   is at v2 (required by the schema), which adds one to the count.
   The 68-entry lock applies to the manifest-shipped set, not to
   every file in the directory; Step 8 (manifest-scan) returned the
   locked 68 and test #8 Part A iterated exactly 68 manifest
   entries. No drift — the session prompt's grep expected value is
   off-by-one for the post-migration state; documenting here so a
   future executor doesn't misinterpret.
3. **`pnpm --filter @legendary-arena/registry build` post-tsc failure.**
   `tsc -p tsconfig.build.json` compiles cleanly and emits
   `dist/theme.schema.js`, `dist/theme.validate.js`, and their
   `.d.ts` files. The post-tsc step `node scripts/build-dist.mjs`
   exits 1 with `dist/cards.json not found — run 'pnpm normalize'
   first` — a pre-existing pipeline artifact unrelated to WP-055
   (confirmed by temporarily stashing the new theme files and
   observing the same failure). The registry test runner uses
   `node --import tsx --test` which compiles TypeScript on the fly
   and is unaffected by the build-dist pipeline. WP-055 does not
   regress this pipeline; the pipeline is documented here so it
   cannot be mis-attributed to WP-055 at pre-commit review.
