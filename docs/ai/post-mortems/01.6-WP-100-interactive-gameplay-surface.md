## 5. Mutation, Aliasing, & Reference Safety (High-Risk)

This is the WP-028 lesson (`cardKeywords` aliasing). Reviewed line-by-line:

- [x] No component mutates the Pinia `uiState` store. Verified by `Determinism & Authority` AC group (binary check).
- [x] No component mutates props received from `PlayView` (children consume props as read-only).
- [x] PlayView's `viewer` computed returns the matching `UIPlayerState` reference directly from the store snapshot — but the consumer (`HandRow`) never mutates `viewer.handCards`. The hand is rendered by `v-for cardId in handCards` which is a read-only iteration. The click handler emits a new `{ cardId }` payload — no aliasing.
- [x] PlayView passes `viewer.handCards ?? []` to HandRow. The `?? []` fallback creates a new empty array per render when handCards is undefined — no shared-reference risk.
- [x] CityRow / HQRow iterate `city.spaces` / `hq.slots` by index and emit `{ cityIndex }` / `{ hqIndex }` — primitives only, no object-reference leaks.
- [x] The `submitMove` closure in App.vue reads `liveClient.value` at call time (not at setup time) — no captured-stale-reference risk.

No aliasing fixes needed. The WP-028 precedent does not apply because no component returns or shares any object slice from the store — they only read scalars and pass-through references that are never mutated downstream.

## 6. Hidden-Coupling Detection

- [x] No engine internals exposed: components consume `UIState` projections only (`UICityState`, `UIHQState`, `UIMastermindState`, `UITurnEconomyState`, `UIPlayerState`).
- [x] No enum widened: `UiMoveName` is a strict subset of `LegendaryGame.moves` (six names, mirrored explicitly).
- [x] No implicit knowledge of upstream implementation: components do not know that `playerView` populates `handCards` only for the viewer — they detect "I am the viewer" by the `handCards !== undefined` predicate, which is exactly what the redaction contract guarantees.
- [x] Ordering assumptions are explicit and documented: `// why:` comments on CityRow / HQRow positional-index iteration cite the engine's positional `cityIndex` / `hqIndex` payload contract.
- [x] No dependency on non-exported functions or internal module state.

The viewer-identification-by-handCards-defined predicate is a forward-stable choice: if WP-111 adds a `handDisplay?: UICardDisplay[]` field that is also redacted for non-viewers, the `handCards !== undefined` predicate still correctly identifies the viewer (handCards remains the viewer-only field). If a future WP changes the redaction contract (e.g., populates handCards for everyone), this predicate would silently break — but that's a contract change that would require a synchronized UI WP anyway.

## 9. Forward-Safety Questions

- [x] **Can this code survive future refactors without touching unrelated layers?** Yes. PlayView is the sole composer; the five interactive children are independently buildable and testable. A bug in one child is localized to its `.vue` + `.test.ts` pair.
- [x] **Can replay/debugging reconstruct behavior from stored data?** Yes. Every UI-visible state change is a pure function of the Pinia snapshot, which is itself a function of server-pushed frames. Replaying frames into the store fully reconstructs UI state.
- [x] **Would removing upstream data still fail safely?** Yes. `snapshot === null` → empty-match placeholder; `viewer === null` → ArenaHud-only render; `phase !== 'play'` → ArenaHud-only render; `handCards: []` → "Hand is empty" placeholder; null City/HQ slots → "Empty slot" placeholders.
- [x] **Is it impossible for this WP's output to influence gameplay unintentionally?** Yes. Components emit intent through the prop-drilled `submitMove` only; the engine validates and decides outcomes. No optimistic UI, no local state mutation.
- [x] **Is the contract stable enough to be referenced by future WPs?** Yes. The `UiMoveName` union is the explicit extension seam. WP-111 will bind these components to richer card display data without changing prop signatures (only prop content). Future target-picker or affordability-gating WPs extend the disabled expression and add additional props; the existing contract is additive-stable.
