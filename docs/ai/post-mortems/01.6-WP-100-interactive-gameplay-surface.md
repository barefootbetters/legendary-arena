## 9. Forward-Safety Questions

- [x] **Can this code survive future refactors without touching unrelated layers?** Yes. PlayView is the sole composer; the seven interactive children are independently buildable and testable. A bug in one child is localized to its `.vue` + `.test.ts` pair.
- [x] **Can replay/debugging reconstruct behavior from stored data?** Yes. Every UI-visible state change is a pure function of the Pinia snapshot, which is itself a function of server-pushed frames. Replaying frames into the store fully reconstructs UI state.
- [x] **Would removing upstream data still fail safely?** Yes. `snapshot === null` → empty-match placeholder; `viewer === null` → ArenaHud-only render; non-lobby/non-play phase → ArenaHud-only render; `handCards: []` → "Hand is empty" placeholder; null City/HQ slots → "Empty slot" placeholders.
- [x] **Is it impossible for this WP's output to influence gameplay unintentionally?** Yes. Components emit intent through the prop-drilled `submitMove` only; the engine validates and decides outcomes. No optimistic UI, no local state mutation. The §Scope J retarget is a deterministic phase transition with the same authority surface as the original (still gated by `validateCanStartMatch`).
- [x] **Is the contract stable enough to be referenced by future WPs?** Yes. The `UiMoveName` union is the explicit extension seam (any future move added to the UI vocabulary extends the union). D-10006 explicitly lays out the two evolution paths for re-architecting the lobby → setup → play transition. Future deck-construction WPs can reroute through setup OR take ownership of the lobby → play seam differently — neither is locked out.
