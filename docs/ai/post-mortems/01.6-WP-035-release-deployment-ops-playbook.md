## 4. Representation & Determinism Audit

- [x] New representations are data-only and JSON-serializable —
      `OpsCounters` is a plain object of four `readonly number` fields;
      `DeploymentEnvironment` and `IncidentSeverity` are closed literal
      string unions. No functions, closures, collection primitives, or
      temporal objects in any of the three shapes. D-1232 satisfied.
- [x] Execution logic is deterministic — no execution logic exists in
      this WP. RS-1 option (a) locks the "no runtime instance" choice,
      so there is no hidden state, no caching, no memoization.
- [x] Unknown or future values fail safely — the two closed unions
      require exhaustive handling by consumers (the TypeScript compiler
      enforces this at the call site). Extending either union is a
      governance change — a new D-entry plus coordinated documentation
      update — not a silent code change.

The module header calls this out in prose: *"The engine never reads or
writes a counter at runtime (RS-1 option (a); D-3501)."*

## 5. Mutation, Aliasing, & Reference Safety (High-Risk)

- [x] All mutation occurs only in authorized lifecycle contexts — no
      mutation logic in scope.
- [x] No helpers mutate `G` or `ctx` — no helpers in scope.
- [x] No mutation during rendering, projection, or evaluation — no
      projection or evaluation in scope.
- [x] Pure helpers are actually pure — vacuously true; none exist.
- [x] No aliasing bugs — the `readonly` prefix on every `OpsCounters`
      field extends the WP-028 aliasing precedent into ops observability:
      consumers that reconstruct a counter snapshot must produce a new
      object rather than mutating a shared instance. The TypeScript
      compiler enforces this on any caller of engine-exported code.

The `readonly` discipline is deliberate. A future consumer that reads
an `OpsCounters` snapshot and wants to "adjust" one field in place
would get a compile error; the correct shape (spread into a new
object) is the forced pattern. This is the exact lesson WP-028's
post-mortem taught applied proactively instead of reactively.

## Lesson Recorded

**Contract-Only WPs with a locked Pre-Flight can execute without mid-
session surprises.** WP-035's pre-flight resolved PS-1, RS-1, RS-2,
RS-3, and RS-4 before a single code file was written. The execution
produced zero mid-flight fixes — in contrast to WP-034, which surfaced
six P6-43 paraphrase-discipline collisions during its first
verification gate run. The delta comes from the WP-035 prompt's
explicit call-out of paraphrase discipline in its Locked Values, which
pre-loaded the constraint into the authoring context. Worth encoding
as a pattern: **whenever a verification gate uses a literal-string
grep, spell out the paraphrase requirements in the session prompt's
Locked Values, not just in the post-mortem lessons.**
