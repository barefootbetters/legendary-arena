## Section 1 — Overview

WP-102 is the first WP to expose handles in HTTP and the first
profile surface in the Vue SPA. The packet ships a public, read-only
`?profile=<handle>` page composing `PublicProfileView` from
`legendary.players` and `legendary.replay_ownership`, plus the
typed `fetchPublicProfile` HTTP wrapper, plus the `getPublicProfileByHandle`
+ `loadPlayerIdByAccountId` library, plus the `registerProfileRoutes`
Koa adapter, plus 8 tests in one new `describe()` block (3 drift +
5 DB-dependent).

**Mid-execution deferral (deliberate scope reduction, NOT a gap).**
The session prompt's §Goal #5 *Pool lifecycle gate* fired when the
executor verified that `apps/server/src/server.mjs` does not create
a long-lived `pg.Pool`. The rules loader at
`apps/server/src/rules/loader.mjs` constructs a short-lived pool,
runs the startup `legendary.rules` + `legendary.rule_docs` queries,
and ends the pool in `finally`. There is no surviving `database`
reference at the `registerHealthRoute(server.router)` call site
to pass into `registerProfileRoutes(server.router, database)`.

WP-102 §H explicitly anticipated this: *"WP-102 does NOT introduce
pool creation — if no pool exists at the point of call, WP-102's
session prompt will surface this as an additional pre-execution
gate."* Two paths were considered (per the §Goal #5 escalation
table):

- **Option A** — defer the `server.mjs` route registration to the
  future request-handler WP that owns long-lived pool lifecycle.
  Ship 7 of the 8 Commit-A files; keep `registerProfileRoutes`
  exported with no production caller; document via WP / EC
  amendment + new D-entry.
- **Option B** — add a minimal pool-creation block under WP-102.
  Forces load-bearing infra decisions (max, idleTimeoutMillis,
  connection error handlers, SIGTERM ordering, observability
  hooks) inside a "read-only profile composition" packet.

The user authorized **Option A** (D-3103-style mid-execution
amendment) for three reasons beyond "smallest diff":

1. **Pool config is genuinely a separate design question.** Values
   for `max`, `idleTimeoutMillis`, `connectionTimeoutMillis`, error
   handlers, SIGTERM ordering vs. boardgame.io's own connection
   lifecycle, and observability hooks are load-bearing for every
   future request handler. Making them under WP-102's read-only
   scope-discipline pressure is bad governance. The future
   request-handler WP that owns authenticated endpoints (WP-112
   successor or a dedicated "Claim Endpoint WP" or similar; the
   D-entry deliberately names the consumer in abstract terms) is
   the natural forcing function.
2. **Project precedent already accepts shipped-but-unwired
   surfaces.** WP-053's `submitCompetitiveScore` ships exactly
   this pattern — the WORK_INDEX row is explicit: *"rejects every
   submission with `par_not_published` until [request-handler]
   wiring lands ... lifecycle prohibition makes this safe — no
   production caller exists today."* WP-102's deferred route
   registration is structurally identical.
3. **The 404 user experience is indistinguishable from an
   unclaimed handle.** `PlayerProfilePage.vue` already renders the
   locked "No player has claimed this handle." empty-state on
   `status === 404`. During the deferral window, an unwired route
   returns whatever the dev server's default 404 handler returns;
   `fetchPublicProfile` sees `{ ok: false, status: 404 }` and the
   page renders the same empty-state UX. No broken-experience
   cliff.

**Lifecycle prohibition (locked, RISK #16) is honored by Option A
with extra strength** — there is no production caller path at all
during the deferral window, so the per-WP allow-list of consumers
(`profile.logic.test.ts`, `profile.routes.ts`, future
`PlayerProfilePage.vue` via `profileApi.ts`) is structurally
enforceable.

## Section 3 — Lessons & Carry-Forward

### 3.1 Pool-lifecycle gap should be a 01.4 pre-flight checklist item

Pre-flight 01.4 §Dependency Contract Verification has bullets for
"functions are exported" and "imports resolve" but no bullet for
"infrastructure singletons (pool, registry, gate) exist at the
wiring site." When a WP adds an HTTP handler that takes a
caller-injected `DatabaseClient` (or any other long-lived
infrastructure object), pre-flight should `git grep` the named
entry-point file (`apps/server/src/server.mjs` for this codebase)
for `new Pool(`, `new pg.Pool(`, or the equivalent constructor
before declaring READY.

WP-102's pre-flight 2026-04-28 self-flagged this gap in the
session-prompt §Goal #5 ("Pool lifecycle gate") but the gap was
authored as a runtime-only check, not a pre-flight gate. The
runtime-only check fired correctly (executor stopped, escalated,
Option A authorized) but the same WP-102 could have been
re-scoped at pre-flight time as "server-only, route registration
deferred" instead of needing a mid-execution amendment.

**Proposal for 01.4 template:** add a "Wiring-site verification"
sub-item to §Dependency Contract Verification:

> If the WP adds a function that takes a caller-injected
> long-lived dependency (`pg.Pool`, registry, gate, etc.) as a
> parameter, grep the named entry-point file (`server.mjs`,
> `index.mjs`, etc.) for the constructor of that dependency. If
> the constructor is absent OR the dependency is short-lived
> (created and closed within a single `try…finally`), surface a
> PS-N "wiring-site availability" note before declaring READY.

This avoids one round-trip of mid-execution escalation while
preserving the runtime-only check as backup. The session-prompt
§Goal #5 pattern (executor STOPs and escalates) remains the
correct fallback when pre-flight didn't catch it.

### 3.2 Comment text vs. grep-gate exactness (continued lesson)

Two grep gates fired against my initial drafts because comment
text contained literal tokens the gates were policing. This is
the **same pattern WP-101 §3.2** documented; the gates have not
been updated upstream.

- `profile.routes.ts` — the `// why:` comment for the
  "intentionally unauthenticated" route handler originally read
  `"no requireAuthenticatedSession call, no Hanko session check"`.
  Hard Stop #17's grep
  `requireAuthenticatedSession|@teamhanko|hanko\.io|'hanko'|"hanko"`
  fires literally on prose mentions. Reworded to `"no
  authenticated-session helper invocation, no auth-broker session
  check"` — preserves intent, drops the literal tokens.
- `profile.logic.ts` — the per-row narrowing `// why:` comment
  originally quoted the SQL fragment `visibility IN ('public',
  'link')` for documentation. The "exactly 1 match" gate fired at
  count 2 (one in SQL, one in comment). Reworded to `"the SQL
  visibility filter above is the authoritative gate"` — preserves
  intent, drops the literal.
- `profileApi.ts` — the encoding-defense `// why:` comment named
  `encodeURIComponent` literally. The "exactly 1 line" gate fired
  at count 2. Reworded to `"percent-encoding the handle defends
  against..."` — preserves intent, drops the literal.

WP-101's §3.2 carry-forward proposed: *"Grep gates that intend
'exactly N occurrences of this literal as code' should ideally
pattern-match against non-comment context (e.g., line not starting
with `--`, `//`, or `*`), or the gate text should explicitly
tolerate prose mentions."* WP-102 confirms this is a recurring
pattern. Until the upstream gate text is updated, the
paraphrase-the-policed-literal discipline remains the executor's
responsibility.

### 3.3 Two-step `loadPlayerIdByAccountId` + `getHandleForAccount` round-trip

WP-101's `findAccountByHandle` returns the WP-052 `PlayerAccount`
shape, which carries `displayName` but **not** `displayHandle`
(the case-preserved form lives in `legendary.players.display_handle`).
The WP-102 `PublicProfileView.displayHandle` field needs the
case-preserved form. Two implementation paths existed (per the
session-prompt §Implementation Task B note):

- **Option (a)** — fold `display_handle` into
  `loadPlayerIdByAccountId`'s SELECT (one round-trip, but
  `loadPlayerIdByAccountId`'s SQL is locked verbatim per §Locked
  Values).
- **Option (b)** — call `getHandleForAccount(accountId, database)`
  separately (extra round-trip, contract-faithful reuse of WP-101
  surface).

WP-102 chose **Option (b)** to preserve the locked
`loadPlayerIdByAccountId` SQL contract. Cost: one extra round-trip
on the success path (4 SELECTs total per profile request:
`findAccountByHandle` → `loadPlayerIdByAccountId` →
`getHandleForAccount` → replay-filter). Mitigation: a defensive
fallback (`displayHandle = handleRecord ?? playerAccount.displayName`)
keeps the response structurally valid if `getHandleForAccount`
ever returns null (forbidden by WP-101's immutability invariant
but defensive against future schema changes).

A future profile-feature WP that wants to collapse to one
round-trip can add a new `getProfileSummaryByAccountId` helper
that JOINs `legendary.players` once and returns the bigint
`player_id` + `display_handle` + `display_name` in a single
SELECT — without modifying the locked WP-101 / WP-102 helpers.

### 3.4 Structural KoaRouter type vs. `@koa/router` direct dep

The session-prompt §Implementation Task C left the Koa types as
executor's choice between (a) `import type Router from
'@koa/router'` and (b) a structural local interface. WP-102 chose
(b) because:

- `apps/server/package.json` does not declare `@koa/router` as a
  dependency (only `boardgame.io ^0.50.0` and `pg ^8.13.0`).
  Adding a direct import would either need a new dep declaration
  or rely on transitive resolution through boardgame.io — fragile
  in either direction.
- The structural shape needed is minimal: `{ get(path, handler):
  unknown }` for the router and `{ params, status, body }` for
  the context. Two interfaces totaling ~12 lines.
- The `registerHealthRoute(router)` precedent at `server.mjs:30`
  uses a JSDoc `@param {import('@koa/router')} router` annotation
  on a plain JS function (no TypeScript). The structural TS
  approach is consistent in spirit.

A future request-handler WP that wires more routes (e.g., the
deferred `server.mjs` registration call site) may either continue
the structural pattern OR add `@koa/router` as a direct dep and
import the canonical types — both compile against the same
runtime `server.router` object. WP-102 does not lock the choice.

### 3.5 Deferred-export safety in this codebase

The user's handoff guidance (Option A confirmation) flagged a
caveat: *"verify the project doesn't have `noUnusedExports` or
equivalent lint enabled."* Verified at execution:

- `apps/server/` has no `tsconfig.json` (the server runs via `tsx`
  for tests; no separate typecheck step). No
  `noUnusedExports` / `noUnusedLocals` enforcement.
- `apps/arena-client/tsconfig.json` is `strict: true` with
  `exactOptionalPropertyTypes` and `noUncheckedIndexedAccess`,
  but no `noUnusedExports` / `noUnusedLocals`.
- The only `.eslintrc` in the repo (excluding `node_modules`) is
  `apps/registry-viewer/.eslintrc.cjs` — out of WP-102 scope.

`registerProfileRoutes` is exported from `profile.routes.ts` with
no production caller and no compile / lint failure. The deferred
export survives until the future request-handler WP imports it.
No `// @ts-expect-error` or inline suppression needed.

A future WP that does introduce `noUnusedExports` (or migrates
the server to a separate `tsconfig.json` with stricter checks)
must inspect `apps/server/src/profile/profile.routes.ts` for
this surface and either suppress with a `// why:` deferral
comment or land alongside the request-handler WP that consumes it.
