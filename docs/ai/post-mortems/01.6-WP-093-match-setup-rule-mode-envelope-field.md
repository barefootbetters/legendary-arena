## Section 3 — New Contracts / Abstractions

### 3.1 `heroSelectionMode` envelope field (v1)

```jsonc
// MATCH-SETUP envelope fragment
{
  ...,
  "expansions": ["base", "dark-city"],
  "heroSelectionMode": "GROUP_STANDARD",   // optional; default when absent; v1 enum = ["GROUP_STANDARD"]
  "composition": { ... }                    // unchanged — 9-field lock preserved
}
```

**JSON Schema fragment (verbatim from MATCH-SETUP-JSON-SCHEMA.json):**

```json
"heroSelectionMode": {
  "type": "string",
  "description": "Optional rule mode indicating how the composition's hero selection is interpreted. v1 allows exactly one value; the field is backward compatible (absent means GROUP_STANDARD).",
  "enum": ["GROUP_STANDARD"]
}
```

Placement: between `expansions` and `composition` in both the example
JSON and the JSON Schema. Rationale (locked by WP-093's inline
`// why:` comment in its Locked Contract Values): identity/provenance
fields first, interpretation intent next, content last — improves
human readability and diff stability when envelopes are archived.

### 3.2 Error code + message template (byte-for-byte consumed)

- **Error code:** `"unsupported_hero_selection_mode"`
- **Error message template (normative, verbatim):**

  `The loadout envelope's heroSelectionMode is <value>, which is not a supported rule mode in v1 of the match setup schema. Supported modes: GROUP_STANDARD. (HERO_DRAFT is reserved for a future release and is not yet implemented.)`

- `<value>` is the only permitted substitution; every other byte is
  normative. WP-091's validator tests and WP-092's parser tests assert
  **exact-string equality** against this template (not substring, not
  regex, not `.includes()`).

### 3.3 Human-readable label mapping

Machine name + short UI label + long explanation per mode; future-notice
UX copy for the reserved `"HERO_DRAFT"` mode. See D-9301 body for the
authoritative strings — reproduced byte-for-byte in
`MATCH-SETUP-SCHEMA.md §Field Semantics / Hero Selection Mode` and
consumed verbatim by WP-091's rule-mode indicator and WP-092's parser
error surfacing.

### 3.4 Reserved future token: `"HERO_DRAFT"`

Named in prose only. Not in the v1 allowed enum. Any v1 document whose
`heroSelectionMode` is `"HERO_DRAFT"` is rejected by Stage 1 envelope
validation with the error code and message template above. Enabling
`"HERO_DRAFT"` requires a future WP that (1) amends WP-093's governance
blocks, (2) defines new composition-level structures as needed, (3)
implements engine-side interpretation. Until then, the reservation is
governance-only.

## Section 4 — New Long-Lived Governance Pattern: Four-Point Naming-Governance Policy

D-9301 enshrines a reusable policy for all future rule-mode work:

1. **Single source of rule-mode names.** WP-093 (and, by extension,
   D-9301) is the sole place where rule-mode machine names,
   human-readable labels, and reserved-future tokens are introduced.
2. **Consumers verbatim, never invent.** WP-091, WP-092, and any
   future UI or parser WP may **consume** the names and labels here
   byte-for-byte — they may never paraphrase, rename, abbreviate, or
   invent alternates.
3. **New modes amend WP-093 first.** A future WP that adds a new rule
   mode must first amend WP-093's governance blocks (enum, label
   mapping, reserved-future prose) and land a new DECISIONS entry
   extending D-9301 before any UI/parser/engine WP is drafted against
   the new name.
4. **Flavor/lore strings are narrative-only.** Lore labels (e.g.,
   `"Contest of Champions"` for `"HERO_DRAFT"` matches) never appear
   in enums, error messages, JSON, schema validation, lookup keys,
   branch conditions, analytics dimensions, telemetry fields, or log
   tokens. The machine distinction is always the locked enum value.

This pattern generalizes to arbitrary rule-mode enums (hypothetical
`"SEALED"`, `"CUBE_DRAFT"`, `"SCENARIO_FIXED"`, etc.). It prevents
schema drift, wording forks, and retroactive renames across the
registry / engine / client / server / leaderboard boundary.

## Section 5 — New Discipline Established: Flavor/Lore Separation

WP-093 is the **first packet** to formally separate in-universe lore
from machine-readable identifiers for a rule-mode surface. Prior WPs
have implicitly treated ext_ids as machine-only and display names as
UI-only, but no WP until WP-093 has (a) introduced a rule-mode enum,
(b) named a reserved-future token alongside it, and (c) pre-emptively
forbidden the in-universe flavor framing ("Contest of Champions") from
appearing in enums, errors, JSON, schema validation, analytics,
telemetry, or log tokens.

This discipline is now the expected pattern for every future rule-mode
surface. If a future WP introduces a rule-mode enum whose lore framing
could plausibly end up in a log token or analytics dimension, that WP
must explicitly forbid the crossover in its DECISIONS entry. D-9301's
"Flavor / lore framing is narrative-only" paragraph is the template.

  per the "Fixes applied during post-mortem" section below. This
  finding validates the 01.6 workflow's separation of
  implementation-session audit (step 4, filename-scoped, runs before
  the parallel session lands) from gatekeeper pre-commit review
  (step 5, hunk-scoped, runs after the working tree has settled) —
  both layers are necessary.

**Fixes applied during post-mortem:**

- **HTML-escaped `<value>` → literal `<value>` in
  `MATCH-SETUP-VALIDATION.md` Stage 1 bullet.** The first draft of
  the Stage 1 rule-mode bullet used `&lt;value&gt;` for the
  placeholder. WP-093 L1 locks the error-message template as
  byte-for-byte verbatim with a literal `<value>` substitution
  token. The HTML escape would have broken byte-identity with
  `DECISIONS.md` (D-9301) and with `MATCH-SETUP-SCHEMA.md §Field
  Semantics / Hero Selection Mode`. Refactored during first draft of
  this post-mortem's verification walk-through to wrap the full
  template in backticks (the §3.2 format the WP uses itself) so
  markdown does not interpret `<value>` as HTML, and the literal
  `<value>` token is preserved. Cross-file byte-identity grep (Step
  11) passes after the fix.

- **Surgical stage to isolate D-9301 from parallel-session D-0005 in
  `docs/ai/DECISIONS.md`.** Because pCloud sync delivered the WP-095
  parallel session's D-0005 entry at line ~115 of `DECISIONS.md`
  while WP-093's D-9301 entry sat at line ~9437 of the same file, a
  naive `git add docs/ai/DECISIONS.md` would have swept D-0005 into
  the WP-093 commit and violated WP-093's §6 Scope Lock at the
  hunk level (causing `git log` to mis-attribute D-0005 to WP-093
  under an `EC-093:` prefix). Resolved at stage time via a
  reversible five-step sequence: (1) back up the "both additions"
  working-tree copy of `DECISIONS.md` to `/tmp/decisions-both.md`;
  (2) `git checkout HEAD -- docs/ai/DECISIONS.md` to reset the
  working tree to the pre-both-additions state; (3) re-apply the
  D-9301 block only via the same `Edit` contract that landed it
  originally (insertion point after the D-9401 `---` separator, before
  the file-ending `## Final Note`); (4) `git add
  docs/ai/DECISIONS.md` to capture D-9301 only in the index; (5)
  `cp /tmp/decisions-both.md docs/ai/DECISIONS.md` to restore the
  parallel session's D-0005 to the working tree unstaged. Final
  `git status --short` shows `MM docs/ai/DECISIONS.md` — M in the
  index column (staged, D-9301 only) and M in the working-tree
  column (unstaged, D-0005 only). `git diff --cached docs/ai/DECISIONS.md
  | grep ^@@` returns exactly one hunk (`@@ -9437,6 +9437,64 @@`);
  `git diff docs/ai/DECISIONS.md | grep ^@@` returns exactly one
  hunk (`@@ -112,6 +112,92 @@`). WP-095's D-0005, VISION.md
  amendments, DESIGN-RANKING.md, and the WP-095 draft remain in the
  working tree for the parallel session to commit separately as its
  own governance bundle (expected under a `SPEC:` prefix). The
  operation is fully reversible at every step: if any re-apply had
  failed, `cp /tmp/decisions-both.md docs/ai/DECISIONS.md &&
  git reset HEAD docs/ai/DECISIONS.md` would have restored the
  pre-surgery state. This pattern is a reusable precedent for
  future governance-only packets that discover mid-session parallel
  drift on a shared append-mostly file (DECISIONS.md, STATUS.md,
  WORK_INDEX.md, EC_INDEX.md).

## Section 12 — Precedent to Establish

### 12.1 Flavor/lore separation as a reusable discipline

Any future WP that (a) introduces a rule-mode enum, (b) names a
reserved-future token, and (c) has plausible in-universe lore framing
for the mode must follow D-9301's four-point naming-governance policy
and must explicitly forbid the lore framing from appearing in enums,
error messages, JSON property names or values, schema validation
constraints, lookup keys, branch conditions, analytics dimensions,
telemetry fields, or log tokens. This precedent is cited from WP-093
as the template.

### 12.2 Four-point naming-governance policy as a reusable pattern

The policy generalizes to any enum where values are consumed
byte-for-byte by multiple downstream WPs. The pattern is: (1) lock
names + labels in a single source WP and its DECISIONS entry; (2)
forbid consumers from inventing, renaming, paraphrasing, or
abbreviating; (3) require new-value additions to amend the source WP
first; (4) separate machine names from narrative flavor. Future WPs
that introduce byte-for-byte-consumed enums (e.g., a new error-code
family, a new telemetry dimension enum) should consider adopting the
same four-point structure.

### 12.3 Governance-only packet with byte-for-byte consumer contract

WP-093 is a template for future governance-only packets that
canonicalize strings for future consumer packets to assert
byte-for-byte equality against. The discipline: (a) lock the strings
verbatim in one DECISIONS entry + one schema reference doc, (b)
include a pre-commit grep checking cross-file byte-identity, (c) add
the locked strings to a session-context file so the next session's
executor can quote them without re-derivation. WP-093 establishes
this template; future packets with similar byte-for-byte coupling
needs should cite WP-093.

## Section 13 — Lessons Learned (Feed Forward)

- **Byte-for-byte string discipline requires a pre-commit grep.** The
  WP-093 session-context §4.3 recommended a cross-file grep as a
  pre-commit sanity check; during execution it caught the
  HTML-escape drift (§10 above) before commit. Future byte-for-byte
  packets should make this grep a mandatory Verification Step, not a
  recommended one.
- **Staleness between WP prose and HEAD is low-risk on governance-only
  packets but should still be noted.** The §8.1/§7 mismatch is
  harmless but worth flagging for the next reviewer. On a code
  packet, the same kind of staleness could hide a real scope miss;
  on governance, the text patterns dominate.
- **Summary counters drift silently across WPs.** The EC_INDEX
  counter has been wrong for several WPs and no packet's governance
  close has fixed it. A dedicated SPEC cleanup is now cheaper than
  continuing to let it drift.
- **The `<value>` token in markdown needs backtick-wrapping.**
  Markdown renderers may strip or mangle `<value>` as a raw HTML
  tag. The fix (wrap the whole error template in single backticks)
  preserves byte-identity while keeping rendering safe. Future error
  template governance should default to this wrapping.

Feed forward to `01.4 §Established Patterns` and
`docs/ai/REFERENCE/00.6-code-style.md` if a dedicated "governance
byte-for-byte string discipline" row is warranted.
