## Section 1 — Overview

WP-111 closes WP-100's `D-10004` deferral by giving the arena-client
enough display data to render real cards (name, image, recruit/fight
cost) without granting it a runtime registry import. The packet adds
`G.cardDisplayData: Readonly<Record<CardExtId, UICardDisplay>>` as a
sibling snapshot to the existing setup-time records (`G.cardStats`,
`G.villainDeckCardTypes`, `G.cardKeywords`), wires it through
`buildInitialGameState`, and surfaces it through `buildUIState` as
four additive projection paths:

- `UICityCard.display` — additive on the existing City projection
- `UIHQState.slotDisplay?` — parallel-array beside the unchanged
  `slots: (string | null)[]` (PS-6 fallback per Q3 written audit)
- `UIPlayerState.handDisplay?` — parallel-array beside the unchanged
  `handCards?: string[]`
- `UIMastermindState.display` — additive on the existing Mastermind
  projection (looked up internally via `gameState.mastermind.baseCardId`
  per PS-5; `id` continues to expose only the qualified group id)

The arena-client UI binding (replacing `{{ cardId }}` with
`{{ display.name }}` plus `<img :src="display.imageUrl" />` in
`HandRow.vue` / `CityRow.vue` / `HQRow.vue` / `MastermindTile.vue`) is
explicitly out of scope and deferred to a follow-up UI WP. WP-100
continues to render bare CardExtId strings, which remains correct
under the additive parallel-array contract.

## Section 3 — Lessons & Carry-Forward

### 3.1 — Locked-spec illustrative example collided with grep gate

The session-prompt §4 `UNKNOWN_DISPLAY_PLACEHOLDER` example wrote
both `extId: '<unknown>'` and `name: '<unknown>'`, but EC-118 §After
Completing locks `<unknown>` to "exactly one match" across the
engine source. The two sources were internally inconsistent. Since
EC-118 is the authoritative execution contract, the constant was
adjusted to `extId: ''` and the `// why:` comments + one test
assertion message were reworded to avoid the literal token.

**Carry-forward:** when an illustrative session-prompt example carries
a literal that an EC grep gate counts, the gate's literal threshold
must include every site the example introduces. Future session
prompts should either (a) use placeholder text that won't trip the
grep ("placeholder name string" rather than literal `<unknown>`), or
(b) lock the EC threshold to the exact count the example produces.
Pre-flight should note this as a Final Gate item when it surfaces.

### 3.2 — 01.5 cascade pre-authorization works as designed

Pre-flight 2026-04-29 §Runtime Readiness Check + session-prompt §5
explicitly authorized "any other existing test fails because of the
additive G-shape change" under 01.5. The `replay.execute.test.ts`
hash-literal cascade was caught by the test runner immediately,
diagnosed against 01.5's four allowance criteria in seconds, and
applied as a one-line value update with a `// why:` comment citing
the authorization chain. No mid-execution amendment was needed
because the cascade was anticipated.

**Carry-forward:** WPs that add fields to `LegendaryGameState`
should expect at least one structural-hash test to surface the
cascade. The session-prompt pre-flight inventory should explicitly
name `replay.execute.test.ts:PRE_WP080_HASH` as a known cascade
target alongside `buildInitialGameState.{shape,determinism,loadout}.test.ts`.

### 3.3 — `parseCostNullable` "guard not parser" pattern

PS-4 introduced the single-line guard `if (value === null || value
=== undefined) return null; return parseCardStatValue(value);` as a
wrapper around the canonical parser. The structure is small enough
to memorize, big enough to matter (preserves the
`null → null` vs `0 → 0` UX distinction), and explicit-by-design (a
reviewer can confirm "guard, not parser" by reading three lines).

**Carry-forward:** the "guard not parser" pattern is a useful
convention any time a wider input domain needs a different
sentinel mapping than the canonical parser provides. Future WPs
that face the same dilemma (e.g., a future `vAttack` projection
that needs to distinguish "no fight cost" from "free fight cost")
should reuse the wrapper-around-canonical structure rather than
forking the parser.

### 3.4 — Sibling-snapshot pattern continues to scale

`G.cardDisplayData` is the fourth setup-time sibling snapshot
(`G.cardStats` WP-018, `G.villainDeckCardTypes` WP-014B,
`G.cardKeywords` WP-025). The pattern continues to scale: local
structural reader + runtime guard + setup-orchestration diagnostic +
empty-record fallback + JSON-serializable record + drift-sanity
test. Future setup-time sibling snapshots (e.g., a future
`G.cardKeywordsExtended` or `G.cardScoringWeights`) can lift this
template wholesale.

**Carry-forward:** the four sibling snapshots now collectively
establish a strong precedent. A future WP introducing a fifth
should explicitly cite WP-014B + WP-018 + WP-025 + WP-111 as the
template, rather than re-deriving the pattern.

### 3.5 — Parallel-array additive-extension pattern (PS-6 fallback)

`UIHQState.slotDisplay?` parallels `UIPlayerState.handDisplay?` (the
latter introduced by WP-029 / WP-089). The pattern preserves
backwards compatibility on the existing string-array shape while
opting new consumers into the richer display payload. Both `null`
positions and length-equality are positional invariants that the
projection enforces.

**Carry-forward:** when widening a `(string | null)[]` projection
risks breaking off-allowlist consumers, the parallel-array fallback
is the established escape hatch. Future WPs facing the same
dilemma should perform a written consumer audit (per Q3 of WP-111),
