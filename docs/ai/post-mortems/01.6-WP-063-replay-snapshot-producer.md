## 5. Mutation, Aliasing, & Reference Safety (High-Risk)

Line-by-line audit of `buildSnapshotSequence` return paths:

**Outer sequence object:**
```
return Object.freeze({
  version: 1 as const,
  snapshots: frozenSnapshots,
  [metadata: Object.freeze(resolvedMetadata)]
});
```
- The outer object is frozen at return site ✓
- `snapshots` is assigned from `frozenSnapshots` which is
  `Object.freeze([...collected])` — spread-copy + freeze ✓
- `metadata` (when present) is `Object.freeze(resolvedMetadata)` where
  `resolvedMetadata` is a locally-constructed `{}` populated only with
  defined sub-fields — no aliasing of any caller input ✓

**Snapshot array elements (each UIState):**
- Each element is produced by `buildUIState(gameState, ctx)` — a pure
  WP-028 helper that returns a fresh projection per call.
- Known WP-028 precedent (cardKeywords aliasing): WP-028's post-mortem
  landed a spread-copy fix for `G.cardKeywords[space]` projection. My
  helper inherits this fix by delegating to `buildUIState`; no new
  aliasing path is introduced in WP-063.
- `UIPlayerState.handCards` is spread-copied from `G.playerZones[id].hand`
  (verified at `uiState.build.ts:207`), so consumers mutating
  `snapshot.players[i].handCards` cannot reach back into G.
- `UIState.log` is spread-copied from `G.messages` (verified at
  `uiState.build.ts:272`), so consumers mutating `snapshot.log` cannot
  reach back into G.
- The snapshots array itself is frozen; the individual UIState elements
  are not recursively frozen. This matches the WP-063 spec verbatim
  ("outer object AND snapshots array are Object.freeze-d") — consumers
  cannot replace slots in the array but can, in theory, mutate a
  UIState's internal fields. Since UIState's mutable fields are all
  spread-copies of G data, such mutation cannot propagate back to G.

**`gameState` aliasing under `applyReplayStep`:**
- `applyReplayStep` returns the same `LegendaryGameState` reference it
  received (Q2 = A, D-6304 — locked by WP-080). The helper's local
  `gameState` variable is the identity of that reference across every
  loop iteration, by design.
- The WP-080 post-mortem explicitly flagged that "consumers wanting
  historical snapshots MUST project via buildUIState after each step"
  — that is exactly what my helper does. No `structuredClone`, no G
  copy retention, no pre-refactor snapshot hoarding.
- There is no aliasing risk at the G-mutation layer because we never
  return G or any sub-field of G. We return UIState projections, which
  have already been audited for spread-copy discipline.

**CLI aliasing:**
- `serializeWithSortedTopLevel(sequence)` constructs a new local
  `projected: Record<string, unknown>` and copies top-level keys in
  sorted order. The original frozen sequence is never mutated.
- No other reference-returning paths in `cli.ts`.

**Verdict:** No aliasing bugs. All spread-copy sites are inherited
from the upstream engine helpers (WP-027, WP-028, WP-080); no new
aliasing path is introduced by WP-063.

## D-6305 (New Decision Surfaced During Execution)

### D-6305 — `ReplayInputsFile` reconciled with WP-027 canonical field names; `buildSnapshotSequence` requires an explicit registry reader parameter

**Context:** WP-063's spec lists `ReplayInputsFile` with `inputs:
readonly ReplayInput[]` and `buildSnapshotSequence(params: {
setupConfig, seed, inputs })` with three params. Both phrasings
conflict with WP-027's canonical surface:
- WP-027 defines `ReplayInput` as the top-level match record (seed,
  setupConfig, playerOrder, moves), not a per-step move record. The
  per-step record is `ReplayMove`.
- `buildInitialGameState` requires a `CardRegistryReader` argument;
  the spec's 3-field helper signature omits it.
- `applyReplayStep` requires `numPlayers`, which is derived from
  `playerOrder.length`; the 3-field signature omits `playerOrder`.

**Decision:** Reconcile as follows without bumping `version`:
- `ReplayInputsFile.moves` is typed `readonly ReplayMove[]` (not
  `ReplayInput[]`), matching WP-027's per-step record type.
  `ReplayInputsFile` also carries `playerOrder: readonly string[]`
  so the file is self-contained enough to drive the engine.
- `BuildSnapshotSequenceParams` is a 6-field interface:
  `setupConfig`, `seed`, `playerOrder`, `moves`, `registry`,
  optional `metadata`. The `seed` field is accepted for traceability
  but ignored by the determinism-only harness per D-0205. The
  `registry` field is explicit because `buildInitialGameState`
  requires it; the CLI defaults to an empty-list mock matching
  `replay.execute.test.ts`.

**Rationale:** The original WP-063 signature was internally
inconsistent — `snapshots.length === inputs.length + 1` requires
each "input" to be a per-step move record, not a top-level match
record. Reconciling with `ReplayMove` preserves WP-027's naming
invariants and avoids a confusing parallel `ReplayInput[]` pattern.
Requiring `registry` explicitly keeps the helper's pure-function
guarantee intact — no module-level registry singleton, no implicit
global state. The CLI's default empty-list reader mirrors the
engine test precedent and sidesteps a runtime registry import (per
D-6301's `cli-producer-app` rules).

**Rejected alternatives:**
- `ReplayInputsFile = ReplayInput & { version; metadata? }` —
  intersection type collapses cleanly but surfaces all of
  `ReplayInput`'s mutable array types (`playerOrder: string[]`,
  `moves: ReplayMove[]`), which the spec asks to be `readonly`.
- Hide `registry` behind a module-scoped singleton — breaks purity
  and makes the helper non-composable under tests.
- Omit `playerOrder` and infer `numPlayers` from `moves` (e.g.,
  `new Set(moves.map(m => m.playerId)).size`) — fragile, fails on
  move lists where only one player has moved so far.

**Scope:** `packages/game-engine/src/replay/replaySnapshot.types.ts`,
`packages/game-engine/src/replay/buildSnapshotSequence.ts`,
`apps/replay-producer/src/cli.ts`.
