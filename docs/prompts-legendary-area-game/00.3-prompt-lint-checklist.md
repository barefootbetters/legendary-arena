# Prompt Lint Checklist

Use this checklist to **review and approve** any prompt in this set before
executing it. A prompt must pass every applicable section.

Run this checklist when:
- Creating a new prompt
- Modifying an existing prompt
- Reviewing a prompt before handing it to Claude Code

---

## 1. Contract Block

- [ ] The prompt opens with a `>` blockquote FULL CONTENTS MODE contract
      (not an HTML comment, not a markdown section — the `>` blockquote format)
- [ ] The contract explicitly requires:
  - [ ] List of files changed
  - [ ] Full contents of every new or modified file (no diffs, no snippets)
  - [ ] Commands to run with expected output
  - [ ] Acceptance checklist
- [ ] The contract explicitly states: ESM only, Node v22+
- [ ] The contract includes any prompt-specific constraints
      (e.g., "Read-only", "No Math.random()", "No WebSocket logic")
- [ ] The contract does not contradict the prompt body

❌ FAIL if partial output is allowed  
❌ FAIL if diffs are permitted  
❌ FAIL if contract uses HTML comments instead of `>` blockquote

---

## 2. Prerequisites (`## Assumes` section)

- [ ] The prompt has an explicit `## Assumes` section
- [ ] The section lists every prior prompt that must be complete
      (e.g., "Prompts 01–03 complete")
- [ ] The section lists every file or module the prompt depends on
      (e.g., "`src/rules/loader.mjs` exists and exports `loadRules()`")
- [ ] The section lists any external state that must exist
      (e.g., "PostgreSQL has rules data", "R2 bucket is publicly readable")
- [ ] No assumption is left implicit

❌ FAIL if the prompt could silently produce wrong output when run out of order

---

## 3. Output Completeness

- [ ] Every deliverable has an explicit file path
- [ ] New files are clearly identified as new
- [ ] Modified files are clearly identified as modified
- [ ] The full updated version of any modified file is required
      (not just the changed section)
- [ ] No ambiguous output language:
  - [ ] No "update this section"
  - [ ] No "show the diff"
  - [ ] No "add the following code"
  - [ ] No "modify the existing function"

❌ FAIL if any deliverable could produce partial or patch-based output

---

## 4. Naming Consistency

- [ ] All parameter names match the canonical names used across the prompt set:
  - [ ] `mastermindsSlug` (not `mastermindsugSlug` or `mastermindsSlug`)
  - [ ] `buildGame(ctx, setupData, rules)` parameter order
  - [ ] `loadRules()` / `getRules()` export names
  - [ ] `game_sessions` table name (not `games` or `sessions`)
  - [ ] `playerCredentials` (boardgame.io standard name)
- [ ] No typos in file paths (check against prior prompts)
- [ ] No stray backtick artifacts (` `` `) at the end of sections
      — this was a recurring bug in this prompt set

❌ FAIL if parameter names or file paths contradict prior prompts

---

## 5. Dependency Discipline

- [ ] The prompt does not introduce new npm dependencies silently
- [ ] If a new dependency is required:
  - [ ] It is explicitly named with exact version or version range
  - [ ] The prompt requires updating `package.json`
  - [ ] The full updated `package.json` is required as output
- [ ] No "check if X is installed, otherwise use Y" logic
      — the prompt must make a hard decision
- [ ] Forbidden packages are explicitly listed where relevant:
  - [ ] No axios or node-fetch (use Node.js built-in `fetch`)
  - [ ] No ORMs (use `pg` only)
  - [ ] No Jest/Vitest/Mocha (use `node:test` only)
  - [ ] No Passport/Auth0/Clerk (use `jsonwebtoken` or credentials-only)

❌ FAIL if a forbidden package could be used without explicit rejection

---

## 6. Architectural Boundaries

### Backend
- [ ] PostgreSQL used only for rules/configuration data — never turn state
- [ ] No `G` or `ctx` stored in the database
- [ ] No database queries inside boardgame.io moves
- [ ] No WebSocket logic outside `boardgame.io Server()`
- [ ] `pg` pool used for all database connections (not single client)

### Game Logic
- [ ] All randomness uses `ctx.random.*` — no `Math.random()`
- [ ] `G` is JSON-serializable (no class instances, no functions)
- [ ] Moves are pure functions (same inputs → same output)
- [ ] Deck builder logic is in `src/game/deckBuilder.mjs`, not in `legendary.mjs`

### Frontend
- [ ] Components do not fetch R2 data independently
- [ ] Card data resolved through shared `cardRegistry.js` module
- [ ] Components contain no game logic
- [ ] boardgame.io client is in `src/game/client.js`, not in Vue components
- [ ] `boardgame.io/react` is never imported
- [ ] All moves dispatched through `gameStore.move()`

### Diagnostics / Scripts
- [ ] Diagnostic scripts are explicitly marked read-only
- [ ] Seeder scripts use upserts only — never `DELETE`
- [ ] Migration scripts are idempotent

❌ FAIL if any boundary is violated or left ambiguous

---

## 7. Windows Compatibility

This project runs on Windows with PowerShell 7+. Check:

- [ ] Shell scripts use `pwsh` (PowerShell), not `bash` or `sh`
- [ ] File path separators use `\` where paths are hardcoded
- [ ] rclone config path uses `process.env.APPDATA` —
      never hardcode `~/.config/rclone` (that is a Linux path)
- [ ] PowerShell scripts use `Get-Command` for PATH checks,
      not `where.exe` or `which`
- [ ] Node scripts that need `.env` use `node --env-file=.env`
      (Node v22 built-in) or `dotenv-cli` — not Linux-only `.env` sourcing

❌ FAIL if any script assumes a Unix/Linux environment

---

## 8. Environment Variable Hygiene

- [ ] All required environment variables are listed
- [ ] Each variable documents its purpose and where it is set
      (local `.env` / Render dashboard / Cloudflare Pages)
- [ ] `.env.example` is updated when new variables are introduced
- [ ] `VITE_` prefix is used for all frontend browser-exposed variables
- [ ] `PORT` is noted as auto-set by Render — not to be set in Render dashboard
- [ ] No real secrets appear anywhere in output
- [ ] `JWT_SECRET` generation command is shown where relevant

❌ FAIL if any required variable is undocumented or a secret is exposed

---

## 9. Authentication Clarity (if applicable)

- [ ] The prompt explicitly commits to ONE identity model:
  - **Option A** — Dual token (boardgame.io credentials + JWT)
  - **Option B** — Credentials-only (boardgame.io credentials only)
- [ ] The chosen option is implemented consistently throughout all deliverables
- [ ] Protected endpoints clearly state which credentials are required
- [ ] A `## Limitations` section is included explaining what the system
      does NOT protect against
- [ ] If JWT is used: `JWT_SECRET` startup guard is included in `server.mjs`

❌ FAIL if both options are described without a clear choice being made  
❌ FAIL if `JWT_SECRET` is used without a startup guard

---

## 10. Test Quality (if applicable)

- [ ] Tests use `node:test` and `node:assert` only
- [ ] `makeMockCtx` includes `ctx.events` stubs (`endTurn`, `endPhase`, `endGame`)
      so game logic can call them without boardgame.io present
- [ ] `makeMockCtx` uses deterministic shuffle (reverse, not identity)
- [ ] Test helpers do not import from boardgame.io
- [ ] Tests run without network access and without database connections
- [ ] A golden/snapshot deterministic test is included for deck construction
- [ ] The golden test includes a comment warning that failure indicates
      a replay-breaking algorithm change

❌ FAIL if tests import from boardgame.io or require network/DB access

---

## 11. Commands and Verification

- [ ] All commands use `pnpm` (not `npm run`)
- [ ] Exact commands are provided — no "run and verify manually"
- [ ] Expected output is shown where it aids verification
- [ ] `node --env-file=.env` is used for scripts that need env vars
- [ ] `pnpm check` is listed as the verification command for connection checks
- [ ] `pnpm validate` is listed as the verification command for R2 data checks

❌ FAIL if verification steps are vague or missing

---

## 12. Acceptance Checklist Quality

- [ ] Acceptance checklist exists and has 6–12 items
- [ ] Every item is:
  - [ ] Binary (pass or fail — no partial credit)
  - [ ] Observable (can be checked without running the whole system)
  - [ ] Specific (references actual file names, function names, or values)
- [ ] No vague items like "works correctly" or "looks good"
- [ ] Checklist aligns exactly with the deliverables — no phantom checks

❌ FAIL if acceptance criteria are subjective or untestable

---

## Final Gate

A prompt **fails lint** if any of the following are true:

| # | Condition | Section |
|---|---|---|
| 1 | Contract is missing or uses wrong format (HTML comment instead of `>`) | §1 |
| 2 | No `## Assumes` section | §2 |
| 3 | Partial file output or diffs are permitted | §3 |
| 4 | Parameter names contradict prior prompts | §4 |
| 5 | A forbidden package could be used silently | §5 |
| 6 | Any architectural boundary is violated or ambiguous | §6 |
| 7 | Any script assumes Linux/Unix environment | §7 |
| 8 | A required env var is undocumented or a secret is exposed | §8 |
| 9 | Identity model is ambiguous or both options are left open | §9 |
| 10 | Tests import boardgame.io or require network/DB | §10 |
| 11 | Verification commands are missing or vague | §11 |
| 12 | Acceptance criteria are subjective | §12 |

Only prompts that pass all applicable sections should be executed.

---

## How to use this checklist with a new prompt

1. Open the prompt file and this checklist side by side
2. Work through sections 1–12, marking each checkbox
3. If any ❌ FAIL condition triggers, fix the prompt before executing
4. Sections 9 and 10 only apply to identity and testing prompts respectively —
   mark them N/A if not applicable
5. After execution, verify the output against the prompt's own acceptance checklist
